Thread + lock 容易出 bug（**race condition（競爭條件）**—— 兩個執行緒同時改一個變數、結果不可預測 / **deadlock（死鎖）**—— 兩邊互等對方放鎖、整個卡死）。**訊息傳遞（message passing）**是另一條路——不共享記憶體、靠「傳訊息」溝通。

> 💡 **這課的價值**：學會這條路、寫多執行緒程式不再為 lock 抓狂。Erlang / Elixir / Go / Akka / WebWorker 全部是這個思維。

  
### 1. 兩個哲學的對決

```text
🔒 Shared Memory（共享記憶體）：
- 多 thread 共用變數
- 用 lock 保護臨界區
- C / Java / C# / Python 主流
- 寫得對效能極高、寫錯就 race condition / deadlock

📬 Message Passing（訊息傳遞）：
- 每個 actor（演員）有自己的狀態
- 互相送訊息溝通（像郵件）
- 沒共享 = 沒鎖 = 沒 race condition
- Erlang / Elixir / Go / Akka（Scala）/ Rust / Pony

Go 的名言：
「不要靠共享記憶體溝通；用溝通共享記憶體。」
(Don't communicate by sharing memory; share memory by communicating.)
```

  
### 2. Actor Model（演員模型）

**核心概念**：每個 **actor（演員）** 是一個獨立小程式、有自己的「信箱（mailbox）」。

```text
Actor 行為三規則：
1. 收到訊息可以「處理」
2. 處理時可以「送訊息給別人」
3. 處理時可以「建立新 actor」

→ 不能直接讀別人的狀態
→ 不能直接改別人的狀態
→ 只能透過送訊息互動
```

**Erlang/Elixir 範例**（Erlang 是這套理論的源頭）：

```text
# 定義一個 actor
defmodule Counter do
  def start(n) do
    spawn(fn -> loop(n) end)
  end

  defp loop(state) do
    receive do
      {:inc, n} ->
        loop(state + n)
      {:get, caller} ->
        send(caller, state)
        loop(state)
      :stop ->
        :ok
    end
  end
end

# 用法
counter = Counter.start(0)
send(counter, {:inc, 5})
send(counter, {:inc, 3})
send(counter, {:get, self()})

receive do
  value -> IO.puts("Counter: #{value}")  # 8
end
```

**Akka（Scala/Java）範例：**

```text
class Counter extends Actor {
  var count = 0
  def receive = {
    case Inc(n) => count += n
    case Get => sender() ! count
  }
}

val counter = system.actorOf(Props[Counter])
counter ! Inc(5)
counter ! Inc(3)
counter ! Get  // 回 8
```

> 💡 **Erlang 為什麼這麼強**：愛立信用 Erlang 寫的電話交換機、達到 **9 個 9 的可靠度**（一年宕機 < 31 毫秒）。靠的就是 actor model + supervisor 樹（每個 actor 有「監督者」、死了自動重啟）。

  
### 3. Channel（通道、Go 的招牌）

Go 用 **channel（通道）** 實現訊息傳遞——像「水管」、一端塞訊息、另一端取訊息。

```text
package main

import "fmt"

func main() {
    ch := make(chan int)  // 建一個 int channel

    // 開一個 goroutine 送資料
    go func() {
        for i := 1; i <= 3; i++ {
            ch <- i  // 送進 channel（<- 是「送」）
        }
        close(ch)
    }()

    // 主程序接資料
    for v := range ch {  // 接到 channel 關閉為止
        fmt.Println(v)  // 1, 2, 3
    }
}
```

**Channel 的兩種類型：**

```text
🚰 Unbuffered（無緩衝）：ch := make(chan int)
   送者會「等」直到有人接、像一手交錢一手交貨

🪣 Buffered（有緩衝）：ch := make(chan int, 10)
   像信箱、最多塞 10 封、滿了才會等
```

**經典 pattern — worker pool（工作池）：**

```text
jobs := make(chan int, 100)
results := make(chan int, 100)

// 開 3 個 worker
for w := 1; w <= 3; w++ {
    go func(id int) {
        for j := range jobs {
            results <- j * 2  // 處理 + 回結果
        }
    }(w)
}

// 派 5 份工作
for j := 1; j <= 5; j++ {
    jobs <- j
}
close(jobs)

// 收 5 個結果
for a := 1; a <= 5; a++ {
    fmt.Println(<-results)
}
```

**3 個 worker 並行處理 5 份工、自動分配**——這就是 Go 招牌。

  
### 4. CSP 理論（Communicating Sequential Processes）

Go channel 背後是 **CSP（通訊式循序程序）** 理論、Tony Hoare 1978 年論文。

```text
CSP 核心：
- 程序（process）獨立、不共享狀態
- 透過「同步訊息傳遞」溝通
- 程序可組合、互相串接

跟 Actor model 的差別：
- Actor：每個 actor 一個信箱、訊息給「對象」
- CSP：訊息給「channel」、誰來接都行

實務上常混用，理論差別。
```

  
### 5. JS / Web 的訊息傳遞：postMessage

瀏覽器 JS 用 `postMessage` 做訊息傳遞——Web Worker 跟主執行緒互傳。

```text
// 主程式
const worker = new Worker('worker.js');
worker.postMessage({ type: 'calc', data: [1,2,3,4,5] });
worker.onmessage = (e) => {
  console.log('結果:', e.data);
};

// worker.js
self.onmessage = (e) => {
  const { type, data } = e.data;
  if (type === 'calc') {
    const sum = data.reduce((a, b) => a + b);
    self.postMessage(sum);
  }
};
```

**好處**：worker 跟主執行緒**完全隔離**、沒共享記憶體、不會 race condition。

  
### 6. 三大典範什麼時候用哪個

```text
🎯 CPU 重度運算（圖片處理、數值計算）：
   → Shared memory + thread + lock
   → C / C++ / Rust + std::thread / OpenMP

🎯 I/O 重度（網路 / 檔案、Web server）：
   → async / await + event loop（L21 教的）
   → Node.js / Python asyncio / Rust tokio

🎯 高並發 + 高可靠（聊天伺服器、遊戲伺服器、電話交換）：
   → Actor model 或 CSP
   → Erlang / Elixir / Go / Akka
```

  
### 7. 5 大常見坑

```text
❌ 1. 把 channel 當「共享變數」用
   → channel 是「過水」、不是儲存
   → 修：要儲存用 actor + 內部 state

❌ 2. 忘了關 channel
   → for-range 等不到 close 永遠卡
   → 修：送完一定 close、不要重複 close

❌ 3. 死等訊息（deadlock）
   → 兩個 actor 互等對方訊息、永遠卡死
   → 修：加 timeout（select + time.After）

❌ 4. Actor 內做 blocking I/O
   → 一個慢 actor 拖累整個系統
   → 修：blocking 動作丟到 worker pool

❌ 5. 過度使用 actor
   → 不是所有東西都要 actor、簡單共享變數就好
   → 修：分清「需要隔離」vs「純資料」
```

  
### 🎯 重點回顧

- **Shared memory + lock** 是傳統解法、寫對效能高、寫錯就死
- **Message passing** 是另一條路、沒共享 = 沒 race condition
- **Actor model**（Erlang / Akka）：每個 actor 一個信箱、彼此送訊息
- **CSP / Channel**（Go）：訊息送進 channel、任何 goroutine 都能接
- **Web Worker postMessage** 是瀏覽器的訊息傳遞
