### 1. 完整對照表（一張看懂）

```text
面向            SQL                NoSQL
────────────────────────────────────────────
Schema         嚴格、先定義        彈性、隨意加
查詢語法        標準 SQL            各 DB 不同
JOIN           完整支援            限制多 / 沒有
Transaction    ACID 完整           部分支援
一致性          強一致              通常最終一致
規模            垂直 + 部分水平     水平擴展強
資料量          ~ TB（單機）        PB（千倍）
學習            一招走天下          各家學各家
工具生態        巨大、成熟          各 DB 不同
業界職缺        90% 都會            分散、看 DB
雲端服務        多、便宜            部分專屬
```

> 💡 **memorize 這張表**——面試 80% 機率被問到、能答出就直接 +50 分。

  
### 2. 該選 SQL 的訊號

```text
✅ 選 SQL（PostgreSQL / MySQL）如果...

   ⃣✓ 結構固定（user / order / product / article）
   ⃣✓ 需要複雜查詢 + JOIN（一次查多表）
   ⃣✓ 需要 ACID transaction（金流、庫存）
   ⃣✓ 資料 < 1 TB（單機可容、不用 sharding）
   ⃣✓ 不想 vendor lock-in（PG 開源、跑哪都行）
   ⃣✓ 團隊有人會 SQL（多數工程師會）
   ⃣✓ 想要成熟工具生態（ORM、監控、backup）

   特別強調：
   ⃣✓ 不知道用什麼 = 直接選 SQL（永遠不會錯）
```

  
### 3. 該選 NoSQL 的訊號

```text
✅ 選 NoSQL 如果...

   ⃣✓ Schema 變動快（events / logs / IoT）
   ⃣✓ 需要水平擴展（10 TB+ 資料量）
   ⃣✓ 特殊資料模型（圖、文件、向量、時序）
   ⃣✓ 已有強需求（即時聊天、地理、快取）
   ⃣✓ 寫入量超大（百萬 / 秒）
   ⃣✓ 跨地區 / 多 DC 部署
```

  
### 4. 真實 stack 範例（業界怎麼配）

```text
🏠 Indie SaaS（個人開發者、月入 < 100k）
   ────────────────────────────────────
   PostgreSQL（Supabase）：99% 業務資料
   Redis（Upstash）：快取 + Session
   + pgvector → AI 應用
   
   月成本：$30-$100
   特點：一人能維護、不用 DevOps

🏢 中型公司（10-100 人）
   ────────────────────────────────────
   MySQL / PostgreSQL：交易資料（user / order）
   Redis：快取 + 排行榜
   MongoDB / DynamoDB：用戶生成內容（評論、活動）
   Elasticsearch：全文搜尋
   
   月成本：$1K-$10K
   特點：有專職 SRE / DBA

🏭 巨型公司（千人以上、e.g. Google / Meta）
   ────────────────────────────────────
   全部都有、按場景拆：
   Cassandra：時序 / IoT / message
   HBase：超大 OLAP
   Neo4j：社交圖、知識圖譜
   Snowflake / BigQuery：分析（OLAP）
   自研 DB：Spanner / DynamoDB
   
   月成本：$100K+
   特點：DB 團隊就 50 人
```

  
### 5. 反 pattern：不要這樣做

```text
❌ 1. 為 NoSQL 而 NoSQL
   症狀：「聽說 NoSQL 快」就轉、結果痛苦
   真相：規模沒大到 = SQL 簡單 10 倍
   修：百萬筆內、PG 都行、不要先進化

❌ 2. 全用 SQL
   症狀：所有東西塞 PG、不用 Redis
   後果：query 慢、DB 被打爆
   修：cache 一定要、Redis 是必修

❌ 3. Mongo 模仿 SQL
   症狀：寫一堆 $lookup（JOIN）、aggregate
   後果：跑超慢、又難維護
   修：用 NoSQL 接受其限制、否則回 SQL

❌ 4. Firestore 大資料分析
   症狀：在 Firebase 跑 ad-hoc 分析 query
   後果：query 沒辦法、要爬一堆 collection
   修：ETL 到 BigQuery / Snowflake 分析

❌ 5. 一個 DB 解決一切
   症狀：堅持只用 PG（或只用 Mongo）
   後果：某些場景 10 倍慢
   修：每種資料用最適合的 DB（polyglot persistence）
```

  
### 6. Polyglot Persistence（多元存儲）

```text
🌐 Polyglot Persistence（多 DB 並存）

不要堅持「一個 DB 解決一切」、
每種資料用最適合的 DB：

   交易資料（user / order）→ PostgreSQL
   Session / 排行榜       → Redis
   全文搜尋               → Elasticsearch / Meilisearch
   社交圖 / 推薦          → Neo4j
   AI 向量                → pgvector / Pinecone
   時序資料 / 監控        → TimescaleDB / InfluxDB
   log / 大量 write       → MongoDB / Cassandra
   分析（OLAP）           → BigQuery / Snowflake

→ 每個 DB 各做最擅長的
→ 不會 over-engineering（過度設計）的前提：規模到了再加
```

  
### 7. CAP 定理（分散式 DB 必懂）

**CAP 是分散式系統的鐵則**——三選二、不能全拿。

```text
🏛️ CAP theorem（CAP 定理）

   C — Consistency（一致性）
       所有節點同一時間看到一樣資料

   A — Availability（可用性）
       永遠回應、不會掛

   P — Partition tolerance（分區容錯）
       網路斷時還能工作

→ 只能選 2、不能 3 全要
→ 實務：P 必選（網路一定會斷）
→ 真正在選的是「C vs A」

🔒 SQL 通常 CP（一致性優先）
   → 寧可掛掉也不錯
   → 銀行轉帳、訂單

⚡ NoSQL 通常 AP（可用性優先）
   → 寧可舊一點也要回應
   → 社交動態、購物車
```

  
### 8. 何時混用（hybrid 架構）

```text
🔀 混用的 3 個常見模式

   1. 主 DB（SQL）+ Cache（Redis）
      → 99% 場景該用、Redis 是「必修」
      → 業務 PG、查詢 Redis、寫穿（write-through）

   2. 主 DB（SQL）+ 搜尋（Elasticsearch）
      → 商品 / 文章 / log 搜尋
      → PG 寫入 → 同步到 ES 索引

   3. 主 DB（SQL）+ AI（pgvector / Pinecone）
      → RAG 應用
      → PG 存原文、Pinecone 存 embedding
      → 或直接 PG + pgvector 一體
```

### 🎯 重點回顧

- **SQL = 結構固定、強一致、JOIN 強** vs **NoSQL = 彈性、水平擴展、特殊資料**
- **不知道用什麼 = 選 PostgreSQL**——80% 場景都對
- **Polyglot Persistence**：每種資料用最適合的 DB（不是 1 個 DB 解決一切）
- **CAP 定理**：分散式 DB 只能 C / A 二選一（P 必選）
- **2026 黃金 stack**：PG + Redis（+ pgvector for AI）—撐到百萬 MAU 沒問題
