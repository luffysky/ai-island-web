**NoSQL 不是「沒 SQL」、是「Not Only SQL」**——SQL 之外的另一條路、各有專長。**Schema-less（無結構）= NoSQL 的最大賣點、也是最大陷阱**。

> 💡 **這課重點**：搞懂 NoSQL 5 大類型各自的長處 + Schema-less 的「自由」會不會變成「失控」。**有了這課、選 DB 就不會被網路炫文洗腦**。

  
### 1. NoSQL 5 大類型完整對照

```text
📄 1. Document（文件型）
   ────────────────────────
   存什麼：JSON-like 物件（巢狀也行）
   長相：{ name: "小明", tags: ["音樂", "程式"] }
   schema：彈性（每筆可以不一樣）
   代表：MongoDB / Firestore / CouchDB
   超強的場景：App 後端、user profile

🔑 2. Key-Value（鍵值對）
   ────────────────────────
   存什麼：key → value（像超大字典）
   長相：set("session:abc", "user_123")
   schema：沒有（value 是字串、blob、什麼都行）
   代表：Redis / DynamoDB / Memcached
   超強的場景：cache、session、排行榜

📊 3. Wide-Column（寬欄式）
   ────────────────────────
   存什麼：類 SQL 但每列可以不同欄
   長相：(user_id, timestamp) → 任意欄位
   schema：彈性（同 column family 內可變）
   代表：Cassandra / HBase / ScyllaDB
   超強的場景：百億級 write（log、IoT、message）

🕸️ 4. Graph（圖式）
   ────────────────────────
   存什麼：節點（node）+ 關係（edge）
   長相：(Alice) -[FRIEND]-> (Bob) -[LIKES]-> (Pizza)
   schema：彈性（節點 / 關係屬性自由）
   代表：Neo4j / Dgraph / ArangoDB
   超強的場景：社交、推薦、知識圖譜

🧠 5. Vector（向量、AI 時代產物）
   ────────────────────────
   存什麼：高維度浮點數陣列（embedding）
   長相：[0.12, -0.45, 0.88, ..., 0.34]（1536 維）
   schema：固定向量維度
   代表：Pinecone / Qdrant / Weaviate / pgvector
   超強的場景：語意搜尋、RAG、推薦
```

> 💡 **embedding（嵌入向量）= 用一串數字表達一段話的「意思」**。例如「便宜手機」跟「平價智慧型手機」的向量很接近——這就是 AI 知道兩句話意思一樣的方法。

  
### 2. Schema-less 的好處

```text
🆓 Schema-less（無結構）的 3 大好處

   1. 改資料超快
      → SQL：要寫 migration、ALTER TABLE、production 緊張
      → Mongo：直接寫入新欄位、馬上能用

   2. 一筆資料一個樣
      → SQL：用戶 A 沒填生日 → null
      → Mongo：直接不存這欄、文件變小

   3. 巢狀資料天然
      → SQL：要 JOIN 多張表才能組出物件
      → Mongo：本來就一個物件、直接讀

→ 早期 MVP 一週改 5 次 schema、Mongo 救命
```

  
### 3. Schema-less 的壞處（沒人說的真相）

```text
💀 Schema-less 的 3 大陷阱

   1. 資料品質失控
      → 沒 schema 防護、誰都能塞奇怪資料
      → 一年後 collection 內 8 種不同 schema
      → 查詢時 .age 可能是 number、string、null、undefined

   2. 沒 JOIN 痛苦
      → 要查 user + post + comment？
      → SQL：1 個 query
      → Mongo：3 個 query 或 $lookup（慢）

   3. 沒 ACID（多數 NoSQL）
      → 銀行轉帳：A 扣 100、B 加 100
      → 如果中間掛掉 → 錢消失
      → SQL 用 transaction 保證、Mongo 4.0+ 才有

→ 用 Mongo 寫久了會懷念 SQL
```

> 💡 **真相**：很多公司 3 年後把 MongoDB 改回 PostgreSQL。**為了短期彈性、付出長期維護成本**。新手別第一個專案就跳 NoSQL。

  
### 4. SQL vs NoSQL 怎麼選（速決版）

```text
📋 場景 → 選什麼

   結構固定（user / order / product）   → SQL
   Schema 變動快（events / logs）       → NoSQL Document
   快取 / Session                       → Redis
   排行榜 / Queue                       → Redis
   超大規模 write（IoT / log）          → Cassandra
   社交關係 / 推薦                      → Graph（Neo4j）
   AI / RAG / 語意搜尋                  → Vector（pgvector）
   全文搜尋                              → Elasticsearch / Meilisearch
   時序資料（監控 / 感測器）            → TimescaleDB / InfluxDB
   不知道用什麼                          → SQL（PostgreSQL）
```

  
### 5. 為什麼 SQL 仍然是主流？

```text
🏆 SQL 不死的 5 大理由

   1. ACID 保證
      → 金流、訂單、庫存的底線
      → 寫了就不會丟、不會錯

   2. 標準語法
      → 一招走天下、Oracle / PG / MySQL 通用
      → NoSQL 每家自己一套（學了 Mongo 不會 Cassandra）

   3. 工具生態巨大
      → ORM（Prisma / Drizzle）/ 監控 / backup 全有
      → NoSQL 工具碎片化

   4. PostgreSQL 內建 NoSQL 功能
      → JSONB（=Mongo 的彈性）
      → pgvector（=向量 DB）
      → 全文搜尋（=Elasticsearch 基本款）
      → 一個 PG 通殺 80% 場景

   5. 工程師都會
      → 8 成工程師會 SQL、找人簡單
      → NoSQL 各家專家難找
```

  
### 6. NoSQL 致命缺點

```text
❌ NoSQL 共同缺點

   1. 沒 JOIN（或很慢）
      → 複雜查詢痛苦
      → 反正只能 denormalize（複製資料、冗餘儲存）

   2. 沒 transaction（或弱）
      → Mongo 4.0+ 有但效能差
      → Redis 用 MULTI/EXEC 但不是真 ACID

   3. CAP 取捨痛
      → 一致性 vs 可用性二選一
      → 業務要懂、不然踩雷

   4. 學習成本散
      → Mongo / Redis / Cassandra 各學一套
      → 每跳一家都重學
```

  
### 7. 2026 主流組合

```text
🏠 PG + Redis
   → 大部分 Indie / SaaS 都這樣
   → PG 做業務、Redis 做快取
   → 簡單、穩、好招人

🤖 PG + pgvector + Redis
   → AI 應用標配（RAG）
   → 一個 PG 內建向量、不用多 DB

🌐 MongoDB + Redis
   → 全 JS / TS 團隊
   → Mongo schema 跟 JS 物件 1:1

🏭 MySQL + Cassandra
   → 超大規模公司（Discord、Apple）
   → MySQL 業務、Cassandra 大資料寫入
```

  
### 8. 不要為 NoSQL 而 NoSQL

```text
💀 新手最常踩的雷

   看到「NoSQL 快、scale 好」就跳 → 80% 後悔

   真相：
   ⃣✓ 規模沒大到那程度 → SQL 簡單 10 倍
   ⃣✓ 寫程式時 SQL 直觀 → NoSQL 要學 aggregate / pipeline
   ⃣✓ JOIN 是好東西 → 別自己手動 join
   ⃣✓ ACID 是保命符 → 別省

→ 規則：百萬筆 + 結構固定 → PostgreSQL 還是首選
→ 真的撐不住才考慮 NoSQL（這天可能永遠不來）
```

  
### 🎯 重點回顧

- **NoSQL = Not Only SQL**——5 大類型各有專長、不是「替代 SQL」
- **Schema-less 是雙面刃**——彈性換失控、長期會痛
- **PostgreSQL 內建 NoSQL 功能**（JSONB / pgvector / FTS）—一個 PG 通殺
- **不知道用什麼 → 選 SQL**——80% 場景都對
- **不要為 NoSQL 而 NoSQL**——規模不到、SQL 簡單 10 倍
