### 1. 完整決策樹（一張看懂）

```text
🌳 你的資料是什麼？
│
├─ 📋 結構化（user / order / product）
│   → PostgreSQL ⭐
│
├─ 📦 半結構化（events / logs）
│   ├─ 結構偶爾變 → PostgreSQL（JSONB 欄位）
│   └─ 完全自由 → MongoDB
│
├─ ⚡ 快取（cache / session / 排行榜）
│   └─ Redis
│
├─ 🔍 全文搜尋
│   ├─ 小規模 → PostgreSQL（GIN index）
│   ├─ 中規模 → Meilisearch
│   └─ 大規模 + log → Elasticsearch
│
├─ 🧠 AI 向量（RAG / 語意搜尋）
│   ├─ 整合 PG → pgvector ⭐
│   └─ 純 vector 大規模 → Pinecone / Qdrant
│
├─ 📈 時序（IoT / metrics / 監控）
│   ├─ 要 SQL → TimescaleDB ⭐
│   ├─ 純 metrics → InfluxDB
│   └─ K8s 監控 → Prometheus
│
├─ 🕸️ 關係網路（社交 / 推薦）
│   └─ Neo4j
│
├─ 🚀 超大寫入（百萬/秒）
│   └─ Cassandra / ScyllaDB
│
├─ ☁️ AWS Serverless
│   └─ DynamoDB
│
└─ ❓ 不知道用什麼
    → PostgreSQL（永遠不會錯）
```

### 4. 9 大不變鐵則（DB 工程師）

```text
🔒 永遠遵守、不分公司大小

   1. 永遠 backup
      → 3-2-1 規則、每月演練 restore
      → 沒測過的 backup ≈ 沒 backup

   2. 永遠 index
      → WHERE / JOIN / ORDER BY 一定加
      → EXPLAIN ANALYZE 找慢 query

   3. 永遠 connection pool
      → 別 new Client() 每個 request
      → max = DB上限 / instance 數

   4. 永遠 prepared statement
      → 防 SQL injection、效能也好
      → ORM 自動有

   5. 永遠 transaction（多步驟）
      → 扣錢 + 加錢 → 一起成功 / 一起失敗
      → 別讓資料不一致

   6. 永遠 monitor
      → DB 連線數 / 慢 query / 磁碟
      → Grafana + Prometheus

   7. 永遠 timeout
      → query timeout / connection timeout
      → 別 hang 死整台 server

   8. 永遠分頁
      → 別 SELECT * 全表（記憶體爆）
      → cursor pagination

   9. 永遠用 ORM / query builder
      → 別手拼 SQL（SQL injection）
      → Prisma / Drizzle / Mongoose
```

  
### 5. 反 pattern 速查（不要這樣做）

```text
❌ 過早優化
   症狀：MVP 還沒上線就 sharding
   修：先讓單台 PG 撐起來

❌ 為 NoSQL 而 NoSQL
   症狀：聽說 Mongo 快、就跳
   修：規模沒到 = SQL 簡單 10 倍

❌ 全用 SQL 不加 cache
   症狀：每次 query 打 DB
   修：Redis cache-aside

❌ 一個 DB 解決一切
   症狀：堅持只用 PG、所有東西塞
   修：不同資料用不同 DB（polyglot）

❌ 5 個 DB 都用上
   症狀：為了「酷」每種 DB 試一個
   修：1-2 個夠用、別 over-engineer

❌ Mongo 模仿 SQL
   症狀：$lookup（JOIN）寫滿
   修：要 JOIN = 回 PG

❌ Firestore 大資料分析
   症狀：ad-hoc query
   修：ETL 到 BigQuery

❌ 沒 backup
   症狀：手抖刪 production
   修：今天就設定 cron + S3
```

### 8. 何時混用 / 不混用

```text
🔀 必混用（強烈推薦）
   ─────────
   ⃣✓ SQL + Redis（99% Indie SaaS）
   ⃣✓ SQL + Elasticsearch / Meilisearch（要搜尋）
   ⃣✓ SQL + pgvector（AI 應用）

🟡 看情況混用
   ─────────
   ⃣? SQL + MongoDB（log 量大時）
   ⃣? SQL + TimescaleDB（時序量大時）

🔴 不該混用
   ─────────
   ⃣✗ 5 個 DB 都用上（維護地獄）
   ⃣✗ MVP 階段混用（過早優化）
   ⃣✗ 沒明確 use case 的混用
```

  
### 9. 一句話心法

```text
💎 「不知道用什麼 → PostgreSQL」

   ⃣✓ JSONB 等於 Mongo 彈性
   ⃣✓ pgvector 等於向量 DB
   ⃣✓ 全文搜尋等於 Meilisearch 基本款
   ⃣✓ TimescaleDB 等於 InfluxDB 替代
   ⃣✓ 開源、跨雲、工具最多

→ 一個 PG 通殺 80% 場景
→ 加上 Redis 通殺 95%
→ 真的不夠才考慮其他 DB
```

  
### 🎯 重點回顧

- **DB 選型決策樹**：結構化 → PG / 半結構 → Mongo / 快取 → Redis / AI → pgvector / 時序 → TimescaleDB / 圖 → Neo4j
- **按規模成長**：Supabase → PG+Redis+Meilisearch → 自架多 DB → polyglot
- **9 大鐵則**：backup / index / pool / prepared / transaction / monitor / timeout / 分頁 / ORM
- **不知道用什麼 = PostgreSQL**——80% 場景都對
- **不要為混用而混用**——1-2 個 DB 夠 95% Indie SaaS
