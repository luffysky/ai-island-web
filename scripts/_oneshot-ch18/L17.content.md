**Cassandra = 超大規模分散式 wide-column 資料庫**——Facebook 寫出來、Netflix / Apple / Discord 都用。**百萬 / 秒寫入、跨地區、永不停機**。

> 💡 **Cassandra 的招牌**：寫入超快、規模無上限、多 data center 互備。**但學習曲線陡、設計錯改不回來**——是「大公司 DB」、不是新手玩具。

  
### 1. Cassandra 是什麼？

```text
🌐 Apache Cassandra（分散式 NoSQL）

   類型：Wide-Column（寬欄式）
   架構：去中心化（每個節點對等、無 master）
   寫入：超快（百萬 / 秒）
   一致性：可調（從 eventual 到 strong）
   CAP：偏 AP（可用性 > 一致性）

→ Facebook 2008 開源
→ 現在 Apache 基金會維護
→ 「永不停機」的代名詞
```

  
### 2. 為什麼 Cassandra 寫入這麼快？

```text
🚀 Cassandra 寫入快的 3 大祕密

   1. 不檢查 unique（直接寫）
      → SQL 寫入要查 unique constraint
      → Cassandra：寫了再說、覆蓋舊資料

   2. 不寫硬碟（先寫 memory + log）
      → 寫到 memtable（記憶體）
      → 同時寫 commit log（追加、超快）
      → 之後背景 flush 到硬碟

   3. 沒有「master」要協調
      → 任何節點都能收寫入
      → 寫完馬上回 OK、之後 gossip 同步

→ 結果：寫入比 SQL 快 100 倍
→ 代價：讀取慢一些、ad-hoc 查詢痛
```

  
### 3. Consistent Hashing（一致性雜湊）

**Cassandra 怎麼決定資料放哪台機器？**

```text
🎯 Consistent Hashing（一致性雜湊）

想像一個圓環（hash ring）：

           [Node A]
          /        \
   [Node D]       [Node B]
          \        /
           [Node C]

   1. 每個 node（機器）分到圓環一段
   2. 資料 key 雜湊後落到圓環某點
   3. 順時針找到下一個 node → 存那
   4. 加 / 減 node 只影響相鄰範圍

→ 加機器不會「全部資料重排」
→ 比傳統 sharding 優雅 100 倍
```

  
### 4. CQL（Cassandra Query Language）

**CQL 長得像 SQL、但限制多**。

📄 寫進程式檔：`schema.cql`

```text
CREATE KEYSPACE myapp
  WITH replication = {
    'class': 'NetworkTopologyStrategy',
    'us-east-1': 3,
    'eu-west-1': 3
  };

USE myapp;

CREATE TABLE events (
  user_id UUID,
  event_time TIMESTAMP,
  event_type TEXT,
  payload TEXT,
  PRIMARY KEY (user_id, event_time)
) WITH CLUSTERING ORDER BY (event_time DESC);
```

🖥️ 在終端機輸入：

```text
cqlsh -f schema.cql
```

💬 電腦會顯示：

```text
KEYSPACE myapp created
TABLE events created
```

**查詢：**

```text
-- 寫入
INSERT INTO events (user_id, event_time, event_type, payload)
VALUES (uuid(), toTimestamp(now()), 'login', '{}');

-- 查詢（必須用 partition key）
SELECT * FROM events
WHERE user_id = 7a3e... LIMIT 100;
```

  
### 5. Partition Key vs Clustering Key

```text
🔑 PRIMARY KEY (user_id, event_time)
                 ↑          ↑
                 PK         CK

📦 Partition Key（user_id）
   → 決定資料放哪台機器
   → 同 PK 資料在同一台
   → 查詢必須帶 PK

📐 Clustering Key（event_time）
   → 同 PK 內怎麼排序
   → 範圍查（WHERE time > xxx）走索引
   → 適合：時間序、版本歷史
```

> 💡 **Cassandra 查詢鐵則**：**永遠帶 partition key、不然全表掃**。沒帶 PK 的 query 在 production 會被擋。

  
### 6. 限制（不能做什麼）

```text
❌ Cassandra 不能做

   ⃣✗ JOIN（沒有、永遠沒有）
   ⃣✗ Transaction（弱、輕量級 LWT 而已）
   ⃣✗ Ad-hoc 查詢（沒 PK 不能查）
   ⃣✗ Foreign Key（沒有 referential integrity）
   ⃣✗ ORDER BY 任意欄位（只能用 clustering key）
   ⃣✗ GROUP BY 任意欄位（限制多）
   ⃣✗ COUNT 全表（超慢、不準）

→ 為已知 query pattern 設計
→ 想用 SQL 那套？回 PostgreSQL
```

  
### 7. 適用場景

```text
✅ 用 Cassandra 的訊號

   ⃣✓ 寫入量極大（10 萬 / 秒以上）
   ⃣✓ 時序資料（log、metrics、events）
   ⃣✓ 訊息系統（chat、notification）
   ⃣✓ IoT 感測器資料
   ⃣✓ 跨地區、多 DC 部署
   ⃣✓ 不能停機（金融基礎建設）

❌ 不要用 Cassandra
   ⃣✗ 小專案（單台 PG 都夠）
   ⃣✗ 要 JOIN、複雜分析
   ⃣✗ 要 ACID transaction
   ⃣✗ 隨意 ad-hoc 查詢
   ⃣✗ 沒有 DB 專家（很難調）
```

### 9. 替代方案

```text
🌟 比 Cassandra 更好的選擇？

   ScyllaDB（推薦）
   ──────────────

   → 商用版價格不便宜

   CockroachDB
   ──────────────
   → SQL + 分散式（NewSQL）

   DynamoDB
   ──────────────
   → AWS Serverless、不用維護
   → 比 Cassandra 簡單、但 lock-in

   TimescaleDB
   ──────────────
   → 純時序資料、SQL 介面
   → 小到中規模、新手友好
```

  
### 10. 常見坑

```text
❌ 1. Partition Key 不均
   症狀：某 PK 流量 100 倍
   後果：hot partition、效能爛
   修：PK 加 bucket 分散

❌ 2. 用 ALLOW FILTERING
   症狀：query 沒 PK、加 ALLOW FILTERING 硬跑
   後果：全 cluster 掃、超慢、可能掛
   修：永遠用 PK 查、不行就 redesign

❌ 3. 用 Cassandra 做 OLAP
   症狀：跑 GROUP BY + aggregate
   後果：超慢、結果不準
   修：ETL 到 Spark / Snowflake 分析

❌ 4. tombstone（墓碑）爆量
   症狀：刪很多資料 → 查超慢
   後果：tombstone 累積、scan 痛
   修：用 TTL 自動清、別 DELETE
```

  
### 🎯 重點回顧

- **Cassandra = 寫入超快的分散式 wide-column DB**——百萬 / 秒寫入
- **Consistent Hashing**：去中心化、加機器不重排
- **PRIMARY KEY = Partition Key + Clustering Key**——查詢必須帶 PK
- **沒 JOIN、沒 transaction、沒 ad-hoc**——為已知 query pattern 設計
- **規模沒到、別用 Cassandra**——學習成本高、單台 PG 都夠
