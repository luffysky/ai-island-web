**SQL vs NoSQL 怎麼選 = 後端工程師最常被問的問題**——選錯了改不回來、選對了一路順風。**這課給你一套決策流程、不是「看心情」**。

> 💡 **新手最常踩的雷**：看到網路文章說「NoSQL 比較快」、就跳坑用 MongoDB。**結果寫到一半發現要 JOIN、要 transaction、痛到改回 SQL**。這課就是要救你。

  
### 1. 決策的 4 大因素

```text
🎯 選 DB 看 4 件事：

   1. 資料的「形狀」固定嗎？
      → 固定（user / order / product）→ SQL
      → 變動（events / logs / 各種 schema）→ NoSQL Document

   2. 要不要強一致？
      → 要（金流、庫存、轉帳）→ SQL（ACID）
      → 可接受最終一致（社交動態）→ NoSQL

   3. 規模多大？
      → < 1 TB → SQL 單機撐
      → > 10 TB → NoSQL 水平擴展

   4. 團隊熟悉度
      → 8 成工程師都會 SQL → 用 SQL 風險低
      → 全隊都會 Mongo → 也行
```

  
### 2. 該用 SQL 的訊號

```text
✅ 用 SQL（PostgreSQL / MySQL）的訊號

   ⃣✓ 資料結構固定（使用者 / 訂單 / 商品 / 文章）
   ⃣✓ 表跟表有複雜關聯（user → order → product）
   ⃣✓ 要 JOIN（一次查多張表）
   ⃣✓ 需要強一致性（金流、庫存、轉帳）
   ⃣✓ 需要 Transaction（多步驟、全成功才算成功）
   ⃣✓ 規模 < 1 TB（單機撐得住）
   ⃣✓ 不想 vendor lock-in（PG 開源、跑哪都行）
   ⃣✓ 不知道用什麼 = 直接選 SQL（不會錯）
```

> 💡 **ACID 是什麼**：Atomicity（全有或全無）/ Consistency（永遠合法）/ Isolation（互不打架）/ Durability（寫了就不會丟）—— 銀行轉帳的必備保護。

  
### 3. 該用 NoSQL Document（MongoDB）的訊號

```text
✅ 用 MongoDB / Firestore 的訊號

   ⃣✓ Schema（資料結構）變化快（早期 MVP 一週改三次）
   ⃣✓ 資料天然是 JSON（log、events、API response）
   ⃣✓ 需要橫向擴展到超大規模（PB 級）
   ⃣✓ 不太需要複雜 JOIN（單一 collection 查就夠）
   ⃣✓ 全隊都是 JS / TS（Mongo 跟 JS 物件 1:1）
   ⃣✓ user-generated content（每個用戶 schema 不同）
```

  
### 4. 該用 Firebase 的訊號

```text
✅ 用 Firebase / Supabase 的訊號

   ⃣✓ 行動 App / Web 即時同步（chat、共筆）
   ⃣✓ 不想自己架後端（auth、DB、push 全包）
   ⃣✓ MVP / 學生作品 / hackathon（一週上線）
   ⃣✓ Google 生態（FCM 推播、Analytics）
   ⃣✓ 小團隊、沒專職 backend

→ Firebase = NoSQL Document（Firestore）+ Auth + Storage
→ Supabase = SQL（PG）+ Auth + Storage（PG 版的 Firebase）
```

  
### 5. 該用 Redis 的訊號

```text
✅ 用 Redis 的訊號

   ⃣✓ 已有主 DB（PG / MySQL）、需要加速
   ⃣✓ Session 儲存（登入狀態、購物車）
   ⃣✓ 排行榜（sorted set 超快）
   ⃣✓ 限流（rate limit、INCR + EXPIRE）
   ⃣✓ Pub/Sub（即時通知、聊天）
   ⃣✓ 短期 key-value 快取（API response）

→ Redis 幾乎是「必加」的、不是「替代」
→ 跟主 DB 共存、各做各的
```

  
### 6. 該用 Vector DB 的訊號

```text
✅ 用 Vector DB（pgvector / Pinecone）的訊號

   ⃣✓ AI 應用（RAG、推薦、搜尋）
   ⃣✓ 語意搜尋（不是關鍵字、是「意思」）
   ⃣✓ 處理 embedding（文字 / 圖 / 音訊）
   ⃣✓ 要做 ChatGPT 客服 / 文件搜尋

→ similarity search（相似度搜尋）= 找「向量距離近」的
→ 例：搜「便宜手機」→ 找到「平價智慧型手機」
→ 不是字一樣、是意思相近
```

  
### 7. 2026 黃金組合（按專案規模）

```text
🎓 學生 / MVP（0-1K 用戶）
   → Firebase 或 Supabase（單一服務搞定）
   → 1 週上線、不用 DevOps

🏠 Indie SaaS（1K-50K 用戶）
   → Supabase + Upstash Redis
   → PG（業務）+ Redis（cache / session）
   → 月成本 $30-$50

🏢 中型 SaaS（50K-500K 用戶）
   → PostgreSQL（自架或 RDS）+ Redis + S3
   → 加 pgvector → AI 功能
   → 加 Meilisearch → 全文搜尋

🤖 AI 應用（任何規模）
   → PostgreSQL + pgvector + Redis（RAG 標配）
   → 或 PG + Pinecone（向量量大時）

🏭 大型 / 多模態（百萬用戶以上）
   → PG（業務、強一致）
   → MongoDB（log、彈性）
   → Redis（cache、即時）
   → Pinecone（vector、AI）
   → Elasticsearch（搜尋、日誌）
```

  
### 8. 真實 case study

```text
📦 Shopify（電商）
   → MySQL 為主（訂單、商品、用戶）
   → Redis（cache + session）
   → Elasticsearch（產品搜尋）
   → 啟示：再大的電商、主 DB 還是 SQL

🎵 Spotify（音樂串流）
   → Cassandra（播放紀錄、超大規模 write）
   → PostgreSQL（用戶、訂閱）
   → BigQuery（分析）
   → 啟示：不同資料、不同 DB（polyglot persistence）

💬 Discord（聊天）
   → Cassandra（訊息、每天百億級寫入）
   → 後來改 ScyllaDB（C++ 重寫的 Cassandra、快 10 倍）
   → 啟示：write-heavy 用 wide-column

🤖 Notion AI
   → PostgreSQL（業務）
   → pgvector（向量搜尋）
   → 啟示：PG 一條龍、不亂加 DB
```

  
### 9. 反模式：不要這樣做

```text
❌ 1. 用 MongoDB 存金流 / 訂單
   → 沒 ACID、可能丟錢
   → 修：金流一律用 SQL

❌ 2. 用 SQL 存大量 log
   → 變動快、JOIN 用不到、浪費 SQL 強項
   → 修：log 用 MongoDB / Elasticsearch

❌ 3. 每查一次資料都直接打 DB
   → DB 會被打爆、回應變慢
   → 修：一定要加 Redis 快取

❌ 4. 為了「酷」用 5 種 DB
   → 維護地獄、運維成本爆表
   → 修：選 1-2 個 DB、夠用就好

❌ 5. 規模沒大就 sharding（分片）
   → 過早優化、複雜度爆炸
   → 修：百萬筆內、單機 PG 都行

❌ 6. 用 NoSQL 又模仿 SQL
   → 寫一堆 $lookup、aggregate JOIN
   → 修：用 NoSQL 就接受它的限制、別硬模仿
```

  
### 🎯 重點回顧

- **選 DB 看 4 件事**：結構固定嗎 / 要強一致嗎 / 規模多大 / 團隊熟什麼
- **不知道選什麼 → 選 PostgreSQL**——80% 場景都對
- **Redis 幾乎必加**——當快取 / session / 排行榜
- **AI 應用 = PG + pgvector + Redis**——2026 RAG 標配
- **真實大公司都用 polyglot**——不同資料用不同 DB、不要堅持「一個 DB 解決一切」
