**Neo4j = 圖式資料庫的代表**——專門存「關係」、不是「表格」。**FB 朋友推薦、IG 推薦你認識的人、Amazon 「買 X 也買 Y」全靠它**。

> 💡 **新手最快理解**：SQL 像 Excel（一格一格）、MongoDB 像 JSON（一個個物件）、**Neo4j 像「心智圖」**——節點 + 連線、找關係超快。

  
### 1. Graph DB 是什麼？

```text
🕸️ Graph Database（圖式資料庫）

   存什麼：
   ⃣◯ Node（節點）= 一個東西（user / product / 文章）
   ⃣→ Relationship（關係）= 節點之間的連結

   範例：
   (Alice) -[:FRIEND]-> (Bob)
   (Bob)   -[:LIKES]->   (Pizza)
   (Alice) -[:BOUGHT]-> (iPhone)

→ 不是表格、不是文件、是「圖」
→ 找關係比 SQL 快 100-1000 倍
→ 代表：Neo4j / Dgraph / ArangoDB / Amazon Neptune
```

  
### 2. 為什麼不用 SQL 做關係？

**SQL 不是有 JOIN 嗎？為什麼還要 Graph DB？**

```text
🐢 SQL 做關係的痛點

   找「Alice 的朋友的朋友」：

   SQL：
   ────
   SELECT f2.user_id
   FROM friendships f1
   JOIN friendships f2 ON f1.friend_id = f2.user_id
   WHERE f1.user_id = 'alice';

   → 2 層 JOIN、慢

   找「Alice 的朋友的朋友的朋友」：
   → 3 層 JOIN、超慢

   找「Alice 跟 Bob 之間最近的關係鏈」：
   → SQL 寫不出來（要 recursive CTE、地獄）

   Neo4j：
   ───────
   MATCH (a {name:"Alice"})-[:FRIEND*1..6]-(b {name:"Bob"})
   RETURN b;

   → 一行、超快、6 層也輕鬆
```

> 💡 **Neo4j 的招牌**：**多層關係查詢、SQL 不可能、Neo4j 一行**。社交網路、推薦系統的命根。

  
### 3. Cypher（Neo4j 的查詢語言）

**Cypher = 「畫圖」式的查詢語法**——括號 `()` 是節點、箭頭 `->` 是關係。

📄 寫進程式檔：`graph.cypher`

```text
-- 建立節點 + 關係
CREATE (a:User {name: "Alice", age: 25}),
       (b:User {name: "Bob", age: 28}),
       (c:User {name: "Carol", age: 22}),
       (a)-[:FRIEND]->(b),
       (b)-[:FRIEND]->(c),
       (a)-[:LIKES]->(:Movie {title: "Inception"});

-- 查詢：Alice 的朋友
MATCH (a:User {name: "Alice"})-[:FRIEND]->(f)
RETURN f.name;

-- 查詢：朋友的朋友（推薦交友）
MATCH (me:User {name: "Alice"})-[:FRIEND]->(f)-[:FRIEND]->(fof)
WHERE NOT (me)-[:FRIEND]->(fof) AND me <> fof
RETURN fof.name, COUNT(*) AS mutual
ORDER BY mutual DESC
LIMIT 10;
```

  
### 4. 用 Node.js 連 Neo4j

📄 寫進程式檔：`neo4j-demo.ts`

```text
import neo4j from "neo4j-driver";

const driver = neo4j.driver(
  "neo4j://localhost:7687",
  neo4j.auth.basic("neo4j", "your_password")
);

const session = driver.session();

// 寫入
await session.run(
  `CREATE (u:User {name: $name, age: $age})`,
  { name: "Alice", age: 25 }
);

// 查詢
const result = await session.run(
  `MATCH (u:User {name: $name}) RETURN u`,
  { name: "Alice" }
);

result.records.forEach(record => {
  console.log(record.get("u").properties);
});

await session.close();
await driver.close();
```

🖥️ 在終端機輸入：

```text
npm install neo4j-driver
node neo4j-demo.ts
```

💬 電腦會顯示：

```text
{ name: 'Alice', age: { low: 25, high: 0 } }
```

  
### 5. 推薦系統（協同過濾）

**「買了 X 的人也買了 Y」怎麼做？**

```text
-- 找「跟我買一樣東西的人」買的其他東西
MATCH (me:User {id: $userId})-[:BOUGHT]->(p:Product)
      <-[:BOUGHT]-(other:User)-[:BOUGHT]->(rec:Product)
WHERE NOT (me)-[:BOUGHT]->(rec)
RETURN rec.name, COUNT(*) AS score
ORDER BY score DESC
LIMIT 10;
```

**這在 SQL 寫起來地獄、Neo4j 一行解決**。Amazon、Netflix 推薦背後都這套。

### 7. 何時不該用 Graph DB？

```text
❌ 不要用 Graph DB

   ⃣✗ 結構固定、不太需要關係查詢
      → 用 SQL（PostgreSQL）

   ⃣✗ 大量寫入、少查詢
      → Graph 寫入比 SQL 慢

   ⃣✗ 大量 ad-hoc 分析
      → 用 BigQuery / Snowflake

   ⃣✗ 規模超大但關係淺（1-2 層）
      → SQL JOIN 還能撐

   ⃣✗ 不知道怎麼用就用
      → Graph DB 學習成本高、團隊要培訓
```

  
### 8. 替代方案 + 託管服務

```text
🌟 Graph DB 選項

   Neo4j（業界標準）
   ─────────────────
   → 最成熟、Cypher 語法業界通用
   → Community 免費 / Enterprise 付費
   → Aura（雲端託管、Neo4j 官方）

   Amazon Neptune
   ─────────────────
   → AWS 託管
   → 支援 Gremlin + SPARQL + openCypher
   → 想 AWS 一條龍選這

   ArangoDB
   ─────────────────
   → Multi-model（document + graph + KV）
   → 用 1 個 DB 想要多種能力時

   Dgraph
   ─────────────────
   → 開源、Go 寫
   → GraphQL 原生（不是 Cypher）

   pgvector + PG 簡單關係
   ─────────────────
   → 小規模、PG 用 recursive CTE 也行
   → 沒到萬筆關係、別搬 Neo4j
```

### 10. 常見坑

```text
❌ 1. 用 Graph 存純資料
   症狀：什麼都塞 Neo4j、沒在用關係
   後果：浪費 Graph 強項、效能差
   修：純資料用 SQL、關係用 Graph

❌ 2. 關係太深（10+ 層）
   症狀：MATCH ()-[:R*1..10]-()
   後果：指數級爆炸、查超久
   修：限制深度（1..3）、加 WHERE 條件

❌ 3. 沒 INDEX
   症狀：MATCH (u:User {email: $e})
   後果：全圖掃
   修：CREATE INDEX FOR (u:User) ON (u.email)

❌ 4. 用 Cypher 取代 SQL
   症狀：每個查詢都 Cypher
   後果：簡單查詢也複雜
   修：Polyglot——SQL 為主、Graph 為輔
```

  
### 🎯 重點回顧

- **Graph DB = 專門存關係**——節點 + 邊、找關係超快
- **Neo4j + Cypher 是業界標準**——畫圖式語法、直觀
- **多層關係查詢比 SQL 快 100 倍**——社交 / 推薦的命根
- **不是 SQL 替代**——polyglot：主 DB 用 SQL、關係用 Graph
- **小規模 / 關係淺 = SQL 就行**——別為 Graph 而 Graph
