**DynamoDB = AWS 的 NoSQL key-value / document 資料庫**——Serverless（無伺服器）、按請求次數收費、自動水平擴展到無限大。**Amazon、Lyft、Netflix 都用它**。

> 💡 **新手記這句**：DynamoDB 是「AWS 版的 MongoDB + Redis」——一個服務搞定 key-value 跟 document、不用自己架伺服器、用多少付多少。

  
### 1. DynamoDB 是什麼？

```text
☁️ DynamoDB（AWS 的 NoSQL DB）

   類型：Key-Value + Document 混合
   架構：Serverless（無伺服器、AWS 全託管）
   定價：按請求次數 + 儲存量收費
   擴展：自動、無上限（理論上）
   一致性：可選 eventual / strong（最終 / 強）

→ 不用裝、不用維護、不用調效能
→ AWS 內 Lambda / API Gateway 標配搭檔
→ 缺點：vendor lock-in（離不開 AWS）
```

  
### 2. 三大核心概念

```text
🔑 Partition Key（分區鍵、PK）
   ─────────────────────────────
   作用：唯一識別 + 決定資料分布到哪台機器
   像 SQL：primary key
   範例：user_id = "u123"

   → 同 PK 的資料在同一台機器
   → 設計時要讓 PK 分散均勻、不要熱點

📐 Sort Key（排序鍵、SK）
   ─────────────────────────────
   作用：搭 PK 變 composite key（複合鍵）
        可以排序、可以範圍查
   範例：order_id = "2026-05-28#001"

   → PK + SK 一起唯一識別一筆資料
   → 同 PK 內按 SK 排序儲存
   → 適合：時間序、版本歷史

🔍 GSI（Global Secondary Index、全域次索引）
   ─────────────────────────────
   作用：另一組 PK + SK、查不同欄位用
   像 SQL：建立 index
   範例：原本 PK=user_id、GSI PK=email

   → 一張表可以建多個 GSI
   → 每個 GSI 都會額外收費（注意）
```

> 💡 **DynamoDB 跟 SQL 最大差異**：**只能用 PK / SK / GSI 查詢**——不能像 SQL 一樣「WHERE 任何欄位」。**設計時就要想好所有 access pattern**。

  
### 3. AWS SDK：基本 CRUD

📄 寫進程式檔：`dynamo.ts`

```text
import { DynamoDBClient } from "@aws-sdk/client-dynamodb";
import {
  DynamoDBDocumentClient,
  PutCommand,
  GetCommand,
  QueryCommand,
  UpdateCommand,
  DeleteCommand,
} from "@aws-sdk/lib-dynamodb";

const client = DynamoDBDocumentClient.from(new DynamoDBClient({}));

// 寫入（Create）
await client.send(new PutCommand({
  TableName: "orders",
  Item: {
    user_id: "u1",
    order_id: "2026-05-28#001",
    amount: 100,
    status: "paid",
  },
}));

// 讀取單筆（Read by PK+SK）
const { Item } = await client.send(new GetCommand({
  TableName: "orders",
  Key: { user_id: "u1", order_id: "2026-05-28#001" },
}));

// 查詢同 PK 多筆（Query）
const { Items } = await client.send(new QueryCommand({
  TableName: "orders",
  KeyConditionExpression: "user_id = :uid",
  ExpressionAttributeValues: { ":uid": "u1" },
}));

// 更新（Update）
await client.send(new UpdateCommand({
  TableName: "orders",
  Key: { user_id: "u1", order_id: "2026-05-28#001" },
  UpdateExpression: "SET #s = :status",
  ExpressionAttributeNames: { "#s": "status" },
  ExpressionAttributeValues: { ":status": "shipped" },
}));
```

🖥️ 在終端機輸入：

```text
npm install @aws-sdk/client-dynamodb @aws-sdk/lib-dynamodb
node dynamo.ts
```

💬 電腦會顯示：

```text

```

  
### 4. Single Table Design（單表設計）

**DynamoDB 進階心法**：所有資料塞一張表、用 PK / SK 命名規則區分。

```text
PK              | SK                  | data
────────────────┼─────────────────────┼─────────────────
USER#u1         | PROFILE             | { name, email }
USER#u1         | ORDER#2026-05-28    | { amount: 100 }
USER#u1         | ORDER#2026-05-27    | { amount: 200 }
PRODUCT#p1      | DETAIL              | { name, price }
PRODUCT#p1      | REVIEW#r1           | { stars: 5 }

→ 一個 Query 抓某用戶的所有資料（USER#u1）
→ 一個 Query 抓某產品的詳情 + 評論
→ 效能爆好、但設計超難
```

> 💡 **Single Table Design 是 DynamoDB 聖經、也是地獄**——設計對效能無敵、設計錯改不回來。**新手別直接玩、先用一張表一個 entity**。

  
### 5. Pricing（價錢怎麼算？）

```text
💰 DynamoDB 兩種計費模式

   1. On-Demand（按用量）
      → 寫入：$1.25 / 百萬次
      → 讀取：$0.25 / 百萬次
      → 適合：流量不固定、新專案

   2. Provisioned（預先買 capacity）
      → 預先買 RCU（讀）/ WCU（寫）
      → 用不完浪費、不夠用會被限流
      → 適合：流量穩定、可預測

📦 儲存：$0.25 / GB / 月
🔍 GSI：每個 GSI 額外算（讀寫 + 儲存）

→ 算下來：1 萬 user、每天 10 query
   → 月成本 ~$5（極便宜）
→ 100 萬 user、每天 100 query
   → 月成本 ~$500-$2000
```

  
### 6. DynamoDB Streams（變更串流）

```text
🌊 DynamoDB Streams = 自動追蹤資料變化

每當 DynamoDB 有寫入 / 更新 / 刪除：
→ 自動推一筆 event 到 Stream
→ Lambda 可以監聽 Stream
→ 做即時通知、同步到 Elasticsearch、稽核 log

→ 用法：CDC（Change Data Capture）
→ 比自己寫「資料變了就推訊息」優雅 1000 倍
```

  
### 7. 何時用 / 不該用

```text
✅ 該用 DynamoDB
   ⃣✓ AWS Serverless 架構（Lambda + API Gateway）
   ⃣✓ 已知 access pattern（查詢方式固定）
   ⃣✓ 巨量、需要彈性擴展
   ⃣✓ 寫入量極大（百萬/秒）
   ⃣✓ 不想管 DB 運維

❌ 不該用 DynamoDB
   ⃣✗ Ad-hoc 查詢（隨意查）
   ⃣✗ 複雜分析（要 JOIN / aggregate）
   ⃣✗ 不在 AWS 上（lock-in）
   ⃣✗ 寫不出 access pattern（schema 沒想清楚）
   ⃣✗ 小專案（單台 PG 都比較便宜）
```

  
### 8. DynamoDB vs MongoDB 對照

```text
面向            DynamoDB         MongoDB
──────────────────────────────────────────
類型           Key-Value + Doc  Document
查詢           只能 PK/SK/GSI   彈性、aggregate
host           AWS only         自架 / Atlas
schema 彈性    高              極高
規模上限       無限（雲）       PB 級
學習曲線       高（要懂單表設計）中
工具生態       AWS 內豐富       獨立生態（Compass）
定價           按請求次數       按伺服器 / Atlas tier
```

  
### 9. 常見坑

```text
❌ 1. PK 設計太集中（hot partition）
   症狀：某 PK 流量 100 倍、其他冷
   後果：那台機器爆、被限流
   修：PK 加 suffix 分散（"user_id#${random}"）

❌ 2. 沒想好 access pattern
   症狀：寫完發現要查的查不到
   後果：要全表 scan（超慢、超貴）
   修：設計表前畫好所有 query

❌ 3. 過度用 GSI
   症狀：每個 query 都建 GSI
   後果：每個 GSI 都算錢、總費爆表
   修：合併 access pattern、少建 GSI

❌ 4. 用 scan 不用 query
   症狀：寫 ScanCommand 全表掃
   後果：超慢 + 超貴
   修：永遠用 query、靠 PK / SK 索引
```

  
### 🎯 重點回顧

- **DynamoDB = AWS Serverless 標配**——不用維護、按用量收費
- **PK + SK + GSI 三大概念**——查詢只能靠這 3 個
- **Single Table Design 是進階心法**——所有資料塞一張表、命名分區
- **設計前先列 access pattern**——不像 SQL 隨意查、要先想好
- **AWS lock-in 風險**——離不開 AWS、選之前想清楚
