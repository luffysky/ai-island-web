**NoSQL（Not Only SQL、不只是 SQL）= SQL 之外的另一條路**——不用先定義表格、不用嚴格欄位、按「資料長什麼樣」選最適合的存法。**SQL 是萬用刀、NoSQL 是專業工具組**。

> 💡 **新手先懂這句**：SQL 像 Excel（表格固定、欄位整齊）、NoSQL 像 Google Drive（什麼檔案都能丟、各自有各自的形狀）。**兩個不是對立、是互補**。

  
### 1. NoSQL 是什麼？跟 SQL 差在哪？

```text
🗂️ SQL（傳統關聯式資料庫）
   → 像 Excel、先畫好欄位（user_id / name / email）才能存
   → 改欄位很麻煩（要 ALTER TABLE）
   → 強項：JOIN（多張表合起來查）、ACID（不會出錯）
   → 代表：PostgreSQL / MySQL / SQLite

📦 NoSQL（非關聯式資料庫）
   → 像 Google Drive、每筆資料長什麼樣都行
   → 加欄位隨便加（不用先宣告）
   → 強項：水平擴展（加機器就行）、特殊資料型態
   → 代表：MongoDB / Redis / Cassandra / Neo4j
```

> 💡 **NoSQL 不是「沒 SQL」**——是「Not Only SQL」、不只用 SQL。很多 NoSQL 也有自己的查詢語言。

  
### 2. NoSQL 的 5 大類型

```text
📄 1. Document（文件式）
   → 每筆資料是一個 JSON 物件
   → 像：{ name: "小明", age: 18, hobbies: ["音樂", "程式"] }
   → 代表：MongoDB / Couchbase / Firestore
   → 適合：App 後端、user-generated content

🔑 2. Key-Value（鍵值對）
   → 像超大 hash map：key → value
   → 像：set("user:123:name", "小明")
   → 代表：Redis / DynamoDB / Memcached
   → 適合：快取（cache）、Session、排行榜

📊 3. Wide-Column（寬欄式）
   → 像 SQL 但欄位可以一列一個樣
   → 一張表可以存幾十億筆
   → 代表：Cassandra / HBase / ScyllaDB
   → 適合：超大規模寫入、IoT、log

🕸️ 4. Graph（圖式）
   → 存「節點」（node）+ 「關係」（relationship）
   → 像社交網路：A 是 B 的朋友、B 追蹤 C
   → 代表：Neo4j / Dgraph / ArangoDB
   → 適合：社交、推薦系統、知識圖譜

🧠 5. Vector（向量、2023+ 新類型）
   → 把文字 / 圖片 / 音訊轉成數字陣列（embedding）
   → 找「意思相近」而不是「字一樣」
   → 代表：Pinecone / Qdrant / Weaviate / pgvector
   → 適合：AI 應用、RAG、語意搜尋
```

> 💡 **embedding（嵌入向量）**= 把一句話「翻譯成」一串數字（例如 1536 個浮點數）、長得像 [0.12, -0.45, 0.88, ...]。**意思相近的句子 → 向量也相近**、這就是 AI 搜尋的原理。

  
### 3. SQL vs NoSQL 完整對照

```text
面向               SQL                NoSQL
────────────────────────────────────────────
Schema（結構）    嚴格、先定義        彈性、隨意加
查詢語法           SQL（一招走天下）   各 DB 不同
JOIN（多表合查）   完整支援            通常沒有
Transaction       ACID 完整           部分支援
一致性             強一致              通常最終一致（BASE）
擴展方式           垂直（升級機器）    水平（加機器）
資料量             單機 ~ TB           PB（千倍以上）
學習成本           一次學會            每個 DB 都要學
```

  
### 4. 什麼是 CAP 定理？

**CAP 是分散式系統的鐵則**——三選二、不能全拿。

```text
🏛️ CAP theorem（CAP 定理）= 分散式資料庫的三角矛盾

   C — Consistency（一致性）
       所有節點同一時間看到一樣的資料

   A — Availability（可用性）
       永遠回應、不會掛掉

   P — Partition tolerance（分區容錯）
       網路斷掉時還能正常工作

→ 只能選 2 個、不能 3 個都要
→ 實務上 P 是必須的（網路一定會斷）
→ 所以真正在選的是「C vs A」
```

**怎麼選：**

```text
🔒 CP（一致性優先）— 寧可掛掉也不要錯
   代表：傳統 SQL、MongoDB（預設）、HBase
   場景：銀行轉帳、訂單金流

⚡ AP（可用性優先）— 寧可慢一點對也要回應
   代表：Cassandra、DynamoDB、Redis（cluster）
   場景：社交動態、購物車、IoT
```

  
### 5. BASE：NoSQL 的設計哲學

SQL 講究 ACID（不能錯）、NoSQL 走另一條路叫 BASE。

```text
🌿 BASE（NoSQL 的鬆散一致性）

   Basically Available — 基本可用
       永遠能回應、就算資料舊一點

   Soft state — 軟狀態
       資料可以暫時不一致

   Eventually consistent — 最終一致
       「過一陣子」會同步、不是立刻

→ 不像 ACID 一分一秒都不能錯
→ 適合：規模超大、容許短暫延遲的場景
```

**舉例：** 你在 Instagram 按 like、可能 1 秒後另一個地區的朋友才看到、但**最終會同步**——這就是 eventually consistent（最終一致）。

  
### 6. 什麼時候該用 NoSQL？

```text
✅ 該用 NoSQL 的場景
   📊 日誌 / 事件：MongoDB / Cassandra（量大、結構變動）
   💬 即時聊天：Firebase / Redis Pub/Sub
   ⚡ 快取 / Session：Redis（必選）
   🤖 AI 向量搜尋：pgvector / Pinecone
   📱 App 後端（schema 變動快）：Firebase / MongoDB
   🗺️ 地理 / 社交網路：MongoDB / Neo4j
   📈 IoT / 時序：TimescaleDB / InfluxDB

❌ 不該用 NoSQL 的場景
   💰 金流 / 訂單（要 ACID）→ 用 SQL
   🔗 多表複雜查詢（要 JOIN）→ 用 SQL
   📝 結構固定的業務資料 → 用 SQL
   🆕 不知道用什麼 → 用 SQL（PostgreSQL）
```

  
### 7. 2026 主流 stack（推薦組合）

```text
🎓 學生 / MVP
   → Supabase（PG + Auth + Storage 一站式）

🏠 Indie SaaS（個人開發者）
   → PostgreSQL（主）+ Redis（快取）
   → 加 pgvector → 做 AI 應用

🏢 中型 SaaS（1K-100K 用戶）
   → PostgreSQL + Redis + Elasticsearch（搜尋）

🏭 大型 / 多模態（百萬用戶以上）
   → PostgreSQL（業務）+ MongoDB（log）
   → + Redis（cache）+ Pinecone（vector）
   → + Cassandra（時序資料）
```

> 💡 **2026 真相**：PostgreSQL 已經太強——JSONB（NoSQL 彈性）+ pgvector（AI）+ Full-text search（搜尋）一個 PG 通殺。**沒大到需求、不要為 NoSQL 而 NoSQL**。

  
### 🎯 重點回顧

- **NoSQL = Not Only SQL**——SQL 之外的另一條路、不是對立
- **5 大類型**：document（文件）/ key-value（鍵值）/ wide-column（寬欄）/ graph（圖）/ vector（向量）
- **CAP 定理**：分散式系統只能 C / A 二選一（P 必選）
- **BASE**：NoSQL 的鬆散哲學（最終一致 vs SQL 的 ACID）
- **沒大規模、結構固定 = 用 SQL**——80% 場景 PostgreSQL 都夠
