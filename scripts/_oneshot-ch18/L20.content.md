**Elasticsearch / Meilisearch = 全文搜尋專用引擎**——SQL `LIKE '%關鍵字%'` 慢到爆、這兩個快 1000 倍。**電商、文章、log、聊天記錄搜尋**全靠它們。

> 💡 **新手最直觀感受**：你在蝦皮搜「藍芽耳機」、馬上跳出 500 個商品 + 篩選 + typo 容錯——背後就是 Elasticsearch / Meilisearch。**沒它們、SQL 慢到不能用**。

  
### 1. 全文搜尋為什麼需要專用引擎？

```text
🐢 用 SQL 做搜尋的痛點

   SELECT * FROM products
   WHERE name LIKE '%藍芽耳機%';

   問題：
   1. 100 萬筆 → 全表掃、3 秒
   2. 沒辦法處理「typo」（藍芽 vs 蓝牙 vs 藍牙）
   3. 不能排序「相關度」（哪個最相關？）
   4. 不能加權（標題比內容重要）
   5. 不支援中文斷詞（中文沒空格）

🚀 用搜尋引擎

   index 結構：
   "藍芽" → [商品1, 商品3, 商品7, ...]
   "耳機" → [商品1, 商品2, 商品3, ...]

   搜「藍芽耳機」：
   → 找兩個 list 的交集（毫秒級）
   → 自動算相關度（TF-IDF / BM25）
   → 支援 typo / 同義詞 / 中文

→ 速度差距：3 秒 → 5 ms（600 倍）
```

  
### 2. Inverted Index（倒排索引）

```text
🔄 Inverted Index 是搜尋引擎的核心

   一般索引（forward）：
   商品1 → [藍芽, 耳機, 耳塞]
   商品2 → [手機, 平板]
   商品3 → [藍芽, 喇叭]

   倒排索引（inverted）：
   藍芽 → [商品1, 商品3]
   耳機 → [商品1]
   手機 → [商品2]
   喇叭 → [商品3]

→ 搜「藍芽」：直接看 list、O(1) 找到
→ 搜「藍芽 AND 耳機」：算交集 [商品1]
→ 比全表掃快 1000 倍
```

  
### 3. Meilisearch（推薦 Indie / SaaS）

**Meilisearch = 「新手友善版 Elasticsearch」**——Rust 寫、開箱即用、設定超少。

🖥️ 在終端機輸入：

```text
docker run -d -p 7700:7700 \
  -e MEILI_MASTER_KEY=your_master_key \
  --name meilisearch \
  getmeili/meilisearch:latest
```

📄 寫進程式檔：`meili-demo.ts`

```text
import { MeiliSearch } from "meilisearch";

const client = new MeiliSearch({
  host: "http://localhost:7700",
  apiKey: "your_master_key",
});

// 1. 加文件
await client.index("products").addDocuments([
  { id: 1, name: "iPhone 17 Pro", brand: "Apple", price: 35900 },
  { id: 2, name: "Galaxy S25", brand: "Samsung", price: 28900 },
  { id: 3, name: "AirPods Pro 3", brand: "Apple", price: 7990 },
]);

// 2. 搜尋（毫秒級）
const res = await client.index("products").search("iphone", {
  filter: 'price < 40000',
  sort: ['price:asc'],
  limit: 20,
});

console.log(res.hits);
```

🖥️ 在終端機輸入：

```text
npm install meilisearch
node meili-demo.ts
```

💬 電腦會顯示：

```text
[
  { id: 1, name: 'iPhone 17 Pro', brand: 'Apple', price: 35900 }
]
```

> 💡 **Meilisearch 的招牌**：**typo tolerance（打錯字也找得到）內建**。打「ipone」也找到 iPhone、打「藍芽耳機」找到「藍牙耳機」。**0 設定**。

  
### 4. Elasticsearch（企業級）

**Elasticsearch = 業界搜尋之王**——Lucene 內核、功能爆多、設定也複雜。

📄 寫進程式檔：`es-demo.ts`

```text
import { Client } from "@elastic/elasticsearch";

const es = new Client({
  node: "http://localhost:9200",
});

// 寫入
await es.index({
  index: "products",
  document: { name: "iPhone 17 Pro", price: 35900, brand: "Apple" },
});

// 複雜查詢（bool + filter + range）
const res = await es.search({
  index: "products",
  query: {
    bool: {
      must: [{ match: { name: "iPhone" } }],
      filter: [
        { range: { price: { lte: 50000 } } },
        { term: { brand: "Apple" } },
      ],
    },
  },
  size: 20,
  sort: [{ price: "asc" }],
});
```

### 6. typo tolerance（容錯）怎麼運作？

```text
🎯 容錯距離（edit distance、Levenshtein）

   "ipone" → "iphone"
   差距：1 個字（多了 h）
   → 視為相同

   "藍芽" → "藍牙"
   差距：1 個字
   → 視為相同（同義詞 + 容錯）

   "ehad" → "head"
   差距：2 個字
   → 仍視為相同

→ Meilisearch 預設：
   < 5 字：容錯 1 個
   ≥ 9 字：容錯 2 個
→ 可調設定
```

  
### 7. 中文斷詞（CJK 支援）

```text
🇨🇳 中文搜尋難點

   英文："I love iPhone" → 空格自然斷詞
   中文："我愛蘋果手機" → 沒空格、怎麼斷？

   斷錯：我 / 愛蘋 / 果手 / 機（無意義）
   斷對：我 / 愛 / 蘋果 / 手機（有意義）

→ Elasticsearch：IK analyzer（最有名中文分詞器）
→ Meilisearch：內建支援中文（v1.0+）
→ PostgreSQL：pg_jieba（中文斷詞）
```

  
### 8. PostgreSQL 全文搜尋（小規模夠用）

**沒到中型規模、PG 內建就行**。

```text
-- 建 GIN index
CREATE INDEX ix_search ON products
  USING GIN(to_tsvector('simple', name || ' ' || description));

-- 查詢
SELECT *,
  ts_rank(to_tsvector('simple', name), query) AS rank
FROM products,
  plainto_tsquery('simple', 'iPhone Pro') query
WHERE to_tsvector('simple', name) @@ query
ORDER BY rank DESC
LIMIT 20;
```

  
### 9. 怎麼選？

```text
📋 場景 → 選什麼

   ⃣🎓 小型（< 100k 筆）→ PG 全文搜尋
   ⃣🏠 中型（Indie）   → Meilisearch ⭐
   ⃣🏢 大型 + log     → Elasticsearch
   ⃣☁️ 不想自架       → Algolia（SaaS、付費）
   ⃣🤖 語意搜尋       → pgvector / Pinecone（向量、L1 介紹）

→ Meilisearch 是 2026 Indie 推薦
→ ES 是企業標配、但運維複雜
→ Algolia 是 SaaS 商品搜尋之王（FB / Stripe 都用）
```

### 11. 常見坑

```text
❌ 1. 用 SQL LIKE 做大規模搜尋
   症狀：100 萬筆 → 3 秒
   修：上 Meilisearch / ES

❌ 2. 沒同步 SQL → ES
   症狀：SQL 改了、ES 還是舊
   修：用 CDC（Debezium）或寫穿（write-through）

❌ 3. 不分 index
   症狀：products / users / posts 全塞同 index
   後果：搜尋慢、難維護
   修：每種資料一個 index

❌ 4. ES heap 太小
   症狀：OOM（記憶體爆）
   修：xmx / xms 設定機器一半 RAM
```

  
### 🎯 重點回顧

- **全文搜尋 = inverted index**——比 SQL LIKE 快 1000 倍
- **Meilisearch = Indie 推薦**——Rust 寫、設定 0、typo 內建
- **Elasticsearch = 企業標配**——功能多、規模大、複雜
- **小規模 PG 全文搜尋夠用**——百萬筆內不用上 ES
- **AI 語意搜尋 = 向量 DB**（pgvector / Pinecone）—不是字一樣、是意思接近
