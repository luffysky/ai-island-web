**Backup（備份）/ Restore（還原）= DB 工程師的命根**——沒備份等於沒命。**手抖刪掉 production table 的故事每年都發生**、有 backup 救你、沒 backup 公司倒閉。

> 💡 **業界鐵則**：**「沒測過的 backup ≈ 沒 backup」**。很多公司每天備份、3 年沒測 restore、真的要救時發現備份壞了——這課教你怎麼做對。

  
### 1. 為什麼一定要備份？

```text
💀 沒備份的災難（真實案例）

   1. GitLab 2017
      → 工程師打錯指令、刪了 production DB
      → 5 個備份方案都壞了
      → 救回來 = 6 小時前的 snapshot
      → 6 小時資料永久丟失

   2. Code Spaces 2014
      → AWS 帳號被駭、所有資料 + 備份刪光
      → 公司當天倒閉

   3. Crypto exchange 2024
      → DB migration 失敗、無法 rollback
      → 賠付數千萬美金

→ 備份不是「以防萬一」、是「100% 會用到」
→ 工程師生涯一定會用到 backup 救命
```

  
### 2. 三個備份等級

```text
🥉 Level 1：邏輯備份（logical backup）
   ──────────────────────────────────
   工具：pg_dump / mysqldump / mongodump
   做法：把資料匯出成 SQL / JSON
   優點：人讀懂、跨版本還原
   缺點：慢、大資料量痛
   適合：小到中型 DB（< 100 GB）

🥈 Level 2：實體備份（physical backup）
   ──────────────────────────────────
   工具：pg_basebackup / xtrabackup / RDS snapshot
   做法：複製 DB 檔案
   優點：快、適合大 DB
   缺點：跨版本可能不相容
   適合：中到大型 DB

🥇 Level 3：PITR（Point-in-Time Recovery）
   ──────────────────────────────────
   工具：PG WAL / MySQL binlog
   做法：實體備份 + 持續 log
   優點：還原到「任意時間點」
   缺點：複雜、磁碟需求大
   適合：金融、不能丟資料
```

> 💡 **PITR（Point-in-Time Recovery、定點還原）**= 不只能還原到「昨晚 3 點」、可以還原到「中午 12:34:56」——所有交易型 DB 必備。

  
### 3. PostgreSQL 備份

🖥️ 在終端機輸入：

```text
# 1. 邏輯備份（SQL 格式）
pg_dump -U postgres mydb | gzip > backup.sql.gz

# 2. 邏輯備份（custom 格式、可平行 restore）
pg_dump -U postgres mydb -F c -f backup.dump

# 3. 還原 SQL 格式
gunzip -c backup.sql.gz | psql -U postgres mydb

# 4. 還原 custom 格式（推薦）
pg_restore -U postgres -d mydb -j 4 backup.dump
   # -j 4 = 4 個平行 thread
```

💬 電腦會顯示：

```text
pg_restore: restoring data for table "users"
pg_restore: restoring data for table "orders"
pg_restore: creating INDEX "users_email_key"
```

  
### 4. 自動化 cron 備份

📄 寫進 `/etc/cron.d/db-backup`：

```text
# 每天凌晨 3 點備份
0 3 * * * postgres pg_dump mydb | gzip > /backup/$(date +\%Y-\%m-\%d).sql.gz

# 4 點上傳 S3
0 4 * * * aws s3 sync /backup/ s3://my-bucket/backups/

# 5 點刪 30 天前的
0 5 * * * find /backup/ -mtime +30 -delete
```

```text
✅ 業界標準 backup 策略

   1. 每天 1 次完整備份
   2. 上傳到異地（S3 / GCS / 不同 region）
   3. 加密（KMS / age）
   4. 保留週期：7 天每日 / 4 週每週 / 12 月每月
   5. 每月演練 restore 一次
```

  
### 5. MongoDB 備份

🖥️ 在終端機輸入：

```text
# 1. mongodump（邏輯備份）
mongodump --uri "mongodb://localhost:27017/mydb" \
  --out=/backup/$(date +%Y-%m-%d)

# 2. mongorestore（還原）
mongorestore --uri "mongodb://localhost:27017/mydb" \
  /backup/2026-05-28/mydb

# 3. 直接複製檔案（要先 lock）
db.fsyncLock()
cp -r /var/lib/mongodb /backup/
db.fsyncUnlock()
```

```text
🌐 MongoDB Atlas（雲端託管）
   ─────────────────────────
   → Continuous backup（持續備份）
   → PITR（任意時間點還原）
   → 自動跨 region 複製
   → 設定 5 分鐘搞定
```

  
### 6. Redis 備份

```text
🔄 Redis 兩種持久化

   1. RDB（snapshot 快照）
      ─────────────────
      → 定期把整個記憶體寫成 dump.rdb
      → 設定：save 900 1（15 分鐘有 1 筆寫入就存）
      → 快、檔案小
      → 缺點：最後一次 snapshot 後的資料會丟

   2. AOF（Append Only File、追加日誌）
      ─────────────────
      → 每個寫入指令都追加到檔案
      → 接近 0 丟失（fsync everysec）
      → 缺點：檔案大、restart 慢

→ 業界推薦：兩個都開（RDB + AOF）
→ 雲端服務：Upstash / Redis Cloud 自動備份
```

📄 寫進 `redis.conf`：

```text
# RDB
save 900 1
save 300 10
save 60 10000

# AOF
appendonly yes
appendfsync everysec
```

  
### 7. DynamoDB 備份（AWS）

```text
☁️ DynamoDB 三種 backup

   1. On-Demand Backup
      ─────────────────
      → 手動觸發、全表 snapshot
      → 不影響效能
      → 想備就備

   2. Point-in-Time Recovery（PITR）
      ─────────────────
      → 開啟後、過去 35 天任意秒還原
      → 自動、不用管
      → 推薦：production 一定要開

   3. AWS Backup
      ─────────────────
      → 跨服務統一 backup（DynamoDB + RDS + S3...）
      → 排程 + 跨 region + 加密
```

🖥️ 在終端機輸入：

```text
# 開啟 PITR
aws dynamodb update-continuous-backups \
  --table-name orders \
  --point-in-time-recovery-specification \
  PointInTimeRecoveryEnabled=true

# 手動 backup
aws dynamodb create-backup \
  --table-name orders \
  --backup-name orders-2026-05-28
```

  
### 8. 雲端 DB 託管 backup

```text
☁️ 雲端 DB 託管服務都自動備份

   Supabase Pro
   ─────────────
   → 每日自動 backup（保留 7 天）
   → PITR（保留 7 天）
   → 一鍵 restore
   → $25 / 月起

   AWS RDS
   ─────────────
   → 自動 backup（保留 0-35 天）
   → PITR（5 分鐘精度）
   → 手動 snapshot（永久保留）

   Google Cloud SQL
   ─────────────
   → 類似 RDS
   → 自動 + 手動 + PITR

   MongoDB Atlas
   ─────────────
   → Continuous backup
   → PITR（任意時間點）
   → Cross-cluster restore

→ 託管服務 = 別自己搞 backup
→ 中型以下強烈推薦
```

  
### 9. 異地 / 多 region 備份

```text
🌍 3-2-1 規則（業界標準）

   3 份資料（原 + 2 份備份）
   2 種媒介（本地 + 雲端）
   1 份異地（不同 region / 不同雲）

   範例：
   ├─ Production DB（AWS us-east-1）
   ├─ 每日 backup 到 S3 us-east-1
   └─ 每週 sync 到 S3 us-west-2（異地）

→ AWS 整個 region 掛 → 還有異地能救
→ 加密：客戶端先加密、再上傳
```

  
### 10. 演練 restore（最重要！）

```text
⚠️ 沒測過的 backup ≈ 沒 backup

   常見故事：
   1. 每天 backup 3 年
   2. 一天工程師打錯指令、刪 production
   3. 要 restore → 發現 backup 壞了 3 個月沒人發現
   4. 公司倒閉

🎯 演練排程
   ───────────
   每月一次：
   1. 抓最新 backup
   2. 還原到 staging DB
   3. 跑驗證 query（count、checksum）
   4. 寫 report
   5. 自動寄信給 team

→ 用 dev / staging 環境演練
→ 寫成 script 自動跑
→ 異常立刻告警
```

  
### 11. 常見坑

```text
❌ 1. 沒 backup
   症狀：直接死、沒辦法救
   修：今天就設 pg_dump + cron

❌ 2. backup 沒上雲（只在同台機器）
   症狀：機器掛、backup 一起掛
   修：S3 / GCS 異地

❌ 3. 沒測 restore
   症狀：要救時發現 backup 壞了
   修：每月演練、寫驗證 script

❌ 4. backup 沒加密
   症狀：S3 bucket 設錯權限、資料外洩
   修：客戶端先加密、用 KMS

❌ 5. 沒分層保留
   症狀：30 天都保留 → 磁碟爆
   修：7 天每日 / 4 週每週 / 12 月每月

❌ 6. 鎖太久 backup
   症狀：pg_dump 在 production 跑、QPS 掉
   修：用 read replica 做 backup
```

  
### 🎯 重點回顧

- **Backup 是 DB 工程師命根**——沒備份等於沒命
- **三個等級**：邏輯（pg_dump）/ 實體（snapshot）/ PITR（任意時間點）
- **3-2-1 規則**：3 份資料、2 種媒介、1 份異地
- **沒測過的 backup ≈ 沒 backup**——每月演練 restore
- **雲端託管自動 backup**（Supabase / RDS / Atlas）—中型以下強烈推薦
