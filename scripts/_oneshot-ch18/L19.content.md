**TimescaleDB / InfluxDB = 時序資料專用 DB**——「每秒記一筆」的場景優化到極致。**IoT 感測器、伺服器監控、股價、健身手錶資料**全靠它。

> 💡 **時序資料是什麼**：跟「時間」綁定的資料——溫度（每秒一筆）、股價（每分鐘一筆）、CPU 用量（每 10 秒一筆）。**特點：寫多、讀少、按時間查、舊資料可降頻保存**。

  
### 1. 為什麼時序資料需要專用 DB？

```text
🚨 時序資料的特點 → 一般 DB 撐不住

   1. 寫入量爆炸
      → 1000 個感測器 × 1 秒 1 筆 = 8.64 億筆 / 天
      → SQL 插入會崩

   2. 99% 查詢按時間範圍
      → SELECT * WHERE time > '...' AND time < '...'
      → 一般 SQL 索引沒對「時間」優化

   3. 舊資料可以「降頻」
      → 最近 1 小時要每秒精度
      → 1 個月前每分鐘就夠
      → 1 年前每小時就夠

   4. 自動刪舊資料
      → 法規 / 成本：30 天前自動丟
      → SQL 沒這個機制

→ 時序 DB 就是專門解決這 4 件事
```

  
### 2. TimescaleDB（PostgreSQL extension）

**TimescaleDB = PostgreSQL + 時序優化**——還是 SQL、新手友善、推薦！

📄 寫進程式檔：`schema.sql`

```text
-- 啟用 extension
CREATE EXTENSION IF NOT EXISTS timescaledb;

-- 建一般 PG table
CREATE TABLE sensor_data (
  time        TIMESTAMPTZ NOT NULL,
  sensor_id   INT,
  temperature DOUBLE PRECISION,
  humidity    DOUBLE PRECISION
);

-- 變成 hypertable（超大表、自動按時間分區）
SELECT create_hypertable('sensor_data', 'time');

-- 索引
CREATE INDEX ix_sensor_time ON sensor_data (sensor_id, time DESC);
```

🖥️ 在終端機輸入：

```text
psql -U postgres -d mydb -f schema.sql
```

💬 電腦會顯示：

```text
CREATE EXTENSION
CREATE TABLE
 create_hypertable
-------------------
 (1,public,sensor_data,t)
```

  
### 3. 時序查詢（time_bucket 是命根）

```text
-- 過去 24 小時、每小時平均溫度
SELECT
  time_bucket('1 hour', time) AS hour,
  sensor_id,
  AVG(temperature) AS avg_temp,
  MAX(temperature) AS max_temp,
  MIN(temperature) AS min_temp
FROM sensor_data
WHERE time > NOW() - INTERVAL '24 hours'
GROUP BY hour, sensor_id
ORDER BY hour DESC;
```

> 💡 **time_bucket** = SQL `GROUP BY hour` 的進化版。`time_bucket('5 minutes', time)`、`'1 day'`、`'1 week'` 都行。**做 dashboard 必用**。

  
### 4. 殺手功能 3 招

```text
🔥 TimescaleDB 3 大殺手功能

   1. Continuous Aggregates（背景算統計）
      ──────────────────────────
      → 自動每分鐘算「過去 1 分鐘平均」
      → query 時直接讀預計算結果
      → 比即時算快 1000 倍

   2. Compression（壓縮）
      ──────────────────────────
      → 7 天前的資料自動壓縮
      → 省 90% 儲存空間
      → 還能查、只是慢一點

   3. Retention（自動刪）
      ──────────────────────────
      → 設定「保留 30 天」
      → 自動 drop 舊 chunk
      → 不用寫 cron
```

📄 寫進程式檔：`policies.sql`

```text
-- Continuous Aggregate（每分鐘預先算）
CREATE MATERIALIZED VIEW sensor_1min
WITH (timescaledb.continuous) AS
SELECT time_bucket('1 minute', time) AS bucket,
       sensor_id,
       AVG(temperature) AS avg_temp
FROM sensor_data
GROUP BY bucket, sensor_id;

-- 自動每分鐘 refresh
SELECT add_continuous_aggregate_policy('sensor_1min',
  start_offset => INTERVAL '2 hours',
  end_offset => INTERVAL '1 minute',
  schedule_interval => INTERVAL '1 minute');

-- Compression（7 天前壓縮）
SELECT add_compression_policy('sensor_data', INTERVAL '7 days');

-- Retention（30 天前刪除）
SELECT add_retention_policy('sensor_data', INTERVAL '30 days');
```

  
### 5. InfluxDB（純時序 DB）

**InfluxDB = 100% 為時序設計**——不是 SQL、是自家 Flux 語法。

📄 寫進程式檔：`influx_demo.flux`

```text
from(bucket: "sensors")
  |> range(start: -24h)
  |> filter(fn: (r) => r.sensor_id == "s1")
  |> aggregateWindow(every: 1h, fn: mean)
```

```text
✅ InfluxDB 強處
   ⃣✓ 純時序設計、寫入超快
   ⃣✓ 內建 retention / downsampling
   ⃣✓ Grafana 完美整合
   ⃣✓ Telegraf（自動收集 metrics）

❌ InfluxDB 弱處
   ⃣✗ 不是 SQL（要學 Flux）
   ⃣✗ 不能 JOIN 一般資料
   ⃣✗ ad-hoc 分析弱
```

  
### 6. Prometheus（監控專用）

```text
📊 Prometheus
   ─────────────
   → 跟 InfluxDB 同類、但專做「系統監控」
   → Pull-based（從 app 主動抓 metrics）
   → 自有 PromQL 查詢語法
   → K8s 預設監控方案

📈 用法：
   App 暴露 /metrics endpoint
     ↓
   Prometheus 每 15s 來抓
     ↓
   Grafana 把資料畫成圖表
     ↓
   AlertManager 異常時發 Slack
```

  
### 7. 經典監控 stack

```text
🏗️ 業界標準監控架構

   方案 A：Prometheus stack
   ────────────────────────
   App
     → Prometheus（抓 metrics）
     → Grafana（畫圖表）
     → AlertManager（告警）

   方案 B：InfluxDB stack
   ────────────────────────
   App + 主機
     → Telegraf（agent 收 metrics）
     → InfluxDB（存）
     → Grafana（畫）

   方案 C：TimescaleDB stack（推薦小團隊）
   ────────────────────────
   App
     → 直接 INSERT 到 TimescaleDB
     → Grafana 接 PG 來源
     → 一個 DB 解決一切
```

  
### 8. 怎麼選？

```text
📋 場景 → 選什麼

   小到中型、要 SQL → TimescaleDB ⭐
   純 metrics / IoT → InfluxDB
   K8s / 系統監控  → Prometheus
   小規模（< 1M 筆）→ 原生 PG（partitioning）
   超大規模（PB）  → Cassandra / ClickHouse

   不知道用什麼 → TimescaleDB（SQL 就會用）
```

### 10. 跟 SQL / NoSQL 比較

```text
面向          時序 DB          一般 SQL        NoSQL
─────────────────────────────────────────────────────
寫入速度       極快            中              快
時間範圍查     極快            慢              中
自動降頻       內建            手動            手動
自動 retention 內建            手動            部分
聚合查詢       極快（預計算）   中              慢
ACID          部分（看實作）   有              通常無
語法          SQL（Timescale） SQL             各家不同
```

  
### 11. 常見坑

```text
❌ 1. 用一般 PG 存超大時序
   症狀：插入 10 億筆、query 變慢
   修：改 TimescaleDB hypertable

❌ 2. 沒 retention policy
   症狀：磁碟爆滿
   修：設定 add_retention_policy

❌ 3. 用 OFFSET 翻頁
   症狀：SELECT ... LIMIT 20 OFFSET 100000
   修：用 time 範圍翻頁（cursor）

❌ 4. 沒用 time_bucket
   症狀：DATE_TRUNC('hour', time) 跑超慢
   修：用 time_bucket（TimescaleDB 優化）
```

  
### 🎯 重點回顧

- **時序 DB = 時間綁定資料的專用 DB**——寫多、按時間查、自動降頻
- **TimescaleDB = PG extension**——SQL 就會用、新手友善
- **InfluxDB = 純時序、Flux 語法**——大規模 metrics 強
- **Prometheus = K8s 監控標配**——pull-based、PromQL
- **小團隊推薦 TimescaleDB**——一個 PG 通殺、不用多 DB
