**Migration（資料庫遷移）= DB schema 的版本控制**——像 git 但管 DB 結構。**沒 migration 工具的團隊改 schema 等於玩命**——改了 production 沒紀錄、誰也不知道現在長怎樣。

> 💡 **新手最常踩的雷**：手動進 pgAdmin 改 production schema、隔天 staging / production 不一致、deploy 出錯——這課教你用 migration 工具、再也不踩。

  
### 1. Migration 是什麼？

```text
🗂️ Migration（資料庫遷移）

   把每次 DB schema 改動寫成檔案：
   migrations/
   ├─ 001_create_users.sql
   ├─ 002_add_email_column.sql
   ├─ 003_create_orders.sql
   └─ 004_add_index_users_email.sql

   工具會記錄「跑到哪了」：
   _migrations table:
   ├─ 001_create_users        ✓ 2026-01-15
   ├─ 002_add_email_column    ✓ 2026-02-03
   ├─ 003_create_orders       ✓ 2026-03-21
   └─ 004_add_index           — 待跑

   每次 deploy：
   → 工具看到 004 沒跑
   → 自動跑
   → 記錄完成
```

```text
✅ Migration 的 3 大好處

   1. 版本控制
      → 每次改 schema 都進 git
      → 誰改的、改什麼、何時改 一清二楚

   2. 同步多環境
      → dev / staging / production 跑同樣 migration
      → 絕對一致

   3. 可 rollback
      → 改錯了能還原（要寫 down migration）
```

  
### 2. Prisma（最熱、新手友善）

**Prisma = TypeScript 生態最受歡迎的 ORM + Migration 工具**——schema 用 DSL 寫、超直覺。

📄 寫進程式檔：`prisma/schema.prisma`

```text
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

model User {
  id        String   @id @default(uuid())
  email     String   @unique
  name      String?
  posts     Post[]
  createdAt DateTime @default(now())
}

model Post {
  id        String   @id @default(uuid())
  title     String
  content   String?
  authorId  String
  author    User     @relation(fields: [authorId], references: [id])
  createdAt DateTime @default(now())

  @@index([authorId, createdAt(sort: Desc)])
}
```

🖥️ 在終端機輸入：

```text
# 1. 建第一個 migration（dev）
npx prisma migrate dev --name init

# 2. 自動產生：
#    prisma/migrations/20260528_init/migration.sql
#    + 跑到 dev DB
#    + 產生 Prisma Client

# 3. production deploy
npx prisma migrate deploy
```

💬 電腦會顯示：

```text
✔ Generated Prisma Client (v5.x)
✔ Applied migration 20260528_init
```

  
### 3. Prisma Client 用法

📄 寫進程式檔：`api/users.ts`

```text
import { PrismaClient } from "@prisma/client";

const prisma = new PrismaClient();

// 建立
const user = await prisma.user.create({
  data: { email: "alice@example.com", name: "Alice" },
});

// 查詢（含關聯）
const usersWithPosts = await prisma.user.findMany({
  include: { posts: true },
  where: { email: { contains: "@example.com" } },
  orderBy: { createdAt: "desc" },
  take: 20,
});

// 更新
await prisma.user.update({
  where: { id: user.id },
  data: { name: "Alice Lin" },
});

// Transaction
await prisma.$transaction(async (tx) => {
  await tx.user.create({ data: { email: "a@x.com" } });
  await tx.post.create({ data: { title: "Hi", authorId: "..." } });
});
```

  
### 4. Drizzle（TypeScript first、輕量）

**Drizzle = 比 Prisma 更輕、更貼近 SQL**——type-safe SQL builder。

📄 寫進程式檔：`db/schema.ts`

```text
import { pgTable, uuid, text, timestamp, index } from "drizzle-orm/pg-core";

export const users = pgTable("users", {
  id: uuid("id").defaultRandom().primaryKey(),
  email: text("email").notNull().unique(),
  name: text("name"),
  createdAt: timestamp("created_at").defaultNow(),
});

export const posts = pgTable("posts", {
  id: uuid("id").defaultRandom().primaryKey(),
  title: text("title").notNull(),
  authorId: uuid("author_id").references(() => users.id),
  createdAt: timestamp("created_at").defaultNow(),
}, (t) => ({
  authorIdx: index("ix_posts_author_time").on(t.authorId, t.createdAt),
}));
```

🖥️ 在終端機輸入：

```text
# 1. 產生 migration
npx drizzle-kit generate

# 2. 跑 migration
npx drizzle-kit migrate
```

📄 寫進程式檔：`db/queries.ts`

```text
import { drizzle } from "drizzle-orm/node-postgres";
import { eq, desc } from "drizzle-orm";
import { users, posts } from "./schema";

const db = drizzle(pool);

// 寫入
await db.insert(users).values({
  email: "alice@example.com",
});

// 查詢
const result = await db.select()
  .from(users)
  .where(eq(users.email, "alice@example.com"));

// JOIN
const withPosts = await db
  .select()
  .from(users)
  .leftJoin(posts, eq(users.id, posts.authorId))
  .orderBy(desc(posts.createdAt));
```

```text
🆚 Prisma vs Drizzle 對比

   面向          Prisma         Drizzle
   ─────────────────────────────────────
   風格          ORM            SQL builder
   schema 語法   自家 DSL       TypeScript
   bundle size   大（~5 MB）    小（~50 KB）
   速度          中             快
   學習           直覺           SQL 知識
   migration     強             較新
   生態          巨大           成長中
   推薦          新手 / 中型    高效能 / serverless
```

  
### 5. Mongoose（MongoDB 標準 ODM）

📄 寫進程式檔：`models/User.ts`

```text
import mongoose from "mongoose";

const userSchema = new mongoose.Schema({
  email: { type: String, required: true, unique: true },
  name: String,
  age: Number,
  hobbies: [String],
  createdAt: { type: Date, default: Date.now },
});

userSchema.index({ email: 1 });

export const User = mongoose.model("User", userSchema);
```

📄 寫進程式檔：`api.ts`

```text
import { User } from "./models/User";

// 建立
const u = await User.create({ email: "a@x.com", name: "Alice" });

// 查詢
const users = await User.find({ age: { $gte: 18 } })
  .sort({ createdAt: -1 })
  .limit(20);

// 更新
await User.updateOne({ _id: u._id }, { name: "Alice Lin" });
```

  
### 6. SQL ↔ NoSQL 遷移

```text
🔄 從 SQL 遷到 MongoDB（不推薦但常見）

   情境：團隊覺得 MongoDB 比較「彈性」要遷

   步驟：
   1. 設計 Mongo schema（denormalize）
      ─ user + 內嵌 posts（不分表）
      ─ order + 內嵌 items

   2. 寫 ETL 腳本
      ─ 從 PG SELECT 出來
      ─ 組成 Mongo document
      ─ insertMany

   3. 雙寫期（dual-write）
      ─ 同時寫 PG + Mongo
      ─ 確認 Mongo 沒問題

   4. 切流量
      ─ 讀切到 Mongo
      ─ 觀察 1-2 週

   5. 停 PG 寫入

→ 風險高、不可逆
→ 90% 後來後悔、又遷回 SQL
```

```text
🔄 從 MongoDB 遷到 PostgreSQL（推薦）

   情境：Mongo 用了 2 年、要 JOIN / transaction

   步驟：
   1. 設計 PG schema
      ─ 從 Mongo embedded 拆出來
      ─ user / posts / comments 分表

   2. 寫 migration script
      ─ Mongo find → 拆解 → PG insert

   3. 同上：雙寫 → 切讀 → 停 Mongo

→ 業界更常見
→ Notion、Figma 都從 Mongo 改 PG
```

  
### 7. Migration 心法（業界規則）

```text
✅ DO（業界鐵則）

   1. 永遠寫 migration（不要手動改 schema）
   2. migration 進 git（PR review）
   3. 一個 migration 做一件事（小、可控）
   4. dev 跑 migrate dev、production 跑 migrate deploy
   5. CI 自動跑 migration（deploy 前）
   6. backup 後再跑 production migration
   7. 大 table 改欄位用 online schema change（zero downtime）

❌ DON'T

   1. 別手動改 production schema
   2. 別跑 migrate dev 在 production（會 reset）
   3. 別刪欄位（先停用、3 個月後再刪）
   4. 別在 migration 內 SELECT 大量資料（會 lock）
   5. 別 rename column（先加新欄位、同步、再刪舊）
   6. 別在 migration 內塞商業邏輯
```

  
### 8. Schema 改動的安全規則

```text
🟢 安全（low risk）
   ──────────────
   ✓ 加新欄位（nullable）
   ✓ 加新 table
   ✓ 加 index（concurrently）
   ✓ 加 constraint（先 NOT VALID、再 VALIDATE）

🟡 中等（medium risk）
   ──────────────
   ⚠ 改 column type（小心 lock）
   ⚠ 加 NOT NULL（要先 set default）
   ⚠ 加 unique constraint（要先 backfill）

🔴 危險（high risk）
   ──────────────
   ✗ DROP column / table
   ✗ RENAME（client 沒改完就 deploy）
   ✗ 改 primary key

→ 真要 DROP：先「停用」3 個月、確認沒 query 用、再 DROP
→ RENAME：加新欄位 → 雙寫 → 切讀 → 移除舊
```

  
### 9. zero-downtime migration

```text
🚀 Production 改 schema 不停機

   範例：要 rename users.email → users.user_email

   ❌ 直接 RENAME（會出事）
      → migration 跑時 client 還在用 .email
      → 5 秒沒人能登入

   ✅ 4 階段 zero-downtime

   階段 1：加新欄位
   ─────────────────
   ALTER TABLE users ADD COLUMN user_email TEXT;

   階段 2：雙寫（app code）
   ─────────────────
   await db.user.update({
     data: { email: x, user_email: x }
   });

   階段 3：backfill 舊資料
   ─────────────────
   UPDATE users SET user_email = email
   WHERE user_email IS NULL;

   階段 4：切讀 + 移除舊
   ─────────────────
   → app 全部讀 user_email
   → 確認 1 週沒問題
   → ALTER TABLE users DROP COLUMN email;

→ Stripe / Shopify 都這套
→ 大團隊必修
```

  
### 10. 常見坑

```text
❌ 1. 手動進 pgAdmin 改 production
   後果：dev / prod 不一致、deploy 爆
   修：永遠用 migration tool

❌ 2. migration 內塞商業邏輯
   後果：rollback 變地獄
   修：純 schema、別動資料

❌ 3. 直接 DROP column
   後果：舊 client 還在用 → 500 error
   修：先停用、再刪

❌ 4. 沒在 CI 自動跑
   後果：忘記跑、production 沒 sync
   修：CI 加 prisma migrate deploy 步驟

❌ 5. migration 不可逆
   後果：跑錯沒辦法 rollback
   修：每個 up 配 down
```

  
### 🎯 重點回顧

- **Migration = DB schema 的 git**——版本控制、多環境同步、可 rollback
- **Prisma 是新手首選**——直覺 schema、強 type、大生態
- **Drizzle 更輕量**——適合 serverless / 高效能
- **Mongoose 是 Mongo 標配**——schema validation + populate
- **Zero-downtime 4 階段**：加欄位 → 雙寫 → backfill → 切讀 + 刪
- **永遠別手動改 production**——schema 進 git、CI 自動跑
