**Connection Pool（連線池）+ DB 效能優化 = 從「能用」到「快」的關鍵**——同樣的 DB、會調的人 query 快 1000 倍、不會調的人慢成狗。

> 💡 **新手最常踩的雷**：寫個 API 每次 new Client()、5 分鐘 DB 連線爆滿、整個 server 卡死。**這課教你 10 招、立刻變高手**。

  
### 1. Connection Pool 是什麼？

```text
🚫 沒 connection pool（地獄）

   每個 request：
   1. new Client() — 連 DB（耗 50 ms）
   2. 跑 query
   3. close() — 斷線

   1000 個 request / 秒
   → 1000 個新連線 / 秒
   → DB 連線上限通常 100
   → 第 101 個 → ECONNREFUSED → 整個 server 掛

✅ 有 connection pool（天堂）

   啟動時：建 20 條連線、不斷線
   每個 request：
   1. 從 pool 借一條（< 1 ms）
   2. 跑 query
   3. 還給 pool（不斷線）

   1000 個 request / 秒
   → 重複用同 20 條連線
   → DB 開心、server 不掛
```

  
### 2. Node + PG Pool 設定

📄 寫進程式檔：`db.ts`

```text
import pg from "pg";

const pool = new pg.Pool({
  host: process.env.DB_HOST,
  port: 5432,
  database: "mydb",
  user: "postgres",
  password: process.env.DB_PASSWORD,

  max: 20,                      // pool 最大連線數
  min: 2,                       // 最少保持
  idleTimeoutMillis: 30_000,    // 30 秒沒用就回收
  connectionTimeoutMillis: 2000,// 連 2 秒拿不到就 throw
});

// 用法
export async function getUser(id: string) {
  const result = await pool.query(
    "SELECT * FROM users WHERE id = $1",
    [id]
  );
  return result.rows[0];
}
```

```text
📐 max（最大連線）怎麼設？

   公式：max = (DB max_connections) / (instance 數量)

   範例：
   - PG max_connections = 100
   - 3 台 server
   - max = 30 / server

→ 別貪心設 max=100（會把 DB 打死）
→ 留一些給 backup / migration
```

  
### 3. 10 大效能優化（從最重要排）

```text
🏆 DB 效能 10 招

   1. Index（最重要、佔 80% 提升）
   2. EXPLAIN ANALYZE（找慢 query 兇手）
   3. 解 N+1 query（殺手等級）
   4. 不要 SELECT *
   5. Cursor pagination（不用 OFFSET）
   6. 批量操作（bulk insert / update）
   7. Read replicas（讀寫分離）
   8. Redis 快取（前置）
   9. Materialized View（預計算）
   10. Partitioning（巨表按時間切）
```

  
### 4. Index 黃金法則

```text
📌 Index 是 80% 慢 query 的解法

   建在哪：
   ⃣✓ WHERE 常用的欄位
   ⃣✓ JOIN ON 的欄位
   ⃣✓ ORDER BY 的欄位
   ⃣✓ unique constraint（自動有）

   不要建：
   ⃣✗ 只有 2-3 種值的欄位（gender、status）
   ⃣✗ 寫多讀少的欄位（每寫一筆要更新 index）
   ⃣✗ 已經有 covering index 的子集

   範例：
   CREATE INDEX ix_orders_user_time
   ON orders(user_id, created_at DESC);

→ 複合 index：欄位順序重要
→ WHERE user_id = ... ORDER BY created_at DESC 完美利用
```

  
### 5. EXPLAIN ANALYZE（找兇手）

🖥️ 在終端機輸入：

```text
EXPLAIN ANALYZE
SELECT * FROM orders
WHERE user_id = 'u1'
ORDER BY created_at DESC
LIMIT 20;
```

💬 電腦會顯示：

```text
Index Scan using ix_orders_user_time on orders
  (cost=0.42..28.5 rows=20 width=120)
  (actual time=0.05..0.12 ms rows=20 loops=1)
Planning Time: 0.15 ms
Execution Time: 0.18 ms
```

```text
🔍 看什麼：
   - Seq Scan = 全表掃（壞、慢）
   - Index Scan = 用索引（好、快）
   - Index Only Scan = 只讀 index、不碰 table（最快）
   - rows= 多少筆掃過
   - Execution Time = 真實時間
```

  
### 6. N+1 query（殺手中的殺手）

```text
💀 N+1 query — Indie SaaS 變慢的 #1 兇手

   情境：列出 10 個 user + 各自的 posts

   ❌ N+1 寫法（地獄）
   ──────────────
   const users = await User.findMany();        // 1 query
   for (const u of users) {
     u.posts = await Post.findMany({
       where: { userId: u.id }
     });                                        // N 個 query
   }
   → 1 + 10 = 11 個 query

   ✅ Eager loading（天堂）
   ──────────────
   const users = await prisma.user.findMany({
     include: { posts: true },
   });
   → 1 或 2 個 query（不管多少 user）

→ Prisma include / Drizzle relations / Mongoose populate
→ 永遠用、never 寫 for loop 內 await DB
```

  
### 7. Cursor pagination（不用 OFFSET）

```text
🐢 OFFSET 翻頁（越深越慢）

   SELECT * FROM posts
   ORDER BY id LIMIT 20 OFFSET 10000;

   → DB 要先掃 10020 筆、再丟掉 10000
   → 10 萬筆時超慢

🚀 Cursor pagination（每頁速度一樣）

   -- 第一頁
   SELECT * FROM posts
   ORDER BY id LIMIT 20;

   -- 第二頁（傳上一頁最後一個 id 來）
   SELECT * FROM posts
   WHERE id > $lastId
   ORDER BY id LIMIT 20;

   → 每頁都用 index、永遠快
→ 缺點：不能「跳到第 100 頁」
→ 適合：infinite scroll
```

  
### 8. 不要 SELECT *

```text
❌ 慢寫法
   SELECT * FROM users WHERE id = $1;

   → 拉回所有欄位（包含大 text、JSON）
   → 浪費網路 / 記憶體

✅ 快寫法
   SELECT id, email, name FROM users WHERE id = $1;

   → 只拉需要的欄位
   → 可能走 Index Only Scan（最快）
```

  
### 9. 批量操作（bulk）

```text
❌ 慢
   for (const item of items) {
     await db.insert(item);  // 1000 個 query
   }

✅ 快
   await db.insertMany(items);  // 1 個 query

→ Prisma createMany / pg-promise multi-insert
→ 1000 倍速度差
```

  
### 10. Redis 快取（前置）

```text
🚀 cache-aside pattern（最常用）

   async function getUser(id) {
     // 1. 先看 cache
     const cached = await redis.get(`user:${id}`);
     if (cached) return JSON.parse(cached);

     // 2. cache miss → 打 DB
     const user = await db.user.findOne({ id });

     // 3. 寫回 cache（5 分鐘過期）
     await redis.setex(`user:${id}`, 300, JSON.stringify(user));

     return user;
   }

→ 95% 流量打 Redis（< 1ms）
→ 5% 才打 DB
→ DB 壓力降 95%
```

  
### 11. NoSQL 連線池注意事項

```text
🍃 MongoDB
   ─────────
   driver 預設有 connection pool
   設定：maxPoolSize = 100（單台）

   const client = new MongoClient(uri, {
     maxPoolSize: 50,
     minPoolSize: 5,
   });

🔴 Redis
   ─────
   單連線就好（IORedis / node-redis 自動）
   別開太多連線（Redis single-thread）

   const redis = new Redis({
     maxRetriesPerRequest: 3,
     enableReadyCheck: true,
   });

☁️ Serverless（Lambda / Vercel）
   ─────────────────────────
   特別小心 connection pool
   每個 Lambda instance 一個 pool
   pool size 必須小（1-5）
   或用 PgBouncer / Prisma Accelerate
```

  
### 12. batch 寫入優化

```text
🚛 大量寫入時別一筆一筆

   ❌ 慢
   for (const row of 1_000_000_rows) {
     await db.insert(row);
   }
   → 1M query、可能 1 小時

   ✅ 快（COPY）
   await pool.query(`
     COPY users (email, name) FROM STDIN WITH CSV
   `, ...);
   → 1M 筆 30 秒搞定

→ PG COPY 比 INSERT 快 10-100 倍
→ 大量 ETL 必用
```

  
### 13. 常見坑

```text
❌ 1. 沒 connection pool
   症狀：流量一大 ECONNREFUSED
   修：用 pg.Pool / mongoose 內建

❌ 2. pool max 太大
   症狀：DB 連線爆滿、其他服務連不上
   修：max = DB上限 / instance 數

❌ 3. 連線沒 release
   症狀：pool 滿了、新 request hang
   修：try/finally release、用 ORM 託管

❌ 4. N+1 query
   症狀：列表頁卡 5 秒
   修：用 include / populate

❌ 5. 沒 index
   症狀：1 萬筆就慢
   修：EXPLAIN ANALYZE 找出、建 index

❌ 6. 沒 cache
   症狀：DB 一直被打
   修：加 Redis cache-aside

❌ 7. SELECT *
   症狀：傳輸量大、記憶體爆
   修：只 SELECT 需要的欄位

❌ 8. OFFSET 翻頁
   症狀：第 100 頁卡 3 秒
   修：cursor pagination
```

  
### 🎯 重點回顧

- **Connection Pool 是 production 必備**——別每個 request 都 new Client
- **10 招效能優化**——Index 是最重要（佔 80% 提升）
- **EXPLAIN ANALYZE 是命根**——找慢 query 的兇手
- **N+1 是殺手**——永遠用 include / populate
- **Cursor pagination 取代 OFFSET**——infinite scroll 必用
- **Redis cache-aside**——95% 流量別碰 DB
