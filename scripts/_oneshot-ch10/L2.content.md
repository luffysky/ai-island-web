「**渲染模式**（rendering mode、把資料變成 HTML 的方式）」是 Next.js / Nuxt 最核心的觀念。5 種模式各有適用場景、**選錯模式網站會慢 5 倍或花 10 倍成本**。

> 💡 **這課的價值**：很多人寫 Next.js 全部用預設、不知道有 5 種模式可選。**懂這 5 種、就懂業界 90% 效能 + SEO + 成本的權衡**。

  
### 1. CSR（Client-Side Rendering、瀏覽器端渲染）

```text
使用者打開網頁
   ↓
伺服器回傳「空 HTML + JS bundle」
   ↓
瀏覽器下載 JS（300ms ~ 3s）
   ↓
JS 跑起來、React / Vue 把畫面畫出來
   ↓
看到內容
```

> 💡 **白話**：「**現場才開始畫畫**」——server 只送畫筆跟顏料、瀏覽器自己畫。純 React / Vue 就是這種。

**優點**：開發簡單、server 負擔小、互動極快（後續操作不用打 server）。

**缺點**：首屏（first paint）白屏久、SEO 差、社群分享預覽爛。

**適合**：後台管理介面、登入後 dashboard、SEO 不重要的工具。

  
### 2. SSR（Server-Side Rendering、伺服器端渲染）

```text
使用者打開網頁
   ↓
伺服器跑 React / Vue 元件、抓資料、產生完整 HTML
   ↓
回傳「裝滿內容的 HTML」
   ↓
瀏覽器立刻顯示（首屏快）
   ↓
JS 載入後「Hydration（注水、把靜態 HTML 接上互動）」接管事件
```

> 💡 **白話**：「**先把畫畫好再寄出去**」——server 已經把內容畫成 HTML、瀏覽器拿到就能看。

**優點**：SEO 好、首屏快、社群分享卡片正確（OG 抓得到）。

**缺點**：每次請求都要 server 算一次、成本高、流量大時 server 壓力大；hydration 那 0.5 秒可能無法點。

**適合**：內容會變的頁面、需要登入狀態的個人化內容（用戶 dashboard、購物車、後台首頁）。

  
### 3. SSG（Static Site Generation、靜態網站生成）

```text
build 時（你 npm run build 那一刻）：
  跑所有頁面、抓資料、產出 HTML 檔
  → /index.html
  → /about.html
  → /blog/post-1.html
  → /blog/post-2.html ...

使用者打開網站時：
  CDN（Content Delivery Network、內容傳遞網路）直接吐 HTML
  → 極快（10-50ms）
  → server 0 計算
```

> 💡 **白話**：「**畫好 1000 張畫放倉庫、要哪張拿哪張**」。CDN 就是全球倉庫。

**優點**：最快、最便宜、安全（沒 runtime 可被攻擊）。

**缺點**：內容變了要重 build、不適合「每秒在變」的資料。

**適合**：部落格、文件站、Landing page、行銷頁、AI 島本身（教程內容不太常變）。

  
### 4. ISR（Incremental Static Regeneration、增量靜態再生）

```text
build 時：產生 HTML（先丟出去）
   ↓
有人訪問 + X 秒過後（例如 60 秒）：
   背景靜悄悄重新 build 那一頁
   ↓
下一個訪客拿到新版
   ↓
使用者永遠看到「靜態檔案」（快）
但內容會「漸進更新」（不用人工 rebuild）
```

> 💡 **白話**：「**SSG 加上自動更新**」——平常拿倉庫的畫、過一段時間自動畫新版替換。

**優點**：又快又新鮮、不用全站重 build。

**缺點**：邏輯比 SSG 複雜、可能看到 1-2 分鐘的舊資料。

**適合**：電商商品頁、新聞、需要新鮮但不要「即時」的資料。

📄 **寫進程式檔**：`app/products/[id]/page.tsx`（Next.js ISR 範例）

```text
// 每 60 秒重新驗證
export const revalidate = 60;

export default async function Product({ params }) {
  const product = await fetch(`https://api.shop.com/products/${params.id}`)
    .then(r => r.json());
  return <h1>{product.name}</h1>;
}
```

  
### 5. RSC（React Server Components、React 伺服器元件）

```text
Server Component（伺服器元件）在 server 跑：
  ✓ 可以 await（直接讀 DB）
  ✓ 不送 JS code 到瀏覽器
  ✓ 結果序列化（serialize、變字串）送到 client
  ✓ bundle size 變小

Client Component（瀏覽器元件）在 browser 跑：
  ✓ 處理互動（onClick、useState）
  ✓ 要在檔案頂端寫 "use client"
```

> 💡 **白話**：「**畫的人可以是 server 工程師、畫筆動的人才是瀏覽器**」。RSC 把 React 切兩半、各做擅長的事。

**優點**：bundle 變小（很多 code 不送到瀏覽器）、能直接讀 DB（不用寫 API）、首屏極快、API key 安全（永遠不會到 client）。

**缺點**：心智模型較複雜、要分清什麼能 server 跑什麼必須 client。

**適合**：現代 SaaS、混合靜態 + 互動的頁面、追求極致效能。

  
### 6. 5 種模式對照表（最重要、背起來）

```text
| 場景             | 模式        | 例子                |
|------------------|-------------|---------------------|
| 部落格 / 文件    | SSG         | Vercel docs         |
| 行銷頁 / Landing | SSG         | Linear 首頁          |
| 電商商品頁       | ISR         | 蝦皮、博客來          |
| 新聞 / 媒體      | ISR         | 報導者              |
| 個人化 dashboard | SSR + RSC   | GitHub 首頁          |
| 後台管理         | CSR         | Admin panel         |
| 互動 SaaS        | RSC + Client| Linear、Vercel UI    |
```

  
### 7. 業界趨勢：混合模式才是王道

```text
2026 業界共識：
  「同一個專案、不同路由用不同模式」

例：一個電商網站
  /             首頁 → SSG（行銷內容不太變）
  /products/[id] 商品 → ISR（每分鐘更新庫存）
  /cart         購物車 → CSR + RSC（個人化）
  /admin        後台 → CSR（不在乎 SEO）
  /api/checkout API → SSR（每次都要算）
```

Next.js 跟 Nuxt 都支援「**單個路由獨立設定**」——Nuxt 用 `routeRules`（L19 教）、Next.js 用 `export const revalidate / dynamic`。

  
### 5 大常見坑

```text
❌ 1. 整站用 SSR、結果 Vercel 帳單爆炸
   → 每次請求都打 function
   → 修：能 SSG 就 SSG、其次 ISR、最後才 SSR

❌ 2. 部落格用 CSR、Google 完全搜不到
   → 爬蟲拿到空 HTML
   → 修：內容站一定用 SSG / SSR

❌ 3. 商品列表用 SSG、結果價格永遠是 build 時的
   → 庫存變了沒更新
   → 修：改 ISR + revalidate: 60

❌ 4. RSC 內加 useState、build 直接 fail
   → Server Component 不能用 hook
   → 修：分離 client component、用 "use client"

❌ 5. 不知道有 ISR、用 cron job 自己手動 rebuild
   → 重新發明輪子
   → 修：用 revalidate / revalidateTag 內建機制
```

  
### 🎯 重點回顧

- **5 種模式**：CSR（瀏覽器畫）/ SSR（每次 server 畫）/ SSG（build 時畫好）/ ISR（SSG + 自動更新）/ RSC（元件層級分 server / client）
- **效能**：SSG > ISR > RSC > SSR > CSR
- **SEO**：SSG / ISR / SSR / RSC 都好、CSR 最差
- **成本**：SSG 最便宜、SSR 最貴
- **業界 2026**：混合模式才是王道、單一專案內混用
