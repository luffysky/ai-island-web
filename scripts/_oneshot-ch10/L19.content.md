**routeRules** 是 Nuxt 比 Next.js 更直覺的設計——「**同一個專案、不同路由用不同渲染策略**、全寫在一個地方」。Next.js 要每個 page 內 export 設定、Nuxt 用 `routeRules` 統一管理。

> 💡 **這課的價值**：真實 SaaS 不會「全 SSG」或「全 SSR」、永遠是「**首頁 SSG、商品 ISR、後台 SPA**」混合。Nuxt 的 routeRules 讓這件事**一個檔案搞定**、看一眼就懂全站策略。

  
### 1. 基本 routeRules

📄 **寫進程式檔**：`nuxt.config.ts`

```text
export default defineNuxtConfig({
  routeRules: {
    // 首頁：ISR（每小時重生靜態檔）
    "/": { isr: 3600 },

    // 部落格全部靜態（build 時產生）
    "/blog/**": { prerender: true },

    // 後台：SPA（不要 SSR、純 client）
    "/dashboard/**": { ssr: false },

    // 管理員區：強制 SSR + 安全 header
    "/admin/**": {
      ssr: true,
      headers: {
        "X-Frame-Options": "DENY",
        "Cache-Control": "no-store"
      }
    },

    // API：CORS 開
    "/api/**": { cors: true },

    // 重定向（redirect）
    "/old-page": { redirect: "/new-page" },

    // 反向代理（proxy）
    "/proxy/**": { proxy: "https://api.external.com/**" }
  }
});
```

  
### 2. 各 rule 的意義

```text
┌──────────────────────────────────────────────┐
│ prerender: true                              │
│   → build 時就產靜態 HTML（SSG）              │
│   → 最快、最便宜、CDN 友善                    │
│   → 適合：部落格、文件、行銷頁                │
└──────────────────────────────────────────────┘

┌──────────────────────────────────────────────┐
│ isr: 3600                                    │
│   → 靜態檔 + 每 3600 秒（1小時）自動重生       │
│   → 又快又新鮮                                │
│   → 適合：商品、新聞                           │
└──────────────────────────────────────────────┘

┌──────────────────────────────────────────────┐
│ isr: true                                    │
│   → 永久靜態、用 revalidate API 手動重生       │
│   → 適合：CMS 後台改了再觸發重生               │
└──────────────────────────────────────────────┘

┌──────────────────────────────────────────────┐
│ ssr: false                                   │
│   → 純 SPA、不在 server 渲染                  │
│   → 適合：後台、登入後 dashboard               │
└──────────────────────────────────────────────┘

┌──────────────────────────────────────────────┐
│ ssr: true（預設、可省略）                      │
│   → 每次請求 server 渲染                       │
│   → 適合：需登入狀態的頁面                     │
└──────────────────────────────────────────────┘

┌──────────────────────────────────────────────┐
│ swr: 3600                                    │
│   → Stale-While-Revalidate（先給舊的、背景更新）│
│   → 適合：個人化但能容忍舊資料                 │
└──────────────────────────────────────────────┘

┌──────────────────────────────────────────────┐
│ cache: { maxAge: 60 }                         │
│   → Cache-Control header（CDN 層快取）         │
└──────────────────────────────────────────────┘

┌──────────────────────────────────────────────┐
│ redirect: "/new"                              │
│   → 301 / 302 轉址                            │
│   → redirect: { to: "/new", statusCode: 301 } │
└──────────────────────────────────────────────┘

┌──────────────────────────────────────────────┐
│ proxy: "https://api.example.com/**"           │
│   → 反向代理、隱藏第三方 API                   │
└──────────────────────────────────────────────┘

┌──────────────────────────────────────────────┐
│ cors: true                                    │
│   → 加 CORS header（讓別的網域能呼叫）         │
└──────────────────────────────────────────────┘

┌──────────────────────────────────────────────┐
│ headers: { ... }                              │
│   → 自訂回應 header                           │
└──────────────────────────────────────────────┘
```

  
### 3. 實戰：完整 SaaS routeRules

📄 **寫進程式檔**：`nuxt.config.ts`

```text
export default defineNuxtConfig({
  routeRules: {
    // ─── 公開內容 ────────────────────
    "/": { prerender: true },
    "/about": { prerender: true },
    "/pricing": { prerender: true },
    "/blog": { isr: 600 },             // 列表每 10 分鐘
    "/blog/**": { prerender: true },    // 文章靜態

    // ─── 商品（ISR）─────────────────
    "/products": { isr: 60 },
    "/products/**": { isr: 60 },

    // ─── 登入後（SSR + 不快取）──────
    "/dashboard/**": {
      ssr: true,
      headers: { "Cache-Control": "private, no-store" }
    },
    "/settings/**": {
      ssr: true,
      headers: { "Cache-Control": "private, no-store" }
    },

    // ─── 後台（SPA）──────────────────
    "/admin/**": {
      ssr: false,
      headers: {
        "X-Frame-Options": "DENY",
        "X-Robots-Tag": "noindex"
      }
    },

    // ─── API（CORS + 安全）──────────
    "/api/public/**": { cors: true },

    // ─── 重定向 ──────────────────────
    "/docs": { redirect: "/blog/docs-intro" },
    "/old-blog/**": { redirect: "/blog/**" },

    // ─── 第三方 proxy ────────────────
    "/api/maps/**": { proxy: "https://maps.googleapis.com/**" }
  }
});
```

  
### 4. 部署時 Nitro 怎麼處理

Nuxt 自動把 routeRules 轉成**對應平台**的部署檔：

```text
部署到 Vercel：
  isr 路由 → Vercel ISR (.vercel/output/functions)
  prerender 路由 → 靜態 .html 檔
  ssr: false → 純 client bundle
  headers → vercel.json 自動產
  redirect → vercel.json 內 redirects

部署到 Cloudflare Pages：
  全部轉成 Worker 路由 + R2 靜態檔
  ISR 用 Cloudflare KV 模擬

部署到 Netlify：
  isr → Netlify Edge Functions
  prerender → 靜態檔
```

**好處**：你寫一份 routeRules、部署到 Vercel / Cloudflare / Netlify / Zeabur 都自動轉、不用每個平台改設定。

  
### 5. 在 page 內覆寫（少用）

如果只想單一 page 設定、用 `definePageMeta`：

📄 **寫進程式檔**：`pages/special.vue`

```text
<script setup>
definePageMeta({
  // 跟 routeRules 同樣的選項
  // 不過建議寫 routeRules 統一管理
});
</script>
```

  
### 6. routeRules 跟 cache 互動

📄 **寫進程式檔**：`pages/products.vue`

```text
<script setup>
// 即使 routeRules 設了 isr、useFetch 也有自己的 cache
const { data: products } = await useFetch("/api/products", {
  // 在 SSG / ISR 時、這個 fetch 結果會被內嵌進 HTML
  // client hydration 時直接用、不重打
  key: "products-list"
});
</script>
```

  
### 7. 手動觸發 ISR 重生

```text
// 在 server/api 內
import { hubBlob } from "@nuxt/hub";

export default defineEventHandler(async (event) => {
  // ... 更新資料 ...

  // 觸發 /products/123 重新生成
  await $fetch("/products/123", {
    headers: { "x-nitro-prerender": "1" }
  });
});
```

或用平台原生 API：Vercel 的 `revalidatePath`、Cloudflare 的 cache purge。

  
### 8. routeRules 跟 middleware 的差別

```text
| 特性         | routeRules                  | middleware                   |
|--------------|------------------------------|------------------------------|
| 設定方式      | nuxt.config 寫死             | 程式碼動態判斷                 |
| 影響範圍      | 渲染策略 / cache / header     | redirect / 加 cookie / 改 URL |
| 跑在何時      | build / 部署時轉成設定        | 每個 request                   |
| 適合          | 全站策略                      | 動態判斷（auth、A/B）          |
```

兩個一起用：routeRules 設大策略、middleware 處理動態邏輯。

  
### 9. 觀念對比：Nuxt vs Next.js

```text
Next.js（每個 page 內 export）：
  // app/blog/page.tsx
  export const revalidate = 600;

  // app/dashboard/page.tsx
  export const dynamic = "force-dynamic";

  // app/admin/page.tsx
  export const runtime = "edge";

Nuxt（一個 config 集中管理）：
  routeRules: {
    "/blog/**": { isr: 600 },
    "/dashboard/**": { ssr: true },
    "/admin/**": { ssr: false }
  }
```

兩種設計各有優點：
- **Next.js**：跟 page code 在一起、改 page 時容易看到設定
- **Nuxt**：集中、一眼看全站策略

  
### 5 大常見坑

```text
❌ 1. prerender 動態路由 [slug] 沒設 generateRoutes
   → build 時不知道要產哪些
   → 修：用 nitro.prerender.routes 列清單

❌ 2. ssr: false 但 page 內用 await useFetch
   → SPA 模式沒 SSR、await 變成 client 才抓
   → 修：用 lazy: true + 顯示 loading

❌ 3. routeRules 寫了 isr、卻部署到不支援的平台
   → 例如 self-host Node、沒 ISR 機制
   → 修：用 swr 替代、或選 Vercel / Cloudflare

❌ 4. /api/** 設 cors: true 但 production 沒指定 origin
   → 全世界都能呼叫、被濫用
   → 修：cors: { origin: ["https://yoursite.com"] }

❌ 5. 多個 rule 衝突
   → /blog/** 跟 /blog/special 怎麼解？
   → 更精確的優先（/blog/special 蓋過 /blog/**）
   → 修：知道優先級、別重疊
```

  
### 🎯 重點回顧

- **routeRules**：一個地方管全站渲染策略
- **9 大選項**：prerender / isr / ssr / swr / cache / redirect / proxy / cors / headers
- **跨平台**：Nitro 自動轉 Vercel / Cloudflare / Netlify 設定
- **配合 middleware**：routeRules 大策略、middleware 動態判斷
- **跟 Next.js 差別**：Nuxt 集中、Next.js 跟 code 同檔
