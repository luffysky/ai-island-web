Next.js 14+ 的 `fetch` 是「**加強版**」——預設會快取、能設定 revalidate（重新驗證）、能用 tag 控制。**理解快取機制 = 理解 Next.js 效能的命脈**。錯一個 cache 設定、網站慢 10 倍或一直顯示舊資料。

> 💡 **這課的價值**：很多人 Next.js 部署完發現「**資料改了但網站還是舊的**」、就以為框架壞了——其實是 cache 設錯。學完這課你會準確控制每筆資料的新鮮度。

  
### 1. fetch 的 4 種快取行為

📄 **寫進程式檔**：`app/page.tsx`

```text
export default async function Page() {
  // ① 預設：強制快取（force-cache、永遠不重抓）
  const a = await fetch("https://api.example.com/data");
  const aData = await a.json();

  // ② 不快取（no-store、每次都打）
  const b = await fetch("https://api.example.com/data", {
    cache: "no-store"
  }).then(r => r.json());

  // ③ ISR：時間驗證（每 X 秒重新抓）
  const c = await fetch("https://api.example.com/data", {
    next: { revalidate: 3600 }  // 每小時
  }).then(r => r.json());

  // ④ 標籤驗證（手動觸發）
  const d = await fetch("https://api.example.com/data", {
    next: { tags: ["posts"] }
  }).then(r => r.json());

  return <pre>{JSON.stringify({ aData, b, c, d }, null, 2)}</pre>;
}
```

```text
| 設定                          | 行為                              | 適合           |
|-------------------------------|-----------------------------------|----------------|
| 預設 / cache: "force-cache"   | 永遠用快取、build 時抓一次         | 真正不變的資料   |
| cache: "no-store"              | 每次請求都重打                     | 即時資料        |
| next: { revalidate: N }        | 每 N 秒重新抓一次（ISR）           | 商品 / 新聞     |
| next: { tags: [...] }          | 手動 revalidateTag 才更新           | 後台改了立刻刷新 |
```

  
### 2. 手動觸發 revalidate

📄 **寫進程式檔**：`app/actions/posts.ts`

```text
"use server";
import { revalidateTag, revalidatePath } from "next/cache";

export async function createPost(formData: FormData) {
  await db.posts.create({ data: { ... } });

  // 重新驗證 tag="posts" 的所有 fetch
  revalidateTag("posts");

  // 或：重新驗證某個路徑
  revalidatePath("/blog");

  // 或：重新驗證一個 dynamic 路徑 pattern
  revalidatePath("/blog/[slug]", "page");
}
```

> 💡 **白話**：「**後台新增一篇文章、立刻通知所有抓 posts 的 fetch：你的快取過期了**」。下次有人訪問 `/blog`、會自動抓新資料。

  
### 3. 整頁快取設定（page 層級）

📄 **寫進程式檔**：`app/blog/page.tsx`

```text
// 整頁靜態化（build 時產生、不可動）
export const dynamic = "force-static";

// 整頁動態（永遠不快取）
export const dynamic = "force-dynamic";

// ISR：整頁每 60 秒重生
export const revalidate = 60;

// 控制這個路由內 fetch 的預設快取
export const fetchCache = "force-cache";   // 或 "default-no-store"

// Edge / Node runtime（L11 講）
export const runtime = "edge";   // 或 "nodejs"
```

  
### 4. unstable_cache：快取任何 async function

不只 `fetch` 能快取、任何 async function 都可以：

📄 **寫進程式檔**：`app/lib/data.ts`

```text
import { unstable_cache } from "next/cache";

export const getCachedPosts = unstable_cache(
  async (category: string) => {
    // 任何 async 邏輯（不只 fetch、可以是 DB query）
    return await db.posts.findMany({ where: { category } });
  },
  ["posts-by-category"],  // 快取 key
  {
    revalidate: 3600,
    tags: ["posts"]
  }
);

// 使用
const posts = await getCachedPosts("tech");
```

  
### 5. Next.js 的 4 層快取（必懂）

```text
┌──────────────────────────────────────────────┐
│ 1. Request Memoization（請求記憶）            │
│    同一次 render、同樣 fetch 只執行一次        │
│    自動、不可關                                │
└──────────────────────────────────────────────┘

┌──────────────────────────────────────────────┐
│ 2. Data Cache（資料快取）                     │
│    fetch 結果持久化（跨多次 deployment）        │
│    revalidate / tags 控制                      │
│    存在 server 檔案系統                        │
└──────────────────────────────────────────────┘

┌──────────────────────────────────────────────┐
│ 3. Full Route Cache（整路由快取）              │
│    整個 HTML + RSC payload 快取                │
│    build 時產生（SSG）或第一次訪問時建立        │
└──────────────────────────────────────────────┘

┌──────────────────────────────────────────────┐
│ 4. Router Cache（瀏覽器路由快取）              │
│    Client side 快取已造訪頁面                  │
│    導航不重打 server                            │
│    使用者重整才失效                             │
└──────────────────────────────────────────────┘
```

> 💡 **白話**：
> 1. **Request Memoization**：「**同一次渲染、別重複問同個問題**」
> 2. **Data Cache**：「**問過的答案存起來、下次直接給**」
> 3. **Full Route Cache**：「**整頁 HTML 快取**」
> 4. **Router Cache**：「**瀏覽器自己存路由切換的歷史**」

  
### 6. 常見抓資料模式

**並行抓（同時抓多個、快）：**

```text
export default async function Page() {
  const [user, posts, stats] = await Promise.all([
    getUser(),
    getPosts(),
    getStats()
  ]);

  return (
    <>
      <UserCard user={user} />
      <PostList posts={posts} />
      <StatsPanel stats={stats} />
    </>
  );
}
```

**序列抓（後者依賴前者、慢）：**

```text
export default async function Page({ params }) {
  const { id } = await params;
  const user = await getUser(id);              // 先抓 user
  const orders = await getUserOrders(user.id); // 拿 user.id 再抓 orders

  return <Dashboard user={user} orders={orders} />;
}
```

**抓 + Suspense 切割（L9 詳教）：**

```text
export default function Page() {
  return (
    <>
      <Suspense fallback={<Loading />}>
        <SlowComponent />  {/* 慢的另外載 */}
      </Suspense>
      <FastContent />     {/* 快的先顯示 */}
    </>
  );
}
```

  
### 7. 實戰：何時用哪個快取設定

```text
🟢 force-static / revalidate: false
   → 部落格文章內容（一發布就不太變）
   → About 頁、條款頁

🟡 revalidate: 60 ~ 3600
   → 商品列表（每分鐘 / 每小時更新庫存）
   → 首頁排行榜
   → 新聞首頁

🟡 tags + revalidateTag
   → 後台改了文章、前台立刻更新
   → 不想等時間到

🔴 no-store
   → 個人化內容（用戶 dashboard、購物車）
   → 包含 cookies / headers 的頁面
   → 即時資料（股價、留言）
```

  
### 5 大常見坑

```text
❌ 1. 所有 fetch 都 cache: "no-store"
   → 失去 Next.js 快取好處、server 帳單高
   → 修：能 cache 就 cache、revalidate 控制新鮮度

❌ 2. 改了 DB 但網站沒更新
   → 忘記 revalidatePath / revalidateTag
   → 修：mutation 後一定 revalidate

❌ 3. 用 cookies() / headers() 後整頁變 dynamic
   → 任何 dynamic API 都會「污染」整頁
   → 修：只在需要的元件用、外面用 Suspense 包

❌ 4. fetch 用 cache: "no-store" 但又設 revalidate
   → 互相矛盾、Next.js 報 error
   → 修：兩個只能擇一

❌ 5. unstable_cache 的 key 沒包含參數
   → 不同參數共用快取、拿到錯資料
   → 修：key 用陣列、包進所有變數
```

  
### 🎯 重點回顧

- **fetch 4 種快取**：force-cache（預設）/ no-store / revalidate / tags
- **手動觸發**：`revalidateTag()` / `revalidatePath()`
- **整頁設定**：`export const dynamic / revalidate / fetchCache`
- **unstable_cache**：包任何 async function
- **4 層快取**：Request Memo / Data Cache / Full Route / Router
- **策略**：能 SSG 就 SSG、其次 ISR、最後才動態
