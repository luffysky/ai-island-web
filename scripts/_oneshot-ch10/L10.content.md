完整的 **錯誤處理（error handling、出錯時不讓網站整個炸掉）** 是 production（線上正式環境）跟 demo（示範）的最大差別。Next.js 提供 3 個特殊檔案：`error.tsx` / `not-found.tsx` / `global-error.tsx`、加上手動觸發 API、組成完整錯誤防線。

> 💡 **這課的價值**：使用者看到「白屏」或「Application error」是最糟的體驗。**每個路由都該有錯誤頁、404 該有友善提示、致命錯誤不能讓整站掛**。學完這課你的網站「**錯了但不死**」。

  
### 1. error.tsx：自動錯誤邊界

📄 **寫進程式檔**：`app/blog/error.tsx`

```text
"use client";  // ⚠️ error.tsx 必須是 Client Component
import { useEffect } from "react";

export default function Error({
  error,
  reset
}: {
  error: Error & { digest?: string };
  reset: () => void;
}) {
  useEffect(() => {
    // 把錯誤記到 Sentry / LogRocket / 自家後端
    console.error(error);
  }, [error]);

  return (
    <div className="error-page">
      <h2>發生錯誤</h2>
      <p>{error.message}</p>
      {error.digest && (
        <p className="text-sm text-gray-500">錯誤代號：{error.digest}</p>
      )}
      <button onClick={reset}>重試</button>
    </div>
  );
}
```

**白話**：放在 `/blog` 資料夾的 `error.tsx`、會「**自動 catch /blog 內所有錯誤**」。下層任何 page / layout 拋錯都不會讓整站掛、只會顯示這個錯誤頁。

> 💡 **重點**：error.tsx **必須是 Client Component**（要在瀏覽器執行 reset / onClick）、所以頂端要 `"use client"`。

  
### 2. global-error.tsx：根層級錯誤（最後一道防線）

如果連 root layout 都壞了、上面的 error.tsx 接不住——這時用 `global-error.tsx`：

📄 **寫進程式檔**：`app/global-error.tsx`

```text
"use client";

export default function GlobalError({ error, reset }) {
  return (
    <html>
      <body>
        <h2>致命錯誤</h2>
        <p>{error.message}</p>
        <button onClick={reset}>重試</button>
      </body>
    </html>
  );
}
```

**注意**：`global-error.tsx` 要自己寫 `<html>` 跟 `<body>`、因為它是「**取代根 layout**」、不是「**被根 layout 包**」。

  
### 3. not-found.tsx：404 頁

📄 **寫進程式檔**：`app/not-found.tsx`（全站 404）

```text
import Link from "next/link";

export default function NotFound() {
  return (
    <div className="not-found">
      <h2>404</h2>
      <p>找不到這個頁面、可能搬走或不存在了。</p>
      <Link href="/">回首頁</Link>
    </div>
  );
}
```

📄 **寫進程式檔**：`app/blog/not-found.tsx`（只給 /blog 路徑用）

```text
export default function BlogNotFound() {
  return (
    <div>
      <h2>找不到這篇文章</h2>
      <Link href="/blog">瀏覽其他文章</Link>
    </div>
  );
}
```

  
### 4. 手動觸發 notFound()

不只訪問不存在的 URL 會跳 404、**程式碼內可以主動觸發**：

📄 **寫進程式檔**：`app/blog/[slug]/page.tsx`

```text
import { notFound } from "next/navigation";

export default async function Post({ params }) {
  const { slug } = await params;
  const post = await getPost(slug);

  if (!post) {
    notFound();  // 立刻轉到最近的 not-found.tsx
  }

  // 已過期文章、也當 404
  if (post.expiredAt && post.expiredAt < new Date()) {
    notFound();
  }

  return <article>{post.content}</article>;
}
```

**好處**：Google 收到 404 status code、不會把這頁當有效內容索引。

  
### 5. 各層 error / not-found 的查找順序

```text
訪問 /blog/some-post 出錯時、Next.js 找：

app/blog/[slug]/error.tsx     ← 最近、先找
  ↓ 沒有 ↓
app/blog/error.tsx
  ↓ 沒有 ↓
app/error.tsx
  ↓ 沒有 ↓
app/global-error.tsx           ← 最後防線

notFound() 觸發時找：

app/blog/[slug]/not-found.tsx  ← 最近、先找
  ↓ 沒有 ↓
app/blog/not-found.tsx
  ↓ 沒有 ↓
app/not-found.tsx
```

  
### 6. try / catch 與 error.tsx 的選擇

```text
🟢 用 try / catch（在元件內處理）
   → 預期可能失敗、想優雅顯示替代內容
   → 例：API 掛掉時顯示「資料暫時無法載入」
   
   const data = await fetchData().catch(() => null);
   if (!data) return <div>暫時無法載入</div>;

🟢 讓 error.tsx 接（拋出去）
   → 嚴重錯誤、無法處理
   → 例：DB 連不上、auth 完全失敗
   → 直接 throw new Error()
```

📄 **範例**：

```text
export default async function ProductList() {
  try {
    const products = await fetchProducts();
    return <ul>{...}</ul>;
  } catch (e) {
    // 預期可能失敗、顯示替代
    return <div>商品暫時無法載入、請稍後再試</div>;
  }
}

// 對比：嚴重錯誤就讓 error.tsx 接
export default async function CheckoutPage() {
  const session = await auth();
  if (!session) throw new Error("Not authenticated");  // 讓 error.tsx 處理
  ...
}
```

  
### 7. Server Action 內錯誤處理

```text
"use server";

export async function createPost(formData: FormData) {
  try {
    const post = await db.posts.create({ ... });
    return { success: true, post };
  } catch (e) {
    // 別 throw、回傳錯誤物件給前端顯示
    return {
      error: e instanceof Error ? e.message : "未知錯誤"
    };
  }
}
```

  
### 8. 錯誤記錄到外部服務（production 必裝）

```text
// app/error.tsx
"use client";
import { useEffect } from "react";
import * as Sentry from "@sentry/nextjs";

export default function Error({ error, reset }) {
  useEffect(() => {
    Sentry.captureException(error);  // 自動上傳到 Sentry dashboard
  }, [error]);

  return <ErrorUI error={error} reset={reset} />;
}
```

**推薦工具**：Sentry（最大宗）/ LogRocket / Highlight / Axiom（免費額度大）。

  
### 9. error.tsx 設計建議

```text
✅ 不要顯示原始 error.message
   → 對使用者沒意義、可能露機敏資訊
   → 改顯示「發生錯誤、已通知工程師」

✅ 顯示 error.digest（Next.js 生成的錯誤代號）
   → 使用者回報問題時可以給代號
   → 工程師查 log 對得上

✅ 提供 reset() 按鈕
   → 讓使用者有掌控感
   → 不用重整整頁

✅ 提供「回首頁」連結
   → 讓使用者有出路

✅ 區分「使用者錯」跟「系統錯」
   → 404 → 你打錯網址
   → 500 → 我們的問題
```

  
### 5 大常見坑

```text
❌ 1. error.tsx 沒加 "use client"
   → Next.js 報錯
   → 修：頂端必加 "use client"

❌ 2. global-error.tsx 沒寫 <html><body>
   → 取代根 layout、必須自己寫
   → 修：加上完整 HTML 結構

❌ 3. notFound() 後又 return UI
   → notFound 已 throw、return 不會跑
   → 修：notFound() 後不用 return

❌ 4. 把 error.message 直接秀給使用者
   → 「PrismaClientKnownRequestError: ...」使用者看不懂
   → 修：error.tsx 顯示友善訊息、log 完整 error 到後端

❌ 5. 整站沒 error.tsx
   → 出錯整頁掛、白屏 + console error
   → 修：至少 app/error.tsx 跟 app/not-found.tsx
```

  
### 🎯 重點回顧

- **error.tsx**：自動 catch 下層錯誤、必須 Client Component
- **global-error.tsx**：最後防線、自己寫 html / body
- **not-found.tsx**：404 頁、可分區段
- **notFound()**：程式內主動觸發 404
- **查找順序**：最近的 error / not-found 先生效
- **production 必裝**：Sentry / LogRocket 記錯誤
- **顯示原則**：友善訊息給用戶、完整 error log 到後端
