### 5 大關鍵設計決策

  
### 決策 1：URL 去重（避免重複爬）

**問題**：1000 萬 URL、不能用 `if url in seen_list` 線性搜尋（O(n) 太慢）。

**解法**：用 Redis SET（O(1) 查重 + 分散式共用）。

```text
package main

import (
    "context"
    "crypto/sha256"
    "fmt"
    "time"
    "github.com/redis/go-redis/v9"
)

func isSeen(ctx context.Context, rdb *redis.Client, url string) bool {
    // === URL 算 SHA256 hash（省記憶體）===
    hash := fmt.Sprintf("%x", sha256.Sum256([]byte(url)))
    exists, _ := rdb.SIsMember(ctx, "seen", hash).Result()
    return exists
}

func markSeen(ctx context.Context, rdb *redis.Client, url string) {
    hash := fmt.Sprintf("%x", sha256.Sum256([]byte(url)))
    rdb.SAdd(ctx, "seen", hash)
    rdb.Expire(ctx, "seen", 24*time.Hour)    // 24h 過期、自動清
}
```

> 💡 **為什麼 hash**：(1) URL 可能很長、hash 32 字元固定 (2) Redis SET 用 hash 省記憶體 (3) hash collision 機率近 0。

  
### 決策 2：Rate Limiting（不打爆對方）

**問題**：50 個 worker 同時打對方、會被擋。

**解法**：每個 domain 各自限流、token bucket 算法。

```text
import "golang.org/x/time/rate"

var (
    limiters = make(map[string]*rate.Limiter)
    mu       sync.Mutex
)

func getLimiter(domain string) *rate.Limiter {
    mu.Lock()
    defer mu.Unlock()
    
    if l, ok := limiters[domain]; ok {
        return l
    }
    
    // === 每秒 5 個請求、burst 10 ===
    l := rate.NewLimiter(5, 10)
    limiters[domain] = l
    return l
}

// === 使用 ===
limiter := getLimiter("example.com")
limiter.Wait(ctx)                            // block 直到 token 可用
fetch(url)
```

> 💡 **token bucket**：(1) 每秒補 5 個 token (2) 最多存 10 個（burst） (3) 沒 token 就等。**對 example.com 限速、不影響其他 domain**。

  
### 決策 3：錯誤分類 + 重試策略

**問題**：4xx 不該 retry（資料問題）、5xx / 網路錯誤該 retry（暫時問題）。

```text
func shouldRetry(err error, statusCode int) bool {
    // === 4xx 不重試（網站說沒這個資源、重試也沒用）===
    if statusCode >= 400 && statusCode < 500 {
        return false
    }
    
    // === 5xx 或網路錯重試 ===
    if statusCode >= 500 || isNetworkError(err) {
        return true
    }
    
    return false
}

// === Exponential backoff 重試 ===
for attempt := 0; attempt < 3; attempt++ {
    err := fetchAndParse(url)
    if err == nil {
        return nil                            // 成功
    }
    
    if !shouldRetry(err, statusCode) {
        markFailed(url, err)
        return err                            // 不該 retry、直接失敗
    }
    
    // 等 2^attempt 秒（1s, 2s, 4s）
    time.Sleep(time.Duration(1<<attempt) * time.Second)
}

// 試了 3 次還是失敗、推 Dead Letter Queue
sendToDeadLetter(url)
```

  
### 決策 4：監控 + 告警（必須的、否則資料默默掉）

```text
=== 關鍵指標 + 告警閾值 ===

📊 Queue 長度
   → 過長（> 100 萬）= scheduler 太快、worker 不夠
   → 過短（< 1000）= worker 空閒、可減 pod

📊 Worker 並發數
   → < 5 個活著 = 緊急告警（快全死）
   → > 80% busy = HPA 加 pod

📊 失敗率
   → > 10% = 立刻通知（網站改版？被擋？）
   → > 50% = 半夜叫醒工程師

📊 寫入速度
   → 24h 沒新資料 = scheduler 或 worker 全死
   → 速度突然降 50% = 對方加反爬

📊 機器資源
   → CPU > 90% = 加 pod
   → 記憶體 > 80% = 檢查 leak
   → 磁碟 > 80% = 清舊 log / 加儲存
```

```text
[告警] → Slack / PagerDuty / Discord
```

### 6 大設計準則（千萬規模必遵守）

```text
1. ⭐ 冪等性（idempotent）
   → 同 URL 爬 2 次結果一樣
   → 用 UPSERT 不用 INSERT
   → worker crash 重啟不會壞資料

2. ⭐ 去耦合
   → Scheduler / Worker / DB 分開
   → 一個壞、其他繼續跑
   → 用 queue 連接

3. ⭐ 觀測性
   → 每個關鍵 metric 都監控
   → 半夜資料停了能被叫醒
   → 不能「跑很久才發現問題」

4. ⭐ 自動 retry + dead letter
   → 暫時錯誤自動 retry
   → 永久錯誤推 dead letter queue
   → 不能「丟失資料」

5. ⭐ 漸進部署
   → 改 code 先 1 個 worker 測
   → 沒問題才推 50 個
   → 不能「全部一起死」

6. ⭐ 成本控制
   → 知道每筆資料成本多少
   → 找最貴的優化
   → 不能「燒錢不知道」
```
