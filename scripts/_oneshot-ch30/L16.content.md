**Streaming 爬蟲 = 邊抓邊處理、不爆記憶體**——傳統「**全爬完再處理**」百萬筆 OOM、streaming「**一筆爬完寫一筆**」記憶體 < 100MB。

> 💡 **何時必用**：(1) 資料量 > 10 萬筆 (2) 不確定何時結束（無限捲動） (3) 即時需要結果（不能等全部）。**Python generator + Node async generator** 是核心技術。

  
### 為什麼要 Streaming

```text
=== 傳統「全部爬完再處理」===

def scrape_all():
    results = []
    for page in range(1, 1000):                   # 爬 1000 頁
        items = scrape_page(page)                 # 每頁 100 筆
        results.extend(items)                     # 累積到 list
    return results

results = scrape_all()                            # 等 1 小時
for item in results:
    save_to_db(item)

# 問題：
# - 10 萬筆全在記憶體 → 1-5 GB
# - 中途 crash → 全沒了
# - 不能即時看到結果
```

  
```text
=== Streaming「邊抓邊處理」===

def scrape_all():
    for page in range(1, 1000):
        for item in scrape_page(page):
            yield item                             # ⭐ 每筆立刻 yield

# 邊抓邊存
for item in scrape_all():
    save_to_db(item)

# 好處：
# - 記憶體永遠只有 1 筆
# - 中途 crash 已存的還在
# - 即時看到結果
```

  
### Python Generator（streaming 核心）

```text
def scrape_all():
    """generator function、用 yield 不用 return"""
    for page in range(1, 1000):
        for item in scrape_page(page):
            yield item                             # yield = 「給一筆、暫停、等下次呼叫」

# === 用法跟一般 list 一樣 ===
for item in scrape_all():
    save_to_db(item)
```

> 💡 **`yield` 是 Python 的魔法**：(1) function 變 generator (2) 一筆一筆給 (3) 不會把全部塞進記憶體。

### JSONL Streaming（最常見的 streaming 儲存）

**JSONL = 每行一個 JSON**、適合 streaming 寫入。

```text
import json

with open("data.jsonl", "w", encoding="utf-8") as f:
    for item in scrape_all():
        # === 每筆寫一行 ===
        f.write(json.dumps(item, ensure_ascii=False) + "\n")
```

**為什麼 JSONL：**

```text
✅ 邊寫邊存（不用結束才寫）
✅ 每行獨立、可逐行讀取
✅ 支援 streaming append
✅ 工具完整（jq / Spark / pandas 都支援）

❌ 不適合
- 需要全文件 JSON（不能逐行）
```

  
### Scrapy Pipeline（自動 streaming）

**Scrapy 內建支援 streaming**：

```text
# === pipelines.py ===
class MongoPipeline:
    """每爬到一筆、立刻寫 MongoDB"""
    
    def open_spider(self, spider):
        from pymongo import MongoClient
        self.client = MongoClient("localhost:27017")
        self.db = self.client["scraper"]
    
    def close_spider(self, spider):
        self.client.close()
    
    def process_item(self, item, spider):
        # ⭐ 這個 method 每筆都會呼叫
        self.db.items.insert_one(dict(item))
        return item
```

> 💡 **Scrapy 設計哲學**：每爬到一筆 item、馬上經過 pipeline、寫 DB、不累積在記憶體。

  
### Node Streams（JS streaming）

```text
import { pipeline } from 'node:stream/promises';
import { createWriteStream } from 'node:fs';

async function* scrapeAll() {
  for (let page = 1; page <= 1000; page++) {
    const items = await scrapePage(page);
    for (const item of items) {
      yield JSON.stringify(item) + '\n';            // 每筆 yield 一行
    }
  }
}

// === pipeline 串到檔案 ===
await pipeline(
  scrapeAll(),
  createWriteStream('data.jsonl')
);
```

> 💡 **Node `async function*`** = async generator、跟 Python 用法一樣。

  
### 批量寫 DB（streaming 變體）

**問題**：每筆寫 DB 慢、batch 寫快。

**解法**：streaming 累積 BATCH_SIZE 才寫 1 次。

```text
BATCH_SIZE = 1000
batch = []

for item in scrape_all():
    batch.append(item)
    
    # === 滿 1000 筆寫一次 ===
    if len(batch) >= BATCH_SIZE:
        db.bulk_insert(batch)
        batch = []                                # 清空 batch

# === 最後不滿 1000 的也要寫 ===
if batch:
    db.bulk_insert(batch)
```

**好處：**

```text
- DB 寫入 100-1000x 快（batch insert vs 單筆）
- 記憶體仍只有 1000 筆
- 即使 crash、最多丟最後 1000 筆
```

  
### 記憶體監控（必加）

```text
import psutil
import os

def check_memory():
    """印當前記憶體用量"""
    process = psutil.Process(os.getpid())
    mb = process.memory_info().rss / 1024 / 1024
    print(f"Memory: {mb:.1f} MB")
    return mb

# === 爬蟲中定期檢查 ===
for i, item in enumerate(scrape_all()):
    save_to_db(item)
    
    if i % 10000 == 0:
        check_memory()
        # 預期：< 100 MB（streaming）
        # 如果一直成長 → 有 memory leak、查 code
```

### Apache Kafka / Redis Stream（大規模 streaming）

**規模 10 萬+ events / 秒：**

```text
=== Kafka（業界標準）===

[Scrapy spider]
    ↓
[Kafka producer.send(items)]
    ↓
[Kafka topic（持久化）]
    ↓
[Multiple consumers（並行處理）]
    ↓
[ElasticSearch / DataLake]
```

**為什麼用 Kafka：**

```text
✅ 高吞吐（10 萬+ events/秒）
✅ 持久化（producer / consumer 中斷不丟資料）
✅ Multiple consumers（一份資料、多個下游）
✅ 業界標準
```

  
### 心法總結

```text
✅ 用 streaming 的時機
- 資料量 > 10 萬筆
- 不確定何時結束（無限捲動）
- 即時要結果
- 記憶體有限

❌ 不用 streaming
- 資料 < 1000 筆（直接 list 就好）
- 需要全部資料才能處理（如要排序）
- script 一次性、跑完就好

🎯 心法
- 「永遠想：這個 list 會不會爆？」
- 「能 streaming 就 streaming、保險」
- 「百萬筆必 streaming、別過度自信」
```
