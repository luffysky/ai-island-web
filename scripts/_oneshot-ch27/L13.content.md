### SQL → pandas（拉資料用）

```text
# === 跑 SQL 直接拿 DataFrame ===
df = pd.read_sql("""
    SELECT date, category, SUM(amount) AS revenue
    FROM orders
    WHERE date >= NOW() - INTERVAL '30 days'
    GROUP BY date, category
    ORDER BY date
""", engine)

# === 直接讀整個表 ===
df = pd.read_sql_table("orders", engine)

# === 帶參數（防 SQL injection、超重要！）===
df = pd.read_sql("""
    SELECT * FROM orders WHERE user_id = %(uid)s
""", engine, params={"uid": 123})
```

> ⚠️ **永遠用 params 帶參數、不要字串拼接 SQL** —— 拼接會被 SQL injection、駭客直接拖庫。

  
### pandas → DB（分析完寫回）

```text
# === 整個 df 寫進 DB（小資料）===
df.to_sql(
    "results",
    engine,
    if_exists="replace",                     # 表存在就取代
    index=False                              # 不要寫 Pandas 的 index 欄
)
```

**`if_exists` 3 個選項：**

| 值 | 行為 |
|---|---|
| `"fail"` | 表已存在就 raise 錯誤（最安全） |
| `"replace"` | 砍掉重建（會丟失原資料、小心！） |
| `"append"` | 接在後面（日報 / 增量寫入用） |

  
**大資料分批寫（避免一次塞爆 DB）：**

```text
df.to_sql(
    "big",
    engine,
    chunksize=10000,                         # 每 10000 列一批
    method="multi"                           # 用 multi-value INSERT、快 10 倍
)
```

  
### 完整 ETL Pipeline 實戰

**ETL** = Extract（抓）/ Transform（轉）/ Load（存）—— 資料工程基本流程。

```text
def daily_etl():
    """每日跑一次的 ETL：撈昨天的訂單 → 算每月每用戶統計 → 寫回 DB。"""
    
    # === Extract：從 source DB 撈昨天的訂單 ===
    df = pd.read_sql("""
        SELECT order_id, user_id, amount, created_at
        FROM orders
        WHERE DATE(created_at) = CURRENT_DATE - 1
    """, engine)
    
    # === Transform：加月份欄 + groupby 算統計 ===
    df["month"] = df["created_at"].dt.to_period("M")
    summary = df.groupby(["month", "user_id"]).agg(
        order_count=("order_id", "count"),
        total_amount=("amount", "sum")
    ).reset_index()
    
    # === Load：寫進 summary 表 ===
    summary.to_sql(
        "daily_user_summary",
        engine,
        if_exists="append",                   # 接後面、不取代
        index=False
    )
    print(f"處理 {len(summary)} 筆")

# === 排程：用 cron / Airflow / Prefect 每天凌晨 1 點跑 ===
```

  
### SQLAlchemy ORM（進階、ch26 L28 詳教）

當「**寫 SQL 太累、要物件化**」時用 ORM：

```text
from sqlalchemy.orm import declarative_base, sessionmaker
from sqlalchemy import Column, Integer, String

Base = declarative_base()

class User(Base):
    __tablename__ = "users"
    id = Column(Integer, primary_key=True)
    name = Column(String)

Session = sessionmaker(bind=engine)
session = Session()

# === 用 Python 物件查詢、不寫 SQL ===
users = session.query(User).filter(User.id > 10).all()
```

> 💡 **ORM vs raw SQL**：分析師通常**用 raw SQL**（更直覺）、工程師寫 production 用 ORM（避免重複）。

  
### 處理大資料（記憶體不夠時）

**1. Chunking（分批讀、最常用）：**

```text
chunks = pd.read_sql(
    "SELECT * FROM huge_table",
    engine,
    chunksize=10000                          # 一次拿 10000 列
)

for chunk in chunks:                          # 每個 chunk 是個小 DataFrame
    process(chunk)                            # 處理完就放掉、記憶體不爆
```

  
**2. Lazy load（用 Dask、分散式 Pandas）：**

```text
df = dd.read_sql_table("huge", engine, index_col="id")

# .compute() 才真正執行（之前都是「畫了個計算圖」）
```

  
### Parquet 格式（比 CSV 快 10 倍、大資料必用）

**Parquet** = 「**欄式儲存格式**」、超適合分析（按欄讀超快、檔案小一半以上）。

```text
# === 寫 Parquet ===
df.to_parquet("data.parquet", compression="snappy")
# snappy = Google 的壓縮演算法、快 + 平衡壓縮率

# === 讀 Parquet ===
df = pd.read_parquet("data.parquet")

# === 直接寫到 S3（雲端、大資料 stack 常見）===
df.to_parquet("s3://bucket/data.parquet")
```
