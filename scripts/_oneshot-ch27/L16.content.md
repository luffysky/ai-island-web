**A/B Test** = 「**用資料、不靠直覺**」決定產品改動。設計 / 文案 / 按鈕顏色 / 演算法、改之前先 A/B 驗證、不要拍腦袋。

> 💡 **A/B test 心法**：不是「找最大值」、是「**降低決策風險**」。80% A/B test 結果是 flat（沒差）——接受 flat = 把資源放真正有效的事。

  
### 為什麼要 A/B Test

```text
=== 沒 A/B test：靠「我覺得」 ===
PM：「按鈕改橘色比較顯眼吧？」
設計：「綠色比較高級耶」
工程：「...都行啦」
→ 改了上線、沒人知道好不好

=== 有 A/B test：靠資料 ===
1. 隨機把 50% 用戶顯示 A（綠色）、50% 顯示 B（橘色）
2. 跑 2 週
3. 統計分析：B 點擊率真的 > A 嗎？還是隨機波動？
```

  
### 範例：按鈕顏色測試

```text
A 組（綠色按鈕）：5000 訪客、240 點擊（4.8%）
B 組（橘色按鈕）：5000 訪客、275 點擊（5.5%）

問題：差距 0.7% 真的有意義？還是隨機波動？
→ 需要統計檢定回答
```

  
### p-value：統計顯著性的核心概念

**p-value < 0.05** = 「**假設兩組沒差、出現這個觀察值的機率 < 5%**」= 通常代表「顯著、不是隨機」。

🖥️ **在終端機輸入**（裝 statsmodels）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

```text
uv add statsmodels
```

**這行做什麼？** 裝 statsmodels（L15 已介紹）、Python 統計分析標準套件、含 A/B test 的 z-test / t-test。

  
```text
from statsmodels.stats.proportion import proportions_ztest

count = [240, 275]                            # 成功數（A, B）
nobs = [5000, 5000]                           # 樣本數（A, B）

stat, pval = proportions_ztest(count, nobs)
print(f"p-value: {pval:.4f}")

if pval < 0.05:
    print("✅ 顯著、B 真的比較好")
else:
    print("❌ 不顯著、可能只是隨機")
```

  
### Power Analysis（事前算需要多少樣本、必做）

**核心問題**：「我要偵測 0.5% 的差異、每組要多少樣本才夠？」

**為什麼要事前算？** 樣本太少 = 看不到差異 = 浪費時間。

```text
from statsmodels.stats.power import zt_ind_solve_power

# === 想偵測 0.5% 的差異（從 5% → 5.5%）===
n = zt_ind_solve_power(
    effect_size=0.025,                        # Cohen's h（效應量）
    alpha=0.05,                               # 顯著水準（false positive 率）
    power=0.8                                 # 統計檢定力（true positive 率）
)
print(f"每組需要 {n:.0f} 樣本")
# 例：算出來 6280、每組要等到 6280 樣本才能做檢定
```

  
### 3 大檢定方式（看資料型態選）

**1. 比例 / 轉換率（離散）：Chi-square 或 Z-test：**

```text
from scipy.stats import chi2_contingency

# === 列聯表 ===
observed = [
    [240, 4760],                              # A：成功 240、失敗 4760
    [275, 4725]                               # B：成功 275、失敗 4725
]
chi2, pval, _, _ = chi2_contingency(observed)
```

  
**2. 平均（連續變數、如客單價）：T-test：**

```text
from scipy.stats import ttest_ind

a_sales = [1000, 1200, 1500, 800, 1100, ...]
b_sales = [1100, 1300, 1450, 950, 1250, ...]

stat, pval = ttest_ind(a_sales, b_sales)
```

  
**3. 多組（不只 2 個變體、如 4 種按鈕顏色）：ANOVA：**

```text
from scipy.stats import f_oneway

stat, pval = f_oneway(a, b, c, d)
# ANOVA 是「**至少有一組跟其他不同**」的檢定
# 顯著後要做事後比較（post-hoc）才知道是哪兩組差
```

  
### 信心區間（比 p-value 更實用）

**「p<0.05 顯著」很抽象、客戶看不懂。改報 95% 信心區間更實用。**

```text
import numpy as np
from scipy import stats

mean = np.mean(data)
sem = stats.sem(data)                         # 標準誤差
ci = stats.t.interval(
    0.95,                                     # 95% 信心
    len(data) - 1,
    loc=mean,
    scale=sem
)
print(f"95% CI: {ci}")
```

**報告寫法對比：**

```text
❌「B vs A 提升 14% (p<0.05)」 ← 客戶不懂 p 是啥
✅「B 比 A 提升 14% (95% CI: 5% to 23%)」 ← 老闆秒懂
```

  
### A/B Test 完整流程（7 步驟）

```text
1. 【定假設】「B 會比 A 高 ≥ 10%」（事前寫下）

2. 【算樣本數】Power analysis 算每組要多少

3. 【隨機分組】注意公平
   - 不要 A 都給老用戶、B 都給新用戶
   - 用 hash(user_id) % 2 分組最公平

4. 【跑滿時間】至少 1-2 週、含週末週間
   - 不同時段行為不同、要含完整週期

5. 【不偷看】Peeking 偏差超大、絕對忍住

6. 【分析】p-value + 信心區間 + 業務意義

7. 【決定】A 還 B、還是兩個都不夠好、要重設計
```

  
### 5 大經典坑（新手必踩、必避）

**1. Peeking（偷看早停、最常犯）：**

```text
中間每天看「好像 B 贏了」就停 → 偽陽性率大增
解：用 sequential testing（允許 peek 的統計方法）
或乖乖跑滿預定樣本
```

**2. 樣本太少（自欺）：**

```text
看到 100 vs 105「B 贏 5%！」 → 其實是隨機
解：事前 power analysis、達到樣本數才停
```

**3. 多重比較（多測多錯）：**

```text
測 20 個指標、有 5% 機率某個指標「假性顯著」
解：Bonferroni 修正 → alpha 0.05 / 20 = 0.0025
   或用 False Discovery Rate (FDR) 控制
```

**4. Novelty effect（新鮮感）：**

```text
新東西短期吸引人、長期沒差
解：跑久一點、看「穩態」表現
```

**5. Selection bias（選擇偏誤）：**

```text
分組不公平（A 都給老用戶、B 都給新用戶）
解：randomization 用 user_id hash、不要靠時間 / 地區分組
```

### 進階：Bayesian A/B Test（更直觀）

**Bayesian** 不用 p-value、用機率：

```text
傳統：「p = 0.03、B 顯著好於 A」 ← 不直覺
Bayesian：「B 比 A 好的機率是 87%」 ← 老闆秒懂
```

**工具**：`PyMC`、`Bayesian-AB` Python 套件。

  
    **☕ 用人話講**
    
A/B test 4 件事背起來：(1) **事前算樣本**（power analysis、不然測爽的） (2) **p-value < 0.05 顯著**（離散用 Z-test、連續用 T-test、多組用 ANOVA） (3) **報信心區間**比 p-value 實用 100 倍 (4) **避 5 大坑**：peeking / 樣本太少 / 多重比較 / novelty / selection bias。**心法**：A/B test 是「**降低決策風險**」、不是「找最大值」、接受 80% flat 結果、把資源放真正有效的事。
