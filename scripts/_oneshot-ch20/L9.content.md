**API Versioning（版本控制）= 你的 API 升級時、不會把舊 client 弄壞**。**沒做版本控制 = 改一個欄位、行動 app 舊版直接 crash**。

> 💡 **這課的價值**：API 上線後改一定要懂 versioning、不然你會被綁死在第一版永遠不能改。學會 = 你能安心演進 API。

  
### 1. 為什麼需要 versioning

```text
情境：你的 API 上線一年

舊：GET /api/users/123
    → { name: "John Doe" }

業務需求：拆 first_name / last_name
新：GET /api/users/123
    → { firstName: "John", lastName: "Doe" }

🔥 災難：
- 行動 app v1.0（5 萬人用）：找不到 name 欄位 → crash
- 第三方 SDK：name is undefined → 整合爆
- 不能 force 大家升級
```

**Versioning 的目標**：
- 舊 client 用舊版、繼續工作
- 新 client 用新版、享受新功能
- 半年到一年的 deprecation 緩衝
- 最後優雅淘汰

  
### 2. 3 種 versioning 方式

```text
1. URL Versioning      /api/v1/users  /api/v2/users
2. Header Versioning   Accept: application/vnd.api.v2+json
3. Query Versioning    /api/users?version=2
```

  
### 3. URL Versioning — 最常見、最直觀

```text
GET /api/v1/users/123
GET /api/v2/users/123
```

**優點**：
- 看 URL 就知道版本
- 瀏覽器 / curl 直接試
- Browser cache 簡單（不同 URL）
- 文件分版本清楚

**缺點**：
- URL 變「不純」（REST 純粹派討厭）
- 每個 endpoint 都要重複 /v1

**業界用 URL versioning 的**：
- Stripe（v1）
- Twilio（v1 / v2）
- GitHub（v3 / v4）
- AWS（多）

  
### 4. Header Versioning — REST 純粹派

```text
GET /api/users/123
Accept: application/vnd.example.v2+json

→ server 看 Accept header 決定回 v1 還是 v2
```

**優點**：
- URL 純（資源不變、版本是「表現方式」）
- 符合 REST 哲學
- 一個 endpoint 多版本好管理

**缺點**：
- 瀏覽器試很麻煩（要改 header）
- cache 要看 Vary header
- 對人類不友善

**業界用的**：
- GitHub API v4 也支援 header
- 銀行 / 金融老 API

  
### 5. Query Versioning — 簡單但少用

```text
GET /api/users/123?version=2
```

**優點**：簡單
**缺點**：URL 有 query 還是不純、cache 容易出問題

**少用、不推薦**。

  
### 6. Semantic Versioning（語意化版本）

```text
v MAJOR . MINOR . PATCH

v1.2.3
↓
MAJOR  破壞性變更（v1 → v2）
MINOR  新增功能、向後相容（v1.2 → v1.3）
PATCH  bug fix（v1.2.3 → v1.2.4）

API 常用：
- 只看 MAJOR：/v1 → /v2
- MINOR 跟 PATCH 在同一個 /v1 內推
- 用 changelog 紀錄
```

  
### 7. Deprecation（淘汰）流程

```text
完整流程（建議 6 ~ 12 個月）：

Phase 1：宣布（month 0）
- changelog 寫明
- email 通知所有 API key 持有者
- docs 加紅色警告
- response header 加：
   Deprecation: true
   Sunset: Sat, 31 Dec 2026 23:59:59 GMT
   Link: <https://docs.api.com/migration>; rel="deprecation"

Phase 2：警告（month 3）
- log 用量、找出還沒升級的
- 主動聯絡

Phase 3：限制（month 9）
- 加 rate limit
- 回應加 X-Deprecation-Warning header
- 偶爾回 299 Warning（HTTP 內建警告）

Phase 4：關閉（month 12）
- 410 Gone
- Body 指向 migration guide
```

📄 **寫進程式檔**：`middleware/deprecation.ts`

```text
app.use('/api/v1', (req, res, next) => {
  res.set('Deprecation', 'true');
  res.set('Sunset', 'Sat, 31 Dec 2026 23:59:59 GMT');
  res.set('Link', '<https://docs.ai-island.com/v2-migration>; rel="deprecation"');
  next();
});
```

  
### 8. 內容協商（Content Negotiation）— Accept 完整版

```text
GET /api/users/123
Accept: application/vnd.ai-island.v2+json, application/vnd.ai-island.v1+json; q=0.5

server：先試 v2、若不支援降到 v1（q=0.5 是優先度）

→ 比硬編 version 彈性
```

  
### 9. Backward Compatibility 技巧

#### a. 加欄位 — 永遠安全

```text
v1：{ name, email }
v2：{ name, email, avatarUrl }   ← 加新欄位

→ v1 client 不知道 avatarUrl 也沒差
→ 不需要新 version
```

#### b. 改欄位 — 用「過渡版」

```text
v1：{ name: "John Doe" }

過渡（兩個都回）：
{ name: "John Doe", firstName: "John", lastName: "Doe" }

v1 client：拿 name（OK）
v2 client：拿 firstName / lastName（OK）

deprecation：等 v1 client 都升級了、刪 name
```

#### c. 刪欄位 — 一定要升 MAJOR

```text
v1：{ name, deprecatedField }
v2：{ name }   ← 刪掉
```

#### d. 改型別 — 一定要升 MAJOR

```text
v1：{ age: "30" }       ← string
v2：{ age: 30 }         ← number
```

  
### 10. GraphQL 的 versioning 哲學

```text
GraphQL：不做版本！

哲學：
- schema 是動態的
- 加欄位永遠安全（client 不選就不影響）
- 刪欄位用 @deprecated 緩衝
- 永遠保留舊欄位（最多警告）

實務上：
- 內部用 GraphQL 不版本
- 公開 API（GitHub v4）還是用 URL versioning
```

  
### 11. 何時該升 MAJOR？

```text
✅ 一定要升 v2：
- 刪欄位
- 改型別
- 改認證方式
- 改 URL 結構
- 改錯誤格式

✅ 可以不升（v1.x）：
- 加欄位
- 加 endpoint
- 加新功能
- bug fix

✅ 用 feature flag 過渡：
- API key 內標 beta
- header X-Experimental: true
- 部分 user 試新功能
```

  
### 12. 5 大常見坑

```text
❌ 1. 沒 versioning 就上線
   → 想改就改、客戶集體爆
   → 修：第一天就 /v1 開始

❌ 2. 升 v2 不留 v1
   → 行動 app 舊版直接 crash
   → 修：v1 留 6 ~ 12 個月、deprecation header

❌ 3. v2 改太多、客戶懶得升
   → 半年了 99% 還在 v1
   → 修：v2 改最小、其他用 v1.x 加

❌ 4. 沒 changelog
   → 客戶不知道有改
   → 修：每次 release 寫、email 通知

❌ 5. URL 跟 Header 混用
   → /v2 配 Accept v1 = 哪個贏？
   → 修：選一個徹底用
```

  
### 🎯 重點回顧

- **URL versioning**（最常見） vs **Header versioning**（最 REST 純）
- **加欄位永遠安全**——多數變更不需要新 MAJOR
- **刪 / 改 → 升 MAJOR**（v1 → v2）
- **Deprecation 6 ~ 12 個月緩衝**：宣布 → 警告 → 限制 → 關閉
- **HTTP header**：Deprecation / Sunset / Link 是業界標準
- **GraphQL 不做版本、用 @deprecated 演化**
