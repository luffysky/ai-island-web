**GraphQL（圖形查詢語言）= 「client 決定要什麼欄位」的 API 設計**。Facebook 2015 開源、為了解決 REST 的「打 5 個 endpoint 才能組一個頁面」。

> 💡 **這課的價值**：GitHub / Shopify / Linear 都用 GraphQL。學會 = 你能設計出「一個 request 拿完所有資料」的 API、前端開發速度 ×3。

  
### 1. REST 的痛點 → GraphQL 怎麼解

**情境：顯示一個 post + author + 5 個 comment**

```text
REST 做法：
GET /api/posts/123              → post 本體
GET /api/users/42               → author（要再打一次）
GET /api/posts/123/comments     → comments
（3 個 round trip、每個都 over-fetch）

GraphQL 做法：
POST /graphql
query {
  post(id: "123") {
    title
    body
    author { name avatar }
    comments(limit: 5) { body author { name } }
  }
}
→ 一個 request、剛好的欄位、剛好的數量
```

  
### 2. 三大操作（Operations）

```text
Query        讀資料（= REST 的 GET）
Mutation     寫資料（= REST 的 POST/PUT/PATCH/DELETE）
Subscription 訂閱（= WebSocket、即時推送）
```

  
### 3. Schema — 整個 GraphQL 的核心

📄 **寫進程式檔**：`schema.graphql`

```text
type Post {
  id: ID!                    # ! = 必有
  title: String!
  body: String
  author: User!              # 巢狀關聯
  comments(limit: Int): [Comment!]!   # 陣列
  createdAt: DateTime!
}

type User {
  id: ID!
  name: String!
  email: String!
  posts: [Post!]!
}

type Comment {
  id: ID!
  body: String!
  author: User!
}

type Query {
  post(id: ID!): Post
  posts(limit: Int = 10, offset: Int = 0): [Post!]!
  user(id: ID!): User
}

type Mutation {
  createPost(input: CreatePostInput!): Post!
  deletePost(id: ID!): Boolean!
}

input CreatePostInput {
  title: String!
  body: String
}

type Subscription {
  postAdded: Post!
}
```

**Schema 內建型別**：
- `Int / Float / String / Boolean / ID`
- `[Type]` 陣列
- `Type!` 不能 null
- `enum`、`input`、`union`、`interface`

  
### 4. Query 範例

```text
query GetMyDashboard($userId: ID!) {
  me: user(id: $userId) {
    name
    posts(limit: 5) {
      title
      comments {
        body
      }
    }
  }
}

variables:
{ "userId": "42" }
```

**特性**：
- 有 name（GetMyDashboard）方便 debug
- 帶 variables、不要塞 URL
- alias（`me: user`）改回應的 key

  
### 5. Mutation 範例

```text
mutation CreatePost($input: CreatePostInput!) {
  createPost(input: $input) {
    id
    title
    createdAt
  }
}

variables:
{ "input": { "title": "Hello", "body": "World" } }

response:
{
  "data": {
    "createPost": {
      "id": "789",
      "title": "Hello",
      "createdAt": "2026-05-28T..."
    }
  }
}
```

  
### 6. Subscription（WebSocket 上）

```text
subscription OnNewPost {
  postAdded {
    id
    title
    author { name }
  }
}

→ 每當有新 post、server 推送
→ 底層用 WebSocket（graphql-ws protocol）
```

  
### 7. Resolver — server 怎麼回應

📄 **寫進程式檔**：`resolvers.ts`

```text
export const resolvers = {
  Query: {
    post: async (_, { id }) => db.posts.findUnique({ where: { id } }),
    posts: async (_, { limit, offset }) =>
      db.posts.findMany({ take: limit, skip: offset }),
  },

  Mutation: {
    createPost: async (_, { input }, { user }) => {
      if (!user) throw new Error('Not authenticated');
      return db.posts.create({ data: { ...input, authorId: user.id } });
    },
  },

  Post: {
    author: (post) => db.users.findUnique({ where: { id: post.authorId } }),
    comments: (post, { limit }) =>
      db.comments.findMany({ where: { postId: post.id }, take: limit }),
  },
};
```

**Resolver 拿 4 個參數**：

```text
(parent, args, context, info) => {...}

parent   上一層 resolver 的回傳值
args     query / mutation 帶的參數
context  全域（auth user / db / ...）
info     query 的 metadata（少用）
```

  
### 8. N+1 問題 — GraphQL 最大坑

```text
query {
  posts {       # 1 個 query → 100 posts
    title
    author {    # 每個 post 又打 1 次 → 100 次 user query
      name
    }
  }
}

= 1 + 100 = 101 個 DB query → 災難
```

**解法：DataLoader（批次 + cache）**

📄 **寫進程式檔**：`dataloader.ts`

```text
import DataLoader from 'dataloader';

const userLoader = new DataLoader<string, User>(async (ids) => {
  const users = await db.users.findMany({ where: { id: { in: [...ids] } } });
  return ids.map(id => users.find(u => u.id === id));
});

// resolver 用
author: (post) => userLoader.load(post.authorId),
```

DataLoader 自動把 100 個 `load()` 合併成 1 個 batch query。

  
### 9. 跟 REST 完整對比

```text
                REST              GraphQL
─────────────────────────────────────────────────
Endpoint        多個（/posts...）   一個（/graphql）
方法            GET/POST/...      永遠 POST
拿資料          server 決定欄位    client 決定欄位
Over-fetch      常發生            不會
Under-fetch     常發生（多個 RT）  不會
Cache           HTTP cache 自動    要 client 自己
File upload     簡單              要 multipart spec
複雜度          低                中高
工具生態        最廣              成熟（Apollo / urql / Relay）
適合            公開 API、簡單     複雜前端、多端共用
```

### 11. 5 大常見坑

```text
❌ 1. 沒 DataLoader 直接上
   → N+1 query 把 DB 打爆
   → 修：所有跨表 resolver 用 DataLoader

❌ 2. 沒查詢深度限制
   → 駭客送無限巢狀 query 把 server 燒掉
   → 修：max depth + complexity scoring

❌ 3. 把 GraphQL 當 REST 用（GET 帶 query）
   → 失去 type-safe + variable
   → 修：用 POST + variables

❌ 4. Mutation 沒回新狀態
   → client 還要再 query 一次
   → 修：mutation return 完整 object

❌ 5. Schema 沒分檔、長到 10000 行
   → 沒人想 maintain
   → 修：按 domain 拆 modules、用 stitching
```

  
### 🎯 重點回顧

- **GraphQL = client 決定要什麼欄位**——一個 request 解所有
- **3 大操作**：Query（讀）、Mutation（寫）、Subscription（即時）
- **Schema 是核心**——server 跟 client 的合約
- **Resolver** 是每個欄位的「怎麼算」
- **N+1 問題**用 DataLoader 解
- **複雜前端建議 GraphQL、公開 API 用 REST**
