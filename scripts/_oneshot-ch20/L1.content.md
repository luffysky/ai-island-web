**API（Application Programming Interface、應用程式介面）= 「程式跟程式對話的方式」**。前端要拿後端的資料、第三方要打你的服務——都靠 API。**選對 API 架構 = 開發效率 ×3、選錯 = 半年後砍掉重練**。

> 💡 **這課的價值**：先把 6 大 API 流派印在腦袋裡——之後選哪個、為什麼選、各自的取捨。不要還沒比較就一頭栽進去。

  
### 1. 為什麼需要 API

```text
情境：你有一個 Notion-like 筆記 app

前端（網頁 / iOS / Android / Desktop）
  ↓ 透過 API 對話
後端（Node.js / Python / Go）
  ↓
資料庫（PostgreSQL）

API = 前後端的「合約」
  - 我要怎麼拿筆記？POST /api/notes
  - 回什麼？{ id, title, body }
  - 出錯怎麼辦？status 400 / 500
```

**沒 API = 前後端綁死、改一邊另一邊全爆**。

  
### 2. 6 大 API 流派

```text
┌─────────────┬─────────────┬──────────────┐
│ 流派         │ 出生年      │ 代表          │
├─────────────┼─────────────┼──────────────┤
│ SOAP         │ 1998        │ 銀行老系統     │
│ REST         │ 2000        │ GitHub / 99% │
│ GraphQL      │ 2015        │ Facebook     │
│ gRPC         │ 2016        │ Google / k8s │
│ WebSocket    │ 2011        │ 聊天 / 遊戲   │
│ SSE          │ 2009        │ AI streaming │
│ tRPC         │ 2021        │ TS 全棧       │
└─────────────┴─────────────┴──────────────┘
```

  
### 3. REST（Representational State Transfer、表現層狀態轉移）

**最主流、99% 公司用**。

```text
GET    /api/posts          → 拿所有 post
GET    /api/posts/123      → 拿某一個
POST   /api/posts          → 建立新 post
PUT    /api/posts/123      → 整個更新
PATCH  /api/posts/123      → 部分更新
DELETE /api/posts/123      → 刪掉
```

**特色**：
- 用 HTTP 動詞 + 名詞
- 回 JSON
- 跟瀏覽器超親
- 容易 cache（GET 可以）

**缺點**：
- 一個頁面常常要打 5 個 endpoint（over-fetch / under-fetch）
- 沒 schema、文件容易過時

  
### 4. GraphQL（圖形查詢語言）

**Facebook 2015 開源、要拿什麼就 query 什麼**。

```text
查詢：
query {
  post(id: 123) {
    title
    author {
      name
      avatar
    }
    comments(limit: 5) {
      body
    }
  }
}

→ 一個 request 拿到所有要的、不多不少
```

**特色**：
- 一個 endpoint（POST /graphql）
- client 決定要什麼
- 內建型別系統（schema）
- 自動 docs

**缺點**：
- server 端複雜（resolver / N+1 問題）
- cache 比 REST 難
- 學習曲線陡

  
### 5. tRPC（TypeScript RPC）— 2021 新銳

**TS 全棧最熱**：把 server function 直接當 client function 呼叫。

📄 **寫進程式檔**：`server/trpc.ts`

```text
export const appRouter = router({
  getPost: publicProcedure
    .input(z.object({ id: z.string() }))
    .query(({ input }) => db.posts.findUnique({ where: { id: input.id } })),
});
export type AppRouter = typeof appRouter;
```

📄 **寫進程式檔**：`client/page.tsx`

```text
const { data } = trpc.getPost.useQuery({ id: '123' });
//      ↑ 型別自動推斷、改 server 那邊 client 立刻知道
```

**特色**：
- 0 code generation（不用 codegen）
- 端到端型別安全
- 比 GraphQL 簡單 10 倍

**缺點**：
- 只給 TS 用（其他語言不行）
- 不適合公開 API

  
### 6. gRPC（Google RPC）

**Google 推的二進位 RPC、用 Protobuf 描述**。

📄 **寫進程式檔**：`post.proto`

```text
service PostService {
  rpc GetPost (GetPostRequest) returns (Post);
  rpc StreamPosts (Empty) returns (stream Post);
}

message Post {
  string id = 1;
  string title = 2;
  string body = 3;
}
```

**特色**：
- 二進位、超快、超省頻寬
- 多語言支援好（Go / Java / Python / Rust）
- 雙向 streaming
- Kubernetes 內部都用 gRPC

**缺點**：
- 瀏覽器要 grpc-web 中介
- 不能直接用 curl debug
- 對人類不友善

  
### 7. WebSocket — 雙向即時

```text
HTTP：問一句答一句、答完就斷
WebSocket：開一條長期連線、雙向都能傳

適合：
✅ 聊天 / Discord / Slack
✅ 多人協作（Figma / Google Docs）
✅ 遊戲
✅ 即時報價（股票 / 加密貨幣）
```

```text
const ws = new WebSocket('wss://api.example.com/chat');
ws.onmessage = (msg) => console.log(msg.data);
ws.send(JSON.stringify({ type: 'chat', text: 'hi' }));
```

  
### 8. SSE（Server-Sent Events、伺服器推送事件）

**HTTP 之上的「server → client 單向 streaming」**。

```text
適合：
✅ ChatGPT / Claude 那種「一個字一個字吐」
✅ 通知、推播
✅ Live log 顯示

優於 WebSocket：
✅ HTTP 之上、不用 upgrade
✅ 自動重連
✅ 簡單

缺於 WebSocket：
❌ 單向（client → server 還是 fetch）
```

```text
const es = new EventSource('/api/chat/stream');
es.onmessage = (e) => appendText(e.data);
```

  
### 9. 怎麼選

```text
公開 API（給第三方 / 行動 app）
   → REST（最通用）

內部 TS 全棧（Next.js + tRPC）
   → tRPC（最爽）

複雜資料、多端共用
   → GraphQL（前端決定要什麼）

微服務間（同公司、多語言）
   → gRPC（最快）

即時聊天 / 多人協作
   → WebSocket

AI streaming / 通知
   → SSE
```

**真實場景常混搭**：
- Linear：tRPC 內部 + REST 公開
- Vercel：REST 公開 + GraphQL 後台
- OpenAI / Anthropic：REST + SSE

  
### 10. 都要懂的 5 個基礎

```text
1. HTTP methods（GET / POST / PUT / PATCH / DELETE）
2. Status code（200 / 201 / 400 / 401 / 403 / 404 / 500）
3. JSON 格式
4. JWT / OAuth 認證
5. CORS / rate limit / pagination
```

  
### 11. 5 大常見坑

```text
❌ 1. 一開始就選 GraphQL 結果 over-engineering
   → 簡單 CRUD 用 REST 就好
   → 修：複雜度有需要再升 GraphQL

❌ 2. tRPC 想拿來做公開 API
   → 非 TS client 用不了
   → 修：tRPC 內部、外部用 REST

❌ 3. gRPC 想直接給瀏覽器用
   → 瀏覽器不支援 HTTP/2 trailers
   → 修：架 grpc-web proxy 或改 REST

❌ 4. 所有東西都用 WebSocket
   → server 連線數爆、scale 痛
   → 修：純讀用 HTTP、必要時才 WS

❌ 5. AI streaming 用 WebSocket
   → 太重、其實 SSE 就夠
   → 修：OpenAI / Anthropic 都用 SSE
```

  
### 🎯 重點回顧

- **6 大流派**：REST / GraphQL / tRPC / gRPC / WebSocket / SSE
- **REST** 是 99% 場景的安全選擇
- **GraphQL** 解決 over-fetch 但複雜
- **tRPC** TS 全棧最爽、但只給 TS
- **gRPC** 微服務間最快、瀏覽器要 proxy
- **WebSocket** 即時雙向、**SSE** 單向 streaming（AI 用）
- **真實系統常混搭** 多種
