**RESTful（合 REST 規範的）= 不只是「會 GET / POST」、有 6 大限制**。**99% 的公司寫的不是 RESTful、是「JSON over HTTP」**——但學懂正統可以讓你的 API 比同行專業 10 倍。

> 💡 **這課的價值**：你寫過 REST、但有沒有想過「為什麼要這樣寫」？這課教你 Roy Fielding（REST 發明者）的原本意思、+ 實務怎麼取捨。

  
### 1. REST 6 大限制（出自 2000 年 Fielding 博士論文）

```text
1. Client-Server（前後端分離）
2. Stateless（無狀態）
3. Cacheable（可快取）
4. Uniform Interface（統一介面）⭐ 最重要
5. Layered System（分層系統）
6. Code on Demand（選用、幾乎沒人做）
```

**達成全部 = 真正的 RESTful**。多數人只做了 1、2、4，叫做 "REST-like"。

  
### 2. Uniform Interface — 統一介面（核心）

#### 子原則 1：Resource（資源）— 用名詞、不用動詞

```text
✅ 對：
   GET    /api/posts           拿所有 post
   GET    /api/posts/123       拿一個
   POST   /api/posts           建立
   PUT    /api/posts/123       整個替換
   PATCH  /api/posts/123       部分更新
   DELETE /api/posts/123       刪掉

❌ 錯（很多人寫成這樣）：
   GET  /api/getPosts
   POST /api/createPost
   POST /api/deletePost/123    ← URL 帶動詞、用 POST 刪
```

> 💡 **白話**：URL 是「東西」、動作用 HTTP method 表達。

#### 子原則 2：URL 用複數、巢狀有限度

```text
✅ /api/posts                            所有 post
✅ /api/posts/123                        post 123
✅ /api/posts/123/comments               post 123 的 comment
✅ /api/posts/123/comments/456           特定 comment
❌ /api/posts/123/comments/456/author/likes  ← 太深、看不懂
```

**規則**：最多 2 層巢狀、再深就拆 endpoint。

#### 子原則 3：HTTP verb 各司其職

```text
GET      讀（safe + idempotent、可 cache）
POST     建立 / 觸發動作（不 idempotent）
PUT      整個替換（idempotent）
PATCH    部分更新（不一定 idempotent）
DELETE   刪除（idempotent）
HEAD     拿 header 不拿 body
OPTIONS  問支援哪些（CORS preflight）
```

**Idempotent（冪等）**：跑 N 次結果相同。`DELETE /posts/123` 跑 5 次跟跑 1 次效果一樣。

  
### 3. Status Code（狀態碼）— 用對的數字

```text
🟢 2xx 成功
   200 OK              GET 成功
   201 Created         POST 成功 + 新資源
   202 Accepted        收到、之後做（async job）
   204 No Content      DELETE 成功、沒 body

🟡 3xx 重定向
   301 Moved Permanently  永久搬家
   304 Not Modified       cache 還有效（配 ETag）

🔴 4xx Client 錯
   400 Bad Request        參數錯
   401 Unauthorized       沒登入
   403 Forbidden          登入了但沒權限
   404 Not Found          資源不存在
   409 Conflict           衝突（重複 email）
   422 Unprocessable      格式對但業務規則錯
   429 Too Many Requests  被 rate limit

🔥 5xx Server 錯
   500 Internal Error     server 爆了
   502 Bad Gateway        proxy 後面壞了
   503 Service Unavail    維護中 / 過載
   504 Gateway Timeout    上游超時
```

> 💡 **常見錯**：把所有錯誤都回 200 + `{ error: "xxx" }`——不對！要用正確 status code、瀏覽器 / proxy / CDN 才能正確處理。

  
### 4. 401 vs 403 — 最多人搞錯

```text
401 Unauthorized
   你「沒有身分」（沒帶 token / token 過期）
   → 該重新登入

403 Forbidden
   你「身分有了、但沒權限」
   → 一般 user 想存取 admin 頁面
```

  
### 5. Stateless（無狀態）

```text
每個 request 自帶所有需要的資訊
server 不在 session 記「你上次做了什麼」

✅ 對：JWT 放 header、server 解析後處理
❌ 錯：server 用 in-memory session（換機器就掉）

好處：
- 水平擴展容易（任何 server 都能處理）
- failover 簡單
```

  
### 6. Cacheable

```text
response 加 header：
- Cache-Control: max-age=3600        快取 1 小時
- ETag: "abc123"                      版本識別
- Last-Modified: Tue, 27 May 2026...

client / CDN 用 ETag 問：
- If-None-Match: "abc123"
- 若沒變、server 回 304 Not Modified（沒 body、省頻寬）
```

  
### 7. HATEOAS（Hypermedia as the Engine of Application State）

**REST 的最高境界、99% 人沒做**：response 帶相關連結。

```text
GET /api/posts/123

{
  "id": 123,
  "title": "...",
  "_links": {
    "self":     { "href": "/api/posts/123" },
    "author":   { "href": "/api/users/42" },
    "comments": { "href": "/api/posts/123/comments" },
    "edit":     { "href": "/api/posts/123", "method": "PUT" },
    "delete":   { "href": "/api/posts/123", "method": "DELETE" }
  }
}
```

**好處**：client 不用 hardcode URL、跟著 link 走。

**壞處**：實作複雜、業界很少做。**GitHub API 有做、看了就懂**。

  
### 8. 完整實戰：post API 設計

```text
📋 列表 + 分頁 + 篩選 + 排序
GET /api/posts?page=2&per_page=20&sort=-created_at&author=42

response：
{
  "data": [...],
  "pagination": {
    "page": 2,
    "per_page": 20,
    "total": 1234,
    "total_pages": 62
  },
  "_links": {
    "self": "/api/posts?page=2",
    "next": "/api/posts?page=3",
    "prev": "/api/posts?page=1"
  }
}
```

📄 **寫進程式檔**：`POST /api/posts`

```text
Request：
  Headers:
    Authorization: Bearer xxx
    Content-Type: application/json
  Body:
    {
      "title": "Hello",
      "body": "World"
    }

Response 201 Created：
  Headers:
    Location: /api/posts/123
  Body:
    {
      "id": 123,
      "title": "Hello",
      "body": "World",
      "created_at": "2026-05-28T..."
    }
```

  
### 9. 錯誤回應格式 — 統一格式

📄 **寫進程式檔**：所有錯誤用同一個 schema

```text
HTTP/1.1 422 Unprocessable Entity
Content-Type: application/problem+json

{
  "type": "https://example.com/errors/validation",
  "title": "Validation Failed",
  "status": 422,
  "detail": "Title is required",
  "errors": [
    { "field": "title", "code": "required", "message": "Title is required" },
    { "field": "body",  "code": "max_length", "message": "Body too long" }
  ]
}
```

**RFC 7807 Problem Details** — 業界標準錯誤格式。

  
### 10. 版本控制（簡述、L9 詳講）

```text
URL：    /api/v1/posts        /api/v2/posts
Header：Accept: application/vnd.myapi.v2+json
Query：  /api/posts?version=2
```

  
### 11. 5 大常見坑

```text
❌ 1. URL 帶動詞
   /api/getPosts、/api/createPost
   → 修：名詞 + HTTP method

❌ 2. 用 200 回所有錯誤
   { status: "error", code: 404 }
   → 修：用對的 HTTP status code

❌ 3. 401 / 403 搞混
   → 修：沒身分 401、有身分沒權限 403

❌ 4. DELETE 用 POST、PATCH 用 PUT
   → cache / proxy 行為錯
   → 修：用對 method

❌ 5. 錯誤格式各 endpoint 不一樣
   → client 處理錯誤超痛苦
   → 修：定一個全站 error schema
```

  
### 🎯 重點回顧

- **6 大限制**：Client-Server、Stateless、Cacheable、Uniform Interface、Layered、Code on Demand
- **URL 用名詞**（resource）、**動作用 HTTP verb**
- **Status code 用對的數字**——別全回 200
- **401 沒身分、403 沒權限**
- **Stateless** — JWT 自帶資訊、不靠 server session
- **錯誤格式統一**（建議 RFC 7807 Problem Details）
