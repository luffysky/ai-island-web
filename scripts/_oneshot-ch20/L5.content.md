**API Docs（API 文件）= 你的 API 怎麼用的「使用手冊」**——沒文件 = 你的 API 沒人會用。**OpenAPI / Swagger = 業界寫 docs 的標準**。

> 💡 **這課的價值**：好文件 = 客戶愛你的 API、bug report 少 80%、SDK 自動產生。Stripe / Twilio 為什麼開發者愛？因為 docs 完美。

  
### 1. 為什麼 docs 比 API 本身還重要

```text
情境：你做了一個超棒的 API、但是…

❌ 沒 docs
   → 客戶不知道怎麼呼叫
   → 沒人用、白寫

❌ 過時 docs
   → 客戶照寫、結果 error
   → 客戶罵爆 + 寫信來問

❌ 文字 docs（README.md）
   → 沒有互動、要試還得自己 curl
   → 學習曲線陡

✅ OpenAPI + 互動 UI
   → 客戶在瀏覽器內直接試
   → 自動生 SDK（10 種語言）
   → 改 schema → docs 自動更新
```

  
### 2. OpenAPI / Swagger — 業界標準

```text
歷史：
- 2010 Swagger 誕生
- 2015 SmartBear 捐給 Linux Foundation
- 改名 OpenAPI Specification（OAS）
- 現在叫 OpenAPI、Swagger 只剩工具名稱

OpenAPI 是什麼：
- 一個 YAML / JSON 檔
- 描述你 API 所有 endpoint、參數、回應、錯誤
- 機器可讀 → 自動生 docs / SDK / 測試
```

  
### 3. OpenAPI 範例

📄 **寫進程式檔**：`openapi.yaml`

```text
openapi: 3.1.0
info:
  title: AI 島 API
  version: 1.0.0
  description: 學程式平台 API

servers:
  - url: https://api.ai-island.com/v1
    description: production
  - url: https://staging-api.ai-island.com/v1
    description: staging

paths:
  /posts:
    get:
      summary: 取得 post 列表
      tags: [Posts]
      parameters:
        - name: page
          in: query
          schema:
            type: integer
            default: 1
        - name: per_page
          in: query
          schema:
            type: integer
            default: 20
      responses:
        '200':
          description: 成功
          content:
            application/json:
              schema:
                type: object
                properties:
                  data:
                    type: array
                    items:
                      $ref: '#/components/schemas/Post'

    post:
      summary: 建立 post
      tags: [Posts]
      security:
        - bearerAuth: []
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/CreatePostInput'
      responses:
        '201':
          description: 建立成功
        '401':
          description: 沒登入

components:
  schemas:
    Post:
      type: object
      required: [id, title]
      properties:
        id:
          type: string
          example: "post_abc123"
        title:
          type: string
        body:
          type: string
        createdAt:
          type: string
          format: date-time

  securitySchemes:
    bearerAuth:
      type: http
      scheme: bearer
      bearerFormat: JWT
```

  
### 4. 自動產生 docs UI

🖥️ **在終端機輸入**：

```text
# Swagger UI（最老、最常見）
docker run -p 8080:8080 -v $(pwd)/openapi.yaml:/api.yaml \
  -e SWAGGER_JSON=/api.yaml swaggerapi/swagger-ui

# Redoc（Stripe 風、更漂亮）
npx redoc-cli serve openapi.yaml

# Scalar（2024 新銳、超快）
# npm i @scalar/api-reference
```

打開瀏覽器 → 互動式 docs、可直接「Try it out」。

  
### 5. Code-first vs Spec-first

#### Code-first（從程式碼產生）

```text
寫程式時加 annotation / decorator
→ 自動產 OpenAPI

工具：
- TypeScript：NestJS / tsoa / Zodios
- Python：FastAPI（最爽）
- Go：swag

優點：✅ 不會跟程式碼脫節
缺點：⚠️ 設計階段沒檔可審
```

📄 **FastAPI 範例**：

```text
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class Post(BaseModel):
    title: str
    body: str | None = None

@app.post("/posts", response_model=Post)
def create_post(post: Post):
    return post

# 自動有：
# - /docs（Swagger UI）
# - /redoc（Redoc）
# - /openapi.json
```

#### Spec-first（先寫 spec 再實作）

```text
1. 先寫 openapi.yaml
2. 設計 review
3. 後端按 spec 實作
4. 前端按 spec mock

優點：✅ 設計穩、團隊合作好
缺點：⚠️ 容易跟實作脫節（要 CI 檢查）
```

  
### 6. 從 OpenAPI 自動產 SDK

🖥️ **在終端機輸入**：

```text
# 用 openapi-generator
openapi-generator-cli generate \
  -i openapi.yaml \
  -g typescript-axios \
  -o ./sdk-ts

# 客戶 npm install sdk-ts → 直接呼叫
```

**支援 50+ 語言**：TypeScript / Python / Go / Java / Swift / Kotlin / Rust ...

  
### 7. GraphQL 的 docs — 內建

```text
GraphQL 不需要 OpenAPI——schema 本身就是 docs。

工具：
- GraphQL Playground（內建）
- Apollo Studio（雲端）
- GraphiQL（瀏覽器內）

特色：
- 自動補全
- type hints
- 直接執行 query
- 看 schema 樹狀圖
```

  
### 8. 高品質 docs 6 大要素（Stripe 風）

```text
1. Quickstart（5 分鐘上手）
   - 註冊 / 拿 API key
   - 第一個成功 call（curl）
   - hello world

2. Authentication
   - JWT / API key / OAuth
   - 在哪拿、怎麼帶

3. Reference（每個 endpoint）
   - 描述、參數、範例 request / response
   - 錯誤代碼

4. Guides（最佳實踐）
   - 「處理 webhooks」
   - 「分頁」
   - 「rate limit」

5. SDK / Client libraries
   - 各語言安裝 + 範例

6. Changelog
   - 版本變更
   - deprecation 警告
```

  
### 9. 完整工作流

```text
1. 設計階段：寫 OpenAPI spec
2. Review：PM / 前端 / 後端一起看
3. 後端：照 spec 實作 + 加 annotation
4. CI：跑 spec vs code 一致性檢查
5. 自動發布：docs site + SDK
6. 版本變更：spec diff → 自動 changelog
```

  
### 10. Versioning — 預告 L9

```text
URL versioning：       /v1/posts、/v2/posts
Header versioning：    Accept: application/vnd.api.v2+json
Query versioning：     /posts?v=2

Deprecation：
- Header：Sunset: Sat, 31 Dec 2026
- 警告：Deprecation: true
- docs 標紅
```

  
### 11. 5 大常見坑

```text
❌ 1. docs 寫在 README、沒互動
   → 沒人想看
   → 修：用 Swagger UI / Redoc / Scalar

❌ 2. spec 跟 code 脫節
   → docs 寫 a 欄位、code 回 b 欄位
   → 修：CI 跑一致性檢查、或 code-first

❌ 3. 沒範例（每個 endpoint 只列參數）
   → 客戶不知道實際長什麼樣
   → 修：每個都加完整 request / response 範例

❌ 4. 錯誤代碼沒列
   → 客戶不知道 422 跟 400 差在哪
   → 修：每個 endpoint 列所有可能 status code

❌ 5. 沒 changelog
   → 客戶不知道版本變更、生產 break
   → 修：每次 release 寫 changelog + 標 breaking
```

  
### 🎯 重點回顧

- **OpenAPI（前 Swagger）** 是 REST docs 業界標準
- **Code-first** 跟程式不脫節、**Spec-first** 設計階段好討論
- **自動生 SDK**（openapi-generator）、50+ 語言
- **GraphQL 用 GraphiQL / Apollo Studio**——schema 本身就是 docs
- **高品質 docs 6 要素**：Quickstart、Auth、Reference、Guides、SDK、Changelog
- **參考 Stripe / Twilio docs**——業界天花板
