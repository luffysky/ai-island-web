**SQL Injection（SQL 注入）= 最古老、最常見、最致命的網站攻擊**—— 一個 `' OR 1=1 --` 能讓駭客清空你的整個 DB。本課教你「攻擊原理」+「prepared statement 為什麼是唯一正解」。

> 💡 **這課的價值**：SQL Injection 從 1998 年到 2026 年還是 OWASP Top 10 前三名——因為「字串拼 SQL」太自然了。本課把這個坑根除、你的後端就少一個破口。


### 1. SQL Injection 怎麼發生？

**反例：字串拼 SQL（地獄）**

```text
// JavaScript 偽 code
const sql = "SELECT * FROM users WHERE email = '" + email + "'";
db.query(sql);
```

**正常使用者：**

```text
email = 'luffy@ai.com'
最終 SQL：
   SELECT * FROM users WHERE email = 'luffy@ai.com';
-- 正常運作
```

**駭客輸入：**

```text
email = "' OR 1=1 --"
最終 SQL：
   SELECT * FROM users WHERE email = '' OR 1=1 --';
-- 1=1 永遠 true、回傳全部 user！登入繞過
```


### 2. 更嚴重的攻擊

**清空整張表：**

```text
email = "'; DROP TABLE users; --"
最終 SQL：
   SELECT * FROM users WHERE email = ''; DROP TABLE users; --';
-- 兩句 SQL 一起跑、users 表被刪
```

**盜資料（UNION 攻擊）：**

```text
search = "' UNION SELECT password_hash, email FROM users --"
最終 SQL：
   SELECT name, bio FROM posts WHERE keyword = ''
   UNION SELECT password_hash, email FROM users --';
-- 把 user 密碼 hash 全撈出來
```

**Blind SQL Injection（盲注、無回顯）：**

駭客連錯誤訊息都看不到、用「等 1 秒回應」當訊號逐字猜密碼——比較慢但能做。


### 3. 為什麼 escape 不夠安全？

**錯誤想法**：「我把單引號 escape 掉就好了」。

```text
function escape(s) {
  return s.replace(/'/g, "''");
}

const sql = "SELECT * FROM users WHERE email = '" + escape(email) + "'";
```

**漏洞：**

```text
1. 漏 escape 一個地方就完蛋
2. 不同 DB escape 規則不同（雙引號、反斜線、空字元、Unicode）
3. 第二階 injection（資料存進去再讀出來）
4. integer 欄位的 injection（` WHERE id = 1 OR 1=1`、根本沒引號可 escape）
```

**結論**：**手動 escape = 治標不治本、永遠有漏網之魚**。


### 4. 唯一正解：Prepared Statement（預備陳述句）

```text
// 用 parameter placeholder（佔位符）、不拼 string
const sql = "SELECT * FROM users WHERE email = $1";
db.query(sql, [email]);
```

**運作方式：**

```text
1. SQL 模板先送 DB、PG 解析 + 計畫好「SELECT * FROM users WHERE email = ?」
2. 參數值「luffy@ai.com」單獨送
3. PG 把參數當「值」插入、絕對不會當「SQL 語法」解析

→ 即使輸入 "' OR 1=1 --"、整串會當成 email 值
→ PG 去找 email = "' OR 1=1 --" 的列、當然找不到
→ 攻擊失敗
```


### 5. Prepared Statement 的 placeholder 語法

```text
PostgreSQL（用 $1, $2, $3...）
   SELECT * FROM users WHERE email = $1 AND age > $2

MySQL（用 ?）
   SELECT * FROM users WHERE email = ? AND age > ?

JDBC / Python（用 %s 或 ?）
   cursor.execute("SELECT * FROM users WHERE email = %s", (email,))
```


### 6. 各語言怎麼寫

**Node.js（pg）：**

```text
import { Pool } from 'pg';
const pool = new Pool();

// ✅ 安全
const result = await pool.query(
  'SELECT * FROM users WHERE email = $1',
  [email]
);

// ❌ 危險（字串拼接）
const result = await pool.query(
  `SELECT * FROM users WHERE email = '${email}'`
);
```

**Python（psycopg）：**

```text
import psycopg

conn = psycopg.connect(...)
cur = conn.cursor()

# ✅ 安全
cur.execute("SELECT * FROM users WHERE email = %s", (email,))

# ❌ 危險
cur.execute(f"SELECT * FROM users WHERE email = '{email}'")
```

**Go（database/sql）：**

```text
// ✅ 安全
rows, _ := db.Query("SELECT * FROM users WHERE email = $1", email)

// ❌ 危險
rows, _ := db.Query(fmt.Sprintf("SELECT * FROM users WHERE email = '%s'", email))
```


### 7. ORM 自動防護

**現代 ORM（Prisma、Drizzle、TypeORM、Eloquent...）預設用 prepared statement**：

```text
// Prisma
const user = await prisma.user.findUnique({
  where: { email: email }
});
// ✅ Prisma 內部用 prepared statement

// Drizzle
const user = await db.select().from(users).where(eq(users.email, email));
// ✅ 安全

// Supabase JS SDK
const { data } = await supabase.from('users').select().eq('email', email);
// ✅ 安全
```

**用 ORM 等於免費拿到 SQL Injection 防護**——別自己拼 string。


### 8. ORM 的「raw SQL」陷阱

```text
// Prisma 的 $queryRawUnsafe ⚠️ 危險
await prisma.$queryRawUnsafe(`SELECT * FROM users WHERE email = '${email}'`);

// 正確：用 $queryRaw 配 ${} 參數
await prisma.$queryRaw`SELECT * FROM users WHERE email = ${email}`;
// Prisma 把 ${email} 當參數、不是字串拼接
```

**規則**：用 raw SQL 時、永遠用 ORM 的「parameter 版」、不要 unsafe 版。


### 9. 動態 schema / table 名怎麼辦？

**Prepared statement 只能 parameterize 值、不能 parameterize 表名 / 欄位名**：

```text
-- ❌ 這寫不出來
SELECT * FROM $1 WHERE id = $2  -- 第一個 $1 不能是表名

-- 替代方法：白名單檢查
const allowedTables = ['users', 'posts', 'orders'];
if (!allowedTables.includes(tableName)) {
  throw new Error('Invalid table');
}
const sql = `SELECT * FROM ${tableName} WHERE id = $1`;
db.query(sql, [id]);
-- tableName 來自白名單、安全
```


### 10. 其他防護層

```text
✅ Principle of least privilege（最小權限）
   應用層用的 DB user 只給必要權限
   不要用 superuser
   寫入用一個帳號、唯讀用另一個

✅ Input validation（輸入驗證）
   email 用 regex 驗、age 限數字、長度限制
   即使 SQL 安全、髒資料也擋一層

✅ Web Application Firewall（WAF）
   Cloudflare WAF 等服務偵測 injection 特徵
   多一層保險

✅ Log & monitor
   異常 query 立刻警示
   PG 開 log_statement = 'all' 留紀錄

✅ Code review + 自動掃描
   GitHub CodeQL、Semgrep 掃出潛在 injection
```


### 11. 完整實戰：登入 API

📄 **寫進程式檔**：`api/login.ts`

```text
import { db } from '@/lib/db';
import { compare } from 'bcrypt';

export async function POST(req: Request) {
  const { email, password } = await req.json();

  // 1. 輸入驗證
  if (!email || !password || typeof email !== 'string' || typeof password !== 'string') {
    return Response.json({ error: 'Invalid input' }, { status: 400 });
  }
  if (email.length > 100 || password.length > 100) {
    return Response.json({ error: 'Too long' }, { status: 400 });
  }

  // 2. Prepared statement 查 user
  const result = await db.query(
    'SELECT id, email, password_hash FROM users WHERE email = $1 LIMIT 1',
    [email]
  );

  if (result.rows.length === 0) {
    return Response.json({ error: 'Invalid credentials' }, { status: 401 });
  }

  const user = result.rows[0];

  // 3. 密碼比對
  const ok = await compare(password, user.password_hash);
  if (!ok) {
    return Response.json({ error: 'Invalid credentials' }, { status: 401 });
  }

  // 4. 發 token...
  return Response.json({ id: user.id, email: user.email });
}
```

**這個 API 做對 5 件事：**

1. Prepared statement（防 SQL Injection）
2. 輸入驗證（型別 / 長度）
3. 失敗訊息一致（'Invalid credentials'、不洩漏「email 是否存在」）
4. 密碼用 bcrypt 雜湊比對（防密碼直接存）
5. LIMIT 1（即使資料有問題、不會回多筆）


### 12. 測試你的應用有沒有 injection 漏洞

🖥️ **在登入欄位輸入這些測試**：

```text
'                           ← 單引號（看會不會 500 錯）
' OR 1=1 --                 ← 經典繞過
'; DROP TABLE users; --     ← 致命攻擊
admin' --                   ← 註解後面
"                           ← 雙引號測試
\                           ← 反斜線
1 OR 1=1                    ← 數字 OR
```

**如果有任何一個能：**
- 直接登入成功
- 回傳全部 user
- 500 錯誤露出 SQL 訊息

**→ 你的應用就有 injection 漏洞、立刻修**。

**自動掃描工具**：sqlmap、Burp Suite。


### 13. 5 大常見坑

```text
❌ 1. 動態拼 SQL 字串
   → 100% 中 injection
   → 修：永遠用 prepared statement

❌ 2. 用 ORM 但又 raw SQL 拼字串
   → ORM 防護被繞過
   → 修：raw SQL 也用參數 placeholder

❌ 3. 表名 / 欄位名直接從 input 來
   → prepared statement 救不了
   → 修：白名單 + 程式內檢查

❌ 4. 錯誤訊息把 SQL 漏出去
   → 駭客知道你用什麼 DB、什麼 schema、攻擊更準
   → 修：Production 統一回「Internal error」、log 留內部

❌ 5. DB 用 superuser 連
   → 一旦被 injection、整庫 DROP
   → 修：應用層用「最小權限」帳號、不能 DROP TABLE
```


### 🎯 重點回顧

- **SQL Injection 從 1998 年到 2026 年還在發生**——因為「字串拼 SQL」太自然
- **絕對不要字串拼 SQL**——任何 `"... WHERE x = '" + var + "'"` 都是炸彈
- **Prepared statement** 是唯一正解、所有現代 DB driver 都支援
- **PG 用 `$1, $2`、MySQL 用 `?`** 當 placeholder
- **ORM 預設用 prepared statement**——選好 ORM 就有自動防護
- **表名 / 欄位名** 不能 parameterize——用白名單檢查
- **最小權限原則 + WAF + log** 多層防護
