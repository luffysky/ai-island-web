**JOIN 進階三招：CROSS JOIN / SELF JOIN / LATERAL JOIN**—— 一般場景用 INNER + LEFT 就 95% 了、但這 3 招遇到特殊問題就是救命招。

> 💡 **這課的價值**：這 3 招業界初學者多半不知道、實務遇到時又非用不可——「產生時間序列」、「員工跟主管同表」、「每個 user 取最新 3 筆訂單」。學會直接超越 80% 工程師。


### 1. CROSS JOIN —— 笛卡兒積（每列配每列）

**3 個 user × 2 個 product = 6 種組合：**

```text
SELECT u.name, p.name
FROM users u
CROSS JOIN products p;
```

💬 **電腦會顯示**（如果 3 users × 2 products）：

```text
   u.name |  p.name
----------+----------
   Luffy  | iPhone
   Luffy  | iPad
   Xueyue | iPhone
   Xueyue | iPad
   Nami   | iPhone
   Nami   | iPad
（6 列）
```

**CROSS JOIN 沒有 ON**——它就是「左表每列 × 右表每列」。

**何時用？**

```text
✅ 產生報表表格（每個城市 × 每個月 = 完整矩陣、即使某些組合 0 單）
✅ 生成測試資料（10 users × 100 products = 1000 筆假訂單）
✅ 跟單列 subquery 結合（讓常數加入每列）

❌ 大表 CROSS JOIN = 災難（10 萬 × 10 萬 = 100 億列）
```


### 2. CROSS JOIN 應用：填補「沒資料的月份」

**問題**：報表要顯示 2026 年 1-12 月每個月的營收、即使某個月 0 單也要顯示「0」。

**單純 GROUP BY 做不到**——沒訂單的月份不會出現。

**用 CROSS JOIN 填補：**

```text
WITH months AS (
  SELECT generate_series(
    '2026-01-01'::DATE,
    '2026-12-01'::DATE,
    '1 month'
  )::DATE AS month
),
cities AS (
  SELECT DISTINCT city FROM users
)
SELECT
  c.city,
  m.month,
  COALESCE(SUM(o.amount), 0) AS revenue
FROM cities c
CROSS JOIN months m
LEFT JOIN orders o
  ON o.city = c.city
  AND DATE_TRUNC('month', o.created_at) = m.month
GROUP BY c.city, m.month
ORDER BY c.city, m.month;
```

**做了 4 件事：**

1. `generate_series` 產生 12 個月
2. `cities` 抓所有不同城市
3. `CROSS JOIN` 兩個 → 「3 城市 × 12 月 = 36 列」完整矩陣
4. `LEFT JOIN orders` 把實際訂單套上去、沒訂單補 0


### 3. SELF JOIN —— 同一張表 JOIN 自己

**最經典例子**：員工表 + boss 名字（L14 提過）。

```text
CREATE TABLE employees (
  id      INT PRIMARY KEY,
  name    TEXT,
  boss_id INT REFERENCES employees(id)
);

INSERT INTO employees VALUES
  (1, 'Luffy', NULL),
  (2, 'Zoro', 1),
  (3, 'Nami', 1),
  (4, 'Sanji', 2);

SELECT
  e.name   AS employee,
  b.name   AS boss
FROM employees e
LEFT JOIN employees b ON e.boss_id = b.id;
```

💬 **電腦會顯示**：

```text
 employee |  boss
----------+--------
 Luffy    | NULL
 Zoro     | Luffy
 Nami     | Luffy
 Sanji    | Zoro
```

**關鍵**：**用別名區分**（`e` 是員工、`b` 是 boss）、SQL 才能分辨兩個 employees 是哪個。


### 4. SELF JOIN 應用：找出「同城市的所有 user 配對」

```text
SELECT
  a.name AS user_a,
  b.name AS user_b,
  a.city
FROM users a
JOIN users b ON a.city = b.city AND a.id < b.id
ORDER BY a.city;
```

`a.id < b.id` 避免「Luffy + Xueyue」跟「Xueyue + Luffy」重複出現。

💬 **電腦會顯示**：

```text
 user_a | user_b | city
--------+--------+------
 Luffy  | Xueyue | 台北
 Luffy  | Sanji  | 台北
 Xueyue | Sanji  | 台北
 Nami   | Robin  | 台中
```


### 5. SELF JOIN 應用：留言巢狀結構

```text
CREATE TABLE comments (
  id        SERIAL PRIMARY KEY,
  content   TEXT,
  parent_id INT REFERENCES comments(id)  -- 回覆哪個留言
);

-- 找出每個回覆 + 它回覆的原文
SELECT
  c.content AS reply,
  p.content AS replied_to
FROM comments c
LEFT JOIN comments p ON c.parent_id = p.id
WHERE c.parent_id IS NOT NULL;
```


### 6. LATERAL JOIN —— 對每列做子查詢

**LATERAL（橫向 JOIN）** = 右側 subquery 可以「引用左側列的值」。

**問題**：每個 user 取「最新 3 筆訂單」。

**普通 JOIN 做不到**——subquery 一次只能跑一遍、無法「對每個 user 各跑一次」。

**LATERAL 救援：**

```text
SELECT u.name, o.id AS order_id, o.amount, o.created_at
FROM users u
JOIN LATERAL (
  SELECT id, amount, created_at
  FROM orders
  WHERE user_id = u.id    -- 引用左側的 u.id
  ORDER BY created_at DESC
  LIMIT 3
) o ON TRUE;
```

**讀法**：對每個 user、跑一次「該 user 的最新 3 筆訂單」、結果 JOIN 回來。

**像「for-loop subquery」**——對左表每一列、跑右邊 subquery 一次。


### 7. LATERAL 應用：top-N per group 經典題

```text
-- 每個城市銷量前 3 名的商品
SELECT
  c.city,
  top.name AS product,
  top.total_sold
FROM (SELECT DISTINCT city FROM users) c
JOIN LATERAL (
  SELECT p.name, COUNT(oi.product_id) AS total_sold
  FROM users u
  JOIN orders o ON u.id = o.user_id
  JOIN order_items oi ON o.id = oi.order_id
  JOIN products p ON oi.product_id = p.id
  WHERE u.city = c.city    -- 引用 c.city
  GROUP BY p.id, p.name
  ORDER BY total_sold DESC
  LIMIT 3
) top ON TRUE
ORDER BY c.city, top.total_sold DESC;
```

**這種題目用 Window Function 也能做（L9）**——LATERAL 是另一種寫法、有時更直覺。


### 8. LATERAL vs 普通 JOIN 差別

```text
普通 JOIN：右側 subquery 跟左表無關、跑一次得到結果集
LATERAL JOIN：右側 subquery 對左表每列跑一次、能用左列的值
```

**簡單測試**：

```text
-- 普通 JOIN：subquery 內不能用 u.id ❌
SELECT * FROM users u
JOIN (SELECT * FROM orders WHERE user_id = u.id) o ON TRUE;
-- 報錯：u 在子查詢內不可見

-- LATERAL JOIN：可以！ ✅
SELECT * FROM users u
JOIN LATERAL (SELECT * FROM orders WHERE user_id = u.id) o ON TRUE;
```


### 9. CROSS JOIN LATERAL —— LATERAL 的 INNER 版

```text
SELECT u.name, o.amount
FROM users u
CROSS JOIN LATERAL (
  SELECT amount FROM orders
  WHERE user_id = u.id
  ORDER BY created_at DESC
  LIMIT 1
) o;
```

**`CROSS JOIN LATERAL`** = subquery 沒結果 → 該列就消失（INNER 行為）。

**`LEFT JOIN LATERAL ... ON TRUE`** = subquery 沒結果 → 補 NULL（LEFT 行為）。


### 10. JOIN 進階綜合題

**「找出每個 user 的最新訂單 + 那筆訂單的所有商品」：**

```text
SELECT
  u.name,
  latest.order_id,
  latest.amount,
  oi.product_id,
  oi.qty
FROM users u
LEFT JOIN LATERAL (
  SELECT id AS order_id, amount, created_at
  FROM orders
  WHERE user_id = u.id
  ORDER BY created_at DESC
  LIMIT 1
) latest ON TRUE
LEFT JOIN order_items oi ON oi.order_id = latest.order_id;
```


### 11. 5 大常見坑

```text
❌ 1. CROSS JOIN 用在大表
   → 10 萬 × 10 萬 = 100 億列、爆炸
   → 修：確認左右表都不大、或先 WHERE 縮小

❌ 2. SELF JOIN 沒用別名
   → SQL 不知道 employees.id 是哪個 employees 的
   → 修：永遠 employees e, employees b 用別名

❌ 3. LATERAL 把它寫成普通 JOIN
   → 報錯「u.id 在 subquery 不可見」
   → 修：JOIN LATERAL（不是 JOIN）

❌ 4. 用 LATERAL 取 top-N 但效能差
   → 每列跑一次、總跑量大
   → 修：考慮 Window Function ROW_NUMBER 改寫

❌ 5. CROSS JOIN LATERAL 期望保留沒對到的左列
   → CROSS 是 INNER 行為、沒結果就消失
   → 修：要保留左列、用 LEFT JOIN LATERAL ... ON TRUE
```


### 🎯 重點回顧

- **CROSS JOIN** = 笛卡兒積、每列配每列、適合「生成完整矩陣」（填補空白月份）
- **SELF JOIN** = 同表 JOIN 自己、用別名區分、適合「員工 + boss」、「留言巢狀」
- **LATERAL JOIN** = subquery 可引用左列、適合「每組 top-N」、「對每列做 subquery」
- **大表慎用 CROSS / LATERAL**——效能爆炸
- **能用 Window Function 改寫的、優先 Window**
