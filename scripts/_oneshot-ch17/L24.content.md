**JSON / JSONB（PostgreSQL 的半結構化神器）**—— 結構彈性又能查詢、解決「欄位太多 / 動態 schema」痛點。**JSONB 比 JSON 強 100 倍**——本課把該知道的全包。

> 💡 **這課的價值**：PG 的 JSONB 把「SQL 嚴格」+「NoSQL 彈性」合一。產品設定、AI 模型參數、metadata——全部塞 JSONB、又能查、又能 Index、Mongo 直接被打。


### 1. JSON vs JSONB 差別（決定一切）

```text
JSON
   原始字串、保留空白順序
   ✅ 寫入快（不解析）
   ❌ 查詢慢（每次都要重新 parse）
   ❌ 不能建 GIN Index
   ❌ 重複 key 保留

JSONB
   解析後的二進位
   ✅ 查詢快（已解析）
   ✅ 可建 GIN Index
   ✅ 自動去重複 key
   ❌ 寫入比 JSON 稍慢（要 parse）
   ❌ 不保留空白順序
```

**業界共識：99% 用 JSONB、不要用 JSON**——除非真的要保留原始格式（很少需要）。


### 2. 建表 + 寫入 JSONB

```text
CREATE TABLE products (
  id     SERIAL PRIMARY KEY,
  name   TEXT,
  meta   JSONB
);

INSERT INTO products (name, meta) VALUES
  ('iPhone', '{"color": "black", "storage": "256GB", "tags": ["smartphone", "apple"]}'),
  ('iPad',   '{"color": "silver", "storage": "512GB", "tags": ["tablet", "apple"]}'),
  ('Pixel',  '{"color": "white",  "storage": "128GB", "tags": ["smartphone", "google"]}');
```


### 3. JSONB 4 大操作子

```text
->     取 JSON 值（回傳還是 JSONB）
->>    取文字值（回傳 TEXT）
#>     按路徑取 JSON（回傳 JSONB）
#>>    按路徑取文字（回傳 TEXT）
```

**範例：**

```text
SELECT
  name,
  meta -> 'color'         AS color_jsonb,    -- 回 JSONB（含引號）
  meta ->> 'color'        AS color_text,     -- 回 TEXT（純值）
  meta -> 'tags'          AS tags,
  meta -> 'tags' ->> 0    AS first_tag,      -- 陣列取第一個（TEXT）
  meta -> 'tags' -> 0     AS first_tag_jsonb,
  meta #> '{tags, 0}'     AS first_tag_path,
  meta #>> '{tags, 0}'    AS first_tag_path_text
FROM products;
```

💬 **電腦會顯示**：

```text
  name   | color_jsonb | color_text | ... | first_tag
---------+-------------+------------+-----+-------------
 iPhone  | "black"     | black      | ... | smartphone
 iPad    | "silver"    | silver     | ... | tablet
 Pixel   | "white"     | white      | ... | smartphone
```


### 4. WHERE 條件查 JSONB

```text
-- 找黑色商品
SELECT name FROM products WHERE meta ->> 'color' = 'black';

-- 找 storage 大於 256GB（注意要轉型）
SELECT name FROM products
WHERE (meta ->> 'storage')::TEXT > '256GB';

-- 找 tags 包含 'apple'
SELECT name FROM products WHERE meta -> 'tags' ? 'apple';

-- @> 包含運算子：找 meta 包含 {"color": "black"} 的
SELECT name FROM products WHERE meta @> '{"color": "black"}';

-- @> 多條件：
SELECT name FROM products WHERE meta @> '{"color": "white", "storage": "128GB"}';
```


### 5. JSONB 運算子大全

```text
->        取值（JSONB）
->>       取值（TEXT）
#>        按路徑取（JSONB）
#>>       按路徑取（TEXT）

?         key 存在？        WHERE meta ? 'color'
?|        任一 key 存在？   WHERE meta ?| array['color', 'size']
?&        全部 key 存在？   WHERE meta ?& array['color', 'size']

@>        包含？           WHERE meta @> '{"color": "black"}'
<@        被包含？

||        合併             '{"a":1}'::JSONB || '{"b":2}'::JSONB → {"a":1,"b":2}
-         刪 key           '{"a":1,"b":2}'::JSONB - 'a' → {"b":2}
#-        按路徑刪
```


### 6. UPDATE JSONB 欄位

**整個替換：**

```text
UPDATE products
SET meta = '{"color": "red", "storage": "1TB"}'
WHERE id = 1;
```

**只改一個 key（用 || 合併）：**

```text
UPDATE products
SET meta = meta || '{"color": "red"}'::JSONB
WHERE id = 1;
-- 其他 key 保留、color 變紅
```

**改深層的值（jsonb_set）：**

```text
UPDATE products
SET meta = jsonb_set(meta, '{specs, ram}', '"32GB"', true)
WHERE id = 1;
-- {specs: {ram: '32GB', ...}} 內部改
-- true = 不存在就建立
```

**刪 key：**

```text
UPDATE products SET meta = meta - 'color' WHERE id = 1;
```


### 7. JSONB Index（GIN）

```text
-- 對整個 meta 建 GIN Index
CREATE INDEX idx_products_meta ON products USING GIN (meta);

-- 之後這些 query 都用得到 Index：
WHERE meta @> '{"color": "black"}'
WHERE meta ? 'color'
WHERE meta ?| array['size', 'color']
```

**對特定 path 建 Index（更小、更快）：**

```text
-- 對 (meta ->> 'color') 建 B-tree Index
CREATE INDEX idx_products_color ON products ((meta ->> 'color'));

-- 對 meta -> 'tags' 建 GIN Index（陣列）
CREATE INDEX idx_products_tags ON products USING GIN ((meta -> 'tags'));
```


### 8. JSONB 內建函式

```text
jsonb_array_elements(jsonb)   把陣列展成多列
jsonb_each(jsonb)              把 object 展成 key/value 多列
jsonb_object_keys(jsonb)       取所有 keys
jsonb_typeof(jsonb)            看型別（'object'、'array'、'string'...）
jsonb_pretty(jsonb)            漂亮印
to_jsonb(any)                  轉成 JSONB
row_to_json(record)            列轉 JSON
```

**範例：把 tags 陣列展成多列：**

```text
SELECT name, jsonb_array_elements_text(meta -> 'tags') AS tag
FROM products;
```

💬 **電腦會顯示**：

```text
  name   |    tag
---------+-------------
 iPhone  | smartphone
 iPhone  | apple
 iPad    | tablet
 iPad    | apple
 Pixel   | smartphone
 Pixel   | google
```


### 9. 何時用 JSONB、何時用獨立欄位

```text
✅ 用 JSONB 的場景
   - schema 動態（每個產品 metadata 不同）
   - 欄位太多（50+ 欄位、不想全建出來）
   - 第三方 API 回傳塞著
   - tags / 設定值 這種陣列
   - AI 模型參數（隨模型變）

❌ 不用 JSONB 的場景
   - 常 WHERE / JOIN 的核心欄位（建獨立欄、加 Index 更好）
   - 數字欄位要 SUM / AVG（轉型麻煩）
   - 強 schema 約束（FK、UNIQUE）
```

**業界共識**：**核心欄位獨立、metadata / settings 用 JSONB**——兩者搭配最好。


### 10. JSONB vs MongoDB

```text
MongoDB
   ✅ 自然 schema-less
   ✅ 大 JSON 處理快
   ❌ 沒 JOIN
   ❌ 沒 transaction（4.0 後有但限制多）

PG JSONB
   ✅ 完整 SQL + JOIN
   ✅ 完整 transaction
   ✅ JSON 操作豐富
   ❌ 純 JSON 場景效能略輸 Mongo
```

**2026 業界趨勢**：很多 Mongo 用戶換 PG + JSONB——大部分需求都涵蓋、又能 JOIN + transaction、減少一個中間層。


### 11. 完整實戰：商品 settings

```text
CREATE TABLE products (
  id        SERIAL PRIMARY KEY,
  name      TEXT NOT NULL,
  price     NUMERIC(10, 2) NOT NULL,          -- 核心欄位
  category  TEXT NOT NULL,                     -- 核心欄位
  settings  JSONB DEFAULT '{}'::JSONB          -- 彈性 metadata
);

CREATE INDEX idx_products_settings ON products USING GIN (settings);

INSERT INTO products (name, price, category, settings) VALUES
  ('iPhone 17', 35000, 'phone', '{
    "color_options": ["black", "white", "blue"],
    "storage_options": ["128GB", "256GB", "512GB"],
    "warranty_months": 12,
    "is_5g": true,
    "ai_features": {"siri": true, "image_gen": true}
  }');

-- 查支援 5G 的商品
SELECT name FROM products WHERE settings @> '{"is_5g": true}';

-- 查保固超過 12 個月
SELECT name FROM products
WHERE (settings ->> 'warranty_months')::INT > 12;

-- 查支援 AI 圖像生成
SELECT name FROM products WHERE settings #> '{ai_features, image_gen}' = 'true'::JSONB;
```


### 12. 5 大常見坑

```text
❌ 1. 把所有欄位塞 JSONB
   → 失去 SQL 強型別優勢、效能下降
   → 修：核心欄位獨立、metadata 才 JSONB

❌ 2. 用 JSON 不用 JSONB
   → 查詢慢、不能 Index
   → 修：永遠用 JSONB（除非有特殊原因）

❌ 3. ->> 取文字、忘了轉型計算
   → meta ->> 'age' > 18 ← 字串比較、'9' > '18' 為真
   → 修：(meta ->> 'age')::INT > 18

❌ 4. 沒建 GIN Index 就大量 @> 查
   → 全表掃、慢
   → 修：CREATE INDEX ... USING GIN

❌ 5. 更新 JSONB 用 = 整個替換
   → 別的 key 全沒
   → 修：用 meta || '{...}' 合併、或 jsonb_set 精準改
```


### 🎯 重點回顧

- **JSONB > JSON**——99% 用 JSONB、查得快、可 Index
- **`->`（JSONB）vs `->>`（TEXT）** 取值兩種、`->>` 配條件最常用
- **`@>` 包含** 是高頻運算子、配 GIN Index 飛快
- **核心欄位獨立、metadata 用 JSONB** 是黃金分工
- **GIN Index** 對 JSONB 大殺四方
- **`jsonb_array_elements`** 展開陣列、`jsonb_set` 改深層值
