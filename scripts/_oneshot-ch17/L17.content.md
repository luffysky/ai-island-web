**UNION / INTERSECT / EXCEPT = SQL 三大集合運算**—— JOIN 是「左右拼」、集合運算是「上下拼」。寫「把多張類似結構的表合一個結果」靠它。

> 💡 **這課的價值**：報表 SQL 常常要「A 公司訂單 + B 公司訂單 合在一起」、「VIP - 已退會 = 真正的活躍 VIP」——這時不是 JOIN、是集合運算。學會了寫起來非常乾淨。


### 1. UNION —— 把兩個 SELECT 結果「上下拼」

```text
SELECT name, email FROM users_jp
UNION
SELECT name, email FROM users_us;
-- 日本 user + 美國 user、合起來
```

**3 大規則：**

```text
✅ 兩邊欄位數量要一樣
✅ 對應欄位型別要相容（int / text 不能拼）
✅ 結果的欄位名照「第一個 SELECT」

❌ SELECT name, email UNION SELECT id, name  ← 欄位錯位、報錯
```


### 2. UNION vs UNION ALL（最大差別、效能差很多）

```text
UNION
   會「去重」、自動 DISTINCT
   慢（PG 要排序找重複）

UNION ALL
   不去重、直接拼
   快、效能差 10 倍以上
```

**例：**

```text
-- 假設 users_jp 有 (Luffy)、users_us 也有 (Luffy)
SELECT name FROM users_jp UNION SELECT name FROM users_us;
-- 結果：Luffy（1 筆、去重了）

SELECT name FROM users_jp UNION ALL SELECT name FROM users_us;
-- 結果：Luffy, Luffy（2 筆、沒去重）
```

> 💡 **業界共識**：**用 UNION ALL、除非真的要去重**。多數場景兩邊沒重複（不同 region 的 user）、UNION ALL 又快又對。


### 3. UNION 應用：分表的合併

**情境**：log 表按月分表（logs_2026_01、logs_2026_02...）、查詢時要合併。

```text
SELECT id, msg, created_at FROM logs_2026_01
UNION ALL
SELECT id, msg, created_at FROM logs_2026_02
UNION ALL
SELECT id, msg, created_at FROM logs_2026_03
ORDER BY created_at DESC
LIMIT 100;
```

**注意**：ORDER BY 跟 LIMIT 寫在最後、套用到整個合併結果。


### 4. UNION 應用：把不同 schema 的相似資料統一

```text
-- 統一三個來源的 user 清單
SELECT id, name, 'staff' AS source FROM staff
UNION ALL
SELECT id, name, 'customer' AS source FROM customers
UNION ALL
SELECT id, name, 'vendor' AS source FROM vendors
ORDER BY name;
```

**多加一欄 'staff' / 'customer' / 'vendor'**——分得清是從哪來的。


### 5. INTERSECT —— 兩個 SELECT 的「交集」

```text
SELECT email FROM newsletter_subscribers
INTERSECT
SELECT email FROM paid_users;
-- 同時是「電子報訂閱者」+「付費 user」的 email
```

**INTERSECT** = 兩邊都有的、會去重。

```text
INTERSECT ALL
   保留重複（A 有 3 次、B 有 5 次 → 交集 3 次）
```

**MySQL 沒有 INTERSECT**（要用 JOIN 改寫）；PG、SQL Server 都有。


### 6. EXCEPT —— A 有、B 沒有（差集）

```text
SELECT email FROM all_users
EXCEPT
SELECT email FROM unsubscribed;
-- 所有 user 但減去已退訂的、剩活躍 user
```

**EXCEPT** = A 集合 - B 集合、自動去重。

```text
EXCEPT ALL
   保留重複的差異（A 有 3 次、B 有 1 次 → 留 2 次）
```

**MySQL 沒有 EXCEPT**（用 LEFT JOIN + IS NULL 改寫）。


### 7. 三大集合運算的順序

當有多個運算時、**INTERSECT 優先於 UNION / EXCEPT**：

```text
SELECT a FROM t1
UNION
SELECT b FROM t2
INTERSECT
SELECT c FROM t3;

-- 實際執行：
SELECT a FROM t1
UNION (
  SELECT b FROM t2
  INTERSECT
  SELECT c FROM t3
);
```

**永遠加括號**——別賴記憶。


### 8. UNION 配 ORDER BY / LIMIT

```text
(SELECT name FROM users_jp ORDER BY name LIMIT 5)
UNION ALL
(SELECT name FROM users_us ORDER BY name LIMIT 5);
-- 各取 5 筆、再合併
```

**最終 ORDER BY 套整體：**

```text
SELECT name FROM users_jp
UNION ALL
SELECT name FROM users_us
ORDER BY name
LIMIT 10;
-- ORDER BY 與 LIMIT 在最後、整體結果排序取 10
```


### 9. 集合運算 vs JOIN

```text
JOIN
   橫向拼（左右）
   一列拼一列、欄位變多
   例：user + 訂單 + 商品

集合運算
   縱向拼（上下）
   列變多、欄位不變
   例：日本 user + 美國 user
```

**錯用對方**：

- 想合併兩張表（同 schema）卻用 JOIN → 結果欄位重複
- 想關聯兩張表卻用 UNION → 報錯（欄位數不一樣）


### 10. UNION 效能技巧

**陷阱**：

```text
SELECT * FROM huge_table_a
UNION
SELECT * FROM huge_table_b;
-- UNION 要找重複、合 1 千萬筆 → 排序爆 disk
```

**改善：**

```text
-- 1. 不需要去重就 UNION ALL
SELECT * FROM huge_table_a
UNION ALL
SELECT * FROM huge_table_b;

-- 2. 先各自過濾再合
SELECT * FROM huge_table_a WHERE created_at > NOW() - INTERVAL '7 days'
UNION ALL
SELECT * FROM huge_table_b WHERE created_at > NOW() - INTERVAL '7 days';
```


### 11. EXCEPT 改寫成 LEFT JOIN

如果用 MySQL（沒 EXCEPT）、或想用 Index 加速、改成 LEFT JOIN：

```text
-- EXCEPT 寫法（PG）
SELECT email FROM all_users
EXCEPT
SELECT email FROM unsubscribed;

-- LEFT JOIN 改寫（任何 DB 都能用）
SELECT a.email
FROM all_users a
LEFT JOIN unsubscribed u ON a.email = u.email
WHERE u.email IS NULL;
```


### 12. 完整實戰：多來源報表

```text
-- 訂單系統有兩個版本（v1、v2）、做合併報表
SELECT
  'v1' AS source,
  id,
  amount,
  created_at
FROM orders_v1
WHERE created_at >= '2026-01-01'

UNION ALL

SELECT
  'v2' AS source,
  id,
  total_amount AS amount,    -- 欄位名不同、用別名對齊
  ordered_at AS created_at
FROM orders_v2
WHERE ordered_at >= '2026-01-01'

ORDER BY created_at DESC
LIMIT 100;
```

**做對的事：**

- 用 `'v1' AS source` 加識別欄位
- v2 的欄位名不同、用 AS 對齊
- ORDER BY 在最外層、套整體


### 13. 5 大常見坑

```text
❌ 1. UNION 沒留意欄位順序
   → SELECT name, age UNION SELECT email, id ← 型別錯亂
   → 修：兩邊嚴格對齊欄位順序

❌ 2. 預設 UNION 沒注意去重的代價
   → 大表 UNION 排序爆
   → 修：能用 UNION ALL 就用

❌ 3. ORDER BY 寫到第一個 SELECT 後
   → 報錯或結果亂
   → 修：ORDER BY 寫在整段最後

❌ 4. MySQL 用 INTERSECT / EXCEPT
   → MySQL 不支援（直到 8.0.31 才加 INTERSECT）
   → 修：用 JOIN 改寫

❌ 5. UNION 列不能用同個欄位名
   → 寫了 SELECT name AS x, ... UNION SELECT name AS y, ...
   → 最終欄位名只取第一個（x）、y 沒用
   → 修：知道規則、第一個 SELECT 決定欄位名
```


### 🎯 重點回顧

- **UNION** 上下拼、自動去重（慢）；**UNION ALL** 不去重（快）——預設用 ALL
- **INTERSECT** 取交集、PG 有 MySQL 沒
- **EXCEPT** 取差集（A - B）、PG 有 MySQL 沒
- **欄位數量、型別、順序要對齊**——順序不對就錯位
- **ORDER BY / LIMIT 寫整段最後**、套到整體結果
- **大表 UNION 改 UNION ALL**、效能差 10 倍
