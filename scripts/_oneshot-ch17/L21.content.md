**Transaction（交易）= 把多個 SQL「綁成一包、要嘛全成、要嘛全退」**—— 轉帳扣 A 加 B、中間失敗的話、絕對不能只扣不加。Transaction 是 SQL 的「核心保證」。

> 💡 **這課的價值**：沒 Transaction 概念、寫複雜業務時 100% 出 bug。轉帳系統、訂單系統、庫存扣減——任何「多步驟、要一致」的操作、都要 BEGIN ... COMMIT。本課把 ACID + 4 種隔離級別講透。


### 1. 為什麼需要 Transaction？

**反例：轉帳沒用 transaction：**

```text
UPDATE accounts SET balance = balance - 1000 WHERE id = 1;
-- ⚠️ 假設這時斷電 / server 死掉
UPDATE accounts SET balance = balance + 1000 WHERE id = 2;
-- 結果：A 扣了 1000、B 沒收到 → 1000 元憑空消失
```

**正解：用 transaction：**

```text
BEGIN;
UPDATE accounts SET balance = balance - 1000 WHERE id = 1;
UPDATE accounts SET balance = balance + 1000 WHERE id = 2;
COMMIT;
-- 中間斷電 → 整包失敗、A 也沒扣
-- 全部成功 → COMMIT 一起生效
```

**Transaction 的保證：要嘛全部成功、要嘛全部沒發生**——絕對不會「只做一半」。


### 2. 基本指令：BEGIN / COMMIT / ROLLBACK

```text
BEGIN;                  -- 開始 transaction
   ... 一堆 SQL ...
COMMIT;                 -- 全部生效

-- 或
BEGIN;
   ... 一堆 SQL ...
ROLLBACK;               -- 全部退回、沒事發生過
```

**範例：**

```text
BEGIN;
INSERT INTO orders (user_id, amount) VALUES (1, 1000) RETURNING id;
-- 結果 id = 123

INSERT INTO order_items (order_id, product_id, qty) VALUES (123, 5, 2);
UPDATE products SET stock = stock - 2 WHERE id = 5;

-- 看起來都對 → COMMIT
COMMIT;
```

**如果中間發現有問題：**

```text
BEGIN;
INSERT INTO orders (...) ...
-- ⚠️ 發現產品庫存不足
ROLLBACK;   -- 整包退回、訂單沒建、庫存沒扣
```


### 3. SAVEPOINT —— 部分 ROLLBACK

```text
BEGIN;
INSERT INTO users (name) VALUES ('A');
SAVEPOINT sp1;

INSERT INTO users (name) VALUES ('B');
SAVEPOINT sp2;

INSERT INTO users (name) VALUES ('C');
-- 發現 C 寫錯

ROLLBACK TO sp2;   -- 退回到 sp2、C 沒寫
                    -- 但 A、B 還在

COMMIT;
-- A、B 寫進去、C 沒
```

**SAVEPOINT 像 transaction 內的「存檔點」**——複雜流程裡留多個退路。


### 4. ACID —— Transaction 的 4 大保證

```text
A — Atomicity（原子性）
   要嘛全成、要嘛全退
   不會「做一半」

C — Consistency（一致性）
   transaction 結束後、DB 必須滿足所有約束
   例：FK 對到、CHECK 過、UNIQUE 不撞

I — Isolation（隔離性）
   多個 transaction 同時跑、互相不干擾
   就像「一個個排隊跑」一樣

D — Durability（持久性）
   COMMIT 後資料寫進 disk
   即使馬上斷電也不會掉
```

**SQL 資料庫的核心競爭力就是 ACID**——NoSQL 多半放棄 C 或 I 換速度、SQL 守得很緊。


### 5. 4 種隔離級別（Isolation Level）

最複雜的是 I（隔離性）——同時很多 transaction 跑、彼此能不能看到對方的資料？

```text
READ UNCOMMITTED
   能讀到別人「還沒 COMMIT」的資料
   → 髒讀（dirty read）
   PG 不支援（其實效果同 READ COMMITTED）

READ COMMITTED   ⭐ PG 預設
   只讀別人 COMMIT 過的
   → 還會有「不可重複讀」（同 transaction 內、兩次讀同列、結果不同）

REPEATABLE READ
   transaction 內、同樣 query 永遠回同樣結果
   → 但會有「phantom read」（其他人 INSERT 新列、你看不到）

SERIALIZABLE
   完全隔離、像排隊跑
   → 最安全、效能最差
```


### 6. 設定隔離級別

```text
-- 整個 session
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- 單個 transaction
BEGIN ISOLATION LEVEL REPEATABLE READ;
... SQL ...
COMMIT;
```

**業界共識**：

- 一般業務：READ COMMITTED（PG 預設、足夠）
- 報表 / 大量讀：REPEATABLE READ
- 財務 / 強一致：SERIALIZABLE


### 7. 同時跑的常見問題

**Dirty read（髒讀）—— READ UNCOMMITTED 才會：**

```text
A 的 transaction：UPDATE balance = 0 WHERE id = 1; （未 COMMIT）
B 讀到 balance = 0
A 之後 ROLLBACK
B 拿到的是「不存在」的資料 ❌
```

**Non-repeatable read（不可重複讀）—— READ COMMITTED 會：**

```text
A 開 transaction
A 第 1 次讀：SELECT balance FROM accounts WHERE id=1; → 1000
B 改了 balance = 500 並 COMMIT
A 第 2 次讀：→ 500
A 同一個 transaction 內、結果不一致
```

**Phantom read（幻讀）—— REPEATABLE READ 還會：**

```text
A 第 1 次 COUNT(*) FROM users WHERE city='台北' → 100
B INSERT 一個台北 user 並 COMMIT
A 第 2 次 COUNT → 101（多了一筆「幻影」）
```


### 8. Lock（鎖）—— Transaction 的內部機制

```text
SELECT ... FOR UPDATE
   鎖住這幾列、其他人想 UPDATE 必須等

SELECT ... FOR SHARE
   鎖住、但允許別人 SELECT、不允許 UPDATE
```

**經典應用：扣庫存防止超賣：**

```text
BEGIN;

SELECT stock FROM products WHERE id = 5 FOR UPDATE;
-- 鎖住這列、別的 transaction 想改要等

-- 應用層檢查 stock >= 想買數量
IF stock >= 2 THEN
  UPDATE products SET stock = stock - 2 WHERE id = 5;
  INSERT INTO orders ...
END IF;

COMMIT;  -- 解鎖
```


### 9. Deadlock（死鎖）—— 最頭痛的問題

```text
Transaction A:                Transaction B:
BEGIN                          BEGIN
UPDATE accounts SET ... WHERE id=1;  UPDATE accounts SET ... WHERE id=2;
UPDATE accounts SET ... WHERE id=2;  UPDATE accounts SET ... WHERE id=1;
                              -- A 等 B、B 等 A → 死鎖
```

**PG 自動偵測 deadlock、丟 error 給其中一方**：

```text
ERROR: deadlock detected
```

**避免方法**：永遠用「固定順序」鎖資源（例：永遠先鎖 id 小的）。


### 10. Optimistic Locking（樂觀鎖）vs Pessimistic Locking（悲觀鎖）

```text
Pessimistic（悲觀）
   先鎖、再操作
   SELECT ... FOR UPDATE
   保險但慢

Optimistic（樂觀）
   不鎖、操作完檢查版本
   加一個 version 欄位、UPDATE 時帶條件
```

**樂觀鎖範例：**

```text
-- 讀：版本 = 5
SELECT * FROM products WHERE id = 1;
-- 結果：stock=10, version=5

-- 更新時帶版本檢查
UPDATE products
SET stock = 8, version = version + 1
WHERE id = 1 AND version = 5;
-- 如果別人改過、version 已經 = 6、本句 update 0 列 → 重試
```


### 11. 完整實戰：訂單系統 transaction

```text
BEGIN;

-- 1. 鎖住產品列
SELECT stock FROM products WHERE id = $1 FOR UPDATE;

-- 2. 檢查庫存
-- （在應用層）

-- 3. 扣庫存
UPDATE products SET stock = stock - $qty WHERE id = $1;

-- 4. 建訂單
INSERT INTO orders (user_id, total) VALUES ($user, $total)
RETURNING id;

-- 5. 建訂單明細
INSERT INTO order_items (order_id, product_id, qty)
VALUES ($order_id, $product_id, $qty);

-- 6. 扣餘額
UPDATE accounts SET balance = balance - $total WHERE user_id = $user;

COMMIT;

-- 任何一步失敗 → ROLLBACK、全沒事發生
```


### 12. Transaction 的長度

```text
✅ 短：幾個 ms、影響少列
❌ 長：分鐘級、鎖很多列

長 transaction 的問題：
   - 鎖住資源、其他人等
   - WAL 累積、檢查點變慢
   - 失敗代價大（要重做整包）

業界共識：transaction 越短越好
```


### 13. 5 大常見坑

```text
❌ 1. 忘了 COMMIT、transaction 卡住
   → 整張表被鎖、其他 query 死等
   → 修：永遠 try-finally 確保 COMMIT 或 ROLLBACK

❌ 2. autocommit 模式忘了 BEGIN
   → 每句獨立 commit、中間失敗無法 rollback
   → 修：明確 BEGIN 開始 transaction

❌ 3. Transaction 內等使用者輸入
   → 使用者去吃飯了、鎖好幾分鐘
   → 修：先讀完所有資料、再開 transaction 寫

❌ 4. Deadlock 沒處理
   → PG 丟 error 給應用層、應用層沒重試
   → 修：catch deadlock error、自動重試 2-3 次

❌ 5. 在 transaction 內呼叫外部 API
   → API 慢 → transaction 卡很久
   → 修：transaction 只放 DB 操作、API 寫到 queue 之後處理
```


### 🎯 重點回顧

- **BEGIN → SQL → COMMIT / ROLLBACK** 是 transaction 基本流
- **ACID 4 字訣**：原子性、一致性、隔離性、持久性
- **READ COMMITTED 是 PG 預設**、99% 業務夠用
- **FOR UPDATE 鎖列**——扣庫存防超賣的經典招
- **Deadlock PG 自動偵測**、應用層要會 retry
- **Transaction 越短越好**、不要在裡面等使用者 / 打外部 API
- **Optimistic Locking（version 欄位）** 高併發場景常用
