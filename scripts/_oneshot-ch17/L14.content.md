**INNER JOIN（內連接）= 把兩張表「拼」起來**—— 用某個共同欄位（通常是 FK）配對、只留兩邊都有對到的列。是 SQL 真正威力的開始。

> 💡 **這課的價值**：單表 SELECT 沒什麼了不起、**JOIN 才是 SQL 的靈魂**。每張訂單要顯示「客戶名 + 商品名 + 數量」——三張表 JOIN 一句搞定。會 JOIN = SQL 真的入門。


### 1. 為什麼需要 JOIN？

**反例：所有資料塞一張表（會死）：**

```text
CREATE TABLE bad_orders (
  id        INT,
  customer_name  TEXT,
  customer_email TEXT,
  customer_phone TEXT,
  product_name   TEXT,
  product_price  NUMERIC,
  qty       INT
);
-- 同一個客戶下 100 單 → 客戶資料重複 100 次
-- 改地址 → 100 筆都要更新
-- 一致性災難
```

**正解：拆三張表、用 FK 關聯：**

```text
users    (id, name, email, phone)
products (id, name, price)
orders   (id, user_id FK, product_id FK, qty)
```

**讀的時候用 JOIN 拼回來——這就是「關聯式資料庫」的核心**。


### 2. 準備練習資料

```text
CREATE TABLE users (
  id   SERIAL PRIMARY KEY,
  name TEXT NOT NULL,
  city TEXT
);

CREATE TABLE orders (
  id      SERIAL PRIMARY KEY,
  user_id INT REFERENCES users(id),
  amount  NUMERIC(10, 2),
  created_at DATE
);

INSERT INTO users (name, city) VALUES
  ('Luffy', '台北'), ('Xueyue', '台北'),
  ('Nami', '台中'), ('Zoro', '高雄');

INSERT INTO orders (user_id, amount, created_at) VALUES
  (1, 1200, '2026-05-01'),
  (1, 850,  '2026-05-15'),
  (2, 3200, '2026-05-03'),
  (3, 750,  '2026-05-22');
-- 注意 user 4 (Zoro) 沒有訂單
```


### 3. 基本 INNER JOIN

```text
SELECT
  users.name,
  orders.amount,
  orders.created_at
FROM users
INNER JOIN orders ON users.id = orders.user_id;
```

💬 **電腦會顯示**（4 筆訂單、Zoro 沒訂單所以不出現）：

```text
  name  | amount  | created_at
--------+---------+------------
 Luffy  | 1200.00 | 2026-05-01
 Luffy  |  850.00 | 2026-05-15
 Xueyue | 3200.00 | 2026-05-03
 Nami   |  750.00 | 2026-05-22
```

**INNER JOIN 的意思**：「只留兩張表都對到的列」。Zoro 沒訂單 → 不出現。

```text
INNER JOIN 結構：
   FROM 表 A
   INNER JOIN 表 B  ON 連接條件
              ↑       ↑
            另一張表   兩張表怎麼配對（通常是 A.id = B.a_id）
```


### 4. 表別名（短名好讀）

JOIN 寫多了表名超長、用別名簡化：

```text
SELECT
  u.name,
  o.amount,
  o.created_at
FROM users u
INNER JOIN orders o ON u.id = o.user_id;
-- 一樣的結果、但短多了
```

**`INNER` 也能省略**（INNER JOIN = JOIN、預設就是 INNER）：

```text
FROM users u JOIN orders o ON u.id = o.user_id;
```


### 5. 三張表 JOIN（最常見實務）

**加一張 products 表：**

```text
CREATE TABLE products (
  id    SERIAL PRIMARY KEY,
  name  TEXT,
  price NUMERIC
);

CREATE TABLE order_items (
  order_id   INT REFERENCES orders(id),
  product_id INT REFERENCES products(id),
  qty        INT,
  PRIMARY KEY (order_id, product_id)
);
```

**4 張表 JOIN：「列出所有訂單、客戶名、商品名、數量」：**

```text
SELECT
  o.id          AS order_id,
  u.name        AS customer,
  p.name        AS product,
  oi.qty,
  p.price * oi.qty AS subtotal
FROM orders o
JOIN users u        ON u.id = o.user_id
JOIN order_items oi ON oi.order_id = o.id
JOIN products p     ON p.id = oi.product_id
ORDER BY o.id;
```

**重點**：JOIN 可以連續寫、一張接一張。


### 6. USING（更簡潔的寫法）

如果兩張表「連接欄位名一樣」、可以用 USING 取代 ON：

```text
-- ON 寫法
SELECT *
FROM users u
JOIN orders o ON u.id = o.user_id;
-- 問題：要把欄位名改一致才能用 USING

-- 假設 orders 的 FK 也叫 id（通常會、是「user_id」這種情況）
-- USING 比較適合 join 兩張表都有同名欄位、例如：
SELECT *
FROM orders o
JOIN order_items oi USING (id);
-- 等同：ON o.id = oi.id
```

**USING 自動「合併」同名欄位、只顯示一次。** 實務上**用 ON 比較清楚**——USING 偶爾遇得到、知道就好。


### 7. NATURAL JOIN（自動配對同名欄位）

```text
SELECT * FROM users NATURAL JOIN orders;
-- 自動找兩表中所有同名欄位、當連接條件
```

**業界忠告**：**永遠不要用 NATURAL JOIN**——加新欄位、突然多一個同名 → JOIN 行為改變、bug 難找。**用 ON 就對了**。


### 8. 自我 JOIN（self join、JOIN 同一張表）

**範例**：員工表有 `boss_id`、想列出「員工 + boss 名字」。

```text
CREATE TABLE employees (
  id      INT PRIMARY KEY,
  name    TEXT,
  boss_id INT REFERENCES employees(id)
);

SELECT
  e.name   AS employee,
  b.name   AS boss
FROM employees e
LEFT JOIN employees b ON e.boss_id = b.id;
-- 同一張表 JOIN 兩次、必須用別名區分
```

**self join 必須用別名**（不然 SQL 不知道兩個 employees 是哪個）。


### 9. JOIN 配 WHERE / GROUP BY

JOIN 後面接 WHERE / GROUP BY / ORDER BY 都正常：

```text
-- 找出「台北客戶的訂單總額」
SELECT
  u.name,
  SUM(o.amount) AS total
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE u.city = '台北'
GROUP BY u.id, u.name
ORDER BY total DESC;
```

💬 **電腦會顯示**：

```text
  name  |  total
--------+---------
 Luffy  | 2050.00
 Xueyue | 3200.00
```


### 10. JOIN 條件不只能用 `=`

```text
-- 範圍 join：找出每個訂單適用的折扣等級
SELECT o.id, o.amount, d.discount_pct
FROM orders o
JOIN discounts d
  ON o.amount BETWEEN d.min_amount AND d.max_amount;

-- 多條件 join
SELECT *
FROM orders o
JOIN coupons c
  ON c.user_id = o.user_id
  AND c.used_at IS NULL
  AND c.expires_at > o.created_at;
```


### 11. JOIN 跟 WHERE 的等價（單表時）

很多時候 INNER JOIN 也能寫成 subquery + WHERE：

```text
-- JOIN 寫法
SELECT u.name, o.amount
FROM users u
JOIN orders o ON u.id = o.user_id;

-- WHERE 寫法（一樣結果、但效能可能差）
SELECT u.name, (SELECT amount FROM orders WHERE user_id = u.id LIMIT 1)
FROM users u;

-- IN 寫法
SELECT name FROM users
WHERE id IN (SELECT DISTINCT user_id FROM orders);
```

**業界共識**：能用 JOIN 就用 JOIN——可讀性 + 效能通常都贏。


### 12. JOIN 後欄位名衝突

兩張表都有 `id`、`created_at`——`SELECT *` 會兩個 id 都出現、混淆。

```text
-- ❌ 模糊
SELECT * FROM users u JOIN orders o ON u.id = o.user_id;

-- ✅ 明確
SELECT
  u.id AS user_id,
  u.name,
  o.id AS order_id,
  o.amount
FROM users u JOIN orders o ON u.id = o.user_id;
```

**永遠加表別名、明確指定欄位**。


### 13. 5 大常見坑

```text
❌ 1. JOIN 忘記 ON
   → 變 CROSS JOIN、笛卡兒積、結果爆炸
   → 修：永遠寫 ON、PG 會擋（MySQL 不擋會出事）

❌ 2. JOIN 同名欄位沒指定別名
   → SELECT id 不知道哪張表的 id
   → 修：永遠 u.id / o.id

❌ 3. JOIN 後 GROUP BY 漏欄位
   → SELECT u.name 但 GROUP BY u.id ← PG 嚴格、報錯
   → 修：GROUP BY 加上 u.id, u.name

❌ 4. INNER JOIN 期望「沒對到也保留」
   → INNER 會把沒對到的全砍、結果少一半
   → 修：用 LEFT JOIN（L15）

❌ 5. JOIN 沒 Index
   → 大表 JOIN 慢到死
   → 修：FK 欄位永遠加 Index（L19）
```


### 🎯 重點回顧

- **INNER JOIN = 只留兩邊都對到的列**
- **`FROM A JOIN B ON A.x = B.y`** 是基本句型
- **`INNER` 可省、`JOIN = INNER JOIN`**
- **多表 JOIN** 連續寫、寫幾張都行
- **永遠用表別名**（`users u`）、欄位明確（`u.id`）
- **不要用 NATURAL JOIN**——schema 改了 bug 難找
- **JOIN 欄位加 Index**——大表必備
