**LEFT JOIN（左連接）= 左表全保留、右表沒對到的補 NULL**—— INNER JOIN 把沒對到的全砍、LEFT JOIN 留下來。寫「找沒下單的客戶」、「沒交作業的學生」這種問題、靠它。

> 💡 **這課的價值**：80% 的「找出沒做 X 的人」問題、答案都是 `LEFT JOIN ... WHERE 右表.id IS NULL`。這是 SQL 面試最愛考的題目、本課練熟它你就無敵。


### 1. LEFT JOIN vs INNER JOIN 對比

延續 L14 練習資料（4 個 user、4 筆 orders、Zoro 沒訂單）。

**INNER JOIN（少 Zoro）：**

```text
SELECT u.name, o.amount
FROM users u
INNER JOIN orders o ON u.id = o.user_id;

  name  | amount
--------+--------
 Luffy  |   1200
 Luffy  |    850
 Xueyue |   3200
 Nami   |    750
-- Zoro 不見
```

**LEFT JOIN（Zoro 也在、但 amount 是 NULL）：**

```text
SELECT u.name, o.amount
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;

  name  | amount
--------+--------
 Luffy  |   1200
 Luffy  |    850
 Xueyue |   3200
 Nami   |    750
 Zoro   |   NULL   ← 保留、右表沒對到、補 NULL
```


### 2. LEFT JOIN 概念圖

```text
左表（users）        右表（orders）

  Luffy        ────┬──→ 訂單 1
                   └──→ 訂單 2
  Xueyue       ───────→ 訂單 3
  Nami         ───────→ 訂單 4
  Zoro                   ← 沒對到

LEFT JOIN：左表全保留、Zoro 也留（amount = NULL）
INNER JOIN：只留有對到的、Zoro 不見
```

**「LEFT」指的是 FROM 後面那張表**——`FROM users LEFT JOIN orders` → 左表是 users。


### 3. 經典題：找出「沒下單的客戶」

**寫法 1：LEFT JOIN + IS NULL（最常用）**

```text
SELECT u.id, u.name
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.id IS NULL;
```

💬 **電腦會顯示**：

```text
 id | name
----+------
  4 | Zoro
```

**邏輯**：
1. LEFT JOIN 把所有 user 撈出來、沒訂單的右邊全 NULL
2. WHERE `o.id IS NULL` → 只留「沒對到 order 的」

**寫法 2：NOT EXISTS（一樣意思）**

```text
SELECT id, name
FROM users u
WHERE NOT EXISTS (
  SELECT 1 FROM orders o WHERE o.user_id = u.id
);
```

**寫法 3：NOT IN（注意 NULL 坑）**

```text
SELECT id, name
FROM users
WHERE id NOT IN (SELECT user_id FROM orders WHERE user_id IS NOT NULL);
-- 要記得排除 NULL、不然全空集合（L6 講過）
```

**業界共識**：**LEFT JOIN + IS NULL** 跟 **NOT EXISTS** 都好用、後者通常稍快。


### 4. LEFT JOIN 配聚合：每個 user 的訂單數（含 0 單的）

```text
SELECT
  u.id,
  u.name,
  COUNT(o.id) AS order_count,
  COALESCE(SUM(o.amount), 0) AS total
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.name
ORDER BY total DESC;
```

💬 **電腦會顯示**：

```text
 id |  name  | order_count | total
----+--------+-------------+--------
  2 | Xueyue |           1 |   3200
  1 | Luffy  |           2 |   2050
  3 | Nami   |           1 |    750
  4 | Zoro   |           0 |      0    ← 0 單也顯示
```

**重點**：
- `COUNT(o.id)` 不是 `COUNT(*)`——因為 `*` 連 NULL 都算、Zoro 會變 1
- `COALESCE(SUM(o.amount), 0)` 把 NULL SUM 換成 0、好看


### 5. LEFT JOIN 多張表

```text
SELECT
  u.name,
  o.id     AS order_id,
  p.name   AS product
FROM users u
LEFT JOIN orders o      ON u.id = o.user_id
LEFT JOIN order_items oi ON o.id = oi.order_id
LEFT JOIN products p    ON oi.product_id = p.id
ORDER BY u.id;
```

**每個 LEFT 都「向右」延伸——左邊保留、右邊沒對到補 NULL**。


### 6. ON 條件放 LEFT JOIN 上 vs WHERE 上（最大坑）

**範例**：找出「2026 年 5 月每個 user 的訂單數（含 0 單）」。

**❌ 錯誤寫法**：

```text
SELECT u.name, COUNT(o.id) AS count
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.created_at >= '2026-05-01'
  AND o.created_at <  '2026-06-01'
GROUP BY u.name;
```

**問題**：`WHERE o.created_at >= '2026-05-01'` 在 JOIN 後過濾——Zoro 的 o.created_at 是 NULL、NULL >= ... 永遠 false → **Zoro 又被砍了**。

**✅ 正確寫法**：把過濾條件放進 ON：

```text
SELECT u.name, COUNT(o.id) AS count
FROM users u
LEFT JOIN orders o
  ON u.id = o.user_id
  AND o.created_at >= '2026-05-01'
  AND o.created_at <  '2026-06-01'
GROUP BY u.name;
```

**重點**：

```text
ON 條件   → 影響「怎麼配對」、不滿足的右邊變 NULL（左邊保留）
WHERE 條件 → JOIN 之後再過濾、會把左邊也砍掉
```

**LEFT JOIN 的右表條件、要放 ON 不是 WHERE**——這是 SQL 最容易踩的坑之一。


### 7. RIGHT JOIN —— 跟 LEFT 對稱、但業界少用

```text
SELECT u.name, o.amount
FROM users u
RIGHT JOIN orders o ON u.id = o.user_id;
-- 右表（orders）全保留、左表沒對到的補 NULL
```

**為什麼少用？**——任何 RIGHT JOIN 都能改寫成 LEFT JOIN、改寫後更直覺：

```text
-- RIGHT JOIN：
FROM users u RIGHT JOIN orders o ON ...

-- 等同 LEFT JOIN（換位置）：
FROM orders o LEFT JOIN users u ON ...
```

**業界共識**：**永遠用 LEFT JOIN、把要保留的表放左邊**。代碼一致、不會混亂。


### 8. FULL OUTER JOIN —— 左右都保留

```text
SELECT u.name, o.amount
FROM users u
FULL OUTER JOIN orders o ON u.id = o.user_id;
```

**結果**：
- 左表有、右表沒對到 → 右表欄位 NULL
- 右表有、左表沒對到 → 左表欄位 NULL
- 兩邊都對到 → 正常

**應用場景**：對帳——A 系統有的訂單、B 系統有的訂單、找出兩邊差異。

```text
SELECT
  COALESCE(a.order_id, b.order_id) AS id,
  a.amount AS amount_a,
  b.amount AS amount_b
FROM system_a a
FULL OUTER JOIN system_b b ON a.order_id = b.order_id
WHERE a.order_id IS NULL OR b.order_id IS NULL;
-- 找出兩邊不一致的訂單
```

**MySQL 不支援 FULL OUTER JOIN**——要用 LEFT UNION RIGHT 拼。PG 直接支援。


### 9. JOIN 類型總覽

```text
INNER JOIN
   兩邊都有對到才留        ← 90% 場景

LEFT JOIN（LEFT OUTER）
   左邊全留、右邊沒對到補 NULL  ← 8% 場景（找「沒做 X 的人」）

RIGHT JOIN（RIGHT OUTER）
   右邊全留、左邊沒對到補 NULL  ← 業界少用（改寫成 LEFT）

FULL OUTER JOIN
   兩邊都留、沒對到都補 NULL    ← 對帳、資料比對

CROSS JOIN
   左右每列全配對（笛卡兒積）   ← 特殊場景（L16）
```


### 10. 完整實戰：找出「過去 7 天沒回購的 VIP」

```text
SELECT
  u.id,
  u.name,
  u.email,
  u.last_login_at,
  MAX(o.created_at) AS last_order_at
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE u.tier = 'vip'
GROUP BY u.id, u.name, u.email, u.last_login_at
HAVING MAX(o.created_at) < NOW() - INTERVAL '7 days'
    OR MAX(o.created_at) IS NULL;
```

**邏輯**：

1. LEFT JOIN orders、所有 vip 都留
2. WHERE 過濾 vip
3. GROUP 每個 user 算最後訂單時間
4. HAVING 留「超過 7 天沒下單」或「從沒下單」的


### 11. 5 大常見坑

```text
❌ 1. LEFT JOIN 的右表條件放 WHERE
   → 左表也被砍、變相 INNER
   → 修：右表條件放 ON、左表條件放 WHERE

❌ 2. 用 COUNT(*) 算 LEFT JOIN 結果
   → 沒對到的右邊 NULL、COUNT(*) 還是算 1
   → 修：COUNT(右表非空欄位)、例 COUNT(o.id)

❌ 3. NOT IN + 內含 NULL
   → 永遠空集合
   → 修：用 NOT EXISTS 或 LEFT JOIN + IS NULL

❌ 4. RIGHT JOIN 混用 LEFT JOIN
   → 邏輯難讀、容易出 bug
   → 修：統一用 LEFT、把要保留的表放左邊

❌ 5. LEFT JOIN 多張表後、忘記前面的 LEFT 效果
   → 中間有一個 INNER、整段又變 INNER
   → 修：要保留左表的話、後面 JOIN 全部 LEFT
```


### 🎯 重點回顧

- **LEFT JOIN = 左表全留、右表沒對到補 NULL**
- **找「沒做 X 的人」** = LEFT JOIN + WHERE 右表.id IS NULL
- **LEFT JOIN 的右表條件放 ON、不放 WHERE**（最大坑）
- **COUNT(*) 改成 COUNT(右表欄位)** 避免算到 NULL
- **業界用 LEFT 不用 RIGHT**——一致性
- **FULL OUTER JOIN** 對帳神器、PG 支援 MySQL 不支援
