**EXPLAIN（執行計畫）= PG 給你看「它打算怎麼跑這 query」**—— 慢查詢的偵探工作從 EXPLAIN 開始。本課把 Seq Scan / Index Scan / cost / actual time 一次看懂。

> 💡 **這課的價值**：DBA 跟一般工程師的差別、就在「能不能看懂執行計畫」。學會 EXPLAIN 你就能說「這 query 為什麼慢」、不再是丟給後端工程師救火。


### 1. EXPLAIN 跟 EXPLAIN ANALYZE 差別

```text
EXPLAIN          只「估算」、不跑 query、安全
EXPLAIN ANALYZE  真的跑、看實際時間（注意 UPDATE/DELETE 也會真的執行！）
```

**安全做法（測 UPDATE / DELETE）：**

```text
BEGIN;
EXPLAIN ANALYZE UPDATE users SET age = 0;
ROLLBACK;  -- 看完計畫就回退、不真的改
```


### 2. 看一個最簡單的 EXPLAIN

```text
EXPLAIN SELECT * FROM users WHERE email = 'luffy@ai.com';
```

💬 **電腦會顯示**：

```text
Seq Scan on users  (cost=0.00..18334.00 rows=1 width=68)
  Filter: (email = 'luffy@ai.com'::text)
```

**逐行翻譯：**

```text
Seq Scan on users     掃描方式：順序掃描 users 表（從頭到尾）
cost=0.00..18334.00   成本估算（不是時間、是相對值）
                      0.00  = 啟動成本（拿第一筆要多久）
                      18334 = 總成本（全部跑完要多久）
rows=1                估計回傳 1 列
width=68              估計每列 68 bytes
Filter:               過濾條件
```


### 3. EXPLAIN ANALYZE 加上實際數字

```text
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'luffy@ai.com';
```

💬 **電腦會顯示**：

```text
Seq Scan on users
  (cost=0.00..18334.00 rows=1 width=68)
  (actual time=123.456..789.012 rows=1 loops=1)
  Filter: (email = 'luffy@ai.com'::text)
  Rows Removed by Filter: 999999
Planning Time: 0.234 ms
Execution Time: 789.123 ms
```

**多出來的關鍵資訊：**

```text
actual time=123.456..789.012   實際時間（毫秒）
                                123.456 = 拿到第一筆的時間
                                789.012 = 跑完所有的時間
loops=1                        這個節點跑了 1 次
Rows Removed by Filter: 999999 ⚠️ 過濾掉的列數（看到這個就要警覺）
Planning Time                  PG 想計畫花的時間
Execution Time                 ⭐ 實際跑的總時間
```

**這個 query 看到 `Rows Removed by Filter: 999999` —— 掃了 100 萬筆只回 1 筆、超浪費 → 該加 Index**。


### 4. 加 Index 後再 EXPLAIN

```text
CREATE INDEX idx_users_email ON users(email);
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'luffy@ai.com';
```

💬 **電腦會顯示**：

```text
Index Scan using idx_users_email on users
  (cost=0.42..8.44 rows=1 width=68)
  (actual time=0.156..0.158 rows=1 loops=1)
  Index Cond: (email = 'luffy@ai.com'::text)
Planning Time: 0.345 ms
Execution Time: 0.234 ms
```

**從 789 ms → 0.23 ms**——快了 3000 倍！這就是 Index 的威力。


### 5. 6 種 Scan 類型

```text
Seq Scan       順序掃描（沒 Index 或全表都要）
   慢、但小表 OK

Index Scan     用 Index 直接定位
   快、適合「查少量列」

Index Only Scan  只用 Index、不查原表
   超快、需要 Index 包含查的所有欄位（covering index）

Bitmap Index Scan + Bitmap Heap Scan
   兩階段：先用 Index 找 page、再回表撈列
   適合「中量結果」、PG 自動選

Tid Scan        直接用 row id 找
   罕見、PG 內部優化

Sort / Merge / Hash 之類
   不是 Scan、是後處理（排序、合併、JOIN）
```


### 6. JOIN 在 EXPLAIN 看起來

```text
EXPLAIN ANALYZE
SELECT u.name, o.amount
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE u.city = '台北';
```

💬 **可能顯示**：

```text
Hash Join  (cost=...)
  Hash Cond: (o.user_id = u.id)
  ->  Seq Scan on orders o
  ->  Hash
        ->  Seq Scan on users u
              Filter: (city = '台北'::text)
```

**3 種 JOIN 演算法：**

```text
Nested Loop Join
   左表每列、掃右表一次
   左表小（< 1000）+ 右表有 Index 時、最快

Hash Join
   把一張表建 hash table、另一張查
   兩邊都大時、最快（PG 預設）

Merge Join
   兩張表都先排序、再合併
   兩邊已排序時、最快
```

**PG 自動選**——選錯時可以手動 hint、但通常不用。


### 7. PG 的 cost 怎麼算

```text
cost = 啟動成本..總成本

PG 用「假想單位」計算：
   讀 1 個 page（8KB）      = 1.0
   隨機讀 1 個 page         = 4.0
   處理 1 列 CPU 成本       = 0.01
   處理 1 個 operator       = 0.0025
```

**cost 不是時間、是「相對複雜度」**——比較同 query 兩個計畫哪個快、絕對值意義不大。**新手只看 `Execution Time`** 比較直覺。


### 8. 常見「不好的」執行計畫訊號

```text
🚨 Seq Scan on huge_table
   大表全掃 → 該加 Index

🚨 Rows Removed by Filter: 大數字
   掃很多、回很少 → Index 沒對齊條件

🚨 Sort 出現在計畫裡、且有大表
   ORDER BY 沒 Index 支援 → 加 Index

🚨 Nested Loop + 左表大
   應該是 Hash Join、PG 選錯了

🚨 actual rows vs estimated rows 差很多
   統計過時、跑 ANALYZE 表名

🚨 Planning Time > Execution Time
   query 太複雜、計畫成本超過跑的成本
```


### 9. EXPLAIN 格式（FORMAT JSON 適合分析工具）

```text
EXPLAIN (ANALYZE, FORMAT JSON)
SELECT * FROM users WHERE email = 'luffy@ai.com';
```

把結果貼到 [https://explain.dalibo.com/](https://explain.dalibo.com/) 或 [https://explain.depesz.com/](https://explain.depesz.com/) —— 視覺化執行計畫、慢的節點紅色標出來。

**Production 排查必備工具**。


### 10. EXPLAIN 完整選項

```text
EXPLAIN (
  ANALYZE,     -- 真的跑
  BUFFERS,     -- 顯示 buffer 命中
  TIMING,      -- 顯示時間（預設開）
  VERBOSE,     -- 顯示完整欄位列表
  FORMAT JSON  -- JSON 格式
) SELECT ...;
```

**`BUFFERS` 超有用：**

```text
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM huge_table;

→ shared hit=1234 read=5678
  hit = cache 命中、read = 從 disk 讀
  讀超多 = IO 瓶頸
```


### 11. 完整實戰：debug 慢 query

**步驟 1：確認慢——**

```text
EXPLAIN ANALYZE
SELECT u.name, COUNT(o.id)
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE u.city = '台北'
GROUP BY u.name;

-- Execution Time: 5234 ms （太慢）
```

**步驟 2：看計畫——**

```text
Hash Aggregate
  -> Hash Right Join
       -> Seq Scan on orders   (cost=...18334)  ← 全表掃
       -> Seq Scan on users    (cost=...18334)
              Filter: (city = '台北'::text)
              Rows Removed by Filter: 990000   ← ⚠️
```

**步驟 3：兩個 Seq Scan 都該變 Index Scan——加 Index：**

```text
CREATE INDEX idx_users_city ON users(city);
CREATE INDEX idx_orders_user_id ON orders(user_id);
```

**步驟 4：再 EXPLAIN 確認——**

```text
-- Execution Time: 23 ms ✅
```

**慢 query 救援流程**：
1. EXPLAIN ANALYZE 找瓶頸
2. 加對的 Index
3. 再 EXPLAIN 確認改善


### 12. PG 統計資訊維護

PG 用「統計資訊」決定執行計畫——統計過時、計畫選錯：

```text
-- 手動更新統計
ANALYZE users;

-- 整個 DB 都更新
ANALYZE;

-- 更激進：VACUUM + ANALYZE（清理 dead tuple + 更新統計）
VACUUM ANALYZE users;
```

**通常 PG 自動做（autovacuum）**——但大表改動後手動跑一次比較保險。


### 13. 5 大常見坑

```text
❌ 1. EXPLAIN（沒 ANALYZE）就下結論
   → cost 只是估算、跟實際差很多
   → 修：用 EXPLAIN ANALYZE 看真實時間

❌ 2. EXPLAIN ANALYZE 在 Production UPDATE / DELETE 跑
   → 真的執行了、資料被改
   → 修：BEGIN + EXPLAIN ANALYZE + ROLLBACK

❌ 3. 看 cost 不看 actual
   → cost 1000 但 actual 0.1 ms = 沒事
   → 修：永遠看 Execution Time / actual time

❌ 4. 加 Index 沒重跑 EXPLAIN 確認
   → 以為加了就用、實際 PG 沒選用
   → 修：永遠 EXPLAIN ANALYZE 驗證

❌ 5. 統計過時導致 PG 選錯計畫
   → 大量 INSERT 後沒 ANALYZE
   → 修：定期 ANALYZE、或開 autovacuum
```


### 🎯 重點回顧

- **EXPLAIN** 看計畫、**EXPLAIN ANALYZE** 真跑看時間
- **Seq Scan = 慢、Index Scan = 快、Index Only Scan = 最快**
- **看 `Execution Time` 比 cost 直覺**
- **Rows Removed by Filter 大** → 該加 Index
- **JOIN 演算法 3 種**：Nested Loop / Hash Join / Merge Join（PG 自動選）
- **explain.dalibo.com / explain.depesz.com** 是視覺化神器
- **ANALYZE 表名** 更新統計、避免 PG 選錯計畫
