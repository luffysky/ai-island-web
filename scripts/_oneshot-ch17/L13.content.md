**DELETE（刪除）= SQL 最危險的指令**—— 跟 UPDATE 一樣、**沒寫 WHERE 整張表變空**。本課除了講語法、更講「soft delete vs hard delete」業界爭論、以及 TRUNCATE 為什麼快 1000 倍。

> 💡 **這課的價值**：刪資料不只是 `DELETE FROM`、是「要不要保留紀錄」、「外鍵怎麼處理」、「大量刪要不要鎖表」——這課把這些坑提早擋掉。


### 1. 基本 DELETE

```text
DELETE FROM 表名 WHERE 條件;
```

**範例：**

```text
DELETE FROM users WHERE id = 5;
```

💬 **電腦會顯示**：

```text
DELETE 1
-- 刪了 1 列
```

**沒寫 WHERE：**

```text
DELETE FROM users;
-- ❌ 全表被刪、6 萬筆變 0 筆
```


### 2. 多條件刪除

```text
-- 刪除一年沒登入的 user
DELETE FROM users
WHERE last_login_at < NOW() - INTERVAL '1 year'
  AND tier = 'free';

-- 刪除 status = 'cancelled' 超過 30 天的訂單
DELETE FROM orders
WHERE status = 'cancelled'
  AND cancelled_at < NOW() - INTERVAL '30 days';
```


### 3. DELETE USING（PG 的 JOIN DELETE）

**範例**：刪除「所有屬於 `inactive` user 的 posts」。

```text
DELETE FROM posts
USING users
WHERE posts.author_id = users.id
  AND users.status = 'inactive';
```

**邏輯**：USING 後面接的表「不被刪、只當條件用」。


### 4. RETURNING（刪除前先拿一份備份）

```text
DELETE FROM users
WHERE id = 5
RETURNING *;
```

💬 **電腦會顯示**（被刪掉的整列資料）：

```text
 id |  name  |     email     | age
----+--------+---------------+-----
  5 | Sanji  | sanji@ai.com  |  30
```

**應用**：把要刪的記到 log table：

```text
WITH deleted AS (
  DELETE FROM users
  WHERE last_login_at < NOW() - INTERVAL '2 years'
  RETURNING *
)
INSERT INTO deleted_users_log
SELECT *, NOW() AS deleted_at FROM deleted;
```


### 5. soft delete vs hard delete（業界爭論）

**Hard delete（真刪）：**

```text
DELETE FROM users WHERE id = 5;
-- 列從 DB 消失
```

**Soft delete（軟刪、加標記）：**

```text
ALTER TABLE users ADD COLUMN deleted_at TIMESTAMPTZ;

UPDATE users SET deleted_at = NOW() WHERE id = 5;
-- 列還在、但 deleted_at 有值

-- 查資料時都加 WHERE deleted_at IS NULL
SELECT * FROM users WHERE deleted_at IS NULL;
```

**兩派比較：**

```text
Hard delete
   ✅ 簡單、DB 乾淨
   ✅ 符合 GDPR「被遺忘權」（要真刪）
   ❌ 誤刪救不回
   ❌ 沒審計軌跡

Soft delete
   ✅ 誤刪可恢復
   ✅ 有審計軌跡（誰刪的、什麼時候刪）
   ✅ 外鍵不會炸（資料還在）
   ❌ 每個 query 要記得 WHERE deleted_at IS NULL
   ❌ DB 越來越大、效能變差
   ❌ GDPR 還是要真刪、麻煩
```

**林董的建議：**

```text
業務上會有「回復」需求 → soft delete
   例：聊天訊息、文章、訂單

完全用不到的紀錄 → hard delete
   例：session、verification code、log

GDPR 個資 → 永遠 hard delete
   user 要求刪除、就真的刪
```


### 6. 外鍵 + DELETE 的連動行為（複習 L4）

```text
ON DELETE CASCADE       父刪、子也刪
ON DELETE SET NULL      父刪、子的 FK 變 NULL
ON DELETE RESTRICT      父被引用、不准刪
```

**範例：**

```text
CREATE TABLE posts (
  id        SERIAL PRIMARY KEY,
  author_id INT REFERENCES users(id) ON DELETE CASCADE,
  title     TEXT
);

DELETE FROM users WHERE id = 5;
-- author_id = 5 的所有 posts 也一起被刪
```

> 💡 **CASCADE 會連鎖**——刪 user → 刪 posts → 刪 comments → 刪 likes ... 全表炸光。**Production 慎用 CASCADE**。


### 7. TRUNCATE —— 清空整張表的最快方法

```text
TRUNCATE TABLE logs;
-- 整張表變空、比 DELETE 快 1000 倍
```

**TRUNCATE vs DELETE 差別：**

```text
DELETE 沒 WHERE
   ✅ 可 ROLLBACK
   ✅ 觸發 trigger
   ❌ 每列都記 log、慢

TRUNCATE
   ✅ 超快（直接 reset table）
   ✅ 可 ROLLBACK（PG 在 transaction 內）
   ❌ 不觸發 trigger（PG 觸發 TRUNCATE trigger）
   ❌ AUTO_INCREMENT / SERIAL 重置回 1
   ❌ 需要更高權限
```

**TRUNCATE 適用場景：**

- 清空測試表
- 清空 log 表
- 每天重建的 staging 表

**TRUNCATE 連同 RESTART：**

```text
TRUNCATE TABLE products RESTART IDENTITY;
-- 同時把 SERIAL 計數器歸零
```


### 8. 大量 DELETE 的效能陷阱

**問題**：`DELETE FROM logs WHERE created_at < NOW() - INTERVAL '1 year';` 影響 1000 萬列——直接卡住整個 DB。

**為什麼？**

```text
1. DELETE 一筆 = 寫一筆 WAL log
2. 1000 萬筆 = WAL 寫爆
3. 鎖整張表、其他 query 全 timeout
4. transaction 太大、回滾風險高
```

**解法 A：分批刪**

```text
DELETE FROM logs
WHERE id IN (
  SELECT id FROM logs
  WHERE created_at < NOW() - INTERVAL '1 year'
  LIMIT 10000
);
-- 用 loop 跑、每次刪 1 萬筆、跑直到沒列
```

**解法 B：用 partition table（PG / MySQL 都支援）**

```text
-- 按月分表
CREATE TABLE logs_2025_01 PARTITION OF logs
  FOR VALUES FROM ('2025-01-01') TO ('2025-02-01');

-- 過期就 DROP 整個 partition（瞬間刪、不掃描）
DROP TABLE logs_2025_01;
```

**Production 大表清理**——一律用 partition。


### 9. 完整實戰：訂閱到期清理

📄 **寫進程式檔**：`scripts/cleanup-expired.sql`

```text
-- 1. 把到期 30 天以上的訂閱標 deleted（soft）
UPDATE subscriptions
SET deleted_at = NOW()
WHERE expires_at < NOW() - INTERVAL '30 days'
  AND deleted_at IS NULL;

-- 2. soft delete 過了 90 天的「真的刪掉」（hard）
DELETE FROM subscriptions
WHERE deleted_at IS NOT NULL
  AND deleted_at < NOW() - INTERVAL '90 days'
RETURNING id, user_id;

-- 3. session 表直接全刪（不需要紀錄）
DELETE FROM sessions
WHERE expires_at < NOW();
```


### 10. CTE + DELETE（複雜邏輯拆解）

```text
WITH duplicate_emails AS (
  SELECT email, MIN(id) AS keep_id
  FROM users
  GROUP BY email
  HAVING COUNT(*) > 1
)
DELETE FROM users u
USING duplicate_emails d
WHERE u.email = d.email
  AND u.id <> d.keep_id;
-- 同 email 多筆、留 id 最小那筆、其他刪
```


### 11. 防呆：先用 SELECT 看影響範圍

**永遠的好習慣：**

```text
-- 第 1 步：用 SELECT 看會影響哪些列
SELECT id, name, email FROM users
WHERE last_login_at < NOW() - INTERVAL '2 years';

-- 第 2 步：覺得對 → 換成 DELETE
DELETE FROM users
WHERE last_login_at < NOW() - INTERVAL '2 years';
```

**或用 BEGIN + ROLLBACK：**

```text
BEGIN;
DELETE FROM users WHERE ...;
SELECT COUNT(*) FROM users;   -- 看剩多少
ROLLBACK;                      -- 試完退回
-- 確認對才真的跑
```


### 12. 5 大常見坑

```text
❌ 1. DELETE 沒寫 WHERE
   → 全表變空、災難
   → 修：永遠先 BEGIN 試、ROLLBACK 回退

❌ 2. CASCADE 過度連鎖
   → 刪一個 user 砍掉 10 張表的資料
   → 修：重要關聯用 RESTRICT、不要無腦 CASCADE

❌ 3. 一次 DELETE 上百萬筆
   → 鎖表、WAL 爆、Production 掛
   → 修：分批刪 / partition table / TRUNCATE

❌ 4. soft delete 忘了過濾
   → 列表頁顯示已刪 user
   → 修：建一個 view 包好 deleted_at IS NULL、所有 query 用 view

❌ 5. TRUNCATE 在 Production 亂用
   → 不記 row-level log、外鍵不檢查 → 出事查不到
   → 修：Production 用 DELETE + 分批、staging 才 TRUNCATE
```


### 🎯 重點回顧

- **DELETE FROM 表 WHERE 條件**——WHERE 永遠先寫、再確認
- **soft delete**（標 deleted_at）保留紀錄、適合業務資料
- **hard delete** 真刪、GDPR / log 用
- **外鍵 CASCADE / SET NULL / RESTRICT** 影響刪除連鎖
- **TRUNCATE** 清空整表、快 1000 倍但 reset SERIAL、不觸發 row trigger
- **大量刪要分批**、不然 lock 整張表
