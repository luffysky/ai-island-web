**View（檢視表）= 把一段 SQL 起個名字、當虛擬 table 用**—— 像程式裡的 function、複用、抽象。**Materialized View（物化檢視表）** 更進一步、把結果存下來、查超快。

> 💡 **這課的價值**：每次寫複雜統計 SQL 都重複貼 5 個 JOIN 太累——包成 View、之後 `SELECT * FROM v` 就拿到結果。報表加速用 Materialized View——預先算好、查詢從 10 秒變 10 ms。


### 1. 什麼是 View？

**View = 一段 SELECT 的「別名」、用的時候像表一樣 SELECT**。

**範例：**

```text
-- 建 view
CREATE VIEW active_users AS
SELECT id, name, email, last_login_at
FROM users
WHERE deleted_at IS NULL
  AND last_login_at > NOW() - INTERVAL '30 days';

-- 用 view（像表一樣）
SELECT * FROM active_users WHERE email LIKE '%@gmail.com';
```

**注意**：View **不存資料**——每次查 view、PG 都會跑底下的 SELECT 一次。**view 是「執行時組裝」**。


### 2. View 解決什麼問題？

**3 大用途：**

```text
1. 邏輯複用
   複雜 query 包一次、之後不用重複貼

2. 權限隔離
   給某些 user 看 view（只顯示部分欄位 / 行）
   原始 table 不給看

3. 抽象 schema 改動
   schema 重構時、view 維持舊介面
   應用層不用改
```


### 3. 完整建 View 範例

```text
CREATE VIEW user_stats AS
SELECT
  u.id,
  u.name,
  u.email,
  COUNT(o.id) AS order_count,
  COALESCE(SUM(o.amount), 0) AS total_spent,
  MAX(o.created_at) AS last_order_at
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.name, u.email;

-- 之後直接用
SELECT * FROM user_stats WHERE total_spent > 100000;
SELECT name, total_spent FROM user_stats ORDER BY total_spent DESC LIMIT 10;
```

**好處**：應用層不用知道有多複雜的 JOIN、只看到「user_stats 表」。


### 4. CREATE OR REPLACE VIEW（修改 view）

```text
CREATE OR REPLACE VIEW user_stats AS
SELECT
  u.id,
  u.name,
  COUNT(o.id) AS order_count,
  -- 加新欄位
  AVG(o.amount) AS avg_order
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.name;
```

**規則**：CREATE OR REPLACE 只能「加欄位、不能改型別、不能刪欄位」。要大改 → DROP VIEW + CREATE。


### 5. 刪 View

```text
DROP VIEW user_stats;
DROP VIEW IF EXISTS user_stats;

-- 刪除有 dependency 的
DROP VIEW user_stats CASCADE;  -- 其他 view 也跟著刪
```


### 6. View 能不能 INSERT / UPDATE？

**簡單 view（單表、無聚合）——可以**：

```text
CREATE VIEW active_users AS
SELECT id, name, email FROM users WHERE deleted_at IS NULL;

INSERT INTO active_users (name, email) VALUES ('A', 'a@x.com');
-- 等同插到 users 表
```

**複雜 view（JOIN / GROUP BY / DISTINCT）——不行**：

```text
INSERT INTO user_stats VALUES (...);
-- ❌ ERROR: cannot insert into view "user_stats"
```

**用 INSTEAD OF Trigger 變通**（L23 講）——能讓複雜 view 也能寫。


### 7. Materialized View —— 預存結果

**普通 View** = 每次查都重算（即時）。
**Materialized View** = 結果存起來、查的時候像查表一樣快。

```text
CREATE MATERIALIZED VIEW daily_revenue AS
SELECT
  DATE(created_at) AS day,
  COUNT(*) AS order_count,
  SUM(amount) AS revenue
FROM orders
GROUP BY DATE(created_at);

-- 查超快（已經算好存著）
SELECT * FROM daily_revenue WHERE day = '2026-05-28';
```

**重點**：Materialized View **不會自動更新**——資料變了、結果還是舊的、要手動 REFRESH：

```text
REFRESH MATERIALIZED VIEW daily_revenue;
```


### 8. CONCURRENTLY REFRESH（不卡查詢）

```text
REFRESH MATERIALIZED VIEW daily_revenue;
-- 預設：建期間其他人不能查 view（鎖）

REFRESH MATERIALIZED VIEW CONCURRENTLY daily_revenue;
-- 不鎖、其他人可以正常查（看到舊資料）
-- 條件：view 必須有 UNIQUE Index
```

**Production 一律用 CONCURRENTLY**——不影響其他查詢。

**先建 UNIQUE Index：**

```text
CREATE UNIQUE INDEX ON daily_revenue (day);
```


### 9. View vs Materialized View 對比

```text
View（即時）
   ✅ 永遠最新
   ✅ 寫入立刻反映
   ❌ 每次查都重算、複雜 view 慢

Materialized View（預存）
   ✅ 查超快（已存結果）
   ✅ 適合「不需要即時、但要快查」的報表
   ❌ 結果可能過時（取決於 REFRESH 頻率）
   ❌ 佔空間（存了一份結果）
   ❌ 要手動 REFRESH
```

**何時用哪個？**

```text
即時資料、查詢不頻繁 → View
即時資料、查詢頻繁但 view 簡單 → View
不需要即時、要查很快 → Materialized View（最常用）
即時 + 快 → 改用 trigger 維護 summary 表（自動更新）
```


### 10. 完整實戰：儀表板報表

```text
-- 建 materialized view、預先算好「過去 30 天每日營收」
CREATE MATERIALIZED VIEW dashboard_revenue AS
SELECT
  DATE(o.created_at) AS day,
  u.city,
  COUNT(DISTINCT o.user_id) AS unique_customers,
  COUNT(o.id) AS order_count,
  SUM(o.amount) AS revenue,
  AVG(o.amount) AS avg_order
FROM orders o
JOIN users u ON u.id = o.user_id
WHERE o.created_at >= NOW() - INTERVAL '30 days'
GROUP BY DATE(o.created_at), u.city;

-- 建 UNIQUE Index 支援 CONCURRENTLY REFRESH
CREATE UNIQUE INDEX ON dashboard_revenue (day, city);

-- 每 5 分鐘跑一次 refresh（用 cron / scheduler）
-- REFRESH MATERIALIZED VIEW CONCURRENTLY dashboard_revenue;

-- 應用層查就超快
SELECT * FROM dashboard_revenue
WHERE day >= NOW() - INTERVAL '7 days'
ORDER BY day DESC, revenue DESC;
```


### 11. View 的權限管理

```text
-- 給 reader 帳號只能看 view、不能直接查 users
GRANT SELECT ON active_users TO reader;
REVOKE ALL ON users FROM reader;

-- reader 只能：
SELECT * FROM active_users;
-- 不能：
SELECT * FROM users;  -- ❌ permission denied
```

**經典應用**：去敏感欄位的 view、給 BI 工具 / 第三方用。

```text
CREATE VIEW public_users AS
SELECT id, name, city FROM users;
-- email、phone 不暴露
```


### 12. WITH CHECK OPTION（防止寫入違反條件）

```text
CREATE VIEW taipei_users AS
SELECT * FROM users WHERE city = '台北'
WITH CHECK OPTION;

-- 透過 view INSERT 必須符合 view 條件
INSERT INTO taipei_users (name, city) VALUES ('A', '台北');  -- OK
INSERT INTO taipei_users (name, city) VALUES ('B', '台中');  -- ❌
-- ERROR: new row violates check option for view
```


### 13. View 的效能

```text
✅ 簡單 view（單表 SELECT）：
   PG 直接 inline、效能 = 直接寫 query

✅ 複雜 view（JOIN / GROUP）：
   也通常 inline、但優化空間小

❌ 巢狀 view（view 包 view 包 view）：
   優化器難以 inline、效能可能變差
   超過 3 層 nested view → 改 CTE 或 materialized
```


### 14. 5 大常見坑

```text
❌ 1. 以為 view 會「快取結果」
   → 普通 view 每次查都重算
   → 修：想快取 → Materialized View

❌ 2. Materialized View 沒 REFRESH 就期望即時
   → 永遠是建立 / REFRESH 當下的 snapshot
   → 修：用 cron 排程 REFRESH、或 trigger 自動

❌ 3. REFRESH 沒 CONCURRENTLY 鎖死 Production
   → 全部查詢卡住等 refresh 完
   → 修：必加 CONCURRENTLY（要先建 UNIQUE Index）

❌ 4. 巢狀 view 越包越多層
   → debug 地獄、效能崩潰
   → 修：超過 2 層 nested、改 CTE

❌ 5. 改 underlying table、view 沒跟著改
   → view 用到舊欄位名 → 查 view 報錯
   → 修：DROP CASCADE 或先 DROP VIEW 再改表再重建
```


### 🎯 重點回顧

- **View** = 一段 SELECT 的別名、像虛擬 table、不存資料、每次查都重算
- **CREATE OR REPLACE VIEW** 改 view（限「加欄位」）
- **Materialized View** = 預存結果、查超快、要手動 REFRESH
- **REFRESH MATERIALIZED VIEW CONCURRENTLY** 不影響其他查詢（要 UNIQUE Index）
- **WITH CHECK OPTION** 防止透過 view 寫入違反條件
- **View 用於權限管理**——去敏感欄位、給 reader
