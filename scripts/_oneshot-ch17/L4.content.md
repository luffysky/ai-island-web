**constraint（約束）= 在 DB 層擋住髒資料的保鑣**—— 上一課的 NOT NULL / DEFAULT 是熱身、本課把 **PRIMARY KEY、UNIQUE、CHECK、FOREIGN KEY** 4 大約束完整講透。

> 💡 **這課的價值**：應用層程式可以有 bug、DB 約束不會放水。在 DB 設好約束、髒資料**永遠進不來**——這是「資料正確性」的最後防線。


### 1. 為什麼一定要約束？

**反例：沒約束的 table——**

```text
CREATE TABLE bad_users (
  id    INT,
  email TEXT,
  age   INT
);

INSERT INTO bad_users VALUES (1, 'luffy@ai.com', 30);
INSERT INTO bad_users VALUES (1, 'luffy@ai.com', -5);    -- id 重複 + 年齡負數！
INSERT INTO bad_users VALUES (NULL, NULL, 200);          -- 全空 + 年齡 200！
```

**全部寫進去**——資料庫不擋、應用層忘了擋 = 髒資料永遠住下來。

**正解：加約束——**

```text
CREATE TABLE good_users (
  id    INT PRIMARY KEY,
  email TEXT UNIQUE NOT NULL,
  age   INT CHECK (age >= 0 AND age <= 150)
);

INSERT INTO good_users VALUES (1, 'luffy@ai.com', 30);   -- OK
INSERT INTO good_users VALUES (1, 'luffy@ai.com', 30);   -- ❌ id 重複
INSERT INTO good_users VALUES (2, 'luffy@ai.com', 30);   -- ❌ email 重複
INSERT INTO good_users VALUES (3, 'new@ai.com', -5);     -- ❌ 違反 CHECK
```

DB 自動擋下、應用層不用寫一堆 if-else。


### 2. PRIMARY KEY（主鍵）—— 每筆資料的身分證

**作用**：保證每筆 row 有「唯一識別」、其他 table 透過它找回來。

```text
✅ 一張表只能有一個 PRIMARY KEY
✅ 自動 NOT NULL（主鍵不能空）
✅ 自動 UNIQUE（主鍵不能重複）
✅ 自動建 Index（查超快）
```

**3 種寫法：**

```text
-- 寫法 1：欄位旁
CREATE TABLE users (
  id    SERIAL PRIMARY KEY,
  email TEXT
);

-- 寫法 2：最後獨立宣告（多欄位主鍵時用）
CREATE TABLE order_items (
  order_id  INT,
  product_id INT,
  qty       INT,
  PRIMARY KEY (order_id, product_id)   -- 複合主鍵
);

-- 寫法 3：UUID 當主鍵（2026 推薦）
CREATE TABLE users (
  id    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email TEXT
);
```

**SERIAL vs UUID 怎麼選？**

```text
SERIAL（自動 +1 的整數）
   優點：短、直覺、好 debug
   缺點：分散式撞號、洩漏「總共有幾筆」
   用在：小型專案、後台系統

UUID（隨機 128-bit）
   優點：分散式不撞、不洩漏總數、客戶端能先生
   缺點：長（36 字元）、肉眼不好認
   用在：SaaS、公開 API、分散式系統
```


### 3. UNIQUE（唯一）—— 這欄不可重複

```text
CREATE TABLE users (
  id    SERIAL PRIMARY KEY,
  email TEXT UNIQUE NOT NULL,   -- email 唯一
  phone TEXT UNIQUE              -- phone 唯一（但可空）
);

INSERT INTO users (email, phone) VALUES ('a@x.com', '0911');
INSERT INTO users (email, phone) VALUES ('a@x.com', '0922');
-- ❌ ERROR: duplicate key value violates unique constraint
```

**多欄位 UNIQUE（兩欄組合不可重複）：**

```text
CREATE TABLE follows (
  follower_id  INT,
  followee_id  INT,
  created_at   TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE (follower_id, followee_id)   -- 同一個人不能追蹤同一個人兩次
);
```

> 💡 **UNIQUE vs PRIMARY KEY 差別**：PRIMARY KEY = 表的「唯一身分證」、一張表只能一個；UNIQUE = 「這欄不可重複」、一張表可以有多個 UNIQUE 欄位。


### 4. CHECK —— 自訂規則

**作用**：寫一個布林表達式、DB 確保每筆 row 都通過。

```text
CREATE TABLE products (
  id     SERIAL PRIMARY KEY,
  name   TEXT NOT NULL,
  price  NUMERIC(10, 2) CHECK (price > 0),
  stock  INT CHECK (stock >= 0),
  status TEXT CHECK (status IN ('draft', 'published', 'sold_out'))
);

INSERT INTO products (name, price, stock, status)
  VALUES ('iPhone', 30000, 50, 'published');   -- OK

INSERT INTO products (name, price, stock, status)
  VALUES ('iPad', -1, 50, 'published');
-- ❌ violates check constraint "products_price_check"

INSERT INTO products (name, price, stock, status)
  VALUES ('AirPods', 5000, 10, 'unknown');
-- ❌ violates check constraint "products_status_check"
```

**多欄位 CHECK：**

```text
CREATE TABLE bookings (
  id         SERIAL PRIMARY KEY,
  starts_at  TIMESTAMPTZ NOT NULL,
  ends_at    TIMESTAMPTZ NOT NULL,
  CHECK (ends_at > starts_at)   -- 結束時間一定晚於開始時間
);
```

**常見 CHECK 模式：**

```text
CHECK (price > 0)
CHECK (age BETWEEN 0 AND 150)
CHECK (email ~ '^[^@]+@[^@]+\.[^@]+$')   -- 正則驗 email
CHECK (status IN ('todo', 'doing', 'done'))
CHECK (length(name) >= 2)
```


### 5. FOREIGN KEY（外鍵）—— 把這欄「指向別張表」

**作用**：保證 `posts.author_id` 一定對應到 `users.id` 中真的存在的人。

```text
CREATE TABLE users (
  id    SERIAL PRIMARY KEY,
  name  TEXT NOT NULL
);

CREATE TABLE posts (
  id        SERIAL PRIMARY KEY,
  author_id INT NOT NULL REFERENCES users(id),
  title     TEXT NOT NULL
);

INSERT INTO users (name) VALUES ('Luffy');   -- id = 1
INSERT INTO posts (author_id, title) VALUES (1, 'Hello');  -- OK
INSERT INTO posts (author_id, title) VALUES (999, 'Bye');  -- ❌
-- ERROR: insert or update on table "posts" violates foreign key constraint
```

**FOREIGN KEY 完整寫法：**

```text
CREATE TABLE posts (
  id        SERIAL PRIMARY KEY,
  author_id INT NOT NULL,
  title     TEXT NOT NULL,
  FOREIGN KEY (author_id) REFERENCES users(id)
);
```


### 6. 重點：ON DELETE / ON UPDATE 行為

**問題**：如果 users.id = 1 的人被刪掉、他寫的 posts 怎麼辦？

```text
CASCADE        users 刪、相關 posts 一起刪（最常用）
SET NULL       users 刪、posts.author_id 變 NULL（要允許 NULL）
SET DEFAULT    users 刪、posts.author_id 變 DEFAULT 值
RESTRICT       users 還有 posts 引用 → 不准刪（預設、安全）
NO ACTION      同 RESTRICT
```

**範例：**

```text
-- 文章跟著作者一起死
CREATE TABLE posts (
  id        SERIAL PRIMARY KEY,
  author_id INT REFERENCES users(id) ON DELETE CASCADE,
  title     TEXT
);

-- 留言保留、但 author_id 變 NULL（匿名）
CREATE TABLE comments (
  id        SERIAL PRIMARY KEY,
  author_id INT REFERENCES users(id) ON DELETE SET NULL,
  content   TEXT
);

-- 訂單一旦下、不准刪客戶（要先處理訂單）
CREATE TABLE orders (
  id          SERIAL PRIMARY KEY,
  customer_id INT REFERENCES users(id) ON DELETE RESTRICT,
  total       NUMERIC(10, 2)
);
```

> 💡 **新手原則**：**會跟父資料同生共死的用 CASCADE、要保留紀錄的用 SET NULL、重要資料用 RESTRICT**。


### 7. 完整範例：電商 schema

📄 **寫進程式檔**：`003_shop_schema.sql`

```text
-- 使用者
CREATE TABLE users (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email      TEXT UNIQUE NOT NULL CHECK (email ~ '^[^@]+@[^@]+\.[^@]+$'),
  name       TEXT NOT NULL CHECK (length(name) >= 2),
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 商品
CREATE TABLE products (
  id     UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  sku    TEXT UNIQUE NOT NULL,
  name   TEXT NOT NULL,
  price  NUMERIC(10, 2) NOT NULL CHECK (price >= 0),
  stock  INT NOT NULL DEFAULT 0 CHECK (stock >= 0),
  status TEXT NOT NULL DEFAULT 'draft'
         CHECK (status IN ('draft', 'published', 'sold_out'))
);

-- 訂單
CREATE TABLE orders (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL REFERENCES users(id) ON DELETE RESTRICT,
  total       NUMERIC(10, 2) NOT NULL CHECK (total >= 0),
  status      TEXT NOT NULL DEFAULT 'pending'
              CHECK (status IN ('pending', 'paid', 'shipped', 'delivered', 'cancelled')),
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- 訂單明細（複合主鍵）
CREATE TABLE order_items (
  order_id    UUID NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
  product_id  UUID NOT NULL REFERENCES products(id) ON DELETE RESTRICT,
  qty         INT NOT NULL CHECK (qty > 0),
  unit_price  NUMERIC(10, 2) NOT NULL CHECK (unit_price >= 0),
  PRIMARY KEY (order_id, product_id)
);
```

**這個 schema 做對的事：**

- UUID 主鍵（適合 SaaS）
- email 用 CHECK 驗格式 + UNIQUE 防重
- 金額 + 庫存 + 數量都 `>= 0`
- status 用 CHECK + IN 限制合法值（白名單）
- 訂單刪除 → 明細 CASCADE（一起刪）
- 商品被引用 → RESTRICT（不准刪）


### 8. 命名約束（讓錯誤訊息看得懂）

PG 自動取的名字像 `users_email_check`——還行、但長表時建議自己命名：

```text
CREATE TABLE users (
  id    SERIAL PRIMARY KEY,
  email TEXT NOT NULL,
  age   INT,
  CONSTRAINT users_email_unique UNIQUE (email),
  CONSTRAINT users_age_range CHECK (age BETWEEN 0 AND 150)
);
```

**好處**：違反時錯誤訊息是 `users_age_range`、一眼看出哪條規則。


### 9. 5 大常見坑

```text
❌ 1. FOREIGN KEY 忘了 ON DELETE
   → 父資料刪不掉、堆積一堆「孤兒」
   → 修：明確選 CASCADE / SET NULL / RESTRICT

❌ 2. CHECK 寫進 NULL 比較
   → NULL = anything 永遠是 NULL（不是 FALSE）
   → 修：CHECK (col IS NOT NULL OR ...) 配合 NOT NULL 處理

❌ 3. UNIQUE 多欄位、漏寫括號
   → UNIQUE col1, col2 變兩個獨立 UNIQUE
   → 修：UNIQUE (col1, col2) 一定要括號

❌ 4. PRIMARY KEY 用 email
   → email 會改、改了所有外鍵都炸
   → 修：永遠用 SERIAL 或 UUID 當主鍵、業務欄位用 UNIQUE

❌ 5. 約束加太晚、線上資料已違反
   → ALTER TABLE ADD CONSTRAINT 直接失敗
   → 修：先清髒資料、再加約束、或加 NOT VALID 跳過舊資料
```


### 🎯 重點回顧

- **PRIMARY KEY** = 每筆 row 的身分證、一表一個、自動 UNIQUE + NOT NULL
- **UNIQUE** = 這欄不可重複、可多個
- **CHECK** = 自訂規則（範圍、白名單、正則）
- **FOREIGN KEY** = 指向別表、配 ON DELETE 行為（CASCADE / SET NULL / RESTRICT）
- **約束在 DB 設一次、應用層少寫 100 行 if-else**
