**Trigger（觸發器）= 「資料變化時、自動跑一段 code」**—— `updated_at` 自動更新、audit log 自動寫、寫進去前自動加密——都靠 Trigger。配合 **Function（函式）** 是 PG 的隱藏神兵。

> 💡 **這課的價值**：Trigger 用得好、業務邏輯藏在 DB、應用層程式碼少一半。但用得太兇、邏輯藏太深、debug 地獄。本課教你「該用」跟「不該用」的分界。


### 1. Trigger 是什麼？

**Trigger = 監聽資料表的事件、發生時自動跑 function**。

**監聽事件：**

```text
INSERT     有新列寫入
UPDATE     有列被改
DELETE     有列被刪
TRUNCATE   表被 truncate

時機（BEFORE / AFTER）：
   BEFORE INSERT   寫入前（可改要寫的資料）
   AFTER INSERT    寫入後（紀錄 log 用）
   BEFORE UPDATE   ...
   AFTER UPDATE    ...
```


### 2. 經典應用 1：自動更新 updated_at

**問題**：每次 UPDATE 都要記得 `SET updated_at = NOW()`、容易忘。

**解法**：用 trigger 自動做。

```text
-- 1. 建一個 function（要做的事）
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- 2. 在 users 表上掛 trigger
CREATE TRIGGER users_updated_at
BEFORE UPDATE ON users
FOR EACH ROW
EXECUTE FUNCTION update_updated_at();
```

**之後不管哪裡 UPDATE users、updated_at 自動變新時間**——應用層不用管。

**逐部分翻譯：**

```text
CREATE OR REPLACE FUNCTION   建一個 function（可覆蓋）
RETURNS TRIGGER              這 function 給 trigger 用
NEW                          代表「即將要寫進去的列」
                              OLD = 改之前的列、DELETE 時用
BEFORE UPDATE                UPDATE 之前跑
FOR EACH ROW                 每改一列、跑一次（也可 FOR EACH STATEMENT）
```


### 3. 經典應用 2：Audit Log（誰改了什麼）

```text
CREATE TABLE audit_log (
  id          SERIAL PRIMARY KEY,
  table_name  TEXT,
  row_id      INT,
  action      TEXT,
  old_data    JSONB,
  new_data    JSONB,
  changed_at  TIMESTAMPTZ DEFAULT NOW(),
  changed_by  TEXT
);

CREATE OR REPLACE FUNCTION audit_users()
RETURNS TRIGGER AS $$
BEGIN
  IF TG_OP = 'INSERT' THEN
    INSERT INTO audit_log (table_name, row_id, action, new_data, changed_by)
      VALUES ('users', NEW.id, 'INSERT', row_to_json(NEW), current_user);
    RETURN NEW;
  ELSIF TG_OP = 'UPDATE' THEN
    INSERT INTO audit_log (table_name, row_id, action, old_data, new_data, changed_by)
      VALUES ('users', NEW.id, 'UPDATE', row_to_json(OLD), row_to_json(NEW), current_user);
    RETURN NEW;
  ELSIF TG_OP = 'DELETE' THEN
    INSERT INTO audit_log (table_name, row_id, action, old_data, changed_by)
      VALUES ('users', OLD.id, 'DELETE', row_to_json(OLD), current_user);
    RETURN OLD;
  END IF;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER users_audit
AFTER INSERT OR UPDATE OR DELETE ON users
FOR EACH ROW EXECUTE FUNCTION audit_users();
```

**現在 users 表的任何變化、都會自動記到 audit_log**。


### 4. NEW / OLD / TG_OP 等特殊變數

```text
NEW              新資料（INSERT / UPDATE 時）
OLD              舊資料（UPDATE / DELETE 時）
TG_OP            操作類型：'INSERT' / 'UPDATE' / 'DELETE' / 'TRUNCATE'
TG_NAME          trigger 名字
TG_TABLE_NAME    哪張表
TG_WHEN          'BEFORE' / 'AFTER'
TG_LEVEL         'ROW' / 'STATEMENT'
```


### 5. BEFORE vs AFTER 差別

```text
BEFORE INSERT/UPDATE
   能改 NEW 的內容（return NEW 改過的）
   能 RETURN NULL 取消這次操作
   例：寫入前加密密碼

AFTER INSERT/UPDATE
   NEW 已經寫進去、不能改了
   適合「紀錄 log」、「通知 webhook」
   例：audit log
```

**範例：BEFORE 自動加密密碼：**

```text
CREATE OR REPLACE FUNCTION hash_password()
RETURNS TRIGGER AS $$
BEGIN
  IF NEW.password_hash IS NOT NULL AND NEW.password_hash <> OLD.password_hash THEN
    NEW.password_hash = crypt(NEW.password_hash, gen_salt('bf'));
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER users_hash_password
BEFORE INSERT OR UPDATE ON users
FOR EACH ROW EXECUTE FUNCTION hash_password();
```


### 6. ROW 級 vs STATEMENT 級

```text
FOR EACH ROW
   每改一列、trigger 跑一次
   有 NEW / OLD 可用
   最常用

FOR EACH STATEMENT
   每句 SQL、trigger 跑一次
   不管影響幾列、只跑 1 次
   沒 NEW / OLD（不知道是哪列）
   適合「整體統計」、「通知一次就好」
```


### 7. Function（PL/pgSQL）基本語法

```text
CREATE OR REPLACE FUNCTION 函式名(參數 型別, ...)
RETURNS 回傳型別 AS $$
DECLARE
  -- 宣告區
  my_var INT;
BEGIN
  -- 邏輯區
  my_var := 5;

  IF my_var > 0 THEN
    RETURN my_var * 2;
  ELSE
    RETURN 0;
  END IF;
END;
$$ LANGUAGE plpgsql;
```

**呼叫：**

```text
SELECT 函式名(參數);
```


### 8. Function 範例：計算 user 等級

```text
CREATE OR REPLACE FUNCTION calc_user_tier(p_total NUMERIC)
RETURNS TEXT AS $$
BEGIN
  IF p_total >= 100000 THEN RETURN 'platinum';
  ELSIF p_total >= 50000 THEN RETURN 'gold';
  ELSIF p_total >= 10000 THEN RETURN 'silver';
  ELSE RETURN 'bronze';
  END IF;
END;
$$ LANGUAGE plpgsql IMMUTABLE;

-- 用法
SELECT name, total_spent, calc_user_tier(total_spent) AS tier
FROM users;
```

**`IMMUTABLE`** 標記讓 PG 知道「同樣 input 永遠同樣 output」——可以快取結果。


### 9. 經典應用 3：自動算金額（INSERT 時帶上）

```text
CREATE OR REPLACE FUNCTION calc_order_total()
RETURNS TRIGGER AS $$
BEGIN
  NEW.total := NEW.qty * NEW.unit_price;
  NEW.tax   := NEW.total * 0.05;
  NEW.grand_total := NEW.total + NEW.tax;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER order_items_calc
BEFORE INSERT OR UPDATE ON order_items
FOR EACH ROW EXECUTE FUNCTION calc_order_total();

-- 應用層只要 INSERT qty + unit_price、total / tax / grand_total 自動算
INSERT INTO order_items (order_id, product_id, qty, unit_price)
VALUES (1, 5, 3, 1000);
-- total=3000、tax=150、grand_total=3150 自動填好
```


### 10. INSTEAD OF Trigger（給 view 用）

普通 view 不能 INSERT / UPDATE—用 INSTEAD OF trigger 變通：

```text
CREATE VIEW user_with_profile AS
SELECT u.id, u.name, p.bio FROM users u LEFT JOIN profiles p ON ...;

-- 透過 view INSERT 不行 → 用 INSTEAD OF trigger 攔截
CREATE OR REPLACE FUNCTION insert_user_with_profile()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO users (name) VALUES (NEW.name) RETURNING id INTO NEW.id;
  INSERT INTO profiles (user_id, bio) VALUES (NEW.id, NEW.bio);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER tr_insert_user_with_profile
INSTEAD OF INSERT ON user_with_profile
FOR EACH ROW EXECUTE FUNCTION insert_user_with_profile();
```


### 11. Trigger 的優缺點

```text
✅ 優點
   - 業務邏輯集中在 DB、應用層不會漏
   - audit、updated_at 自動化
   - 多個應用程式共用同樣邏輯

❌ 缺點
   - 邏輯藏在 DB、新人不知道有 trigger
   - debug 困難（query 跑出意料外結果、是 trigger 改的）
   - 大量 trigger 拖效能
   - 跨 DB migration 麻煩
```

**業界共識**：

```text
✅ 用 trigger 的好場景
   - updated_at 自動更新
   - audit log
   - 計算欄位自動填
   - 維護 cache 表

❌ 不要用 trigger 的場景
   - 複雜業務邏輯（放應用層、易維護）
   - 外部呼叫（HTTP、send mail）
   - 跨 transaction 的事
```


### 12. 刪 Trigger / Function

```text
DROP TRIGGER trigger_name ON table_name;
DROP FUNCTION function_name(arg_types);

-- IF EXISTS
DROP TRIGGER IF EXISTS users_updated_at ON users;
```


### 13. 5 大常見坑

```text
❌ 1. Trigger 寫得太重、拖慢 INSERT/UPDATE
   → 每筆寫入 +50ms
   → 修：trigger 邏輯精簡、複雜邏輯放應用層

❌ 2. 多個 trigger 互相觸發、無限遞迴
   → trigger A 改 B 表觸發 trigger B 改 A 表 ...
   → 修：trigger 不要互相寫表、加終止條件

❌ 3. AFTER trigger 試圖改 NEW
   → AFTER 階段資料已寫入、改不了
   → 修：要改用 BEFORE

❌ 4. BEFORE trigger return NULL 沒留意
   → return NULL = 取消這次操作
   → 修：永遠 return NEW、除非真的要擋

❌ 5. Trigger 內呼叫外部 API
   → API 慢 → INSERT 卡住、transaction 拖長
   → 修：trigger 寫到 queue table、之後 worker 處理
```


### 🎯 重點回顧

- **Trigger = 資料變化時自動跑 function**、最常用 `updated_at` 跟 audit log
- **BEFORE / AFTER + INSERT / UPDATE / DELETE** 組合出 6 種時機
- **NEW / OLD / TG_OP** 是特殊變數
- **FOR EACH ROW** 最常用（每列跑）
- **PL/pgSQL** 是 PG 的 procedural 語言、寫 function 用
- **INSTEAD OF Trigger** 讓複雜 view 也能寫入
- **Trigger 不要太重、不要呼叫外部 API**
