**CTE（Common Table Expression、共同資料表表達式）= `WITH` 開頭、把複雜 SQL 拆成多段**—— 像是程式裡的「先存變數、再用」。長 SQL 一旦學會 CTE、可讀性立刻飆升。

> 💡 **這課的價值**：實務 SQL 動不動 50 行、嵌 3 層 subquery 像義大利麵。CTE 一加上、整段 SQL 變成「步驟 1 → 步驟 2 → 步驟 3」、別人秒懂、自己半年後也讀得懂。


### 1. CTE 是什麼？

**CTE = 「給這段 query 取個名字、後面當 table 用」**。

**基本語法：**

```text
WITH 名字 AS (
  SELECT ...
)
SELECT ... FROM 名字;
```

**範例：**

```text
WITH active_users AS (
  SELECT id, name, email
  FROM users
  WHERE last_login_at > NOW() - INTERVAL '30 days'
)
SELECT COUNT(*) FROM active_users;
```

**讀法**：「把活躍使用者命名為 `active_users`、然後算他們的數量」。


### 2. 為什麼 CTE 比 subquery 好讀？

**寫法 A：嵌套 subquery（像義大利麵）：**

```text
SELECT *
FROM (
  SELECT user_id, COUNT(*) AS orders, SUM(amount) AS total
  FROM (
    SELECT * FROM orders
    WHERE created_at >= '2026-01-01'
  ) recent_orders
  GROUP BY user_id
) user_stats
WHERE total > 10000
ORDER BY total DESC;
```

**寫法 B：CTE 拆解（清楚步驟）：**

```text
WITH recent_orders AS (
  SELECT * FROM orders
  WHERE created_at >= '2026-01-01'
),
user_stats AS (
  SELECT user_id, COUNT(*) AS orders, SUM(amount) AS total
  FROM recent_orders
  GROUP BY user_id
)
SELECT * FROM user_stats
WHERE total > 10000
ORDER BY total DESC;
```

**B 的好處：**

```text
✅ 由上而下讀、像看食譜「步驟 1 → 步驟 2 → 最後組合」
✅ 每段都有名字、知道在做什麼
✅ debug 時可以單獨跑某一段（SELECT * FROM recent_orders）
✅ 同一段可以被後面多次引用
```


### 3. 多個 CTE 串接

CTE 可以一個接一個、用逗號分隔：

```text
WITH
  step1 AS (
    SELECT ... FROM ...
  ),
  step2 AS (
    SELECT ... FROM step1 ...     -- 可以引用 step1
  ),
  step3 AS (
    SELECT ... FROM step2 ...     -- 可以引用 step1 和 step2
  )
SELECT * FROM step3;
```

**思考方式**：像寫程式的「變數、函式」、一步步建構最後答案。


### 4. CTE vs subquery 何時用哪個？

```text
CTE
   ✅ 步驟超過 2 段
   ✅ 同一段要被引用多次
   ✅ 寫 Recursive query（後面講）
   ✅ 給別人讀的、需要可讀性

Subquery（簡單一行的）
   ✅ 一次性、簡單
   ✅ 在 WHERE / SELECT 裡用單值
   例：WHERE user_id IN (SELECT id FROM users WHERE city='台北')
```

**業界共識**：**長 query、用 CTE；單行子查詢、用 subquery**。


### 5. Subquery 三大類型（複習一下）

```text
1. Scalar subquery（傳回 1 列 1 欄）
   SELECT name, (SELECT COUNT(*) FROM posts WHERE author_id = u.id) AS posts
   FROM users u;

2. Row subquery（傳回 1 列多欄）
   WHERE (city, age) IN (SELECT city, age FROM vips);

3. Table subquery（傳回多列多欄）
   SELECT * FROM (SELECT ... FROM ...) AS sub;
```

**SQL 也認 Correlated subquery（相關子查詢）—— 內層用到外層的值：**

```text
SELECT name,
  (SELECT COUNT(*) FROM posts p WHERE p.author_id = u.id) AS post_count
FROM users u;
-- 內層 u.id 用到外層的 u、是 correlated
```

**注意**：correlated subquery 對外層每列都跑一次、大表會慢、能用 JOIN 改寫就改寫。


### 6. Recursive CTE（遞迴 CTE）—— 處理階層資料

**問題**：員工表有 boss_id 指向 boss、要找出「Luffy 底下所有員工（含下下層）」。

```text
CREATE TABLE employees (
  id      INT PRIMARY KEY,
  name    TEXT,
  boss_id INT REFERENCES employees(id)
);

INSERT INTO employees VALUES
  (1, 'Luffy',  NULL),
  (2, 'Zoro',   1),
  (3, 'Nami',   1),
  (4, 'Sanji',  2),
  (5, 'Robin',  3);
```

**Recursive CTE 寫法：**

```text
WITH RECURSIVE org AS (
  -- 起點：Luffy 自己
  SELECT id, name, boss_id, 0 AS level
  FROM employees
  WHERE id = 1

  UNION ALL

  -- 遞迴：找下屬
  SELECT e.id, e.name, e.boss_id, org.level + 1
  FROM employees e
  JOIN org ON e.boss_id = org.id
)
SELECT * FROM org;
```

💬 **電腦會顯示**：

```text
 id |  name  | boss_id | level
----+--------+---------+-------
  1 | Luffy  |    NULL |     0
  2 | Zoro   |       1 |     1
  3 | Nami   |       1 |     1
  4 | Sanji  |       2 |     2
  5 | Robin  |       3 |     2
```

**Recursive CTE 結構：**

```text
WITH RECURSIVE 名字 AS (
  起始查詢              ← 起點（base case）
  UNION ALL
  遞迴查詢              ← 用上次結果再查
)
```

**應用場景：**

- 公司組織樹
- 留言層級（巢狀 reply）
- 商品分類樹
- 路徑追蹤（社群網路找朋友的朋友）


### 7. CTE 在 INSERT / UPDATE / DELETE 也能用

**範例：把「30 天沒登入的」標記為 inactive：**

```text
WITH inactive AS (
  SELECT id FROM users
  WHERE last_login_at < NOW() - INTERVAL '30 days'
)
UPDATE users
SET status = 'inactive'
WHERE id IN (SELECT id FROM inactive);
```

**範例：用 CTE 同時 INSERT 兩張表（PG 專屬 `WITH ... INSERT`）：**

```text
WITH new_user AS (
  INSERT INTO users (name, email) VALUES ('Luffy', 'l@ai.com')
  RETURNING id
)
INSERT INTO profiles (user_id, bio)
SELECT id, '海賊王' FROM new_user;
```


### 8. CTE 效能注意（PG 12+ 的優化）

**PG 12 之前**：CTE 是「優化柵欄（optimization fence）」——優化器不會跨 CTE 優化、有時反而慢。

**PG 12+**：預設 inline（內聯到 main query）、跟 subquery 一樣快。

**強制 inline / materialized：**

```text
WITH stats AS NOT MATERIALIZED ( ... )   -- 強制 inline（PG 自動會這樣）
WITH stats AS MATERIALIZED ( ... )       -- 強制獨立執行 + 暫存結果
```

**何時用 MATERIALIZED？**這段被引用多次、計算很重時——讓它只跑一次、結果重用。


### 9. 完整實戰：算出「過去 30 天、消費前 10 名 user」+ 占比

```text
WITH recent AS (
  SELECT user_id, amount
  FROM orders
  WHERE created_at >= NOW() - INTERVAL '30 days'
),
user_totals AS (
  SELECT user_id, SUM(amount) AS spent
  FROM recent
  GROUP BY user_id
),
total_revenue AS (
  SELECT SUM(amount) AS grand_total FROM recent
)
SELECT
  ut.user_id,
  ut.spent,
  ROUND(ut.spent / tr.grand_total * 100, 2) AS pct
FROM user_totals ut, total_revenue tr
ORDER BY ut.spent DESC
LIMIT 10;
```

**讀起來像論文章節：**
1. recent：近 30 天訂單
2. user_totals：每個 user 的消費
3. total_revenue：總營收
4. 最後：算占比、排名取前 10


### 10. 5 大常見坑

```text
❌ 1. CTE 寫了沒用（孤兒 CTE）
   → 不會報錯、但浪費效能
   → 修：最後 SELECT 一定要 FROM 用到的 CTE

❌ 2. Recursive 沒終止條件
   → 無限遞迴、PG 直接吃光記憶體
   → 修：base case 要明確、遞迴步驟要逐漸收斂

❌ 3. 把 CTE 當「臨時 table」期待多次查詢
   → CTE 只在當前 query 內有效、下一句 SELECT 看不到
   → 修：要跨 query 用、改成 TEMP TABLE 或 VIEW

❌ 4. 大量 CTE + correlated subquery
   → 對大表效能炸裂
   → 修：先用 EXPLAIN 看執行計畫、必要時改 JOIN

❌ 5. CTE 名字撞到 table 名字
   → CTE 會「遮蔽」同名 table（看 CTE 不看 table）
   → 修：CTE 名字加前綴 cte_ 或意義詞
```


### 🎯 重點回顧

- **CTE = WITH 開頭、把複雜 SQL 拆成步驟**、可讀性飆升
- **多個 CTE 串接**用逗號、後面可引用前面
- **CTE vs subquery**：長 query 用 CTE、單行用 subquery
- **Recursive CTE** 處理階層資料（組織樹、留言層級）
- **CTE 也能配 INSERT / UPDATE / DELETE** 用
- **PG 12+ CTE 跟 subquery 速度一樣**、放心用
