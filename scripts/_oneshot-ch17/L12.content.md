**UPDATE（更新）= 修改已存在的列**—— 看似簡單、實則最危險。**沒寫 WHERE 的 UPDATE = 整張表被改**——本課把這個地雷拆乾淨。

> 💡 **這課的價值**：每個工程師職涯都會有一次「UPDATE 忘了 WHERE、整張表炸了」的恐怖故事。本課教你 safe update mode、JOIN UPDATE、RETURNING——既能更新得快又能更新得安全。


### 1. 最基本：UPDATE + WHERE

```text
UPDATE 表名
SET 欄位 = 新值
WHERE 條件;
```

**範例：**

```text
UPDATE users
SET age = 31
WHERE id = 1;
```

💬 **電腦會顯示**：

```text
UPDATE 1
-- 影響 1 列
```


### 2. 沒寫 WHERE 的災難

```text
UPDATE users SET age = 0;
-- ❌ 全表的 age 都變 0
-- ❌ 沒辦法 undo（除非有備份）
```

**這是 DBA 噩夢**。**永遠先寫 WHERE、再寫 SET**——練成肌肉記憶。


### 3. Safe Update Mode（保險開關）

**Supabase / pgAdmin / DBeaver 都可以開「safe update mode」**——沒有 WHERE 直接拒絕執行。

**手動模擬：先用 BEGIN + ROLLBACK 試水溫：**

```text
BEGIN;                                           -- 開始 transaction

UPDATE users SET age = 0;                        -- 試跑
SELECT COUNT(*) FROM users WHERE age = 0;        -- 看影響幾筆

-- 看起來不對 → 退回：
ROLLBACK;

-- 看起來沒問題（含 WHERE）→ 提交：
COMMIT;
```

L21 會完整講 Transaction、這裡先學「BEGIN → 試 → ROLLBACK 或 COMMIT」這個保命招。


### 4. UPDATE 多個欄位

```text
UPDATE users
SET
  name = 'Luffy Lin',
  age = 31,
  city = '東京',
  updated_at = NOW()
WHERE id = 1;
```

**用逗號分隔多個 SET、永遠記得更新 `updated_at`**。


### 5. UPDATE 用「自身值」

```text
-- 全表 views +1
UPDATE posts SET views = views + 1;

-- 特定文章 views +1
UPDATE posts SET views = views + 1 WHERE id = 5;

-- 把 email 轉小寫
UPDATE users SET email = LOWER(email);

-- 名字加前綴
UPDATE users SET name = '王 ' || name WHERE city = '台北';
```


### 6. UPDATE 配 CASE WHEN（多條件分支）

**範例：根據訂單金額升級會員等級：**

```text
UPDATE users
SET tier = CASE
  WHEN total_spent >= 100000 THEN 'platinum'
  WHEN total_spent >= 50000  THEN 'gold'
  WHEN total_spent >= 10000  THEN 'silver'
  ELSE 'bronze'
END;
```

**一句 SQL 處理 4 種情況**——比 4 個 UPDATE 跑 4 次快很多。


### 7. UPDATE ... FROM（JOIN UPDATE、PG 寫法）

**問題**：把每個 user 的 `total_spent` 算出來、更新到 users 表。

```text
UPDATE users u
SET total_spent = sub.total
FROM (
  SELECT user_id, SUM(amount) AS total
  FROM orders
  GROUP BY user_id
) sub
WHERE u.id = sub.user_id;
```

**結構：**

```text
UPDATE 目標表 別名
SET 欄位 = 別表.欄位
FROM 別表（或 subquery）
WHERE 目標表.key = 別表.key;
```

**MySQL 不同寫法**：

```text
UPDATE users u
JOIN orders o ON u.id = o.user_id
SET u.total_spent = u.total_spent + o.amount;
-- PG 不支援這種寫法、要用上面的 FROM
```


### 8. RETURNING（更新後立刻拿結果）

```text
UPDATE orders
SET status = 'shipped', shipped_at = NOW()
WHERE id = 1234
RETURNING id, status, shipped_at;
```

💬 **電腦會顯示**：

```text
  id  | status  |        shipped_at
------+---------+----------------------------
 1234 | shipped | 2026-05-28 14:30:00.123+00
```

**API 寫法**：UPDATE + RETURNING 回給前端、不用多一次 SELECT。


### 9. UPDATE 配 subquery

**例：把「30 天沒登入的」標記 inactive：**

```text
UPDATE users
SET status = 'inactive'
WHERE last_login_at < NOW() - INTERVAL '30 days';
```

**例：用 EXISTS 判斷：**

```text
UPDATE posts
SET is_hot = TRUE
WHERE EXISTS (
  SELECT 1 FROM likes
  WHERE likes.post_id = posts.id
  GROUP BY likes.post_id
  HAVING COUNT(*) > 100
);
```


### 10. WITH ... UPDATE（CTE + UPDATE）

```text
WITH top_users AS (
  SELECT id FROM users
  ORDER BY total_spent DESC
  LIMIT 100
)
UPDATE users
SET tier = 'vip'
WHERE id IN (SELECT id FROM top_users);
```

**好處**：複雜更新邏輯拆步驟、更好讀。


### 11. UPDATE 影響筆數的回傳

```text
UPDATE users SET age = 31 WHERE id = 1;
-- "UPDATE 1"

UPDATE users SET age = 31 WHERE city = '不存在';
-- "UPDATE 0"  ← 0 筆受影響（不報錯）
```

**注意**：影響 0 筆不會報錯。**應用層要自己判斷**：

```text
const result = await db.query(
  'UPDATE users SET name = $1 WHERE id = $2',
  ['NewName', userId]
);

if (result.rowCount === 0) {
  throw new Error('使用者不存在或沒權限');
}
```


### 12. 完整實戰：訂單狀態流轉

```text
-- 把「待付款超過 7 天」的訂單取消
UPDATE orders
SET
  status = 'cancelled',
  cancelled_at = NOW(),
  cancel_reason = '超時未付款'
WHERE status = 'pending'
  AND created_at < NOW() - INTERVAL '7 days'
RETURNING id, user_id;
```

**回傳取消的訂單 id + user_id**——可以接著發通知信給用戶。

```text
-- 配合 transaction（L21）：
BEGIN;

UPDATE products SET stock = stock - 1
WHERE id = $1 AND stock > 0
RETURNING stock;

-- 如果回 0 筆 → 庫存不夠、ROLLBACK
-- 否則繼續：
INSERT INTO orders (...) VALUES (...);

COMMIT;
```


### 13. soft update（軟更新、不真的刪）

**經典做法**：不真的 DELETE、改設 `deleted_at`：

```text
UPDATE users
SET deleted_at = NOW()
WHERE id = 123;

-- 查資料時都加：
SELECT * FROM users WHERE deleted_at IS NULL;
```

**好處**：誤刪可救、有審計軌跡。**壞處**：每個 query 都要記得加 `deleted_at IS NULL`——容易忘。L13 會詳講 soft vs hard delete。


### 14. 5 大常見坑

```text
❌ 1. UPDATE 沒寫 WHERE
   → 全表被改、災難
   → 修：先 BEGIN、跑完 ROLLBACK、確認對才 COMMIT

❌ 2. UPDATE ... = 一個 subquery 撞 NULL
   → SET col = (SELECT ... WHERE not exists)、col 變 NULL
   → 修：COALESCE 給預設值、或加 EXISTS 判斷

❌ 3. 大量 UPDATE 沒分批
   → 一次 UPDATE 100 萬列、lock 整張表
   → 修：分批 UPDATE（每次 1000 筆配 WHERE id BETWEEN）

❌ 4. UPDATE 改 PRIMARY KEY
   → 外鍵 CASCADE 一連串更新、可能踩坑
   → 修：盡量不改主鍵、要改用 ON UPDATE CASCADE

❌ 5. 忘了更新 updated_at
   → audit 紀錄不準
   → 修：用 Trigger 自動更新（L23）或全部 query 記得加
```


### 🎯 重點回顧

- **UPDATE 表 SET 欄位 = 值 WHERE 條件**——WHERE 是保命關鍵
- **沒 WHERE = 全表被改**——永遠先 BEGIN 試水溫
- **SET 多欄位**用逗號分隔、別忘 `updated_at = NOW()`
- **UPDATE ... FROM** 是 PG 的 JOIN UPDATE 寫法
- **RETURNING** 拿回更新後的值、省一次 SELECT
- **CASE WHEN** 一句處理多分支邏輯
- **大表分批 UPDATE**、不要一次改百萬筆
