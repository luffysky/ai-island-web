**Index（索引）= 資料庫的「目錄」**—— 沒 Index 查 100 萬筆要掃 100 萬筆、有 Index 直接跳到位、效能差 1000 倍。本課把 B-tree 原理、加 Index 時機、EXPLAIN 看實際用了沒一次講透。

> 💡 **這課的價值**：Production 慢查詢 80% 的原因是「沒加 Index」、20% 是「Index 加錯」。會這課你的 DB 速度直接從烏龜變高鐵——而且不用改一行業務 code。


### 1. 為什麼需要 Index？

**反例：1000 萬筆 user、找 email = 'luffy@ai.com'：**

```text
SELECT * FROM users WHERE email = 'luffy@ai.com';
-- 沒 Index：從第 1 筆掃到第 1000 萬筆、找完所有 → 10 秒
-- 有 Index：直接跳到目標位置 → 10 毫秒
```

**比喻**：
- 沒 Index = 翻一本 1000 頁的字典、每頁看「有沒有這個字」
- 有 Index = 用注音目錄、直接翻到 P 開頭那頁


### 2. Index 怎麼運作（B-tree）

**B-tree（B 樹）** 是 99% 的 Index 用的資料結構。簡化長這樣：

```text
                  [50]
                 /     \
              [25]    [75]
             /   \    /   \
          [10] [30][60] [90]
          /\   /\  /\   /\
        ... ... ... ... ...
```

**找 email = 'luffy@ai.com' 過程：**

```text
1. 從根節點開始
2. 跟中間值比較、決定往左還往右
3. 每次「分一半」、log₂(1000 萬) ≈ 23 步就找到
4. 100 萬筆只要 20 步、1 億筆 27 步

→ 這就是為什麼 Index 快得誇張
```

**結論**：B-tree 讓「找特定值」變成 O(log n)、而不是 O(n)。


### 3. 加 Index 語法

```text
CREATE INDEX idx_users_email ON users(email);
```

**逐部分翻譯：**

```text
CREATE INDEX     建索引
idx_users_email  自取的索引名（慣例：idx_表_欄位）
ON users(email)  針對 users 表的 email 欄位
```

**確認**：

```text
\d users           -- psql 看 table、會顯示 Indexes 區段
```


### 4. 複合 Index（多欄位）

```text
CREATE INDEX idx_orders_user_created
ON orders(user_id, created_at);
```

**這個 Index 對哪些 query 有用？**

```text
✅ WHERE user_id = 1
✅ WHERE user_id = 1 AND created_at > '2026-01-01'
✅ WHERE user_id = 1 ORDER BY created_at

❌ WHERE created_at > '2026-01-01'   ← 沒帶 user_id、用不上
```

**「左前綴原則（leftmost prefix）」**：複合 Index 只能用「從左邊開始的欄位組合」。

**順序很重要**：

```text
INDEX (user_id, created_at)
   user_id 為主、created_at 為輔

INDEX (created_at, user_id)
   created_at 為主、user_id 為輔
   → 適合 WHERE created_at = '...' 為主的 query
```


### 5. 什麼欄位該加 Index？

```text
✅ 加 Index 的欄位
   - PRIMARY KEY（PG 自動加）
   - FOREIGN KEY（用來 JOIN）
   - 常出現在 WHERE 的（email、status、city...）
   - 常用來 ORDER BY 的（created_at）
   - UNIQUE 約束欄位（自動有 Index）

❌ 不要加 Index 的
   - 值很少種類的（性別 M/F 只兩個值、Index 沒用）
   - 經常更新的欄位（每次 UPDATE 也要更新 Index、慢）
   - 小表（PG 直接掃表就行、Index 反而慢）
   - 永遠不在 WHERE 出現的（不會用到）
```

> 💡 **新手原則**：**FK 永遠加 Index、常 WHERE 的加 Index、不確定不要加**。Index 多 = 寫入慢、佔空間。


### 6. EXPLAIN —— 看 PG 「打算怎麼跑」這 query

```text
EXPLAIN SELECT * FROM users WHERE email = 'luffy@ai.com';
```

💬 **電腦會顯示**（沒 Index 時）：

```text
Seq Scan on users  (cost=0.00..18334.00 rows=1 width=...)
  Filter: (email = 'luffy@ai.com')
```

**`Seq Scan`** = 順序掃描、從頭掃到尾——慢。

💬 **電腦會顯示**（加 Index 後）：

```text
Index Scan using idx_users_email on users  (cost=0.42..8.44 rows=1 width=...)
  Index Cond: (email = 'luffy@ai.com')
```

**`Index Scan`** = 用 Index 直接定位——快。


### 7. EXPLAIN ANALYZE —— 真的跑、看實際時間

```text
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'luffy@ai.com';
```

💬 **電腦會顯示**：

```text
Index Scan using idx_users_email on users
  (cost=0.42..8.44 rows=1 width=...)
  (actual time=0.123..0.125 rows=1 loops=1)
Planning Time: 0.234 ms
Execution Time: 0.156 ms
```

**重點看 `Execution Time`**——真實跑的時間。

L20 整課講 EXPLAIN、這裡先有概念。


### 8. UNIQUE Index（兼具約束 + 加速）

```text
CREATE UNIQUE INDEX idx_users_email_unique ON users(email);
-- 等同 ALTER TABLE users ADD CONSTRAINT ... UNIQUE (email);
-- 同時：
--   1. 確保 email 唯一
--   2. 查 email 變快
```


### 9. Partial Index（條件 Index）

只對「符合條件的列」建 Index、省空間：

```text
-- 只對「未刪除」的 user 建 Index
CREATE INDEX idx_users_active_email
ON users(email)
WHERE deleted_at IS NULL;

-- 只對「pending」訂單建 Index（其他狀態不查）
CREATE INDEX idx_orders_pending
ON orders(created_at)
WHERE status = 'pending';
```

**好處**：Index 小、查詢更快、省記憶體。


### 10. 函式 Index / 表達式 Index

對「函式結果」建 Index：

```text
-- email 大小寫不敏感查詢
CREATE INDEX idx_users_lower_email ON users(LOWER(email));

-- 之後查：
SELECT * FROM users WHERE LOWER(email) = 'luffy@ai.com';
-- 用得到 Index
```


### 11. 其他 Index 類型（PG 專屬）

```text
B-tree（預設）
   95% 場景用這個
   範圍查、等於查、ORDER BY 都行

GIN
   給 JSONB / Array / 全文檢索用
   CREATE INDEX ... USING GIN (tags)

GiST
   給地理空間 / 範圍型別用
   PostGIS 必備

BRIN
   超大表（億級）、單調遞增的欄位（時間戳）
   省空間、不太精準

Hash
   等於查超快、但範圍查不行
   B-tree 通常勝出、Hash 少用
```


### 12. 刪 Index

```text
DROP INDEX idx_users_email;

-- 不存在也不報錯
DROP INDEX IF EXISTS idx_users_email;
```

**何時要刪？**

- Index 加錯欄位（用不到）
- Index 太多拖慢寫入
- Migrate schema 時暫時刪、之後重建


### 13. Index 的代價

```text
✅ 讀變快（SELECT 大幅加速）

❌ 寫變慢
   INSERT / UPDATE / DELETE 都要更新 Index
   一張表 10 個 Index = 寫入慢 10 倍

❌ 佔空間
   每個 Index 都是一份「複製排序」
   大表 10 個 Index 可能 = 表本身大小

❌ 維護成本
   schema 改、Index 也要改
```

**業界共識**：**只加會用到的 Index、定期 review 沒用的刪掉**。


### 14. 完整實戰：電商 schema 加 Index

```text
-- users 表
CREATE INDEX idx_users_email ON users(email);           -- 登入查
CREATE INDEX idx_users_city ON users(city);             -- 篩選

-- products 表
CREATE INDEX idx_products_status ON products(status)
  WHERE status = 'published';                            -- partial
CREATE INDEX idx_products_price ON products(price);     -- 價格篩選排序

-- orders 表
CREATE INDEX idx_orders_user_created
  ON orders(user_id, created_at DESC);                   -- 列出 user 的訂單
CREATE INDEX idx_orders_status ON orders(status)
  WHERE status IN ('pending', 'processing');             -- 後台撈待處理
CREATE INDEX idx_orders_created ON orders(created_at);  -- 時間範圍報表
```


### 15. 5 大常見坑

```text
❌ 1. 每個欄位都加 Index
   → 寫入慢死、表變超大
   → 修：只加會 WHERE / JOIN / ORDER BY 的欄位

❌ 2. 沒 ANALYZE、Index 統計過時
   → 優化器選錯計畫、不用 Index
   → 修：ANALYZE 表名（PG）或 autovacuum 自動跑

❌ 3. 複合 Index 順序錯
   → INDEX (a, b)、WHERE b=... 用不上
   → 修：常 WHERE 的欄位放第一個

❌ 4. WHERE col + 1 = 100
   → Index 用不上（左側函式運算）
   → 修：改成 col = 99 或建函式 Index

❌ 5. LIKE '%abc%' Index 用不上
   → 前綴 % 阻擋 B-tree
   → 修：用全文檢索 GIN（L25）、或反向 Index
```


### 🎯 重點回顧

- **Index = DB 的目錄、B-tree 結構、查特定值 O(log n)**
- **慣例命名 `idx_表_欄位`**、複合 Index 用 `idx_表_欄位 1_欄位 2`
- **FK / 常 WHERE / 常 ORDER BY** 加 Index
- **複合 Index 從左前綴開始**——順序很重要
- **Partial Index** 只索引符合條件的列、省空間
- **EXPLAIN** 看 PG 怎麼跑、`Index Scan` 比 `Seq Scan` 快
- **Index 不是越多越好**——寫入會變慢、佔空間
