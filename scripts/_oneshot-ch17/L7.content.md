**ORDER BY（排序）+ LIMIT（取前 N 筆）+ OFFSET（跳過 N 筆）= SQL 三大分頁神器**—— 所有列表頁、無限滾動、API pagination 全靠這 3 招。

> 💡 **這課的價值**：每個有「列表頁」的網站都用 ORDER BY + LIMIT + OFFSET。Instagram 動態、蝦皮搜尋、Notion 文件清單——全是 SELECT ... ORDER BY ... LIMIT N OFFSET M。


### 1. ORDER BY 基本

```text
SELECT name, age FROM users ORDER BY age;
```

💬 **預設由小到大（ASC）**：

```text
  name  | age
--------+-----
 Nami   |  26
 Xueyue |  28
 Robin  |  28
 Luffy  |  30
 Sanji  |  30
 Zoro   |  32
```

**由大到小用 DESC：**

```text
SELECT name, age FROM users ORDER BY age DESC;
```

```text
ASC   ascending  由小到大、A→Z、舊→新（預設）
DESC  descending 由大到小、Z→A、新→舊
```


### 2. 多欄位排序

```text
SELECT name, age, city
FROM users
ORDER BY city ASC, age DESC;
-- 先按 city 由 A 到 Z、同 city 內再按 age 由大到小
```

**理解方式**：先看第一個欄位、相同時才看第二個。**像 Excel 多層排序**。

```text
 name   | age | city
--------+-----+------
 Zoro   |  32 | 高雄    ← 高雄
 Luffy  |  30 | 台北    ← 台北、age 由大到小
 Sanji  |  30 | 台北
 Xueyue |  28 | 台北
 Robin  |  28 | 台中    ← 台中、age 由大到小
 Nami   |  26 | 台中
```


### 3. 按計算結果排序

ORDER BY 後面能放表達式、不只欄位名。

```text
SELECT name, age FROM users ORDER BY age * 2 DESC;
SELECT name FROM users ORDER BY LENGTH(name) DESC;  -- 名字長到短
```

**也能用 SELECT 的別名（跟 WHERE 不一樣！）：**

```text
SELECT name, age * 2 AS double_age
FROM users
ORDER BY double_age DESC;   -- ✅ 可以！
```

**為什麼 ORDER BY 能用別名、WHERE 不行？**因為 ORDER BY 在 SELECT 之後跑、別名已經存在了。


### 4. NULLS FIRST / NULLS LAST

NULL 在排序時擺哪？PG 預設：

```text
ASC   → NULL 在最後（NULLS LAST）
DESC  → NULL 在最前（NULLS FIRST）
```

**手動指定：**

```text
SELECT name, age
FROM users
ORDER BY age ASC NULLS FIRST;   -- NULL 排在最前
```

**實務常見**：「沒填截止日的任務排最後」：

```text
SELECT * FROM tasks ORDER BY due_date ASC NULLS LAST;
```


### 5. LIMIT —— 只取前 N 筆

```text
SELECT name FROM users ORDER BY age DESC LIMIT 3;
-- 取最年長的 3 個
```

💬 **電腦會顯示**：

```text
  name
--------
 Zoro
 Luffy
 Sanji
```

> 💡 **重要**：ORDER BY + LIMIT 是黃金組合。**沒 ORDER BY 的 LIMIT 結果不固定**——資料庫想給誰就給誰、每次跑可能不一樣。


### 6. OFFSET —— 跳過前 N 筆

```text
-- 第 1 頁（前 3 筆）
SELECT name FROM users ORDER BY id LIMIT 3 OFFSET 0;

-- 第 2 頁（跳過 3 筆、再取 3 筆）
SELECT name FROM users ORDER BY id LIMIT 3 OFFSET 3;

-- 第 3 頁（跳過 6 筆）
SELECT name FROM users ORDER BY id LIMIT 3 OFFSET 6;
```

**通用公式：**

```text
LIMIT  每頁筆數
OFFSET (頁碼 - 1) × 每頁筆數
```

**例**：第 5 頁、每頁 20 筆 → `LIMIT 20 OFFSET 80`。


### 7. 完整分頁 API 範例

📄 **寫進程式檔**：`api/users.ts`

```text
// Next.js API route
export async function GET(req: Request) {
  const url = new URL(req.url);
  const page = parseInt(url.searchParams.get('page') || '1');
  const pageSize = 20;
  const offset = (page - 1) * pageSize;

  const result = await db.query(`
    SELECT id, name, email, created_at
    FROM users
    ORDER BY created_at DESC
    LIMIT $1 OFFSET $2
  `, [pageSize, offset]);

  // 另外撈總數（顯示「共 N 筆、第 X / Y 頁」）
  const { rows: [{ count }] } = await db.query(
    'SELECT COUNT(*) FROM users'
  );

  return Response.json({
    page,
    pageSize,
    total: parseInt(count),
    totalPages: Math.ceil(count / pageSize),
    data: result.rows,
  });
}
```


### 8. OFFSET 的效能陷阱（大表必看）

```text
SELECT * FROM posts ORDER BY id LIMIT 20 OFFSET 1000000;
-- PG 還是要先掃過前 1,000,000 筆、再丟掉
-- 越往後越慢
```

**解法：Cursor-based pagination（用上次最後一筆的 id）：**

```text
-- 第 1 頁
SELECT id, title FROM posts
ORDER BY id DESC
LIMIT 20;
-- 假設最後一筆 id = 12345

-- 第 2 頁（不用 OFFSET）
SELECT id, title FROM posts
WHERE id < 12345
ORDER BY id DESC
LIMIT 20;
```

**好處**：永遠只掃 20 筆、不管在第幾頁都一樣快。Twitter / Instagram 的無限滾動都這樣做。


### 9. 三招綜合應用

**🔍 場景：找出「台北人、年齡 25-35」、按年齡由大到小、第 1 頁（每頁 5 筆）：**

```text
SELECT name, age, email
FROM users
WHERE city = '台北'
  AND age BETWEEN 25 AND 35
ORDER BY age DESC, name ASC
LIMIT 5 OFFSET 0;
```

**SQL 完整骨架**（這課把後面 4 個塊補齊了一半）：

```text
SELECT  ...
FROM    ...
WHERE   ...
ORDER BY ...
LIMIT   ...
OFFSET  ...
```


### 10. 標準寫法 vs FETCH FIRST（SQL 標準語法）

ANSI SQL 標準的寫法（PG 也支援）：

```text
SELECT * FROM users
ORDER BY age DESC
FETCH FIRST 10 ROWS ONLY
OFFSET 0 ROWS;
```

**現實**：大家還是寫 `LIMIT 10 OFFSET 0`（短、習慣）。**MySQL 沒 FETCH FIRST、只能 LIMIT**——所以 LIMIT 是事實標準。


### 11. 隨機排序（小心用）

```text
-- PG / MySQL
SELECT * FROM users ORDER BY RANDOM() LIMIT 1;
-- 隨機抓一筆
```

**注意**：大表用 `ORDER BY RANDOM()` 超慢——PG 要算每筆的隨機值再排序。

**大表替代方案：**

```text
SELECT * FROM users
WHERE id = (
  SELECT floor(random() * (SELECT MAX(id) FROM users) + 1)::INT
);
```


### 12. 5 大常見坑

```text
❌ 1. LIMIT 沒配 ORDER BY
   → 結果不固定、每次跑可能不一樣
   → 修：永遠 ORDER BY + LIMIT 一起寫

❌ 2. 用 OFFSET 做大表分頁
   → 越往後越慢、OFFSET 100000 等於白掃 10 萬筆
   → 修：cursor-based pagination（WHERE id < ?）

❌ 3. ORDER BY 後面接欄位編號
   → ORDER BY 2（按第 2 欄排）能跑但難讀、SELECT 改順序就壞
   → 修：用欄位名

❌ 4. 中文排序看起來亂
   → 預設按字元編碼排、不是按筆畫 / 注音
   → 修：用 COLLATE "zh-TW-x-icu"（PG）或之後處理

❌ 5. 排序欄位沒加 Index
   → 大表排序拖死
   → 修：常排序的欄位加 Index（L19 整課）
```


### 🎯 重點回顧

- **ORDER BY 欄位 ASC/DESC** 排序、預設 ASC
- **多欄位排序** `ORDER BY a ASC, b DESC` 像 Excel 多層
- **NULLS FIRST / LAST** 控制 NULL 擺哪
- **LIMIT N** 取前 N 筆、永遠配 ORDER BY 一起寫
- **OFFSET M** 跳過 M 筆、大表慎用（效能差）
- **Cursor-based pagination**（用上次 id 當錨點）是大表分頁正解
