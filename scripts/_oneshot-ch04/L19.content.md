**design pattern（設計模式）= 老一輩工程師整理出的「常見問題解法套路」**。每個模式對應一種「這種情境就這樣寫」的最佳實踐。JS 因為靈活、又跑在前後端、有自己一套適合的模式——這課介紹 7 個 2026 還常用的。

> 💡 **這課的價值**：不是「為設計而設計」、而是看懂後在寫 code 時自然會用——code review 也看得懂別人為什麼這樣寫。

  
### 1. **Singleton（單例）** —— 全應用只有一個

當「整個 app 只該有一份」的東西：logger、設定、資料庫連線、購物車。

📄 **寫進程式檔**：`Logger.js`

```text
class Logger {
  static #instance;     // 私有靜態欄位（ES2022）

  static getInstance() {
    if (!Logger.#instance) {
      Logger.#instance = new Logger();
    }
    return Logger.#instance;
  }

  log(msg) {
    console.log(`[${new Date().toISOString()}] ${msg}`);
  }
}

// 全局只有一個
const a = Logger.getInstance();
const b = Logger.getInstance();
a === b;     // true
```

**ESM 內更簡單的單例**：

```text
// logger.js（一個 module 自動是單例）
class Logger { log(m) { console.log(m); } }
export default new Logger();

// 任何地方
import logger from "./logger.js";
```

  
### 2. **Factory（工廠）** —— 統一建立物件的入口

不要在 100 個地方用 `new Xxx()`、把建立邏輯集中在 factory。

📄 **寫進程式檔**：`button-factory.js`

```text
function createButton(type, label) {
  switch (type) {
    case "primary":
      return { color: "blue", label, click: () => {} };
    case "danger":
      return { color: "red", label, click: () => alertDanger() };
    case "ghost":
      return { color: "transparent", label, click: () => {} };
    default:
      return { color: "gray", label, click: () => {} };
  }
}

const submit = createButton("primary", "送出");
const cancel = createButton("ghost", "取消");
```

**好處**：

```text
✅ 集中變更（改 button 樣式只改一處）
✅ 隱藏建立細節（呼叫者不用懂 new 什麼）
✅ 容易換實作（switch case 變 lookup table 一行改）
```

  
### 3. **Observer（觀察者）** —— 發布 / 訂閱模式

「事件發生」時通知所有「訂閱者」——pub-sub。Redux / Vue 響應式 / DOM event 底層都是這個。

📄 **寫進程式檔**：`EventBus.js`

```text
class EventBus {
  #listeners = new Map();

  on(event, fn) {
    if (!this.#listeners.has(event)) {
      this.#listeners.set(event, []);
    }
    this.#listeners.get(event).push(fn);
    // 回傳「取消訂閱」的函式
    return () => this.off(event, fn);
  }

  emit(event, data) {
    this.#listeners.get(event)?.forEach(fn => fn(data));
  }

  off(event, fn) {
    const arr = this.#listeners.get(event);
    if (arr) arr.splice(arr.indexOf(fn), 1);
  }
}

// 用法
const bus = new EventBus();
const unsub = bus.on("login", user => console.log("hi", user.name));
bus.emit("login", { name: "招財" });
unsub();   // 退訂
```

  
### 4. **State Machine（狀態機）** —— 用「狀態」管理流程

按鈕 → loading → 成功 / 失敗 → 重試——這些「有限狀態」用 state machine 寫最清楚。

📄 **寫進程式檔**：`fsm.js`

```text
const states = {
  idle:    { start: "loading" },
  loading: { success: "done", fail: "error" },
  done:    { reset: "idle" },
  error:   { retry: "loading", reset: "idle" }
};

function transition(current, action) {
  return states[current]?.[action] ?? current;
}

let state = "idle";
state = transition(state, "start");     // "loading"
state = transition(state, "success");   // "done"
state = transition(state, "reset");     // "idle"
state = transition(state, "invalid");   // "idle"（不存在的 action 不變）
```

> 💡 **業界推薦**：複雜流程用 [XState](https://xstate.js.org/) library——可視化狀態圖、防止錯誤狀態。

  
### 5. **Module Pattern** —— 用 closure 模擬 private

ES2022 有 `#` 之前、就靠這招藏資料。

📄 **寫進程式檔**：`counter.js`

```text
const Counter = (() => {
  let count = 0;          // 私有

  return {
    inc() { count++; },
    dec() { count--; },
    get() { return count; }
  };
})();

Counter.inc();
Counter.get();   // 1
Counter.count;   // undefined ← 外部摸不到
```

> 💡 **白話**：IIFE 包起來、只回傳「該對外的方法」、`count` 變數從外面拿不到——這就是 closure 玩私有屬性。

  
### 6. **Proxy** —— 攔截物件的所有操作

Vue 3 的響應式系統就是這個。

📄 **寫進程式檔**：`proxy.js`

```text
const reactive = new Proxy({ count: 0 }, {
  get(target, key) {
    console.log("讀取", key);
    return target[key];
  },
  set(target, key, value) {
    console.log("寫入", key, "=", value);
    target[key] = value;
    render();   // 自動重新渲染畫面
    return true;
  }
});

reactive.count;          // 印「讀取 count」
reactive.count = 5;      // 印「寫入 count = 5」、自動 render
reactive.count++;        // 觸發 get + set
```

**Proxy 可攔截**：

```text
get / set / has / deleteProperty / ownKeys / 
getPrototypeOf / setPrototypeOf / apply / construct
```

> 💡 **真實應用**：Vue 3、MobX、SolidJS、Immer 都用 Proxy 做響應 / 不可變狀態。

  
### 7. **Strategy（策略）** —— 換演算法不換接口

同一個任務、有多種做法、能換來換去。

📄 **寫進程式檔**：`strategy.js`

```text
const validators = {
  email: (v) => /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(v),
  phone: (v) => /^09\d{8}$/.test(v),
  url:   (v) => /^https?:\/\//.test(v),
  required: (v) => v.length > 0
};

function validate(value, strategy) {
  return validators[strategy]?.(value) ?? false;
}

validate("luffy@gmail.com", "email");   // true
validate("0912345678", "phone");        // true
validate("", "required");               // false
```

> 💡 **跟 Factory 像、但定位不同**：Factory 是「建立物件」、Strategy 是「換演算法」。

  
### 8. **Decorator** —— 包裝既有函式 / class

加 log、加快取、加權限檢查——不改原函式、只在外面包一層。

📄 **寫進程式檔**：`decorator.js`

```text
// HOF 版（function decorator）
function withLog(fn) {
  return function(...args) {
    console.log("呼叫", fn.name, args);
    const result = fn(...args);
    console.log("回傳", result);
    return result;
  };
}

function add(a, b) { return a + b; }
const loggedAdd = withLog(add);
loggedAdd(1, 2);   // 印 log + 回傳 3

// class 版（ES2022 Stage 3、TypeScript 已支援）
function log(value, context) {
  return function(...args) {
    console.log("呼叫", context.name, args);
    return value.apply(this, args);
  };
}

class Calc {
  @log
  multiply(a, b) { return a * b; }
}
```

  
### 9. **Repository（資料層）** —— 把資料存取藏起來

UI 不該知道資料從 API 還是 localStorage 來——抽一層 repository。

📄 **寫進程式檔**：`user-repo.js`

```text
class UserRepo {
  async getById(id) {
    // 先看快取
    const cached = localStorage.getItem(`user:${id}`);
    if (cached) return JSON.parse(cached);

    // 再去 API
    const r = await fetch(`/api/users/${id}`);
    const u = await r.json();

    // 存快取
    localStorage.setItem(`user:${id}`, JSON.stringify(u));
    return u;
  }
}

// UI 只用 repo、不知道有沒有 cache
const repo = new UserRepo();
const u = await repo.getById(1);
```

  
### 10. 過度設計（over-engineering）警告

```text
⚠️ 不要為了用 pattern 而用 pattern

❌ 只有兩個 if 就建 Strategy
❌ 一個 logger 套三層 abstraction
❌ 簡單 fetch 包 Repository + Service + Adapter

✅ 規則：先寫簡單的、痛了再 refactor
✅ 「YAGNI」(You Aren't Gonna Need It)
✅ 寫 200 行才開始考慮設計、不是 20 行就過設計
```

  
### 5 個常見坑

```text
❌ 1. 到處 new 同一個 class、忘了 Singleton
   → 重複初始化、資源浪費
   → 修：用 Singleton 或 ESM default export

❌ 2. EventBus 沒處理「退訂」
   → 元件 unmount 後 listener 還在、memory leak
   → 修：on 回傳 unsubscribe function

❌ 3. State Machine 隨便用 if / else
   → 狀態爆炸後地獄、無效轉換滿天飛
   → 修：用 lookup table 或 XState

❌ 4. Module Pattern 過度藏東西
   → 連測試都摸不到、debug 困難
   → 修：藏實作細節、保留 API 可測

❌ 5. Proxy 全寫
   → 效能差（每次存取都過攔截器）
   → 修：只攔需要的、不要 wrap 所有物件
```

  
### 🎯 重點回顧

- **Singleton**：全 app 一份（logger / config）—— ESM default export 自動單例
- **Factory**：集中物件建立邏輯
- **Observer / EventBus**：pub-sub、解耦合
- **State Machine**：複雜流程改用「狀態 + 轉換」
- **Module Pattern**：closure 模擬 private（ES2022 後改用 `#`）
- **Proxy**：攔截操作、做響應 / 驗證 / log
- **Strategy**：換演算法不換接口
- **不要過度設計**——簡單先、痛了再 refactor

> 💡 **下一課（L20）**：學 **效能優化**——debounce / throttle / lazy load / virtualization 五大實戰招式。
