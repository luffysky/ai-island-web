**event loop（事件迴圈）= JS 處理「等很久才回來的事情」的核心機制**。JS 是**單執行緒**（single-thread、一次只能做一件事）、但又能 fetch 資料、計時、回應點擊——怎麼辦到的？答案就是 event loop。

> 💡 **這課的價值**：搞懂 event loop 後、你寫的 async / await / setTimeout / Promise 才會「為什麼順序這樣」變得清楚。**大廠面試 100% 考**。

  
### 1. 為什麼 JS 是「單執行緒」？

```text
🧵 thread（執行緒）= CPU 同時能跑的「一條線」

多執行緒語言（Java / Go）：
  ┌──┐  ┌──┐  ┌──┐
  │  │  │  │  │  │  ← 同時做 3 件事
  └──┘  └──┘  └──┘

JS（單執行緒）：
  ┌────────────────┐
  │  一條線跑全部   │  ← 一次只能做一件事
  └────────────────┘
```

**好處**：簡單、不用鎖、不會 race condition。
**壞處**：一件事卡住、整個頁面就死。

**那 fetch / setTimeout 怎麼辦？** → **交給瀏覽器處理、自己繼續跑下面**。

  
### 2. event loop 架構（簡化版）

```text
┌──────────────────────────┐
│  Call Stack（呼叫堆疊）  │  ← 同步程式碼在這跑
├──────────────────────────┤
│  Microtask Queue         │  ← Promise.then / queueMicrotask
├──────────────────────────┤
│  Task Queue（macrotask）│  ← setTimeout / 點擊 / fetch 回來
└──────────────────────────┘
              ↓
        Event Loop
   不停做這 3 件事：
   1. 跑完 call stack
   2. 把 microtask queue 全清光
   3. 從 task queue 取 1 個來跑
   → 回 1
```

> 💡 **白話**：JS 像一個只有一個窗口的便利店員——一次只服務一個客人。客人說「我等微波爐」、店員不會傻等、會去接下一個客人、微波叮了再回來。**「等微波爐」這部分交給瀏覽器（不是 JS 引擎）**。

  
### 3. 經典面試題：誰先印？

📄 **寫進程式檔**：`order.js`

```text
console.log("1");

setTimeout(() => console.log("2"), 0);

Promise.resolve().then(() => console.log("3"));

console.log("4");
```

🖥️ **在終端機輸入**：

```text
node order.js
```

💬 **電腦會顯示**：

```text
1
4
3
2
```

**為什麼？**

```text
步驟 1：同步「1」直接印
步驟 2：setTimeout 進「task queue」
步驟 3：Promise.then 進「microtask queue」
步驟 4：同步「4」直接印
步驟 5：call stack 空了 → 先清 microtask queue → 印「3」
步驟 6：再從 task queue 拿一個 → 印「2」
```

> 💡 **規則**：**microtask 優先級高過 macrotask**——`Promise.then` 永遠比 `setTimeout(_, 0)` 早跑。

  
### 4. 兩種 queue 的差別

```text
🟦 Microtask Queue（高優先）
   Promise.then / .catch / .finally
   queueMicrotask(...)
   MutationObserver
   async function await 後的程式

🟧 Task Queue / Macrotask（低優先）
   setTimeout / setInterval
   setImmediate（Node）
   I/O（檔案讀寫、網路）
   UI 事件（click、scroll）
   postMessage
```

**規則**：每跑完一個 macrotask、就把所有 microtask 清光、才繼續下一個 macrotask。

  
### 5. async / await 怎麼接上 event loop？

📄 **寫進程式檔**：`async-order.js`

```text
async function demo() {
  console.log("A");
  await Promise.resolve();
  console.log("B");   // ← 這行其實是 microtask
}

console.log("start");
demo();
console.log("end");
```

💬 **輸出**：

```text
start
A
end
B
```

**為什麼 B 在 end 後面？**

```text
1. console.log("start")
2. 進 demo() → 印 A
3. 遇到 await → 後面的程式變成 microtask
4. 退出 demo()、回到外面 → 印 end
5. call stack 空了 → 清 microtask → 印 B
```

> 💡 **白話**：`await` 像是「我先回去、有結果再叫我」。`await` 後面的程式都會被排到 microtask queue。

  
### 6. 為什麼大迴圈會「卡頁面」

📄 **寫進程式檔**：`block.js`

```text
// ✗ 一直跑大迴圈 → 卡死
function heavy() {
  for (let i = 0; i < 1e9; i++) {
    // 算很久...
  }
}

button.addEventListener("click", () => {
  heavy();   // 點下去頁面 lag 5 秒、無法點別的按鈕
});
```

**為什麼？** call stack 上一直有東西、event loop 沒辦法處理「點擊」「滾動」「渲染」——通通排隊等。

  
### 7. 長任務優化 —— 拆批處理

📄 **寫進程式檔**：`chunk.js`

```text
// ✓ 拆批：每處理一批就讓出控制權
async function processBig(arr) {
  const CHUNK = 100;
  for (let i = 0; i < arr.length; i += CHUNK) {
    const batch = arr.slice(i, i + CHUNK);
    batch.forEach(item => heavy(item));

    // 讓 event loop 喘氣：把後面的工作排到下一輪
    await new Promise(r => setTimeout(r, 0));
  }
}

// 2024+ 更現代的方式：scheduler.yield()
async function processBig2(arr) {
  for (const item of arr) {
    heavy(item);
    await scheduler.yield();   // 主動讓出時間片
  }
}
```

> 💡 **白話**：在大迴圈中間「插一個 `await`」、就像店員處理完一批工作後抬頭問「有沒有新客人」——讓 UI 有機會更新。

### 10. setTimeout(_, 0) 真的是 0 嗎？

```text
setTimeout(() => console.log("hi"), 0);

// 實際上：
// → 最少 4ms（HTML5 規範）
// → 嵌套超過 5 層、瀏覽器強制至少 4ms
// → 如果 call stack 卡著、可能等更久
```

**白話**：`setTimeout(_, 0)` 不是「立刻」、是「**最快的下一個 task queue 機會**」。

  
### 5 個常見坑

```text
❌ 1. 以為 setTimeout(_, 0) 立刻執行
   → 至少要等 call stack 清空 + microtask 清完
   → 修：要真的立刻用 queueMicrotask 或不用 setTimeout

❌ 2. 大迴圈卡 UI 不知道為什麼
   → 同步任務占住 call stack
   → 修：拆批 + await new Promise(r => setTimeout(r))

❌ 3. await 後的 console.log 順序怪
   → 那行其實是 microtask、會等所有同步跑完
   → 修：知道規則就好

❌ 4. forEach 內用 await 沒效果
   → forEach 不認 await
   → 修：for...of 或 Promise.all(arr.map(...))

❌ 5. 連續 await 但其實可以並行
   → const a = await fetchA(); const b = await fetchB(); 是 A+B 時間
   → 修：const [a, b] = await Promise.all([fetchA(), fetchB()]);
```

  
### 🎯 重點回顧

- **JS 單執行緒**——一次一件事、靠 event loop 處理 async
- **microtask 優先級高過 macrotask**——`Promise.then` 比 `setTimeout(_, 0)` 早
- **`await` 後面的程式是 microtask**——會等同步全跑完才跑
- **大迴圈會卡 UI**——拆批 + `await` 讓出時間片
- **`setTimeout(_, 0)` 至少 4ms**、不是真的 0
