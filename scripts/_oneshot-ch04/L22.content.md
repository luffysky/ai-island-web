**testing（測試）= 寫程式檢查你的程式是不是真的對**。改一行 code 不會跑壞別的 90 行——這就是測試的價值。2026 主流：**Vitest（單元）+ Testing Library（元件）+ Playwright（E2E）**。

> 💡 **這課的價值**：「不寫測試 = 每次改 code 都在賭博」。學會這三件套、你寫的 code 比 80% 同事更靠譜。

  
### 1. 三種測試 —— 測試金字塔

```text
                   E2E（10%）
                  /          \
              整合測試（20%）
              /              \
          單元測試（70%）

🟦 unit test（單元測試）
   → 測「一個函式」/「一個小模組」
   → 快、多、覆蓋邊角案例

🟧 integration test（整合測試）
   → 測「多個模組合起來」
   → 例：API + DB / 元件 + state

🟥 E2E（End-to-End、端到端）
   → 開瀏覽器跑「完整使用者流程」
   → 慢、少、只測關鍵流程
```

> 💡 **規則**：底層測得越多越好（快、便宜）、頂層精選關鍵流程（慢、貴）。**不要 100% E2E**——跑半小時還不一定穩定。

  
### 2. **Vitest** —— 2026 單元測試主流

取代 Jest 的新一代——更快、更現代、跟 Vite 完美整合。

🖥️ **在終端機輸入**：

```text
npm install -D vitest
```

📄 **寫進程式檔**：`math.js`

```text
export function add(a, b) {
  return a + b;
}

export function divide(a, b) {
  if (b === 0) throw new Error("除以零");
  return a / b;
}
```

📄 **寫進程式檔**：`math.test.js`

```text
import { describe, it, expect } from "vitest";
import { add, divide } from "./math.js";

describe("add", () => {
  it("加兩個正數", () => {
    expect(add(1, 2)).toBe(3);
  });

  it("處理負數", () => {
    expect(add(-1, 1)).toBe(0);
  });

  it("處理零", () => {
    expect(add(0, 0)).toBe(0);
  });
});

describe("divide", () => {
  it("除以零會 throw", () => {
    expect(() => divide(10, 0)).toThrow("除以零");
  });
});
```

🖥️ **在終端機輸入**（跑測試）：

```text
npx vitest
```

💬 **電腦會顯示**：

```text
✓ math.test.js (4)
  ✓ add (3)
    ✓ 加兩個正數
    ✓ 處理負數
    ✓ 處理零
  ✓ divide (1)
    ✓ 除以零會 throw

Test Files  1 passed (1)
     Tests  4 passed (4)
```

  
### 3. **常用 assertion（斷言）**

```text
expect(value).toBe(3);                    // === 嚴格相等
expect(obj).toEqual({ a: 1 });            // 深度相等
expect(arr).toContain(2);                 // 陣列含 2
expect(arr).toHaveLength(3);
expect(fn).toThrow();                     // 會丟錯
expect(fn).toThrow("特定訊息");
expect(fn).toHaveBeenCalled();            // mock 被呼叫
expect(fn).toHaveBeenCalledTimes(2);
expect(fn).toHaveBeenCalledWith("arg");
expect(promise).resolves.toBe(42);         // Promise 成功
expect(promise).rejects.toThrow();         // Promise 失敗
expect(str).toMatch(/regex/);
expect(num).toBeGreaterThan(0);
expect(value).toBeNull();
expect(value).toBeUndefined();
expect(value).toBeTruthy();
```

  
### 4. **mock（模擬）** —— 用假的取代真的

測試時不要打真的 API / DB——用 mock 假裝有回應。

📄 **寫進程式檔**：`mock.test.js`

```text
import { vi, describe, it, expect } from "vitest";

// 1. 建假函式
const fn = vi.fn().mockReturnValue(42);
fn();
expect(fn).toHaveBeenCalledTimes(1);

// 2. 假回傳 Promise
const fetchUser = vi.fn().mockResolvedValue({ id: 1, name: "招財" });

// 3. mock 整個模組
vi.mock("./api", () => ({
  fetchUser: vi.fn().mockResolvedValue({ id: 1 })
}));

import { fetchUser } from "./api";

it("呼叫 fetchUser", async () => {
  const u = await fetchUser();
  expect(u.id).toBe(1);
  expect(fetchUser).toHaveBeenCalled();
});
```

  
### 5. **Testing Library** —— 元件測試標準

測 React / Vue / Svelte 元件——「**像使用者那樣**」操作。

🖥️ **在終端機輸入**：

```text
npm install -D @testing-library/react @testing-library/jest-dom
```

📄 **寫進程式檔**：`Counter.test.jsx`

```text
import { render, screen, fireEvent } from "@testing-library/react";
import { describe, it, expect } from "vitest";
import Counter from "./Counter";

describe("Counter", () => {
  it("初始為 0", () => {
    render(<Counter />);
    expect(screen.getByText("0")).toBeInTheDocument();
  });

  it("點按鈕 + 1", () => {
    render(<Counter />);
    fireEvent.click(screen.getByRole("button", { name: "+" }));
    expect(screen.getByText("1")).toBeInTheDocument();
  });
});
```

**Testing Library 哲學**：

```text
✅ 測「使用者看到 / 做到」的事
✅ 用 getByRole / getByText / getByLabelText
✅ 不測 className / id / 內部 state
```

  
### 6. **Playwright** —— E2E 測試 2026 主流

取代 Cypress 的後浪——更快、API 更現代、跨瀏覽器（Chromium / Firefox / WebKit）。

🖥️ **在終端機輸入**：

```text
npm init playwright@latest
```

📄 **寫進程式檔**：`tests/login.spec.js`

```text
import { test, expect } from "@playwright/test";

test("登入流程", async ({ page }) => {
  await page.goto("/login");

  await page.fill('input[name="email"]', "luffy@gmail.com");
  await page.fill('input[name="password"]', "12345678");
  await page.click("button[type=submit]");

  await expect(page).toHaveURL("/dashboard");
  await expect(page.getByText("歡迎，林董")).toBeVisible();
});
```

🖥️ **在終端機輸入**（跑測試）：

```text
npx playwright test
npx playwright test --ui    # 視覺化 UI mode
```

  
### 7. **Playwright 厲害的功能**

```text
✅ 自動等元素（不用手動 sleep）
✅ 跨瀏覽器（Chromium / Firefox / Safari）
✅ 跨裝置（mobile emulation）
✅ Trace viewer（測試失敗後重播）
✅ Codegen（操作網頁自動產生測試 code）
✅ 截圖 / 錄影（每個 test step）
✅ 平行執行（測試超快）
✅ Network mocking（攔截 API）

🔧 npx playwright codegen https://example.com
   → 開瀏覽器操作、自動產生 code
```

  
### 8. **TDD vs BDD**

```text
🟦 TDD（Test-Driven Development、測試驅動）
   1. 先寫測試（必失敗）
   2. 寫剛好讓測試過的 code
   3. refactor
   → 適合：演算法 / 純邏輯 / library

🟧 BDD（Behavior-Driven Development、行為驅動）
   1. 用「使用者故事」描述需求
   2. 轉成 Given/When/Then 句式
   3. 寫測試
   → 適合：產品功能 / E2E
   → 工具：Cucumber、Vitest 也支援

範例 BDD：
  Given 我是已登入使用者
  When  我點「加入購物車」
  Then  購物車數量 +1
```

  
### 9. **coverage（涵蓋率）** —— 測了多少 code

```text
🖥️ npx vitest --coverage

→ 產生報告：
   Statements: 87%
   Branches:   72%
   Functions:  91%
   Lines:      85%
```

**警告**：

```text
⚠️ 100% coverage ≠ 100% 沒 bug
   → coverage 高代表「行被跑過」、不代表「結果正確」
⚠️ 不要追 100% coverage、追「關鍵邏輯都覆蓋」
⚠️ 配置工具：Vitest 內建、Jest 用 c8 / istanbul
```

### 11. 實戰建議：怎麼開始寫測試？

```text
🎯 不要追 100% coverage、追「最痛的地方」

✅ 1. 邊角邏輯先測（折扣、稅、權限判斷）
✅ 2. 修 bug 時順手寫測試（防止再回來）
✅ 3. 新功能寫測試（一邊寫一邊測）
✅ 4. legacy code「重構前」先寫測試保護
✅ 5. CI 卡住測試不過、不准 merge
```

  
### 5 個常見坑

```text
❌ 1. 測試靠 console.log
   → 沒自動化、改 code 不知道哪壞了
   → 修：寫 unit test、CI 強制跑

❌ 2. 測太多實作細節
   → 改實作就壞、嚇得不敢 refactor
   → 修：測「行為 / 輸出」、不測內部 state

❌ 3. mock 過度
   → 全部 mock 後測不到真實互動
   → 修：純邏輯 mock 依賴、整合測試別 mock

❌ 4. E2E 寫太多
   → 慢、不穩定、CI 跑半小時
   → 修：金字塔比例（單元 70 / 整合 20 / E2E 10）

❌ 5. 測試之間互相依賴
   → 順序錯就爆、跨 file 共用 state
   → 修：每個 test 獨立、beforeEach reset
```
