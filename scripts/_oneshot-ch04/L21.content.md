**Web 資安 = 防止壞人偷資料 / 假冒使用者 / 注入惡意碼**。前端三大經典攻擊：**XSS（跨站腳本）**、**CSRF（跨站請求偽造）**、**Prototype Pollution（原型污染）**。這三個 2026 還是天天有人中招、新手必看必會。

> 💡 **這課的價值**：寫 code 不懂資安 = 開後門給壞人。**這課的每個範例都是真實案例**——lodash、jQuery、Express 都曾經中過。

  
### 1. **XSS（Cross-Site Scripting、跨站腳本攻擊）**

攻擊者把惡意 JS 注入網頁、別人開啟就執行——可以偷 cookie / 改畫面 / 假冒身份。

📄 **寫進程式檔**：`xss-bad.js`

```text
// ✗ 危險：直接把 user input 塞 HTML
const userInput = "<script>alert('XSS');</script>";
div.innerHTML = userInput;
// → 攻擊者真的能執行任何 JS

// 真實場景：留言系統
const comment = req.body.text;
commentEl.innerHTML = comment;   // user 留 <script> 全網中招
```

**防禦 1：用 `textContent`**

```text
// ✓ 安全：當純文字、不解析 HTML
div.textContent = userInput;
// → "<script>..." 直接變成顯示字、不執行
```

**防禦 2：必須允許 HTML 時 sanitize**

🖥️ **在終端機輸入**：

```text
npm install dompurify
```

📄 **寫進程式檔**：`xss-fix.js`

```text
import DOMPurify from "dompurify";

const dirty = "<img src=x onerror=alert(1)>";
const clean = DOMPurify.sanitize(dirty);
// "<img src=\"x\">" ← 移除危險屬性

div.innerHTML = clean;
```

  
### 2. **XSS 三種類型**

```text
1️⃣ Stored XSS（儲存型）
   攻擊者把 <script> 存到 DB
   → 所有看到該頁的人都中招
   → 最嚴重、留言板 / 個人簡介常見

2️⃣ Reflected XSS（反射型）
   URL 帶惡意參數、被網頁反射回 HTML
   → /search?q=<script>... 被 echo 到頁面
   → 釣魚連結

3️⃣ DOM-based XSS
   全在前端發生（不經過 server）
   → 例：location.hash 取值後 innerHTML
   → SPA 容易踩
```

  
### 3. **CSP（Content Security Policy、內容安全政策）** —— 瀏覽器層級擋 XSS

在 server response header 或 HTML meta 設：

📄 **寫進程式檔**：`index.html`

```text
<meta http-equiv="Content-Security-Policy"
      content="default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:;">
```

```text
✅ default-src 'self'        只允許自家網域
✅ script-src 'self'         禁止 inline JS（高安全等級）
✅ img-src https:           圖片只允許 HTTPS
🚫 'unsafe-inline'           禁掉就是高安全（但很多 library 會壞）
```

> 💡 **2026 推薦**：用 [Report-Only 模式](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Content-Security-Policy-Report-Only) 先試水溫——壞了不擋、只回報、確認沒問題再開正式。

  
### 4. **CSRF（Cross-Site Request Forgery、跨站請求偽造）**

使用者已登入 A 網站、攻擊者讓他在 B 網站點連結、瀏覽器自動帶 cookie 發請求到 A——A 以為是本人。

```text
情境：
1. user 登入 bank.com → bank.com 給了 cookie
2. user 不登出、又開了 evil.com
3. evil.com 內有：
   <img src="https://bank.com/transfer?to=hacker&amount=10000">
4. 瀏覽器自動帶 bank.com 的 cookie 發請求
5. bank.com 以為是 user 本人轉帳
```

**防禦 1：CSRF token**

```text
✅ Server 每個 form 給一個隨機 token
   <input type="hidden" name="csrf" value="xyz123">

✅ 提交時驗證 token
   → 攻擊者拿不到 token、發不出有效請求
```

**防禦 2：SameSite cookie**

```text
Set-Cookie: token=xxx; SameSite=Strict; Secure; HttpOnly

→ Strict: 完全不跨站帶 cookie
→ Lax:    導航類 GET 才帶（2026 預設）
→ None:   全帶（要 Secure 配合、用於跨域 API）
```

**防禦 3：檢查 Origin / Referer header**

```text
// server 端
if (req.headers.origin !== "https://my-site.com") {
  return res.status(403).end();
}
```

  
### 5. **Prototype Pollution（原型污染）**

攻擊者改 `Object.prototype`、**所有物件**都被污染。

📄 **寫進程式檔**：`pollution-bad.js`

```text
// ✗ 危險：deep merge 任意使用者資料
function merge(target, source) {
  for (const key in source) {
    if (typeof source[key] === "object") {
      merge(target[key] = target[key] || {}, source[key]);
    } else {
      target[key] = source[key];
    }
  }
}

// 攻擊
const attack = JSON.parse('{"__proto__": {"isAdmin": true}}');
merge({}, attack);

// 所有物件都被污染
const u = {};
u.isAdmin;   // true ← 任意物件都有 isAdmin
```

**真實案例**：lodash `_.merge` / jQuery `$.extend` / Moment 都被攻擊過、CVE 一堆。

**防禦**：

```text
✅ 1. 禁止危險 key
   if (key === "__proto__" || key === "constructor" || key === "prototype") {
     continue;
   }

✅ 2. 用 Object.create(null) 建純物件
   const safe = Object.create(null);
   // 沒有 __proto__、攻擊者改不到

✅ 3. 用 Map 取代物件當 key-value
   const data = new Map();
   data.set(userInput, value);

✅ 4. 用 schema validator（Zod / Yup）
   const schema = z.object({ name: z.string() });
   const safe = schema.parse(userInput);
```

  
### 6. **其他常見前端漏洞**

```text
🔴 Open Redirect
   /go?to=https://evil.com 直接跳過去
   → 修：白名單檢查 url

🔴 Clickjacking
   把你的網站 iframe 在 evil.com、user 以為點別的
   → 修：X-Frame-Options: DENY 或 CSP frame-ancestors

🔴 Insecure localStorage
   把 token / 信用卡 / 密碼存 localStorage
   → 修：敏感資料用 httpOnly cookie

🔴 SQL injection（後端）
   user 輸入 ' OR 1=1 -- 拼進 SQL
   → 修：永遠用參數化查詢（prepared statement）

🔴 SSRF（Server-Side Request Forgery）
   server fetch user 給的 URL、可能打到內網
   → 修：URL 白名單 + 禁止內部 IP
```

  
### 7. **fetch 跟 CORS 的「資安」面**

```text
✅ CORS 不是 user 的安全機制、是「server 控制誰能 cross-origin 拿資料」
✅ Browser 預設擋 cross-origin response body 給 JS 讀
✅ 想跨域要 server 帶 Access-Control-Allow-Origin

⚠️ 不要用 Access-Control-Allow-Origin: *
   配合 credentials: include 是大漏洞
```

  
### 8. **passwords / token 保管**

```text
✅ 密碼絕對不能 plain text 存 DB
   → 用 bcrypt / argon2 hash + salt

✅ JWT / session token
   → 短時效 + refresh token
   → HttpOnly cookie 保管

✅ API key 不可寫前端
   → 任何人 view source 都看得到
   → 改後端代理或 short-lived token
```

  
### 9. 實戰 checklist —— 寫網站前看一遍

```text
✅ user input 進 DOM 一律 textContent 或 sanitize
✅ session token 用 HttpOnly + Secure + SameSite cookie
✅ 所有 form 加 CSRF token
✅ CSP header 開啟
✅ HTTPS 強制（HSTS header）
✅ 密碼 hash 用 bcrypt / argon2
✅ deep merge / Object assign 擋 __proto__
✅ external 連結加 rel="noopener noreferrer"
✅ 圖片 / 字型 / 媒體用 CDN + HTTPS
✅ 後端 input validation（永遠別信前端）
```

### 5 個常見坑

```text
❌ 1. innerHTML 直接吃 user input
   → XSS 漏洞王
   → 修：textContent 或 DOMPurify

❌ 2. localStorage 存 session token
   → XSS 一爆全洩漏
   → 修：HttpOnly cookie

❌ 3. deep merge 沒擋 __proto__
   → prototype pollution 入口
   → 修：用 schema validation 或自己寫白名單

❌ 4. CSP 一次全開最嚴
   → 整站爆紅、所有 library 壞
   → 修：先用 Report-Only 模式收集、再逐步收緊

❌ 5. API key 寫前端 .js
   → 任何人都看得到
   → 修：搬到後端、前端只透過自家 API 取
```

  
### 🎯 重點回顧

- **XSS**：user input 一律 `textContent` 或 DOMPurify
- **CSRF**：CSRF token + SameSite cookie + 檢查 Origin
- **Prototype Pollution**：擋 `__proto__` / `constructor` / `prototype` key
- **敏感 token 永遠 HttpOnly cookie**——不放 localStorage
- **API key 不能寫前端**——用後端 proxy
- **2026 工具**：Snyk / DOMPurify / Helmet / CSP
