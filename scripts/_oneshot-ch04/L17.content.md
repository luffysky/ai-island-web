**WebAssembly（WASM）= 讓 C / C++ / Rust / Go 程式碼也能在瀏覽器跑的「二進位格式」**。速度**接近原生**（比 JS 快 10-100 倍）、是 JS 跑不夠快時的終極武器。Photoshop Web、Figma、Google Earth 都靠它。

> 💡 **這課的價值**：你不一定會寫 WASM、但要知道**「什麼時候 JS 不夠用、該找 WASM 救援」**。2026 邊緣運算（Cloudflare Workers / Vercel Edge）也都接受 WASM。

  
### 1. WASM 是什麼？

```text
🟦 JS（直譯式）
   .js 檔 → JS 引擎一邊讀一邊解釋執行
   → 啟動快、但執行慢

🟧 WASM（二進位）
   .wasm 檔 → 已經是接近機器碼的格式
   → 載入後直接跑、跟原生程式速度相近
```

**白話**：JS 像「現場翻譯員」、邊講邊翻；WASM 像「先寫好稿子的翻譯」、上場直接念。**速度差 10-100 倍**。

  
### 2. WASM 怎麼產生？

```text
C / C++ / Rust / Go / AssemblyScript / Zig
       ↓ 編譯器
   xxx.wasm（二進位檔）
       ↓ 瀏覽器載入
   能跑！
```

**最熱門的編譯器**：

```text
🦀 Rust + wasm-pack          ← 2026 最熱
🌐 Emscripten (C/C++)         ← 老牌但成熟
📜 AssemblyScript             ← TypeScript-like、新手友善
🐹 TinyGo（Go 的精簡版）
```

  
### 3. 從 JS 載入 WASM 模組

📄 **寫進程式檔**：`load-wasm.js`

```text
// 1. 老式寫法
const buffer = await fetch("/math.wasm").then(r => r.arrayBuffer());
const { instance } = await WebAssembly.instantiate(buffer);

// 2. 現代寫法（推薦、streaming）
const { instance } = await WebAssembly.instantiateStreaming(
  fetch("/math.wasm")
);

// 呼叫 WASM 匯出的函式
const result = instance.exports.fibonacci(40);
console.log(result);   // 比 JS 版快 10-100 倍

// 看有哪些匯出
console.log(instance.exports);
// { fibonacci: fn, multiply: fn, memory: WebAssembly.Memory }
```

### 5. **WASI（WebAssembly System Interface）** —— 跨平台標準

2024 後 WASM 不只在瀏覽器跑——**WASI** 讓它變成「**跨語言、跨平台**」標準。

```text
✅ Cloudflare Workers：接受 WASM 部署
✅ Vercel Edge Functions：接受 WASM
✅ Fastly Compute@Edge：以 WASM 為主
✅ wasmtime / wasmer：server 端 WASM runtime
✅ Docker：可以跑 WASM container
```

> 💡 **白話**：以前 WASM 只能在瀏覽器跑、現在連 server / 邊緣機房都能跑——**真正的「一次寫、到處跑」**。

  
### 6. WASM 適合什麼？

```text
✅ 適合
   - CPU 密集運算（圖像處理、加密、編解碼）
   - 已有 C/C++/Rust 程式碼想搬到 Web
   - 遊戲引擎、CAD、3D 模擬
   - AI 模型推論（onnxruntime-web）
   - 嚴格效能要求的演算法

❌ 不適合
   - 簡單邏輯（JS 已經很快）
   - DOM 操作（WASM 不能直接動 DOM）
   - API 串接（純 JS 就好）
   - 短期專案（學習成本高）
```

  
### 7. WASM 限制

```text
🚫 不能直接動 DOM
   → 要透過 JS 橋接

🚫 啟動時間有成本
   → 載入 + 編譯 + instantiate 要時間
   → 小東西用 WASM 反而慢

🚫 學習曲線陡
   → 要懂 Rust / C++ 才能寫
   → debug 比較痛

🚫 SEO 不友善
   → search engine 不執行 WASM
```

  
### 8. 簡單範例：Rust → WASM

🖥️ **在終端機輸入**：

```text
cargo install wasm-pack
cargo new --lib hello-wasm
cd hello-wasm
```

📄 **寫進程式檔**：`src/lib.rs`（Rust）

```text
use wasm_bindgen::prelude::*;

#[wasm_bindgen]
pub fn add(a: i32, b: i32) -> i32 {
    a + b
}

#[wasm_bindgen]
pub fn fibonacci(n: u32) -> u64 {
    if n <= 1 { return n as u64; }
    let mut a: u64 = 0;
    let mut b: u64 = 1;
    for _ in 0..n - 1 {
        let temp = a + b;
        a = b;
        b = temp;
    }
    b
}
```

🖥️ **在終端機輸入**：

```text
wasm-pack build --target web
```

📄 **寫進程式檔**：`index.html`

```text
<script type="module">
  import init, { add, fibonacci } from "./pkg/hello_wasm.js";
  await init();
  console.log(add(1, 2));         // 3
  console.log(fibonacci(40));     // 102334155、瞬間出來
</script>
```

  
### 9. WASM 跟 JS 互通

```text
✅ JS → WASM
   傳數字（i32 / i64 / f32 / f64）很快
   傳字串 / 物件要透過 memory buffer + 編碼

✅ WASM → JS
   呼叫 import 進來的 JS 函式
   讀 / 寫共享 memory

🔧 用 wasm-bindgen / wasm-pack 自動產生膠水
   → 寫起來像普通 import、底層是 buffer 操作
```

### 5 個常見坑

```text
❌ 1. 用 WASM 跑簡單邏輯
   → 啟動成本反而拖慢
   → 修：只有重運算才搬 WASM

❌ 2. WASM 想直接修改 DOM
   → WASM 看不到 DOM
   → 修：postMessage 給 JS、由 JS 改

❌ 3. 大 WASM 檔案沒做 streaming
   → 用 fetch().then(r => r.arrayBuffer()) 要等載完
   → 修：用 WebAssembly.instantiateStreaming()

❌ 4. 字串傳遞效能爛
   → 字串要編碼進 buffer
   → 修：減少字串傳遞、批次處理

❌ 5. 沒設 mime type
   → server 沒回 application/wasm、streaming 失敗
   → 修：server 設正確 Content-Type
```

  
### 🎯 重點回顧

- **WASM = 比 JS 快 10-100 倍的二進位格式**——適合密集運算
- **常用編譯來源**：Rust（最熱）/ C++ / AssemblyScript / Go
- **不能直接動 DOM**——一律透過 JS 橋接
- **WASI 讓 WASM 跨平台**——Cloudflare / Vercel Edge / Docker 都支援
- **不會寫 WASM 沒關係**——用現成套件就夠（ffmpeg.wasm / sql.js / onnxruntime-web）
