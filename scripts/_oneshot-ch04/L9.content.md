**prototype chain（原型鏈）= JS 物件互相「繼承功能」的家族族譜**。每個物件都有一條「往上找的鏈條」—— 自己沒有的方法、就往爸爸找、爸爸沒有就往爺爺找、找到 `null` 為止。class 只是這條鏈的糖衣、本質還是 prototype。

> 💡 **這課的價值**：90% 的 JS 怪 bug 都跟 prototype 有關。看懂這層、才看得懂為什麼「`[].push` 從哪來」「`obj.toString` 為什麼一定有」。

  
### 1. 從「為什麼陣列會有 push？」開始

📄 **寫進程式檔**：`why-push.js`

```text
const arr = [1, 2, 3];

arr.push(4);    // 為什麼可以？arr 物件本身沒寫 push 啊

// 答案：JS 沿著 __proto__（原型）鏈找
arr.__proto__ === Array.prototype;             // true
Array.prototype.__proto__ === Object.prototype; // true
Object.prototype.__proto__ === null;            // 終點

// arr.push 實際是從 Array.prototype.push 找來的
```

**白話**：陣列 `arr` 自己不會 `push`、但它的「爸爸」(`Array.prototype`) 會、所以呼叫 `arr.push()` 時 JS 沿著鏈條往上找、找到了就用。

  
### 2. 原型鏈的視覺圖

```text
arr → Array.prototype → Object.prototype → null
       (push, map,        (toString,
        filter ...)        hasOwnProperty ...)

user → User.prototype → Object.prototype → null
        (greet ...)       (toString ...)
```

**規則**：

```text
1. 每個物件都有一個 [[Prototype]]（也叫 __proto__）
2. 存取屬性時、先看自己、沒有就看 __proto__
3. 一路找到 null 還沒就回 undefined
4. 這就叫「prototype chain」
```

  
### 3. **`__proto__` vs `prototype` 差別**—— 最常搞混的兩個名字

```text
🟦 __proto__（每個「物件」都有）
   → 指向「我爸爸是誰」
   → 是查找用的指標

🟧 prototype（只有「function / class」有）
   → 用 new 它時、被當作「下一代的爸爸」
   → 是建立用的設計圖
```

📄 **寫進程式檔**：`proto.js`

```text
class Animal {
  greet() { return "hi"; }
}

Animal.prototype;          // { greet: fn, constructor: Animal }
Animal.prototype.greet;    // function

const a = new Animal();
a.__proto__;               // 等於 Animal.prototype
a.__proto__ === Animal.prototype;   // true
```

**口訣**：**class 有 prototype（給後代用的）、instance 有 \_\_proto\_\_（指向爸爸）**。

  
### 4. 手動建立原型 —— `Object.create`

不用 class、也能玩繼承。

📄 **寫進程式檔**：`create.js`

```text
// 父原型
const animalProto = {
  greet() { return `Hi, I'm ${this.name}`; }
};

// 用 animalProto 當「爸爸」建一個物件
const dog = Object.create(animalProto);
dog.name = "小白";

dog.greet();     // "Hi, I'm 小白"
// greet 從 animalProto 找來的

dog.__proto__ === animalProto;   // true
```

> 💡 **`Object.create(null)` 的特殊用法**：建一個「沒爸爸」的純物件、沒有 `toString` / `hasOwnProperty` 等內建方法、適合當「乾淨字典」用、避免 prototype pollution（L21 會講）。

  
### 5. ES5 老式繼承（class 出現前的寫法）

📄 **寫進程式檔**：`old-extends.js`

```text
// 父建構函式
function Animal(name) {
  this.name = name;
}
Animal.prototype.greet = function() {
  return `Hi, I'm ${this.name}`;
};

// 子建構函式
function Dog(name, breed) {
  Animal.call(this, name);   // ← 等同 super(name)
  this.breed = breed;
}

// 建鏈：Dog.prototype.__proto__ = Animal.prototype
Dog.prototype = Object.create(Animal.prototype);
Dog.prototype.constructor = Dog;

Dog.prototype.bark = function() { return "汪"; };

const d = new Dog("小白", "柴犬");
d.greet();   // "Hi, I'm 小白"
d.bark();    // "汪"
```

> 💡 **看完這段就懂 class 解決什麼問題**——以前要 4 行才能繼承、現在 `extends` 一個字。**class 只是糖衣**。

  
### 6. 檢查原型關係

```text
const arr = [];

arr instanceof Array;                        // true
Array.prototype.isPrototypeOf(arr);          // true
Object.getPrototypeOf(arr) === Array.prototype;   // true

// 拿原型
Object.getPrototypeOf(arr);                  // Array.prototype

// 設原型（慢、不推薦）
Object.setPrototypeOf(arr, null);            // 切斷鏈
```

  
### 7. **prototype pollution（原型污染）**—— 真實資安漏洞

如果攻擊者能改 `Object.prototype`、**所有物件都被污染**。

📄 **寫進程式檔**：`pollution.js`

```text
// 危險的 deep merge（接受任何使用者輸入）
function merge(target, source) {
  for (const key in source) {
    if (typeof source[key] === "object") {
      merge(target[key] = target[key] || {}, source[key]);
    } else {
      target[key] = source[key];
    }
  }
}

// 攻擊
const attack = JSON.parse('{"__proto__": {"isAdmin": true}}');
merge({}, attack);

// 結果：所有物件都被污染
const u = {};
u.isAdmin;   // true ← 任意物件都有 isAdmin: true 了
```

**防禦**：

```text
✅ 禁止 key 為 __proto__ / constructor / prototype
✅ 用 Object.create(null) 建純物件
✅ 用 Map 代替當 key-value
✅ 用 schema validator（Zod / Yup）
✅ 凍結 Object.prototype（極端情況）
```

> 💡 **真實案例**：lodash、jQuery、moment 都被這個攻擊過。**新手寫的 deep merge 90% 有漏洞**。

  
### 8. 改 prototype 的「副作用」—— 影響全部

```text
// ✗ 危險：改全部陣列的行為
Array.prototype.last = function() {
  return this[this.length - 1];
};

[1, 2, 3].last();   // 3
// 但每個陣列都有 .last 了、可能跟其他 library 衝突

// ✓ 改用「靜態方法」或「工具函式」
function last(arr) { return arr[arr.length - 1]; }
```

**規則**：**不要動內建的 prototype**（Array / Object / String / ...）—— 容易跟其他 library 撞、升級踩坑。

  
### 9. 「new 一個建構函式」實際發生什麼

```text
const d = new Dog("小白", "柴犬");

// new 做 4 件事：
// 1. 建立空物件 {}
// 2. 把這物件的 __proto__ 設成 Dog.prototype
// 3. 把 this 綁定到這物件、呼叫 Dog("小白", "柴犬")
// 4. 如果 Dog 沒明確 return 物件、就回傳這個 this
```

  
### 10. ES2022 之後常用 prototype 動作

```text
// 拿自己的 keys（不含繼承來的）
Object.keys(obj);

// 連繼承的也要
for (const key in obj) {}

// 檢查屬性是「自己的」還是「繼承的」
Object.hasOwn(obj, "name");        // ES2022 新寫法、推薦
obj.hasOwnProperty("name");        // 舊寫法、有 pollution 風險
```

  
### 5 個常見坑

```text
❌ 1. 改 Array.prototype / Object.prototype
   → 跟所有 library 撞、地獄
   → 修：寫工具函式、別動內建原型

❌ 2. for...in 沒過濾繼承屬性
   → 拿到 prototype 上的 key
   → 修：用 Object.keys() 或 if (Object.hasOwn(obj, key))

❌ 3. obj.hasOwnProperty 被污染 / 被覆寫
   → user input 把 hasOwnProperty 改掉
   → 修：用 Object.hasOwn(obj, key)（ES2022）

❌ 4. deep merge 沒擋 __proto__
   → prototype pollution 漏洞
   → 修：擋 __proto__ / constructor / prototype 三個 key

❌ 5. instanceof 跨 iframe 失敗
   → 不同 window 的 Array 不同
   → 修：用 Array.isArray()（跨 frame 安全）
```

  
### 🎯 重點回顧

- **每個物件都有 `__proto__` 指向爸爸**——找不到屬性就往上找
- **`prototype` 是 class / function 才有**、給後代當「爸爸」用
- **class 是 prototype 的糖衣**——本質還是這套機制
- **prototype pollution 是真實漏洞**——deep merge 不可信資料要擋 `__proto__`
- **不要動內建 prototype**——影響全域、難 debug
