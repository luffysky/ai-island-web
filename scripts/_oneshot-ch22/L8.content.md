**Zero-downtime Deploy（零停機部署）= deploy 新版本時、使用者感覺不到任何中斷**。**4 大策略**：藍綠（blue-green）、金絲雀（canary）、滾動（rolling）、feature flag。**沒做 = 每次 deploy 客戶都看到 502**。

> 💡 **這課的價值**：上線到一定流量後一定要懂。學會 = 你能凌晨 3 點 deploy、客戶毫無感覺。Netflix / Google / Vercel 都這樣做。

  
### 1. 傳統 deploy 為什麼會 down

```text
naive deploy：
1. 停舊版（5 秒～ 1 分鐘）
2. 換 code
3. 啟動新版（10 秒 ～ 1 分鐘）

→ 中間這段使用者看到：
   - 502 Bad Gateway
   - "Service Unavailable"
   - 連線斷掉

→ 流量大時：每次 deploy 影響上萬人
```

  
### 2. 4 大策略

```text
策略           原理                     風險   成本   速度
────────────────────────────────────────────────────────
Rolling        逐台替換                 中     低     中
Blue-Green     雙環境切換                低     高     快
Canary         先給 1% 試               低     中     慢
Feature Flag   code 上線 + 開關控制      最低   低     可分段
```

  
### 3. Rolling Update（滾動更新）

**最常見、k8s 預設**：1 台 1 台慢慢替換。

```text
Before：v1 v1 v1 v1 v1     (5 台都是舊)
Step 1：v2 v1 v1 v1 v1     (殺第 1 台、起新版)
Step 2：v2 v2 v1 v1 v1
Step 3：v2 v2 v2 v1 v1
Step 4：v2 v2 v2 v2 v1
After： v2 v2 v2 v2 v2     (全新版)
```

**優點**：✅ 不用雙倍機器、簡單
**缺點**：⚠️ deploy 過程中有「v1 + v2 共存」——要相容
        ⚠️ 出包要 rolling back、慢

📄 **寫進程式檔**：`k8s-deployment.yaml`

```text
spec:
  replicas: 5
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1   # 一次最多 1 台不可用
      maxSurge: 1         # 一次最多多開 1 台
```

  
### 4. Blue-Green Deployment（藍綠部署）

**兩個完全一樣的環境、切換流量**。

```text
Before：
   Blue  (v1)  ← 所有流量
   Green (none)

Deploy：
   Blue  (v1)  ← 流量
   Green (v2)  ← 部署完成、跑 smoke test

Switch（DNS / Load Balancer 切換）：
   Blue  (v1)  ← 0 流量、留著當 rollback
   Green (v2)  ← 所有流量

完成（24 小時後）：
   Blue → 升級成 v2 待命
   Green (v2)
```

**優點**：
- ✅ 切換瞬間完成
- ✅ Rollback 也是瞬間（切回 Blue）
- ✅ 不用 v1 + v2 相容

**缺點**：
- ⚠️ 要雙倍機器（貴）
- ⚠️ DB schema migration 處理麻煩

**工具**：
- AWS：CodeDeploy 內建
- k8s：用 Service selector 切
- Fly.io：fly deploy --strategy bluegreen

  
### 5. Canary Deployment（金絲雀部署）

**先給 1% 流量試、沒事再放大**——名字來自礦工帶金絲雀下礦（牠先死你才知道有毒）。

```text
Phase 1：1% 流量 → v2
        99% 流量 → v1
        
        觀察 10 分鐘
        - error rate？
        - latency？
        - business metrics（轉換率）？

Phase 2（沒問題）：10% → v2、90% → v1
Phase 3：25% → v2、75% → v1
Phase 4：50% → v2、50% → v1
Phase 5：100% → v2

Phase X（有問題）：立刻拉回 0%
```

**優點**：
- ✅ 風險最低
- ✅ 真實流量驗證
- ✅ 自動 rollback（metric 不對就停）

**缺點**：
- ⚠️ deploy 時間長（小時～天）
- ⚠️ 要好的 metric / observability
- ⚠️ v1 + v2 要相容

**工具**：
- k8s + Argo Rollouts / Flagger
- Istio / Linkerd（service mesh）
- AWS App Mesh

📄 **Argo Rollouts 範例**：

```text
spec:
  strategy:
    canary:
      steps:
        - setWeight: 1     # 1% 流量到新版
        - pause: { duration: 10m }
        - setWeight: 10
        - pause: { duration: 10m }
        - setWeight: 25
        - pause: { duration: 10m }
        - setWeight: 50
        - pause: { duration: 10m }
        - setWeight: 100
      analysis:           # 自動看 metric
        templates:
          - templateName: error-rate
        args:
          - name: service-name
            value: my-app
```

  
### 6. Feature Flag（功能旗標）

**code 上線、但功能用「旗標」控制**——分離 deploy 跟 release。

```text
傳統：
   code change ── deploy ── 上線

Feature Flag：
   code change ── deploy（all users 都 deploy）
                       ↓
                  flag = off → 沒人看到新功能
                       ↓
                  flag = 1% → 1% 看到
                       ↓
                  flag = 100% → 全開
                       ↓
                  flag = off → 出問題立刻關
```

📄 **寫進程式檔**：`feature.ts`

```text
import { useFlag } from '@vercel/flags/next';

function CheckoutPage() {
  const newPayment = useFlag('new-payment-flow');

  return newPayment
    ? <NewPaymentUI />
    : <OldPaymentUI />;
}
```

**工具**：
- **LaunchDarkly**（業界龍頭）
- **Unleash**（開源）
- **PostHog**（含 analytics）
- **Vercel Flags**
- **GrowthBook**

**好處**：
- ✅ deploy 跟 release 解耦
- ✅ A/B test 同一條 pipeline
- ✅ 立刻 rollback（關 flag、不用 redeploy）
- ✅ 漸進式發布

  
### 7. DB Migration — 零停機的最大挑戰

**問題**：v1 跟 v2 對 schema 要求不一樣。

**Expand → Migrate → Contract**（L19.10 講過）：

```text
Phase 1：Expand（擴展）
   - 加新欄位 / 表
   - v1 不用、v2 用
   - 兩版本都能跑

Phase 2：Migrate（遷移）
   - 把 v1 流量切到 v2
   - 回填舊資料

Phase 3：Contract（收縮）
   - 確認沒問題後
   - 拿掉舊欄位 / 表
```

**例子：把 `name` 拆成 `firstName` / `lastName`**：

```text
v1 deploy：只用 name
↓
DB migration：加 firstName / lastName 欄位（NULL 允許）
↓
v2 deploy：v2 寫 firstName / lastName + 維持寫 name
↓
backfill：UPDATE 舊資料、name → firstName + lastName
↓
v3 deploy：v3 只用 firstName / lastName
↓
DB migration：DROP name 欄位
```

  
### 8. Health Check + Graceful Shutdown

**Health Check（健康檢查）**：

📄 **寫進程式檔**：`server.ts`

```text
app.get('/health', (req, res) => res.json({ ok: true }));

app.get('/ready', async (req, res) => {
  // 檢查 DB / cache 都正常才回 200
  try {
    await db.query('SELECT 1');
    await redis.ping();
    res.json({ ready: true });
  } catch {
    res.status(503).json({ ready: false });
  }
});
```

**load balancer / k8s**：定期 ping、不健康就停送流量。

**Graceful Shutdown（優雅關閉）**：

```text
process.on('SIGTERM', async () => {
  console.log('Received SIGTERM, shutting down...');

  // 1. 停接新 request
  server.close();

  // 2. 等現有 request 結束（最多 30 秒）
  await new Promise(r => setTimeout(r, 30000));

  // 3. 關 DB / redis 連線
  await db.end();
  await redis.quit();

  process.exit(0);
});
```

**對 k8s**：`terminationGracePeriodSeconds: 30`

  
### 9. Rollback — 出包怎麼救

```text
策略         Rollback 時間   方法
─────────────────────────────────────
Rolling      5 ~ 30 分鐘     redeploy 舊版（慢）
Blue-Green   瞬間            切回 Blue
Canary       立刻            自動偵測停 promotion
Feature Flag 瞬間            關 flag（不用 redeploy）
```

**rollback 演習**：每月做一次！不演習真出事手忙腳亂。

  
### 10. 5 大常見坑

```text
❌ 1. 沒 health check 直接 deploy
   → app 還沒 ready 流量就打進來、5xx
   → 修：永遠加 /health + /ready

❌ 2. SIGTERM 不處理
   → 進行中的 request 被斬斷
   → 修：listen SIGTERM、graceful shutdown

❌ 3. DB migration 跟 code deploy 同一步
   → schema 改了、舊 code 還在跑、爆
   → 修：Expand → Migrate → Contract

❌ 4. Canary 沒設自動 rollback
   → 半夜出包、沒人盯著
   → 修：error rate > 1% 自動拉回

❌ 5. 從沒演習過 rollback
   → 真出事手忙腳亂
   → 修：每月做一次 rollback drill
```

  
### 🎯 重點回顧

- **4 大策略**：Rolling / Blue-Green / Canary / Feature Flag
- **Rolling** 簡單便宜、**Blue-Green** 快但貴、**Canary** 風險最低、**Feature Flag** 最彈性
- **Health Check + Graceful Shutdown** 是基本盤
- **DB migration 用 Expand → Migrate → Contract** 三段式
- **Rollback 必須演習**——沒練過 = 真出事抓瞎
- **Feature Flag** 把 deploy 跟 release 解耦、業界趨勢
