**部署（Deploy）= 把你的程式從「自己電腦」搬到「網路上其他人也能用」**。**世界上一半工程師卡在這關**——code 寫得很好、但不會把它放到線上。

> 💡 **這課的價值**：先把整個部署地圖印在腦袋裡——dev → build → artifact → runtime → 監控。之後每一課你會知道自己在哪一段。

  
### 1. 部署「從電腦到全世界」的 5 個階段

```text
🟢 Stage 1：Dev（開發）
   - 你電腦上跑（localhost:3000）
   - 只有你看得到

🟡 Stage 2：Build（建置）
   - source code → 可執行的 artifact
   - 編譯 / 打包 / 壓縮
   - 例：tsc → .js、npm run build → dist/

🔵 Stage 3：Artifact（產物）
   - 一份「打包好的程式」
   - 可以是：tarball、Docker image、static folder
   - 存到 Container Registry / S3

🟣 Stage 4：Runtime（執行環境）
   - 真正在跑的機器
   - VM / Container / Serverless / Edge

🔴 Stage 5：監控 + 維運
   - log / metric / alert
   - 出事要知道 + 救得回來
```

  
### 2. 6 種 deploy 方式（演進史）

```text
1990s  傳統 server
       自己買機器、放機房

2006   AWS EC2 = 虛擬機（VM）
       租 AWS 的機器、自己裝

2010s  PaaS（Heroku）
       push code、Heroku 跑全部
       不用管 OS

2014   Docker = container
       打包 OS + 程式、到處都能跑

2015   Kubernetes（k8s）
       管很多 container

2018   Serverless（Vercel / Lambda）
       連 server 都不用管
       上 code、按用量算錢

2020s  Edge Compute（Cloudflare Workers）
       跑在離使用者最近的機器
```

  
### 3. 環境（Environment）— 至少 3 套

```text
🟢 Dev（local）
   你電腦
   - 假資料
   - 隨便壞
   - 快速 iterate

🟡 Staging / Preview
   仿正式的測試環境
   - 用真資料 sample
   - PR 自動部署
   - 給 QA / PM 測

🔴 Production（線上）
   真正服務客戶
   - 真資料、真錢
   - 不能掛
   - 不能隨便改
```

**有些公司**還會加：
- `Dev` → `Test` → `UAT` → `Pre-prod` → `Prod`（金融常見、5 層）
- `Canary`（先給 1% 流量試）

  
### 4. 12-Factor App — 經典原則

Heroku 提出、業界公認最佳實踐：

```text
1. Codebase    一個 repo
2. Dependencies 明確宣告（package.json / requirements.txt）
3. Config      環境變數、不寫死
4. Backing Services 把 DB / Redis 當資源
5. Build / Release / Run  三階段分離
6. Processes   無狀態
7. Port binding 自帶 web server
8. Concurrency 用 process 模型擴展
9. Disposability 啟動快、優雅關閉
10. Dev / Prod parity  環境差異小
11. Logs       stdout（不寫檔案）
12. Admin processes 一次性 script 跟 app 一起
```

**白話**：每條都是被「半夜叫醒救機」的工程師寫的血淚教訓。

  
### 5. CI / CD — 自動化部署

```text
CI（Continuous Integration、持續整合）
   - push code → 自動跑 test / lint / build
   - 確認沒爆

CD（Continuous Deployment、持續部署）
   - test 過 → 自動 deploy
   - 不用人按按鈕
```

📄 **完整流程**：

```text
1. 工程師 push code 到 PR branch
2. GitHub Actions 觸發：
   - lint
   - typecheck
   - test
   - build
   - 部署到 Preview URL
3. PR review pass
4. merge 到 main
5. GitHub Actions：
   - 再跑一次 test
   - build production artifact
   - 部署到 staging
   - 跑 smoke test
   - 部署到 production
   - 通知 Slack「上線了」
```

  
### 6. 4 種 runtime 取捨

```text
🟢 PaaS（Vercel / Railway / Render）
   ✅ 零設定、push 就跑
   ✅ 自動 HTTPS / CDN / Preview
   ❌ 鎖死平台、貴
   適合：90% web app

🔵 Container（Docker / k8s）
   ✅ 控制權最大
   ✅ 任何雲都能跑
   ❌ 要會寫 Dockerfile / yaml
   適合：中大型、多 service

🟣 Serverless（Lambda / Cloudflare Workers）
   ✅ 按量計費、自動 scale
   ✅ 沒 idle 成本
   ❌ cold start
   ❌ 執行時間有限（Lambda 15 分鐘）
   適合：API / 偶爾跑的任務

🟡 VM（EC2 / GCE / DigitalOcean Droplet）
   ✅ 最便宜（大量時）
   ✅ 最自由
   ❌ 自己裝、自己 patch
   適合：成本敏感、特殊需求
```

  
### 7. 監控（Observability、可觀測性）

```text
📊 Metrics（指標）
   - 數字：QPS、延遲、CPU、記憶體
   - 工具：Prometheus、Grafana、DataDog

📜 Logs（日誌）
   - 文字：每個 request、每個 error
   - 工具：CloudWatch、Loki、Logtail

🔍 Traces（追蹤）
   - request 在多個 service 間的旅程
   - 工具：OpenTelemetry、Jaeger、Honeycomb

🚨 Alerts（警報）
   - 出事打電話 / 發 Slack
   - 工具：PagerDuty、Opsgenie
```

  
### 8. 5 大常見坑

```text
❌ 1. 沒有 staging、直接 deploy production
   → 一推上線就壞、客戶看到
   → 修：永遠 staging 先

❌ 2. config 寫死在 code 裡
   → API key / DB URL 都 hardcode
   → 修：用環境變數、機密管理

❌ 3. 沒 CI/CD、手動 deploy
   → 每次都漏步驟、出包
   → 修：第一天就接 GitHub Actions

❌ 4. logs 寫進 file
   → 機器掛了 logs 也沒了
   → 修：永遠 stdout、外部蒐集

❌ 5. 沒監控、出事才知道
   → 客戶 email 罵了才發現掛 2 小時
   → 修：metric + alert + uptime monitor
```

  
### 9. 一個典型 startup 的部署棧

```text
🏗️ source 控制：GitHub
🔄 CI/CD：     GitHub Actions
📦 build：     Docker / Vercel build
☁️ runtime：   Vercel（frontend）+ Railway（API + DB）
🗄️ DB：        Supabase / Neon
🔐 secrets：   Vercel env / Doppler
📊 monitor：   Sentry（errors）+ Vercel Analytics
📝 logs：      Logtail / BetterStack
🚨 alerts：    Slack webhook
```

### 🎯 重點回顧

- **部署 5 階段**：Dev → Build → Artifact → Runtime → 監控
- **6 種 deploy 方式**——從機房到 Edge、各有適用
- **環境**：Dev / Staging / Production 至少 3 套
- **12-Factor App** 是業界血淚教訓
- **CI / CD 自動化**——push 就 deploy
- **4 種 runtime**：PaaS / Container / Serverless / VM
- **監控四要素**：Metrics / Logs / Traces / Alerts
