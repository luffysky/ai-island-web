**Kubernetes（k8s）= Google 開源的「container 大規模管家」**——管 100 台機器上的 1000 個 container、自動調度、自動恢復、自動 scaling。**業界標準、學習曲線最陡**。

> 💡 **這課的價值**：先給警告——99% 的 startup 不需要 k8s、用了反而拖垮 team。但知道它存在、何時該用、概念怎麼運作很重要。

  
### 1. 警告：你可能不需要 k8s

```text
❌ 不要用 k8s 如果：
   - team < 5 人
   - 沒專職 DevOps
   - 月流量 < 100 萬
   - 沒做微服務
   
   → 用 Railway / Fly.io / Render 就好

✅ 適合 k8s 如果：
   - 流量 / 服務數量大（10+ 微服務）
   - 跨雲、跨 region
   - 有專職 SRE team
   - 法規要求自建
```

**口號**：**「k8s 是 Google 的問題、不是你的問題」**。

  
### 2. k8s 解什麼問題

```text
情境：100 個 container 在 50 台機器
- 哪台機器 down 了、上面的 container 要遷
- 流量大了、自動加 container
- 升級不能 down
- network / storage / secrets 統一管理

裸 Docker：自己刻一個 6 個月 + 永遠在救火
k8s：寫 yaml 描述「我要什麼」、它幫你達成
```

**核心理念**：**Declarative（宣告式）**——你說「我要 5 個 web 容器」、k8s 自動維持 5 個。掛 1 個自動補 1 個。

  
### 3. 五大核心物件

```text
🧊 Pod（豆莢）
   - k8s 最小單位
   - 1 個 Pod = 1+ 個 container 共享網路 + 儲存
   - 通常一個 Pod 跑一個 app
   - Pod 是 ephemeral（死了不會回來）

📦 Deployment（部署）
   - 管 Pod 的「desired state」
   - 「我要 5 個 nginx」
   - Pod 掛了自動補
   - 升級用 rolling update

🌐 Service（服務）
   - 給 Pod 一個穩定的網址
   - Pod IP 會變、Service 不會
   - load balance 流量到多個 Pod

🚪 Ingress（入口）
   - 把外部流量路由到 Service
   - 處理 HTTPS、domain routing
   - 像 nginx / load balancer

🔐 ConfigMap / Secret（設定 / 密鑰）
   - 環境變數、設定檔
   - Secret 加密儲存（base64 + 加密）
```

  
### 4. 第一個 k8s yaml

📄 **寫進程式檔**：`deployment.yaml`

```text
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3              # 我要 3 個
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
        - name: web
          image: ghcr.io/me/my-app:1.2.3
          ports:
            - containerPort: 3000
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: my-secrets
                  key: db-url
          resources:
            requests:
              memory: "128Mi"
              cpu: "100m"
            limits:
              memory: "256Mi"
              cpu: "500m"
          livenessProbe:
            httpGet:
              path: /health
              port: 3000
            periodSeconds: 30
          readinessProbe:
            httpGet:
              path: /ready
              port: 3000
            periodSeconds: 5
---
apiVersion: v1
kind: Service
metadata:
  name: my-app-svc
spec:
  selector:
    app: my-app
  ports:
    - port: 80
      targetPort: 3000
  type: ClusterIP
---
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-app-ing
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt
spec:
  tls:
    - hosts: [my-app.com]
      secretName: my-app-tls
  rules:
    - host: my-app.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: my-app-svc
                port:
                  number: 80
```

🖥️ **在終端機輸入**：

```text
kubectl apply -f deployment.yaml
```

💬 **電腦會顯示**：

```text
deployment.apps/my-app created
service/my-app-svc created
ingress.networking.k8s.io/my-app-ing created
```

  
### 5. 常用 kubectl 指令

🖥️ **在終端機輸入**：

```text
# 看 cluster 狀態
kubectl get nodes
kubectl get pods
kubectl get deploy
kubectl get svc
kubectl get ingress

# 看細節
kubectl describe pod my-app-xxx
kubectl logs my-app-xxx
kubectl logs -f my-app-xxx        跟著看
kubectl logs --previous my-app-xxx  上次 crash 的 log

# 進去 container
kubectl exec -it my-app-xxx -- bash

# 改 replica
kubectl scale deployment my-app --replicas=10

# 看歷史
kubectl rollout history deployment my-app
kubectl rollout undo deployment my-app   ← rollback

# 刪
kubectl delete pod my-app-xxx
kubectl delete -f deployment.yaml
```

  
### 6. Auto-scaling — HPA

📄 **寫進程式檔**：`hpa.yaml`

```text
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: my-app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-app
  minReplicas: 2
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
```

**作用**：CPU > 70% 自動加 Pod、降下來自動減。

### 9. 真實成本

```text
☁️ Managed k8s 月費：
   - control plane：$70 ~ $100（GKE / EKS）
   - 3 個 node（4core/16GB）：$300 ~ $500
   - load balancer：$20
   - 總計：~$500/月（最小可用）

👥 人力成本：
   - 至少要 1 個 SRE / DevOps
   - 學習曲線：3 ~ 6 個月才上手

🎯 ROI 怎麼算？
   - 服務 < 5 個：不划算
   - 月流量 < 1000 萬：不划算
   - 月營收 < $10000：不划算
```

### 11. 5 大常見坑

```text
❌ 1. startup 小團隊就上 k8s
   → 全 team 在學 yaml、業務沒推進
   → 修：先用 Railway / Fly.io、流量到了再說

❌ 2. 沒設 resource limits
   → 一個 pod 吃光記憶體、整台 node 倒
   → 修：每個 container 設 requests + limits

❌ 3. 沒 liveness / readiness probe
   → crash pod 還在 routing、500 滿天飛
   → 修：永遠加 /health + /ready

❌ 4. secrets 直接 commit
   → base64 不是加密！
   → 修：用 Sealed Secrets / External Secrets

❌ 5. 自己裝 k8s（不用 managed）
   → control plane 維護地獄
   → 修：用 EKS / GKE / AKS
```

  
### 🎯 重點回顧

- **k8s = container 編排系統**——管 1000+ container 自動調度
- **5 大物件**：Pod / Deployment / Service / Ingress / ConfigMap+Secret
- **宣告式**——yaml 寫「我要什麼」、k8s 自動維持
- **HPA** 自動 scaling、根據 CPU / metric
- **99% startup 不需要 k8s**——先用 Railway / Fly / Render
- **若一定要用**：用 managed（EKS / GKE）、別自己裝
