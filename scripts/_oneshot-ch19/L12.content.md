**Multi-tenant（多租戶）= 一個系統服務多個客戶、彼此資料完全隔離**。**SaaS 的核心架構抉擇**——選錯會讓你「A 公司看到 B 公司資料」直接被告。

> 💡 **這課的價值**：做 SaaS（Notion / Slack / Linear）一定要懂。4 種模式各有取捨、選對能省 70% 成本、選錯會出大事。

  
### 1. 什麼是 Tenant

```text
🏢 Tenant（租戶）= 你的客戶（一家公司）
👤 User（使用者）= 客戶內的人

例：
- Slack 的 tenant = 一個 workspace（Anthropic / Apple / ...）
- Notion 的 tenant = 一個 team
- Linear 的 tenant = 一個 organization
```

**核心需求**：
- A workspace 的訊息、B workspace 永遠看不到
- A 想付錢升級、B 不受影響
- A 出問題、B 不受拖累

  
### 2. 4 種架構模式

```text
模式                      隔離度  成本   複雜度  適用
─────────────────────────────────────────────────────
1. Shared Schema（共用表）  ★★    ★      ★★      小客戶多、SaaS 早期
2. Schema per Tenant       ★★★   ★★     ★★★    中型、需要隔離
3. Database per Tenant     ★★★★  ★★★    ★★★★   大客戶、強合規
4. Hybrid（混合）           ★★★★  ★★★    ★★★★★  Notion / Slack 這種規模
```

  
### 3. 模式 1：Shared Schema（共用表 + tenant_id）

**最簡單**：所有 tenant 共用同一張表、每筆加 `tenant_id` 欄位。

📄 **寫進程式檔**：`schema.sql`

```text
CREATE TABLE tenants (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  plan TEXT NOT NULL DEFAULT 'free'
);

CREATE TABLE posts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID NOT NULL REFERENCES tenants(id),
  title TEXT NOT NULL,
  body TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
);

-- 每張表都加這個 index
CREATE INDEX idx_posts_tenant ON posts(tenant_id);

-- 永遠先 WHERE tenant_id =
SELECT * FROM posts WHERE tenant_id = $1 AND ...;
```

**優點**：
- 簡單、便宜（1 個 DB）
- 加新客戶 = INSERT 一筆 tenants
- 升級 schema 一次搞定

**缺點**：
- 漏寫一個 `WHERE tenant_id` = 大災難
- 大客戶吃掉所有資源
- 備份 / 還原單一客戶很難
- 合規（GDPR）難——刪客戶要 DELETE WHERE tenant_id

  
### 4. RLS（Row Level Security）— 救命的安全網

**Postgres 內建**：每張表設「policy」、自動加 WHERE。

📄 **寫進程式檔**：`rls.sql`

```text
-- 開啟 RLS
ALTER TABLE posts ENABLE ROW LEVEL SECURITY;

-- 設 policy：只能看到自己 tenant 的 row
CREATE POLICY tenant_isolation ON posts
  FOR ALL
  USING (tenant_id = current_setting('app.tenant_id')::uuid);

-- 每個 request 先設
SET app.tenant_id = 'xxx-xxx-xxx';

SELECT * FROM posts;
-- 自動變 SELECT * FROM posts WHERE tenant_id = 'xxx-xxx-xxx';
```

**Supabase 的賣點之一**：RLS + JWT 結合、前端可以直接打 DB 而不洩漏。

  
### 5. 模式 2：Schema per Tenant

**每個 tenant 一個 schema、同一個 DB**。

```text
DB: mydb
  ├── schema: tenant_apple
  │     ├── posts
  │     ├── users
  │     └── ...
  ├── schema: tenant_google
  │     ├── posts
  │     ├── users
  │     └── ...
```

📄 **寫進程式檔**：`provision.sql`

```text
-- 新客戶 sign up
CREATE SCHEMA tenant_apple;

CREATE TABLE tenant_apple.posts (
  id UUID PRIMARY KEY,
  title TEXT,
  -- ... 同樣的 schema
);

-- application 切換
SET search_path TO tenant_apple, public;
```

**優點**：
- 不會漏 WHERE（每個 tenant 自己一份）
- 單客戶備份簡單（pg_dump -n tenant_apple）
- 客戶要客製 schema 容易

**缺點**：
- 升級 schema 要跑 N 次（每個 tenant）
- 1000 個 tenant = 1000 個 schema = PG 元資料肥
- 跨 tenant 統計很難

  
### 6. 模式 3：Database per Tenant

**每個 tenant 一個完整 DB**。

```text
PG cluster:
  ├── db: tenant_apple
  ├── db: tenant_google
  ├── db: tenant_netflix
```

**優點**：
- 最強隔離（系統層）
- 大客戶可獨立升級 / 備份 / 還原
- 合規最容易（刪 DB 就行）

**缺點**：
- 成本最高（每個 DB 一份 PG instance / 連線）
- 跨 tenant 完全不能 JOIN
- 1000 tenant = 1000 DB = 管理地獄

**適合**：
- 大企業客戶（每家付 $50,000+/年）
- 醫療 / 金融（HIPAA / PCI 強合規）
- 客戶要求「資料在我自己機房」

  
### 7. 模式 4：Hybrid（混合）

**Notion / Slack 的真實做法**：

```text
免費 / 小客戶 → Shared Schema（節省成本）
中型客戶     → Schema per Tenant
大客戶 / 企業 → 獨立 DB
超大客戶     → 獨立整個 cluster
```

**好處**：成本跟隔離度都最佳化。

**壞處**：架構複雜、要寫 router 把 request 導到對的地方。

  
### 8. 怎麼選

```text
問題清單：
1. 你預期最大客戶會多大？
   - 小 → Shared OK
   - 大 → DB per Tenant

2. 你預期客戶數會多少？
   - < 1000 → Schema per Tenant 還能管
   - > 10000 → Shared 才省得起

3. 你客戶有合規需求嗎？
   - HIPAA / PCI / GDPR 嚴格 → DB per Tenant

4. 你能投入多少 ops？
   - 1 人 team → Shared + RLS（最省心）
   - 有 DBA → 可上 Hybrid

5. 客戶差距大不大？
   - 均勻 → 一種模式即可
   - 差很大 → Hybrid
```

  
### 9. Tenant 認證流程

```text
1. User 登入 → 拿到 JWT
   {
     "sub": "user_abc",
     "tenant_id": "tnt_apple",
     "role": "admin",
     ...
   }

2. 每個 request 帶 JWT

3. middleware 解析 JWT、設 tenant_id
   SET app.tenant_id = 'tnt_apple';

4. RLS policy 自動加 WHERE
   → 永遠只看到 tnt_apple 的資料
```

📄 **寫進程式檔**：`middleware.ts`

```text
export async function tenantMiddleware(req, res, next) {
  const jwt = req.headers.authorization?.replace('Bearer ', '');
  const payload = verifyJWT(jwt);

  // 設 PG session variable
  await db.query(`SET app.tenant_id = '${payload.tenant_id}'`);
  await db.query(`SET app.user_id = '${payload.sub}'`);

  req.tenant = payload.tenant_id;
  next();
}
```

  
### 10. 5 大常見坑

```text
❌ 1. 沒用 RLS、靠 application 加 WHERE
   → 漏一個 query 就 cross-tenant 洩漏
   → 修：永遠開 RLS、application 是第二層

❌ 2. tenant_id 不在 index 開頭
   → 變超慢
   → 修：所有 index 第一欄都是 tenant_id

❌ 3. Shared 模式下大客戶吃光資源
   → 中小客戶被拖慢
   → 修：partial migrate 大客戶到 dedicated

❌ 4. 跨 tenant 統計用 service role 繞 RLS
   → 寫錯 bug 一樣會洩漏
   → 修：跑在獨立 read-only replica + audit log

❌ 5. 新客戶 onboarding 沒自動化
   → 1000 個 tenant 手動建 schema = 累死
   → 修：寫 provision script、客戶 sign up 自動跑
```

  
### 🎯 重點回顧

- **4 種模式**：Shared / Schema / Database / Hybrid——隔離 vs 成本的權衡
- **Shared Schema + RLS** 是 80% SaaS 的起點
- **RLS（Row Level Security）** 是救命安全網、永遠開
- **JWT 帶 tenant_id** → middleware 設 PG session → RLS 自動篩
- **Hybrid 是大廠的真實做法**——免費客戶 Shared、大客戶獨立 DB
- **所有 index 第一欄永遠是 tenant_id**
