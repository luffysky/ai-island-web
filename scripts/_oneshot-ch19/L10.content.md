**Migration（遷移）= 改 schema（資料表結構）的「正式流程」**——加欄位、改型別、加 index、改約束都算。**改錯 = 線上服務 lock 半小時、客戶罵到爆**。

> 💡 **這課的價值**：90% 工程師「直接 ALTER TABLE」結果把服務搞掛。學會 Migration 策略 = 你能在不停機的情況下、改任何 schema。

  
### 1. Migration vs Data Migration

```text
🏗️ Schema Migration（結構遷移）
   改「表的結構」：
   - CREATE TABLE
   - ADD / DROP COLUMN
   - ALTER TYPE
   - ADD INDEX
   - ADD CONSTRAINT

📦 Data Migration（資料遷移）
   改「裡面的資料」：
   - 把 phone 從 string 拆成 country_code + number
   - 把舊的 status='1' 改成 'active'
   - 批次匯入 / 匯出
```

通常一起做：先 schema migration 加新欄位、再 data migration 填值。

### 3. Migration 檔長什麼樣

📄 **寫進程式檔**：`supabase/migrations/20260528_add_user_role.sql`

```text
-- Up：要做什麼
ALTER TABLE users ADD COLUMN role TEXT NOT NULL DEFAULT 'user';
CREATE INDEX idx_users_role ON users(role);

-- Down（rollback）：怎麼撤
-- DROP INDEX idx_users_role;
-- ALTER TABLE users DROP COLUMN role;
```

**特性**：
- 檔名含時間戳記、自動排序
- 工具記錄「跑過哪些」（schema_migrations 表）
- 重跑只會跑新的

  
### 4. 危險動作：哪些 ALTER 會 lock 整桌

PG 9.6 之前幾乎所有 ALTER 都會 lock：

```text
🔴 整桌 lock（其他 query 全卡住）：
   ALTER TABLE ADD COLUMN ... NOT NULL   ⚠️ 沒 default
   ALTER TABLE ADD COLUMN ... DEFAULT 某值  ⚠️ PG 11- 會 rewrite
   ALTER TABLE ALTER COLUMN TYPE ...     ⚠️ 大多會 rewrite
   ALTER TABLE ADD PRIMARY KEY
   ALTER TABLE ADD UNIQUE CONSTRAINT
   CREATE INDEX（不加 CONCURRENTLY）

🟢 不 lock 或秒殺：
   ALTER TABLE ADD COLUMN ... NULL     ✅ 秒殺
   ALTER TABLE ADD COLUMN ... DEFAULT（PG 11+）  ✅ 秒殺
   CREATE INDEX CONCURRENTLY ...        ✅ 不擋讀寫
   ALTER TABLE DROP COLUMN              ✅ 秒殺（內部標記）
```

  
### 5. 零停機改 schema — 三段式策略

**情境：把 `users.name` 拆成 `first_name` + `last_name`**

#### Phase 1：加新欄位（不破壞舊的）

```text
ALTER TABLE users ADD COLUMN first_name TEXT;
ALTER TABLE users ADD COLUMN last_name TEXT;
```

application 同時寫舊 `name` 跟新欄位。

#### Phase 2：回填舊資料

```text
-- 批次處理、不要 UPDATE 整桌（會 lock）
UPDATE users
SET first_name = split_part(name, ' ', 1),
    last_name  = split_part(name, ' ', 2)
WHERE first_name IS NULL
  AND id BETWEEN 1 AND 10000;
-- 跑 N 次
```

#### Phase 3：切讀寫 + 拿掉舊欄位

```text
-- application 改讀 first_name / last_name
-- 確認沒問題後

ALTER TABLE users DROP COLUMN name;
```

**口訣**：**「Expand → Migrate → Contract」**（展開 → 遷移 → 收縮）。

  
### 6. CREATE INDEX CONCURRENTLY

❌ **錯**（會 lock 整桌寫入）：

```text
CREATE INDEX idx_users_email ON users(email);
```

✅ **對**：

```text
CREATE INDEX CONCURRENTLY idx_users_email ON users(email);
```

**注意**：
- CONCURRENTLY 不能在 transaction 內
- 失敗會留下 INVALID index、要 DROP 重建
- 慢 5 ~ 10 倍、但不擋讀寫

  
### 7. NOT NULL + DEFAULT 的陷阱

❌ **PG 10 以下會 rewrite 整桌**：

```text
ALTER TABLE users ADD COLUMN status TEXT NOT NULL DEFAULT 'active';
```

✅ **安全做法（PG 11+ 已經修了、但保險寫法）**：

```text
-- Step 1: 加 nullable
ALTER TABLE users ADD COLUMN status TEXT;

-- Step 2: 加 default（PG 11+ 秒殺）
ALTER TABLE users ALTER COLUMN status SET DEFAULT 'active';

-- Step 3: 批次回填
UPDATE users SET status = 'active' WHERE status IS NULL AND id < 10000;
-- ... 重複

-- Step 4: 加 NOT NULL（PG 12+ 用 ADD CONSTRAINT NOT VALID 再 VALIDATE）
ALTER TABLE users ADD CONSTRAINT users_status_notnull
  CHECK (status IS NOT NULL) NOT VALID;
ALTER TABLE users VALIDATE CONSTRAINT users_status_notnull;

-- Step 5：最後改成 NOT NULL
ALTER TABLE users ALTER COLUMN status SET NOT NULL;
```

  
### 8. Lock timeout — 防呆

📄 **寫進程式檔**：`migration.sql`

```text
SET lock_timeout = '3s';
SET statement_timeout = '10s';

ALTER TABLE users ADD COLUMN xxx TEXT;
```

**作用**：拿不到 lock 就直接失敗、不會卡死整個 DB。

  
### 9. Data Migration — 大表分批處理

❌ **錯**：

```text
UPDATE users SET phone = format_phone(phone);
-- 1000 萬筆、一次跑、lock 半小時
```

✅ **對**：

```text
DO $$
DECLARE
  batch_size INT := 1000;
  affected   INT;
BEGIN
  LOOP
    UPDATE users
    SET phone = format_phone(phone)
    WHERE id IN (
      SELECT id FROM users
      WHERE phone_formatted = false
      LIMIT batch_size
    );
    GET DIAGNOSTICS affected = ROW_COUNT;
    EXIT WHEN affected = 0;
    PERFORM pg_sleep(0.1);  -- 喘息
    COMMIT;                  -- 每批一個 transaction
  END LOOP;
END $$;
```

### 11. 5 大常見坑

```text
❌ 1. 直接在 production 跑 ALTER
   → 沒 lock_timeout、卡死整個 DB
   → 修：永遠 SET lock_timeout = '3s' 開頭

❌ 2. 一次改 schema + 改 application
   → schema 改了、舊 app 還在跑、爆 error
   → 修：Expand → Migrate → Contract 三段式

❌ 3. CREATE INDEX 沒加 CONCURRENTLY
   → lock 寫入 30 分鐘
   → 修：production 永遠加 CONCURRENTLY

❌ 4. Migration 沒有 rollback
   → 改錯了沒辦法救
   → 修：每個 migration 寫 -- Down 註解或 down.sql

❌ 5. Data migration 一次跑全表
   → lock 半小時
   → 修：批次 1000 ~ 10000 一輪、PERFORM pg_sleep
```

  
### 🎯 重點回顧

- **Migration 工具**：Prisma / Drizzle / Supabase / Flyway 都很好
- **Expand → Migrate → Contract** 三段式 = 零停機改 schema
- **CREATE INDEX 永遠加 CONCURRENTLY**
- **PG 11+ 加欄位 NOT NULL DEFAULT 才安全**、之前要 3 步
- **SET lock_timeout** 防呆、卡住就失敗
- **大表 data migration 分批處理**、每批 1000 ~ 10000 + sleep
