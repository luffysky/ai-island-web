**Replication（複製）= 一份資料複製成好幾份、放在不同機器上**。**1 台寫、N 台讀**——讀流量瞬間 ×10、還能在主機掛掉時自動切備機。**所有上線 DB 必備**。

> 💡 **這課的價值**：學會 Replication = 你的 DB 撐住的人數 ×10、半夜掛掉服務不中斷。Supabase / RDS 預設都有、但你要看得懂它在幹嘛。

  
### 1. 為什麼需要 Replication

```text
單台 DB 的問題：
❌ 撐不住流量（1 台機器有 CPU / RAM 上限）
❌ 機器掛掉 = 服務全停
❌ 跨地區慢（台灣使用者打美國 DB → 200ms 延遲）
❌ 沒備援（停電 = 資料全沒）

有 Replication：
✅ 讀流量分到多台
✅ 主機掛 → 備機接手（failover）
✅ 多地區 read replica
✅ 天然備份
```

  
### 2. 主從架構（Primary-Replica）

```text
┌──────────────┐
│   Primary    │  ← 所有寫入（INSERT / UPDATE / DELETE）
│   （主機）    │
└──────┬───────┘
       │ 資料同步（WAL log stream）
       ↓
   ┌───┴───┬───────┬───────┐
   ↓       ↓       ↓       ↓
 Replica  Replica  Replica  Replica
 （備機）  （備機） （備機） （備機）
   ↑       ↑       ↑       ↑
   └───────┴───────┴───────┘
        所有讀取分散到這裡
```

**舊名 Master-Slave、新名 Primary-Replica**——意思一樣、政治正確改了名。

  
### 3. Sync vs Async Replication

#### Async（非同步）— PG 預設

```text
Primary 寫完 → 立刻回應 client「OK」
      └─→ 同時非同步傳給 Replica

優點：✅ 快、寫入沒延遲
缺點：⚠️ Primary 寫完還沒同步、Primary 突然掛
        → 那筆資料可能掉
```

#### Sync（同步）

```text
Primary 寫 → 等至少一台 Replica 也寫到 → 才回 client「OK」

優點：✅ Primary 掛 Replica 一定有
缺點：⚠️ 每次寫都要等網路、慢
```

**實務**：
- 金流 / 訂單 → Sync（不能掉資料）
- 一般 web app → Async（夠用、夠快）

  
### 4. 讀寫分離（Read / Write Split）

📄 **寫進程式檔**：`db.ts`

```text
import { Pool } from 'pg';

// 寫去 Primary
export const writeDb = new Pool({
  host: 'primary.db.local',
});

// 讀去 Replica（負載平衡）
export const readDb = new Pool({
  host: 'replica.db.local',  // DNS 後面接 3 台
});

// 用法
async function createUser(name: string) {
  return writeDb.query('INSERT INTO users(name) VALUES($1) RETURNING *', [name]);
}

async function getUser(id: number) {
  return readDb.query('SELECT * FROM users WHERE id = $1', [id]);
}
```

> 💡 **Replication Lag（複製延遲）**：Async 模式下、剛寫完去 Replica 讀可能讀不到。解法：剛寫完那個 request 強制讀 Primary、或等 1 秒。

  
### 5. HA（High Availability、高可用）+ Failover

```text
情境：Primary 機器突然掛掉

❌ 沒做 HA：
   service 直接 down、半夜起來救機

✅ 有 HA：
   1. Health check 每秒 ping Primary
   2. 連續 3 次失敗 → 觸發 failover
   3. 選一台 Replica 升為新 Primary
   4. 其他 Replica 改指向新 Primary
   5. 5 ~ 30 秒內服務恢復
```

**工具**：
- **Patroni**（PG 最常用、開源）
- **pg_auto_failover**（CitusData / Microsoft）
- **AWS RDS Multi-AZ**（按鈕一鍵）
- **Supabase**（自動處理、你不用管）

  
### 6. WAL（Write-Ahead Log、預寫日誌）— 同步的基礎

```text
1. client 送 INSERT 給 Primary
2. Primary 先把這筆動作寫到 WAL（一個 append-only 檔）
3. 確認 WAL 寫進磁碟
4. 才真的改 table 資料
5. 回 client「OK」

Replica 怎麼跟上？
- 讀 Primary 的 WAL
- 重播每一筆動作
- 結果 = 跟 Primary 一樣
```

**為什麼用 WAL**：
- 寫 WAL（append）比改 table（隨機寫）快 10 倍
- WAL 也是還原工具（後面 L11 PITR）

  
### 7. Replication 拓樸

#### a. 單一 Primary + 多 Replica（最常見）

```text
1 Primary → 3 ~ 5 Replica
適合 99% 場景
```

#### b. Cascading Replication（串聯）

```text
Primary → Replica A → Replica B
                   → Replica C

省 Primary 頻寬、但延遲變雙倍
```

#### c. Multi-Primary（多主、PG 原生不支援）

```text
2 ~ N 個 Primary 都能寫、互相同步
用工具：BDR（2ndQuadrant）、Bucardo

⚠️ 衝突解決超複雜、實務上少用
```

  
### 8. 雲端託管的 Replication

```text
☁️ Supabase
   - 自動有 Replica
   - 升 Pro plan 才能 read replica
   - failover 自動

☁️ AWS RDS / Aurora
   - Multi-AZ（同區跨機房）= 自動 failover
   - Read Replica = 跨區唯讀
   - Aurora 一個 Primary + 最多 15 Replica

☁️ Google Cloud SQL
   - 跟 RDS 類似

☁️ Railway / Render / Fly.io
   - 都有按鈕加 Replica
```

🖥️ **手動加 PG Replica**（自架）：

```text
# 在 Primary 編輯 postgresql.conf
wal_level = replica
max_wal_senders = 10
wal_keep_size = 1GB

# 在 Replica 跑
pg_basebackup -h primary.host -U replica -D /var/lib/pg -P -R
systemctl start postgresql
```

  
### 9. 5 大常見坑

```text
❌ 1. 以為「有 Replica = 有備份」
   → Replica 完全同步、你 DELETE 它也 DELETE
   → 修：Replica + 定期 pg_dump 雙保險

❌ 2. 讀寫分離後讀不到剛寫的資料
   → Async 延遲（lag）50 ~ 500ms
   → 修：剛寫完那 request 強制讀 Primary

❌ 3. failover 後 application 沒重連
   → 連線池握著舊 Primary IP
   → 修：用 DNS / Proxy 隔開（PgBouncer）

❌ 4. Replica 跟不上（lag 越來越大）
   → Primary 寫太快、Replica I/O 不夠
   → 修：升級 Replica 機器、或減少寫入

❌ 5. 想做 Multi-Primary 不研究就上
   → 衝突解決超難
   → 修：99% 場景 Primary-Replica 就夠
```

  
### 🎯 重點回顧

- **Replication = 1 寫 N 讀**——撐流量 + 備援
- **Async**（PG 預設）快但可能掉資料、**Sync** 慢但安全
- **讀寫分離**讓讀流量分散、注意 lag
- **HA / Failover**：Primary 掛了自動切 Replica
- **WAL** 是 Replication 跟 Backup 的共同基礎
- **雲端託管**（Supabase / RDS）會幫你處理 90%
