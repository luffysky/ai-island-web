**Backup（備份）+ DR（Disaster Recovery、災難復原）= 世界毀滅你也能把 DB 救回來**。**沒測試過還原 = 等於沒備份**。

> 💡 **這課的價值**：工程師最容易忽略、但只要出一次事就毀公司。學會 PITR（Point-In-Time Recovery）= 能還原到 1 秒前的狀態。

  
### 1. 為什麼備份不是「複製一份」

```text
❌ 「我有 Replica 就是備份」
   → Replica 完全同步、你 DELETE 它也 DELETE
   → 工程師手滑刪光 → Replica 也跟著沒

❌ 「我每月手動 export」
   → 中間 30 天的資料都會掉
   → 30 天前的備份救不回今天的客戶

✅ 真正的備份策略：
   - 全量備份（定期）
   - 增量備份（每小時 / 每分鐘）
   - WAL 持續歸檔（每筆寫入）
   - 異地保存（不同雲、不同區）
   - 定期演習還原
```

  
### 2. PG 三大備份方式

#### a. pg_dump — 邏輯備份（最常用）

```text
✅ 跨版本（PG 14 dump 出來能 restore 到 PG 16）
✅ 可只 dump 一張表 / 一個 schema
✅ 檔案小、可讀
❌ 慢（大表幾小時）
❌ 還原時要重建 index、超慢
```

🖥️ **在終端機輸入**：

```text
# 整個 DB dump
pg_dump -h localhost -U postgres mydb > mydb_backup.sql

# 壓縮 + 自訂格式（建議）
pg_dump -h localhost -U postgres -Fc mydb > mydb.dump

# 只 dump 一張表
pg_dump -h localhost -U postgres -t users mydb > users.sql

# Restore
pg_restore -h localhost -U postgres -d mydb_new mydb.dump
psql -h localhost -U postgres mydb_new < mydb_backup.sql
```

#### b. pg_basebackup — 物理備份

```text
✅ 快、整個 data 目錄複製
✅ Restore 也快、不用重建 index
❌ 同版本 PG 才能用
❌ 整桌備、不能挑表
```

```text
pg_basebackup -h localhost -U replica -D /backup/2026-05-28 -Fp -Xs -P
```

#### c. WAL archiving — 連續備份

```text
✅ 持續歸檔每筆寫入
✅ 配 pg_basebackup → PITR（還原到任意時間點）
✅ RPO（資料損失上限）可到秒級
```

📄 **寫進程式檔**：`postgresql.conf`

```text
wal_level = replica
archive_mode = on
archive_command = 'aws s3 cp %p s3://my-bucket/wal/%f'
```

每產生一個 WAL 檔（16MB）就上傳到 S3。

  
### 3. RPO vs RTO — 兩個關鍵指標

```text
🕐 RPO（Recovery Point Objective、恢復點目標）
   能容忍掉多少資料
   - 每天備份 = RPO 24 小時
   - 每小時 = RPO 1 小時
   - WAL 連續歸檔 = RPO 幾秒

🕑 RTO（Recovery Time Objective、恢復時間目標）
   多久要恢復服務
   - 重建機器 + restore dump = RTO 幾小時
   - 切換到 hot standby = RTO 幾秒
```

**不同業務需求不同**：

```text
個人 blog       RPO 1 天、RTO 1 天      （每天 dump 到 S3）
中小企業       RPO 1 小時、RTO 1 小時   （+ WAL + Replica）
金融 / 醫療    RPO 0 秒、RTO 30 秒      （+ Sync Replica + 多區）
```

  
### 4. PITR（Point-In-Time Recovery、時間點恢復）

**情境**：今天下午 2:35 工程師手滑 `DELETE FROM users;`、你要還原到 2:34:59。

```text
前提：你有
1. 每週一次 pg_basebackup
2. 連續的 WAL archive
```

🖥️ **在終端機輸入**：

```text
# 1. 找最近的 basebackup（假設上週日的）
aws s3 cp s3://my-bucket/basebackup/2026-05-24/ /var/lib/pg-restore/ --recursive

# 2. 寫 recovery 設定
echo "restore_command = 'aws s3 cp s3://my-bucket/wal/%f %p'" >> postgresql.conf
echo "recovery_target_time = '2026-05-28 14:34:59'" >> postgresql.conf

# 3. 啟動
touch /var/lib/pg-restore/recovery.signal
pg_ctl start -D /var/lib/pg-restore
```

PG 會：
1. 用 basebackup 還原到 5/24
2. 重放 WAL 一直到 2:34:59
3. 停在那一刻

💬 **電腦會顯示**：

```text
LOG:  starting point-in-time recovery to 2026-05-28 14:34:59
LOG:  restored log file "000000010000001A000000B3" from archive
LOG:  ... (一直跑)
LOG:  recovery stopping before commit ... at 2026-05-28 14:34:59
```

  
### 5. 雲端託管的備份

```text
☁️ Supabase
   - Free：每天 backup、保留 7 天
   - Pro：每天 + 保留 30 天
   - 付費可加 PITR

☁️ AWS RDS
   - 自動每天 snapshot
   - PITR（連續 WAL）內建、最長 35 天
   - Multi-AZ 同步副本

☁️ Google Cloud SQL
   - 自動備份 + PITR 7 天（可付費延長）

☁️ Neon / PlanetScale
   - 內建 branching（像 git）
   - Branch = 一個瞬間 snapshot
```

  
### 6. 異地備份 / 跨雲

```text
災難情境：
- 機房火災
- 雲廠商整個 region 倒
- DDoS / Ransomware

解法：
✅ 主備分 region（東京 + 新加坡）
✅ 跨雲（AWS + GCP / Cloudflare R2）
✅ Cold storage（S3 Glacier）—— 便宜、慢
```

📄 **寫進程式檔**：`backup.sh`

```text
#!/bin/bash
DATE=$(date +%Y%m%d_%H%M)

pg_dump -h primary -Fc mydb > /tmp/mydb_$DATE.dump

# 主：AWS S3 東京
aws s3 cp /tmp/mydb_$DATE.dump s3://backup-tokyo/db/

# 副：Cloudflare R2
aws s3 cp /tmp/mydb_$DATE.dump s3://backup-r2/db/ --endpoint=https://xxx.r2.cloudflarestorage.com

# 冷藏：S3 Glacier（30 天後自動）
aws s3 cp /tmp/mydb_$DATE.dump s3://backup-glacier/db/ --storage-class GLACIER

rm /tmp/mydb_$DATE.dump
```

加進 cron：

```text
0 3 * * * /opt/scripts/backup.sh
```

  
### 7. 演習！— 沒測過 = 沒備份

**每季至少一次**：

```text
1. 隨機抽一個備份
2. Restore 到新機器
3. 跑健康檢查 query
4. 驗證資料量、最新一筆時間
5. 記錄花了多久（測 RTO）
```

📄 **寫進程式檔**：`drill.sh`

```text
#!/bin/bash
START=$(date +%s)

# 拿最新 backup
LATEST=$(aws s3 ls s3://backup-tokyo/db/ | sort | tail -1 | awk '{print $4}')
aws s3 cp s3://backup-tokyo/db/$LATEST /tmp/

# Restore 到 staging
createdb mydb_drill
pg_restore -d mydb_drill /tmp/$LATEST

# 驗證
ROWS=$(psql -d mydb_drill -tA -c "SELECT COUNT(*) FROM users;")
LATEST_TS=$(psql -d mydb_drill -tA -c "SELECT MAX(created_at) FROM users;")

END=$(date +%s)
echo "RTO: $((END - START)) 秒"
echo "Users: $ROWS"
echo "Latest record: $LATEST_TS"
```

  
### 8. 5 大常見坑

```text
❌ 1. 「有 Replica 就是備份」
   → Replica 完全同步、你刪它也刪
   → 修：Replica + 獨立 backup 雙保險

❌ 2. 沒測試過還原
   → 真正出事才發現 backup 壞了 / 不完整
   → 修：每季演習一次

❌ 3. backup 跟 DB 放同一台 / 同一帳號
   → 整台機器毀了一起沒
   → ransomware 把所有檔案都加密
   → 修：跨 region、跨雲、跨帳號

❌ 4. 不知道自己 RPO / RTO 是多少
   → 出事才在算
   → 修：白紙黑字寫清楚、業務知情

❌ 5. WAL archive 失敗沒監控
   → 偷偷停了 3 個月、PITR 失效
   → 修：監控 archive_command 成功率
```

  
### 🎯 重點回顧

- **三大備份**：pg_dump（邏輯）、pg_basebackup（物理）、WAL archive（連續）
- **RPO** 容忍掉多少資料、**RTO** 多久恢復——先想清楚要多嚴
- **PITR** = basebackup + WAL，能還原到任意時間點
- **異地備份**（跨 region / 跨雲）防整個機房毀
- **沒演習 = 沒備份**——每季 restore 一次驗證
- **雲端託管 90% 幫你做了**，但你要知道它做到哪
