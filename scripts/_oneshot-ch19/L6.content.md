**Sharding（分片）= 把一張超大表切成好幾份、放到不同機器**。**Partitioning（分區）= 同一台機器內、把一張表切成好幾塊**。當 Replication 也不夠時、就要 Sharding。

> 💡 **這課的價值**：到 10 億筆資料、單台寫不動、加 Replica 也沒用（寫還是只有一台）——這時就要 Sharding。但 99% 的人不需要、會搞死自己。學會「何時不該 shard」也很重要。

  
### 1. 三種「切資料」方式

```text
🟢 Replication（複製、L5）
   同一份資料、複製 N 份在不同機器
   解：讀流量、HA

🟡 Partitioning（分區）
   同一台機器、一張表內部切成多塊
   解：管理大表、批次刪舊資料

🔴 Sharding（分片）
   不同機器各放一部分資料
   解：寫入太大、單台 RAM / CPU / 磁碟撐不住
```

  
### 2. Horizontal vs Vertical

#### Vertical（垂直分割）

```text
原本一張 users 表有 30 個欄位

切：
- users_core（id, email, name）
- users_profile（id, bio, avatar, ...）
- users_settings（id, prefs JSON）

→ 不同欄位放不同地方
→ 用得不多、實務上多用 normalization 解
```

#### Horizontal（水平分割）= Sharding 主角

```text
原本一張 users 表 10 億筆

切：
- shard_1：id 1 ~ 333,333,333
- shard_2：id 333,333,334 ~ 666,666,666
- shard_3：id 666,666,667 ~ 1,000,000,000

→ 同樣 schema、不同 row
→ 每台機器只管 1/3
```

  
### 3. 3 種 Shard Key 分配策略

#### a. Range Sharding（範圍）

```text
shard 1: user_id 1 ~ 1,000,000
shard 2: user_id 1,000,001 ~ 2,000,000
shard 3: user_id 2,000,001 ~ 3,000,000

優點：✅ 範圍查詢快
缺點：⚠️ Hot Spot（熱點）—— 新註冊都打 shard 3
```

#### b. Hash Sharding（雜湊）

```text
shard = hash(user_id) % 3

優點：✅ 分布超均勻、沒 hot spot
缺點：⚠️ 範圍查詢慢（要打所有 shard）
       ⚠️ 加新 shard 要 rehash 大半資料
```

#### c. List Sharding（清單）— 多租戶常用

```text
shard 1: tenant in ('apple', 'google')
shard 2: tenant in ('netflix', 'spotify')
shard 3: tenant in ('其他小客戶')

優點：✅ 商業需求對齊（大客戶獨享）
缺點：⚠️ 手動管理
```

  
### 4. Partitioning（PG 原生）

📄 **寫進程式檔**：`partition.sql`

```text
-- 按月分區
CREATE TABLE logs (
  id BIGSERIAL,
  created_at TIMESTAMPTZ NOT NULL,
  message TEXT,
  PRIMARY KEY (id, created_at)
) PARTITION BY RANGE (created_at);

CREATE TABLE logs_2026_05 PARTITION OF logs
  FOR VALUES FROM ('2026-05-01') TO ('2026-06-01');

CREATE TABLE logs_2026_06 PARTITION OF logs
  FOR VALUES FROM ('2026-06-01') TO ('2026-07-01');
```

**好處**：
```text
✅ 刪舊資料：DROP TABLE logs_2024_01;（秒殺、不用慢慢 DELETE）
✅ Query 自動 partition pruning（只掃對應分區）
✅ 各分區獨立 VACUUM / index
✅ 不算 sharding：還在同一台機器
```

**3 種 PARTITION BY**：

```text
RANGE  範圍（最常用：時間、id）
LIST   清單（country = 'TW' / 'US' / 'JP'）
HASH   雜湊（自動均分）
```

  
### 5. Sharding 怎麼做

#### a. Application 層分片（最常見）

```text
function getShard(userId: number) {
  return userId % 3;  // shard 0, 1, 2
}

async function getUser(id: number) {
  const shard = getShard(id);
  return shardClients[shard].query('SELECT * FROM users WHERE id=$1', [id]);
}
```

**優點**：✅ 控制權在你手上
**缺點**：⚠️ JOIN 跨 shard 自己處理、超痛苦

#### b. 用工具：Citus（PG extension）

```text
-- 把 users 表變 distributed
SELECT create_distributed_table('users', 'user_id');

-- 從 application 看就是一張表
SELECT * FROM users WHERE user_id = 12345;
-- Citus 自動路由到對的 shard
```

#### c. 雲端託管

```text
☁️ Vitess（YouTube / GitHub 用、MySQL）
☁️ CockroachDB（PG wire-compat、自動 sharding）
☁️ YugabyteDB（PG wire-compat）
☁️ AWS Aurora Limitless Database
```

  
### 6. Sharding 的 3 大痛點

```text
😭 1. Cross-shard JOIN
   user 在 shard 1、order 在 shard 2
   → JOIN 要先 fetch 兩邊再 application 層 merge

😭 2. Cross-shard Transaction
   A 在 shard 1 扣錢、B 在 shard 2 加錢
   → 2PC（兩階段提交）超慢又複雜

😭 3. Rebalancing
   加第 4 台 shard、要把資料重新分配
   → hash sharding：1/4 資料要搬
   → 搬的時候不能停機很難
```

  
### 7. 何時該 / 不該 Sharding

```text
✅ 該 shard：
   - 表 > 10 億筆
   - 單台 RAM / 磁碟撐不住
   - 寫入 QPS > 5 萬
   - 多租戶大客戶要獨立 instance

❌ 不該 shard（直接放棄）：
   - 表 < 1 億筆 → 加 index 就解
   - 讀多寫少 → 加 Replica 就解
   - 你會 JOIN 很多表 → 自虐
   - 你 team 沒 DBA → 99% 會搞死
```

> 💡 **業界共識**：能不 shard 就不 shard。Instagram 撐到 4 億用戶才 shard、Notion 等到 200 億 block 才開始。

  
### 8. 替代方案（在 shard 之前先試）

```text
1. 加 index（90% 慢都靠這個解）
2. 加 RAM / 升級機器（vertical scaling）
3. 加 Replica 分讀流量
4. 用 Partitioning（管大表）
5. 改用 CockroachDB / Yugabyte（自動 shard）
6. 把冷資料搬走（log → S3 + Athena）
7. Cache（Redis）擋 80% 讀
```

  
### 9. 5 大常見坑

```text
❌ 1. 太早 sharding
   → 100 萬筆就 shard、自找麻煩
   → 修：1 億筆以下別想

❌ 2. Shard key 選錯
   → 例：用 country shard、台灣用戶 95% 都打同一 shard
   → 修：用 user_id hash 或 UUID

❌ 3. Application 層 shard 沒 abstraction
   → 每個 query 都手寫路由、改 shard 數要改 1000 處
   → 修：包一層 ShardRouter

❌ 4. 跨 shard JOIN 寫得很糟
   → N+1 query、爆炸慢
   → 修：用 denormalization 把要 JOIN 的資料複製到同 shard

❌ 5. Partitioning 跟 Sharding 搞混
   → 以為 PG PARTITION BY 就是 sharding
   → 其實還在同一台機器、解的問題不同
```

  
### 🎯 重點回顧

- **Partitioning** 同一台機器內切表、**Sharding** 跨機器切資料
- **3 種策略**：Range（範圍）、Hash（雜湊）、List（清單）
- **PG 原生 PARTITION BY** 解 90% 大表問題（不用真的 shard）
- **Sharding 痛點**：跨 shard JOIN / Transaction / Rebalancing
- **99% 的場景不需要 sharding**——先試 index / Replica / Cache
- **若一定要 shard**：考慮 CockroachDB / Citus、別自己刻
