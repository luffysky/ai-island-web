**Index（索引）= 資料表的「目錄」**——沒 index 找一筆要翻整本書、有 index 直接跳到那一頁。**100 萬筆資料、加對 index 速度可以快 1000 倍**。

> 💡 **這課的價值**：加 index 是 90% DB 慢的解法。但「加錯 index」會讓「寫入」變慢還佔空間。學會選對才是真功夫。

  
### 1. 為什麼 index 會快

```text
沒 index：Seq Scan（Sequential Scan、順序掃描）
   - 整張表一筆一筆翻
   - 100 萬筆要看 100 萬次
   - SELECT * FROM users WHERE email='a@b.com' → 慢

有 index：Index Scan（索引掃描）
   - 像查字典：a→b→@→c → 直接定位
   - 100 萬筆只要看 20 次（log₂ 100萬 ≈ 20）
   - 同樣 query → 飛快
```

**白話**：index 就是 DB 偷偷幫你維護一份「排好序的目錄」、找東西不用翻整本書。

  
### 2. 五大 Index 類型

#### a. B-tree（Balanced Tree、平衡樹）— 99% 場景用這個

```text
適合：
✅ =、<、>、BETWEEN、ORDER BY、LIKE 'abc%'（前綴）
✅ 整數、文字、日期、UUID

不適合：
❌ LIKE '%abc'（後綴）→ 用 GIN
❌ 全文搜尋 → 用 GIN
❌ JSON 內查 → 用 GIN
```

📄 **寫進程式檔**：`schema.sql`

```text
-- 最常用：在 email 欄加 B-tree（PG 預設 type 就是 btree）
CREATE INDEX idx_users_email ON users(email);

-- 多欄位（複合索引）
CREATE INDEX idx_orders_user_created
ON orders(user_id, created_at DESC);
```

#### b. GIN（Generalized Inverted Index、廣義倒排索引）— 全文搜尋 / 陣列 / JSON

```text
適合：
✅ 全文搜尋（tsvector）
✅ 陣列欄位（tags TEXT[]）
✅ JSONB 內部查詢

像 Google 的「倒排索引」：每個字 → 哪些文章包含它
```

```text
-- JSON 內部欄位查詢
CREATE INDEX idx_posts_metadata ON posts USING GIN(metadata);

-- 全文搜尋
CREATE INDEX idx_posts_search
ON posts USING GIN(to_tsvector('english', title || ' ' || body));

-- 陣列
CREATE INDEX idx_posts_tags ON posts USING GIN(tags);
```

#### c. GiST（Generalized Search Tree、廣義搜尋樹）— 地理 / 範圍

```text
適合：
✅ PostGIS 地理座標（找「附近的店」）
✅ 範圍類型（tsrange / int4range）
✅ 全文搜尋（GIN 不行的場景）
```

```text
CREATE INDEX idx_shops_location
ON shops USING GIST(location);

-- 找半徑 1km 內的店
SELECT * FROM shops
WHERE ST_DWithin(location, ST_MakePoint(121.5, 25.0)::geography, 1000);
```

#### d. BRIN（Block Range Index、區塊範圍索引）— 超大表、時序

```text
適合：
✅ 億筆以上、按時間遞增的 log 表
✅ 資料天然有序（時間、序號）

優點：
✅ 空間極小（只佔 B-tree 的 1/1000）

缺點：
❌ 精確度低、只適合範圍查詢
```

```text
CREATE INDEX idx_logs_created_brin
ON logs USING BRIN(created_at);
```

#### e. Hash — 只支援 =、用得少

```text
PG 10+ 才完整支援、實務上幾乎都用 B-tree。
B-tree 也支援 =、還支援範圍、所以 hash 沒贏太多。
```

  
### 3. Partial Index（部分索引）— 只 index 一部分

```text
情境：99% 的訂單是 status='completed'、只有 1% 是 'pending'
你 query 永遠在找 pending → index 全部沒意義

解法：Partial Index 只 index pending 那 1%
```

```text
CREATE INDEX idx_orders_pending
ON orders(created_at)
WHERE status = 'pending';
```

**好處**：index 大小 1/100、查得更快。

  
### 4. Covering Index（覆蓋索引）— query 不用回表

```text
一般 index：找到 row 位置 → 還要回 table 拿其他欄位
Covering Index：index 本身就含所有需要的欄位 → 不用回表
```

```text
-- query 常用：SELECT name, email FROM users WHERE id = ?
CREATE INDEX idx_users_id_covering
ON users(id) INCLUDE (name, email);
```

**INCLUDE 那些欄位也存進 index、但不用來排序**。Postgres 11+ 支援。

  
### 5. 多欄位 index 的順序很重要

```text
CREATE INDEX idx_orders ON orders(user_id, status, created_at);

✅ 能用到 index：
   WHERE user_id = ?
   WHERE user_id = ? AND status = ?
   WHERE user_id = ? AND status = ? AND created_at > ?

❌ 用不到 index：
   WHERE status = ?               -- 跳過 user_id
   WHERE created_at > ?           -- 跳過前兩個

口訣：「最左前綴」——從第一欄開始連續才有效。
```

  
### 6. 加 index 之前先問自己

```text
問題清單：
1. 這個欄位 query 真的常用嗎？（看 pg_stat_statements）
2. 這個欄位 cardinality 高嗎？
   → 性別（M/F）= 低、不該加 index
   → email = 每筆唯一、適合加
3. 這個表寫入頻率高嗎？
   → 每秒 1000 次 INSERT 又加 5 個 index → 寫入變超慢
4. 表多大？
   → < 1 萬筆通常 Seq Scan 比 Index Scan 快
```

  
### 7. 怎麼看 index 有沒有被用到

🖥️ **在終端機輸入**：

```text
psql -d mydb -c "EXPLAIN ANALYZE SELECT * FROM users WHERE email='a@b.com';"
```

💬 **電腦會顯示**：

```text
✅ 有用 index：
   Index Scan using idx_users_email on users
     Index Cond: (email = 'a@b.com')
     Execution Time: 0.123 ms

❌ 沒用 index：
   Seq Scan on users
     Filter: (email = 'a@b.com')
     Rows Removed by Filter: 999999
     Execution Time: 523.456 ms
```

  
### 8. 5 大常見坑

```text
❌ 1. 什麼欄位都加 index
   → 寫入變慢 5 倍、磁碟爆炸
   → 修：先 EXPLAIN ANALYZE、確定慢再加

❌ 2. 在 cardinality 很低的欄位加 index
   → 性別、布林、status（只有 3 種值）
   → 修：用 Partial Index 或不加

❌ 3. 多欄位 index 順序錯
   → query 從第二欄開始 → 用不到 index
   → 修：把最常 WHERE 的欄位放第一

❌ 4. LIKE '%abc' 期待用 B-tree
   → 後綴查詢 B-tree 沒用
   → 修：用 GIN + pg_trgm extension

❌ 5. 加了 index 但沒 VACUUM / ANALYZE
   → PG 統計過時、planner 不用 index
   → 修：跑 ANALYZE table_name; 或開 autovacuum
```

  
### 🎯 重點回顧

- **B-tree** 99% 場景用（=、<、>、ORDER BY、前綴 LIKE）
- **GIN** 用在 JSON / 陣列 / 全文搜尋
- **GiST** 用在地理座標 / 範圍
- **BRIN** 用在億筆 log 等天然有序的超大表
- **Partial Index** 只 index 你常查的那一小部分、超省空間
- **永遠先 EXPLAIN ANALYZE、再決定加不加**
