**DB（Database、資料庫）進階 = 把「會寫 SELECT / INSERT」升級到「會在 100 萬筆資料下還跑很快」**。這一章不是教你新語法、是教你「資料變多 / 流量變大 / 不能停機」時要會的所有招式。

> 💡 **這課的價值**：先把整章地圖印在腦袋裡——之後每一課你會知道「我現在學的是哪一塊、解什麼問題」。沒有地圖、學了會散。

  
### 1. 入門 vs 進階差在哪

```text
🟢 SQL 入門（ch18）
   - 會寫 SELECT / JOIN / GROUP BY
   - 會 CREATE TABLE / 加欄位
   - 100 筆資料、什麼都很快

🔴 SQL 進階（ch19）
   - 100 萬筆資料 query 還要 < 100ms
   - 多台 server 同時寫資料、不能搞壞
   - 半夜 DB 掛掉、5 分鐘要救回來
   - 一個資料庫服務 1000 個客戶、不能互看
```

> 💡 **白話**：入門是「教你怎麼開車」、進階是「教你怎麼在塞車 + 下雨 + 載貨時還能準時到」。

  
### 2. 8 大主題互相關係

```text
速度問題 ───┬─→ Index（索引、加速查詢的「目錄」）
            └─→ EXPLAIN ANALYZE（解釋分析、看 query 怎麼跑的）

資料正確性 ─┬─→ Transaction（交易、一組動作全成功或全失敗）
            └─→ Isolation Level（隔離等級、多人同時改怎麼處理）

撐住流量 ───┬─→ Replication（複製、主從架構）
            ├─→ Sharding（分片、把資料切到多台機器）
            └─→ Connection Pool（連線池、共用 DB 連線）

不出意外 ───┬─→ Migration（遷移、改 schema 不停機）
            └─→ Backup / Restore（備份 / 還原、世界毀滅也救得回）

商業模式 ───→ Multi-tenant（多租戶、一個系統服務多家公司）
```

  
### 3. 學完這章你會的

```text
✅ 看到慢 query 會用 EXPLAIN ANALYZE 找出問題
✅ 知道何時加 index、加哪種 index（B-tree / GIN / GiST / BRIN）
✅ 看懂為什麼有時候轉帳會「卡住」（Transaction + Isolation）
✅ 設計可以撐 100 萬人的架構（Replication + Sharding）
✅ 改 schema 不會把線上服務搞掛（Migration 策略）
✅ DB 掛了 5 分鐘救回來（Backup / PITR）
✅ 做 SaaS 時不會讓 A 公司看到 B 公司資料（Multi-tenant）
```

  
### 4. 真實案例：為什麼需要進階

**案例 1：你的網站越來越慢**

```text
1. 一開始 1000 個會員、SELECT 很快
2. 半年後 100 萬會員、登入要等 5 秒
3. 沒加 index → 每次都 Seq Scan（順序掃描、整張表翻一遍）
4. 加了 B-tree index → 0.01 秒搞定
```

**案例 2：雙 11 流量爆炸**

```text
1. 平常 100 人同時在線、1 台 DB 撐得住
2. 雙 11 1 萬人搶購、1 台 DB 直接燒掉
3. 解法：Replication（1 台寫、3 台讀）
4. + Connection Pool（每個請求不要開新連線）
```

**案例 3：半夜被 PM 電話叫起來**

```text
1. 工程師手滑：DELETE FROM users; （忘了 WHERE）
2. 100 萬會員瞬間消失
3. 有 Backup（每天備份）+ WAL（記錄每筆寫入）→ 還原到 5 分鐘前
4. 沒有 Backup → 公司倒了
```

  
### 5. PostgreSQL 為什麼是主角

```text
🐘 PostgreSQL（簡稱 Postgres）
   - 開源、免費、企業級
   - 支援最完整：JSON / 全文搜尋 / GIS / 向量
   - Supabase / Railway / Render 預設都用它
   - AWS / Google Cloud / Azure 都有託管版

🐬 MySQL
   - 也很強、WordPress / WooCommerce 預設
   - 但「進階功能」沒 PG 多

🪶 SQLite
   - 一個檔案就是一個 DB
   - 適合本地 / 桌面 app
   - 不適合多人同時寫

🍃 MongoDB（NoSQL）
   - 文件型、結構鬆散
   - 大數據 / log 用
   - 但「Transaction / JOIN」很弱
```

> 💡 **這一章 90% 範例用 PostgreSQL**——因為 AI 島本身就跑在 Supabase（= PostgreSQL）。

### 7. 常見錯誤的迷思

```text
❌ 1. 「我會 SELECT 就夠了」
   → 100 萬筆下不加 index = 你的網站永遠很慢

❌ 2. 「DB 慢就升級機器」
   → 加 CPU / RAM 一陣子有效、但 query 寫爛多少錢都救不回

❌ 3. 「備份 = 還原」
   → 沒測過還原 = 等於沒備份；定期演習才算數

❌ 4. 「Migration 改 schema 隨便改」
   → 直接 ALTER TABLE 線上 lock 半小時、服務掛半小時

❌ 5. 「Multi-tenant 共用 table 就好」
   → 一個 bug 全部客戶資料外洩、被告賠死
```

### 🎯 重點回顧

- **DB 進階 = 解 4 大問題**：速度、正確性、流量、不出意外
- **8 個主題互相關係**：Index/EXPLAIN 解速度、Transaction 解正確性、Replication/Sharding/Pool 解流量、Migration/Backup 解意外、Multi-tenant 解商業模式
- **PostgreSQL 是業界主角**——Supabase / Railway / Render 都用它
- **不要迷信「升級機器」**——query 寫爛多少錢都救不回
- **這章學完 = 你會做出能撐 100 萬人的 SaaS 後端**
