**Connection Pool（連線池）= 預先開好一堆 DB 連線、大家共用、不要每次都重開**。**沒 pool = 你的網站上線 5 分鐘就出現「too many connections」**。

> 💡 **這課的價值**：Serverless（Vercel / Netlify）+ 傳統 PG 必踩的雷。學會 PgBouncer / Supavisor 等於救了你的線上服務。

  
### 1. 為什麼需要 Pool

**沒 Pool 的世界**：

```text
每個 request 來：
1. TCP 三向交握（30ms）
2. PG 認證 / SSL（70ms）
3. 跑你的 query（5ms）
4. 關連線（10ms）

→ 100 個 request = 開 100 個連線 = 11 秒 overhead

PG 預設 max_connections = 100
→ 第 101 個 request 直接被拒絕
→ 錯誤：FATAL: too many connections
```

**有 Pool 的世界**：

```text
Pool 預先開 20 個連線、一直保持 alive

request 來：
1. 從 Pool 借一個連線（0.1ms）
2. 跑你的 query
3. 還回 Pool（0.1ms）

→ 1000 個 request 共用 20 個連線 → 不爆
```

  
### 2. Pool 的兩層

```text
👇 Application 層 Pool
   - Node.js 的 pg.Pool / Prisma / Drizzle
   - 在你的 server 內、每個 instance 自己一份
   - 適合：傳統長期 server

👇 Database 層 Pool（PgBouncer / Supavisor）
   - 獨立 proxy、所有 application 共用
   - 適合：serverless / 多 instance
```

  
### 3. Application Pool — Node.js 範例

📄 **寫進程式檔**：`db.ts`

```text
import { Pool } from 'pg';

export const pool = new Pool({
  host: 'localhost',
  user: 'postgres',
  database: 'mydb',
  password: '...',
  max: 20,              // pool 大小：最多 20 個連線
  min: 2,               // 最少保留 2 個 idle
  idleTimeoutMillis: 30000,   // idle 30 秒就關
  connectionTimeoutMillis: 5000,  // 5 秒拿不到就 timeout
});

// 用法 1：自動借還
const result = await pool.query('SELECT * FROM users');

// 用法 2：transaction 借單一連線
const client = await pool.connect();
try {
  await client.query('BEGIN');
  // ...
  await client.query('COMMIT');
} finally {
  client.release();  // 重要！還回 pool
}
```

> 💡 **重要**：transaction 必須用 `pool.connect()` 借同一個 client、不能用 `pool.query()`（會借不同連線、transaction 失效）。

  
### 4. Pool size 怎麼算

```text
公式：pool size = (核心數 × 2) + 磁碟數
（來自 PG 官方建議）

實務：
- 小 web app：10 ~ 20
- 中型：20 ~ 50
- 大型：50 ~ 100

⚠️ 不是越大越好！
- 太大：PG 內部 context switch、反而慢
- 太小：request 排隊、慢
```

**全 application 加起來不能超過 PG max_connections**：

```text
PG max_connections = 100
你有 5 個 server instance
→ 每個 pool 最多 100/5 = 20
→ 留一些給 admin、實際 15 ~ 18
```

  
### 5. Serverless 的災難

```text
Vercel / Netlify / Lambda：
- 每個 request 開一個新 function instance
- 每個 instance 都想開 pool
- 1000 個 concurrent request = 1000 個 pool 
  = 1000 × 5 連線 = 5000 個 PG 連線
- PG 直接爆 ❌
```

**解法**：在 PG 前面架 **PgBouncer / Supavisor**。

  
### 6. PgBouncer — 業界標準

```text
[你的 N 個 serverless function]
        ↓ 上萬個短暫連線
   [PgBouncer]
        ↓ 100 個長連線
    [PostgreSQL]
```

📄 **寫進程式檔**：`pgbouncer.ini`

```text
[databases]
mydb = host=127.0.0.1 port=5432 dbname=mydb

[pgbouncer]
listen_port = 6432
listen_addr = *
auth_type = md5
auth_file = /etc/pgbouncer/userlist.txt

pool_mode = transaction
max_client_conn = 10000
default_pool_size = 25
```

**3 種 pool_mode**：

```text
🟢 session mode（預設）
   - client 整個 session 獨佔一條 PG 連線
   - 跟沒用 PgBouncer 差不多、不省

🟡 transaction mode（最常用）
   - 一個 transaction 內獨佔
   - transaction 結束就還
   - ⚠️ 不能用 SET / LISTEN / prepared statement

🔴 statement mode
   - 一句 SQL 結束就還
   - 不支援 transaction、實務上少用
```

🖥️ **在終端機輸入**：

```text
pgbouncer -d /etc/pgbouncer/pgbouncer.ini
```

application 改連 `localhost:6432`（PgBouncer）而非 `localhost:5432`（PG）。

  
### 7. Supavisor — Supabase 自製、雲端友善

```text
- Supabase 給每個專案 2 個 endpoint：
  1. Direct（直連 PG）—— 給 migration 用
  2. Pooler（透過 Supavisor）—— 給 application 用

- Supavisor 用 Elixir 寫、效能很好
- 支援 transaction / session 兩種 mode
- 跟 PgBouncer 概念一樣、雲原生
```

```text
DATABASE_URL=postgresql://user:pass@db.xxx.supabase.co:5432/postgres
                                                       ↑ Direct（migration）

DATABASE_URL=postgresql://user:pass@aws-pooler.supabase.com:6543/postgres
                                                              ↑ Pooler（app）
```

  
### 8. Pool 監控

🖥️ **看 PG 當前連線**：

```text
psql -c "SELECT state, COUNT(*) FROM pg_stat_activity GROUP BY state;"
```

💬 **電腦會顯示**：

```text
   state               | count
-----------------------+-------
 active                |    12
 idle                  |    34
 idle in transaction   |     2   ⚠️ 警訊
```

```text
⚠️ idle in transaction：
   有人 BEGIN 之後忘了 COMMIT
   → 鎖一直握著、其他人卡死
   → 修：找出 application bug、加 statement_timeout
```

**PgBouncer 監控**：

```text
psql -p 6432 pgbouncer -c "SHOW POOLS;"
psql -p 6432 pgbouncer -c "SHOW STATS;"
```

  
### 9. Statement timeout — 防呆

📄 **寫進程式檔**：`schema.sql`

```text
-- session level
SET statement_timeout = '10s';

-- user level（永久）
ALTER ROLE web_user SET statement_timeout = '10s';

-- transaction 內 idle timeout
ALTER ROLE web_user SET idle_in_transaction_session_timeout = '60s';
```

**防止**：
- 慢 query 卡死所有連線
- 有人 BEGIN 沒 COMMIT 鎖整桌

  
### 10. 5 大常見坑

```text
❌ 1. Serverless 沒接 PgBouncer / Supavisor
   → 5 分鐘就 too many connections
   → 修：用 Supavisor / PgBouncer / Neon 內建 pooler

❌ 2. pool.query() 跑 transaction
   → 不同 query 借不同連線、transaction 失效
   → 修：永遠用 pool.connect() + client.release()

❌ 3. 忘了 client.release()
   → 連線洩漏、pool 滿了不還
   → 修：用 try/finally

❌ 4. PgBouncer transaction mode 用 prepared statement
   → 跨 transaction 失效
   → 修：關 prepared statement 或改 session mode

❌ 5. pool size 開太大
   → PG 反而變慢、CPU 100%
   → 修：(cores × 2 + disks) 公式為準
```

  
### 🎯 重點回顧

- **Connection Pool = 預開連線共用**——省 100ms / request
- **Application Pool** 給傳統 server、**DB Pool（PgBouncer / Supavisor）** 給 serverless
- **Pool size 不是越大越好**：(cores × 2 + disks)
- **PgBouncer transaction mode** 是業界標準
- **Supabase 用 Supavisor**——記得用 Pooler endpoint
- **永遠設 statement_timeout** 防呆
