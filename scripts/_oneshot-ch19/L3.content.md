**EXPLAIN ANALYZE = DB 的「X 光機」**——把你的 SQL 拿去跑、然後告訴你「實際怎麼跑、每一步花多少時間、讀了多少資料」。**所有 DB 效能調校的起點**。

> 💡 **這課的價值**：90% 的人加 index 是用猜的、然後加錯。會看 EXPLAIN ANALYZE = 你看得到問題的真實樣子、不是亂槍打鳥。

  
### 1. EXPLAIN vs EXPLAIN ANALYZE 差在哪

```text
EXPLAIN（解釋）
   - 只「估計」query 會怎麼跑
   - 不真的執行
   - 速度快、安全

EXPLAIN ANALYZE（解釋 + 分析）
   - 真的執行 query
   - 給「實際」時間 + 實際 row 數
   - ⚠️ DELETE / UPDATE 會真的改資料！
```

> 💡 **危險**：`EXPLAIN ANALYZE DELETE FROM users;` 會真的刪光。若要分析寫入類，包進 transaction 再 ROLLBACK。

  
### 2. 最基礎用法

🖥️ **在終端機輸入**：

```text
EXPLAIN ANALYZE
SELECT * FROM users WHERE email = 'a@b.com';
```

💬 **電腦會顯示**：

```text
Seq Scan on users  (cost=0.00..18334.00 rows=1 width=120)
                   (actual time=0.012..123.456 rows=1 loops=1)
  Filter: (email = 'a@b.com')
  Rows Removed by Filter: 999999
Planning Time: 0.123 ms
Execution Time: 123.456 ms
```

  
### 3. 怎麼讀執行計畫

```text
🔍 第一行 = node type（節點類型）
   Seq Scan / Index Scan / Hash Join / Nested Loop ...

📊 cost=0.00..18334.00
   - 第一個（0.00）= 取得第一筆的代價
   - 第二個（18334.00）= 全部取完的代價
   - 「代價」是相對值、沒單位、越小越好

📈 rows=1
   - 估計會回幾筆

📏 width=120
   - 每筆平均多少 bytes

⏱️ actual time=0.012..123.456
   - 實際時間（毫秒）

📉 Rows Removed by Filter: 999999
   - ⚠️ 看到大數字 = 警訊！讀了 100 萬筆才篩出 1 筆
```

  
### 4. 8 種常見 node type

```text
✅ Index Scan
   - 用 index 找、最快
   - 看到這個基本上沒問題

✅ Index Only Scan
   - 連 table 都不用回、直接從 index 拿
   - 比 Index Scan 更快

✅ Bitmap Heap Scan + Bitmap Index Scan
   - 先用 index 抓一堆位置、再批次回表
   - 適合「中等量」結果（幾千筆）

⚠️ Seq Scan（Sequential Scan）
   - 整張表翻
   - 小表（< 1 萬筆）OK
   - 大表 = 災難、要加 index

🔗 Nested Loop（巢狀迴圈）
   - JOIN 的方式：外層每一筆 × 內層每一筆
   - 小資料快、大資料災難

🔗 Hash Join
   - JOIN 的方式：先建 hash table 再對
   - 大資料 OK

🔗 Merge Join
   - 兩邊都排好序再 merge
   - 大資料、有排序時最快

📥 Sort
   - ORDER BY 時出現
   - 「Sort Method: external merge」= 排不下記憶體、用磁碟（慢）
```

  
### 5. 加 buffers 看磁碟讀取

🖥️ **在終端機輸入**：

```text
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM users WHERE email = 'a@b.com';
```

💬 **電腦會顯示多一段**：

```text
Buffers: shared hit=42 read=1234

shared hit = 從記憶體拿（快、好）
shared read = 從磁碟讀（慢、不好）

→ read 數字越大 = 磁碟 I/O 越多 = 越慢
→ 想辦法讓 hit 多、read 少（加 index / 加 RAM）
```

  
### 6. 看「估計 vs 實際」差多少

```text
rows=1 (actual ... rows=999999)
        ↑           ↑
       估計         實際

→ 差超過 10 倍 = PG 統計過時、planner 算錯
→ 解法：ANALYZE table_name;（重新統計）
→ 或開 autovacuum（自動）
```

  
### 7. 完整實戰：找出慢 query

📄 **寫進程式檔**：`debug_slow.sql`

```text
-- Step 1: 開啟詳細模式
EXPLAIN (ANALYZE, BUFFERS, VERBOSE, FORMAT TEXT)
SELECT u.name, COUNT(o.id) AS order_count
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
WHERE u.country = 'TW'
GROUP BY u.id
ORDER BY order_count DESC
LIMIT 10;
```

💬 **電腦會顯示**：

```text
Limit (cost=... actual time=2345.678..2345.700 rows=10)
  -> Sort (cost=... actual time=2345.677..2345.690 rows=10)
       Sort Key: (count(o.id)) DESC
       Sort Method: top-N heapsort  Memory: 25kB
       -> HashAggregate (cost=... actual rows=50000)
            -> Hash Right Join (cost=... actual rows=200000)
                 Hash Cond: (o.user_id = u.id)
                 -> Seq Scan on orders o (cost=... rows=1000000)   ⚠️
                      Buffers: shared read=12345
                 -> Hash (cost=... rows=50000)
                      -> Seq Scan on users u (cost=... rows=50000) ⚠️
                           Filter: (country = 'TW')
                           Rows Removed by Filter: 950000          ⚠️
Execution Time: 2345.678 ms
```

**診斷**：
- `users.country` 沒 index → Seq Scan 掃 100 萬筆
- `orders.user_id` 沒 index → Seq Scan 掃 100 萬筆

**解法**：

```text
CREATE INDEX idx_users_country ON users(country);
CREATE INDEX idx_orders_user_id ON orders(user_id);
```

跑完再 EXPLAIN ANALYZE 一次 → 應該變 Index Scan + 時間 < 100ms。

### 9. 5 大常見坑

```text
❌ 1. 用 EXPLAIN ANALYZE 跑 DELETE / UPDATE 沒包 transaction
   → 真的會改資料
   → 修：BEGIN; EXPLAIN ANALYZE ...; ROLLBACK;

❌ 2. 只看 cost、不看 actual time
   → cost 是估的、可能很不準
   → 修：永遠看 actual time

❌ 3. 看到 Seq Scan 就慌
   → 小表（< 1 萬）Seq Scan 比 Index Scan 快
   → 修：先看資料量

❌ 4. 加了 index 但沒生效
   → PG 統計過時、planner 不用
   → 修：ANALYZE table_name;

❌ 5. 看不懂 Nested Loop 為什麼慢
   → 外層 10000 筆 × 內層 1000 筆 = 1000 萬次迴圈
   → 修：給內層加 index、或讓 PG 用 Hash Join
```

  
### 🎯 重點回顧

- **EXPLAIN** 估計、**EXPLAIN ANALYZE** 真執行（DELETE / UPDATE 要包 transaction）
- **看 actual time、不要只看 cost**
- **Rows Removed by Filter 大數字 = 警訊**（讀了一堆才篩出一點）
- **Seq Scan 大表 = 該加 index**
- **加 BUFFERS 看磁碟讀寫量**
- **explain.dalibo.com 視覺化**比文字好讀十倍
