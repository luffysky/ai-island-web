**Transaction（交易）= 一組動作「全成功或全失敗」**——轉帳「A 扣 100、B 加 100」這兩個動作要綁在一起、不能只做一半。**Isolation Level（隔離等級）= 多人同時改資料時要多嚴格**。

> 💡 **這課的價值**：不懂 Transaction = 你的金流系統會「扣錢沒到帳」、「庫存超賣」、「重複下單」。這是「上線必修」、不是進階。

  
### 1. 為什麼需要 Transaction

**情境：A 轉 100 給 B**

```text
UPDATE accounts SET balance = balance - 100 WHERE id = 'A';
-- ⚠️ 這裡如果 server 突然掛掉 ⚠️
UPDATE accounts SET balance = balance + 100 WHERE id = 'B';
```

**結果**：A 少 100、B 沒收到 → 100 元憑空消失。

**用 Transaction 包起來**：

```text
BEGIN;
  UPDATE accounts SET balance = balance - 100 WHERE id = 'A';
  UPDATE accounts SET balance = balance + 100 WHERE id = 'B';
COMMIT;
```

中間掛掉 → 自動 ROLLBACK → A 沒扣、B 沒加 → 沒事。

  
### 2. ACID — Transaction 的 4 個保證

```text
🅰️ Atomicity（原子性）
   全部成功 OR 全部失敗、不會做一半。

🆑 Consistency（一致性）
   交易前後資料合理（餘額 ≥ 0、外鍵不破）。

🅸 Isolation（隔離性）
   多個交易同時跑、看起來像「一個一個跑」。

🅳 Durability（持久性）
   COMMIT 之後資料就算停電也不會掉。
```

> 💡 **白話**：Atomicity = 不做一半；Consistency = 不破壞規則；Isolation = 不互相干擾；Durability = 不掉資料。

  
### 3. 基本語法

📄 **寫進程式檔**：`transfer.sql`

```text
BEGIN;                                    -- 開始交易

UPDATE accounts SET balance = balance - 100 WHERE id = 'A';
UPDATE accounts SET balance = balance + 100 WHERE id = 'B';

-- 中間可以做任何檢查
SELECT balance FROM accounts WHERE id = 'A';
-- 如果不對，ROLLBACK；如果對，COMMIT

COMMIT;   -- 確定 OR
ROLLBACK; -- 全撤
```

**Node.js 範例**：

```text
import { Pool } from 'pg';
const pool = new Pool();

async function transfer(from: string, to: string, amt: number) {
  const client = await pool.connect();
  try {
    await client.query('BEGIN');
    await client.query('UPDATE accounts SET balance = balance - $1 WHERE id = $2', [amt, from]);
    await client.query('UPDATE accounts SET balance = balance + $1 WHERE id = $2', [amt, to]);
    await client.query('COMMIT');
  } catch (e) {
    await client.query('ROLLBACK');
    throw e;
  } finally {
    client.release();
  }
}
```

  
### 4. 多人同時改 → 4 個經典問題

```text
👤 Dirty Read（髒讀）
   讀到別人「還沒 COMMIT」的資料
   → 對方 ROLLBACK 後、你拿到的是「不存在的資料」

🔁 Non-Repeatable Read（不可重複讀）
   同一個 transaction 內、讀兩次同一筆、值不一樣
   → 中間別人改了並 COMMIT

👻 Phantom Read（幻讀）
   同一個 transaction 內、第二次 SELECT 多出 / 少了一些 row
   → 中間別人 INSERT 了新 row

🔒 Lost Update（更新丟失）
   兩人同時讀 balance=100、各加 50、最後 balance=150（不是 200）
   → 後寫的覆蓋前寫的
```

  
### 5. 4 個 Isolation Level（隔離等級）

```text
等級              Dirty Read  Non-Repeat  Phantom   Lost Update
─────────────────────────────────────────────────────────────
Read Uncommitted    ✗           ✗           ✗          ✗
Read Committed      ✅          ✗           ✗          ✗
Repeatable Read     ✅          ✅          ✅(PG)     ✅(PG)
Serializable        ✅          ✅          ✅          ✅

✅ = 擋掉、✗ = 還會發生
```

#### Read Uncommitted（讀未提交）

```text
- 最寬鬆、最快、什麼都讀得到
- PostgreSQL 沒實作（自動升 Read Committed）
- 實務上幾乎沒人用
```

#### Read Committed（讀已提交）— PG 預設

```text
- 只讀「已 COMMIT」的資料
- 同一 transaction 內、每次 SELECT 看到的可能不同
- 適合：一般 web app（讀多寫少）
```

#### Repeatable Read（可重複讀）

```text
- transaction 開始時拍 snapshot、整個過程看同一份
- PG 在這級就把幻讀也擋掉了（SSI 機制）
- 適合：報表、批次處理
```

#### Serializable（序列化）

```text
- 最嚴格、像「一個一個跑」
- PG 會偵測衝突、衝突就讓某個交易失敗（要重試）
- 適合：金流、訂票、庫存
```

  
### 6. 設定 Isolation Level

```text
BEGIN ISOLATION LEVEL SERIALIZABLE;
  -- ... 你的 SQL ...
COMMIT;

-- 或整個 session
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
```

  
### 7. 顯式鎖 — 解 Lost Update

**情境：搶限量商品**

```text
BEGIN;
  -- 鎖住這一 row、別人要改要等
  SELECT stock FROM products WHERE id = 1 FOR UPDATE;

  -- 確認 stock > 0
  UPDATE products SET stock = stock - 1 WHERE id = 1;
  INSERT INTO orders (...) VALUES (...);
COMMIT;
```

**3 種鎖**：

```text
FOR UPDATE         獨佔鎖：別人讀寫都要等
FOR NO KEY UPDATE  輕量、外鍵 join 不擋
FOR SHARE          共享鎖：別人能讀、不能寫
```

  
### 8. Deadlock（死鎖）

```text
Transaction A：鎖 row 1、想鎖 row 2
Transaction B：鎖 row 2、想鎖 row 1
→ 互相等對方、永遠不會結束 = 死鎖

PG 偵測到會殺掉其中一個、丟 error：
"deadlock detected"

解法：
✅ 永遠用「相同順序」鎖（例：先小 id 再大 id）
✅ application 層 try / retry
✅ 用 SERIALIZABLE 讓 PG 幫處理
```

  
### 9. 5 大常見坑

```text
❌ 1. 用 ORM 不知道有沒有開 transaction
   → 失敗只 rollback 一半
   → 修：明確 client.transaction(async (tx) => {...})

❌ 2. transaction 內 await fetch 外部 API
   → 鎖一直握著、其他人卡住
   → 修：把外部 call 放 transaction 外

❌ 3. 用 Repeatable Read 跑金流
   → PG 不保證 Lost Update（其他 DB 也類似）
   → 修：用 SERIALIZABLE 或 SELECT FOR UPDATE

❌ 4. 忘了 application 層 retry SERIALIZABLE 衝突
   → 偶發 error、使用者看到「請重試」
   → 修：用 lib 自動 retry 3 次

❌ 5. transaction 太大、跑 10 分鐘
   → 鎖一堆、其他人全卡死
   → 修：拆小、批次處理
```

  
### 🎯 重點回顧

- **ACID = 原子 / 一致 / 隔離 / 持久** 4 大保證
- **BEGIN → 動作 → COMMIT / ROLLBACK** 是基本三件套
- **4 種隔離等級**：越嚴格越慢、但保護越多
- **PG 預設 Read Committed**——多數 web app 夠用
- **金流 / 庫存 / 訂票** → 用 SERIALIZABLE 或 SELECT FOR UPDATE
- **小心 deadlock**：相同順序鎖 + retry
