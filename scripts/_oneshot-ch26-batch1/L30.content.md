非即時（背景任務）+ 即時通訊（WebSocket / SSE）——Python 後端進階能力。

> 💡 **3 個關鍵字先記住**：
> - **WebSocket** = 雙向即時（聊天 / 多人協作 / 線上遊戲）
> - **SSE（Server-Sent Events）** = 單向「伺服器推給客戶端」（AI 串流、通知）
> - **背景任務（background tasks）** = 「跑得慢的事丟給別人做、API 立刻回」（寄信、影片轉檔）

  
### WebSocket：雙向即時通訊（聊天 / 協作必用）

**WebSocket** = 跟 HTTP 不一樣的協定、建立連線後**雙向都可以隨時送訊息**、不用重複握手。FastAPI 內建支援、不用額外裝。

```text
from fastapi import FastAPI, WebSocket, WebSocketDisconnect

app = FastAPI()

# === 連線管理器（追蹤所有目前連著的 client）===
class ConnectionManager:
    def __init__(self):
        self.active: list[WebSocket] = []
    
    async def connect(self, ws: WebSocket):
        await ws.accept()                      # 接受連線
        self.active.append(ws)
    
    def disconnect(self, ws: WebSocket):
        self.active.remove(ws)
    
    async def broadcast(self, msg: str):       # 廣播給所有人
        for connection in self.active:
            await connection.send_text(msg)

manager = ConnectionManager()

# === WebSocket endpoint（注意：用 @app.websocket、不是 @app.get）===
@app.websocket("/ws/{client_id}")
async def websocket_endpoint(websocket: WebSocket, client_id: int):
    await manager.connect(websocket)
    try:
        while True:
            data = await websocket.receive_text()        # 等對方傳訊息
            await manager.broadcast(f"Client {client_id}: {data}")
    except WebSocketDisconnect:
        manager.disconnect(websocket)
        await manager.broadcast(f"Client {client_id} left")
```

  
**Client 端（瀏覽器 JavaScript）：**

```text
const ws = new WebSocket("ws://localhost:8000/ws/1");
ws.onmessage = (e) => console.log(e.data);    // 收訊息
ws.send("Hello!");                             // 送訊息
```

> 💡 **WebSocket 適合**：聊天 / 多人協作（Figma / Google Docs）/ 線上遊戲 / 即時通知。**不適合**：單向推送（用 SSE 簡單 100 倍）。

  
### SSE：Server-Sent Events（單向推送、AI 串流首選）

**SSE** = HTTP 基礎上的「**伺服器持續推資料給客戶端**」。比 WebSocket 簡單、單向、自動重連。**ChatGPT / Claude 那種「一字一字出來」就是 SSE**。

```text
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import asyncio

@app.get("/stream")
async def stream():
    async def generate():
        for i in range(10):
            yield f"data: {{\"count\": {i}}}\n\n"     # SSE 格式：data: xxx\n\n
            await asyncio.sleep(1)                     # 模擬慢慢推
    
    return StreamingResponse(generate(), media_type="text/event-stream")
```

  
**AI 串流範例（呼叫 OpenAI、把回應一字字推給前端）：**

```text
@app.post("/chat")
async def chat(message: str):
    async def generate():
        async for chunk in openai_stream(message):
            yield f"data: {chunk}\n\n"
    return StreamingResponse(generate(), media_type="text/event-stream")
```

  
**Client 端（瀏覽器 JS）：**

```text
const evt = new EventSource("/stream");
evt.onmessage = (e) => console.log(JSON.parse(e.data));
```

> 💡 **SSE vs WebSocket** —— 只需要「伺服器推給客戶端」（如 AI 串流 / 通知）就用 SSE、客戶端也要送（聊天 / 協作）才需要 WebSocket。

  
### BackgroundTasks（FastAPI 內建簡單背景）

**FastAPI 內建 `BackgroundTasks`**——適合「**簡單一次性背景任務、不需重試**」。例如：使用者註冊後寄歡迎信。

```text
from fastapi import BackgroundTasks

def send_welcome_email(email: str):
    # 慢操作（要 1-3 秒）
    send_email(email, "Welcome!")

@app.post("/register")
async def register(
    user: UserCreate,
    background_tasks: BackgroundTasks,        # FastAPI 自動注入
    db: AsyncSession = Depends(get_db)
):
    new_user = User(**user.model_dump())
    db.add(new_user)
    await db.commit()
    
    # === 寄信丟背景、API 立刻回 ===
    background_tasks.add_task(send_welcome_email, user.email)
    
    return {"user": new_user}
    # ↑ API 回應立刻送出（< 100ms）
    # ↑ 寄信在背景慢慢跑（不影響 user）
```

  
**BackgroundTasks 適用 vs 不適用：**

```text
✓ 適合：
- 簡單一次性背景（寄信 / log / 簡單通知）
- 不需要重試（失敗就失敗）
- 不需要跨 process（只在同一個 FastAPI 內）

✗ 不適合：
- 重 task（影片轉檔 / 大量資料處理）
- 需要重試（失敗自動重跑）
- 需要監控 / 結果追蹤
- 跨 process（多 worker 共用 queue）
```

→ 上述情境用 **Celery / RQ**（見下）

  
### Celery：production-grade 任務隊列

**Celery** = Python **業界標準任務隊列**——支援重試、排程、跨 process、優先級、結果追蹤。但**設定複雜**、新手覺得難。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🖥️ **在終端機輸入**（裝 Celery + Redis client）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

```text
uv add celery redis
```

**這 2 個套件做什麼？**
- `celery` —— Celery 主套件（任務佇列框架）
- `redis` —— Redis Python 客戶端（Celery 用 Redis 當 broker、暫存任務）

> ⚠️ 還要另外**安裝 Redis server**（Mac: `brew install redis`、Docker: `docker run -d -p 6379:6379 redis`）。

  
**寫 `tasks.py`：**

```text
from celery import Celery

celery_app = Celery(
    "myapp",
    broker="redis://localhost:6379/0",        # broker = 任務暫存（誰要做啥）
    backend="redis://localhost:6379/1"        # backend = 結果暫存（做完啥結果）
)

# === 任務：寄信（含自動重試）===
@celery_app.task(bind=True, max_retries=3)
def send_email_task(self, email: str, subject: str, body: str):
    try:
        email_service.send(email, subject, body)
    except Exception as exc:
        # 失敗自動重試、用指數退避（exponential backoff、1s → 2s → 4s）
        raise self.retry(exc=exc, countdown=2 ** self.request.retries)

# === 任務：影片處理（重 task、必丟背景）===
@celery_app.task
def process_video(video_id: int):
    video = db.query(Video).get(video_id)
    # 轉檔、產生縮圖、上傳 CDN（可能跑 30 分鐘）
    process(video)
```

  
**從 FastAPI 觸發 Celery task：**

```text
@app.post("/upload-video")
async def upload(video_data: ...):
    video = Video(...)
    db.add(video)
    await db.commit()
    
    # === 丟 Celery 隊列、立刻回 ===
    process_video.delay(video.id)              # .delay() = 非同步觸發
    
    return {"video_id": video.id}              # 不等影片處理完、立刻回
```

  
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🖥️ **在另一個終端機輸入**（啟動 Celery worker、實際執行 task）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

```text
celery -A tasks worker --loglevel=info
```

**這行做什麼？逐個解釋：**
- `celery -A tasks` —— 用 `tasks.py` 內定義的 celery_app
- `worker` —— 啟動 worker 模式（從 broker 拉任務執行）
- `--loglevel=info` —— 印 INFO 等級 log（看到任務跑的訊息）

> 💡 **架構**：FastAPI 觸發任務 → 寫進 Redis broker → Celery worker 拉任務 → 執行 → 結果寫回 Redis backend。worker 可以開很多 process / 多台機器、scale 起來。

🖥️ **在終端機輸入**（裝 + 啟動 Flower）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### Celery Beat：定時任務（cron 在 Python）

```text
from celery.schedules import crontab

celery_app.conf.beat_schedule = {
    # === 每天凌晨 3 點跑日報 ===
    "daily-report": {
        "task": "tasks.generate_daily_report",
        "schedule": crontab(hour=3, minute=0),
    },
    # === 每 5 分鐘 health check ===
    "every-5-min": {
        "task": "tasks.health_check",
        "schedule": 300.0,                     # 秒數
    },
}
```

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🖥️ **在另一個終端機輸入**（啟動 Beat、定時觸發）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

```text
celery -A tasks beat --loglevel=info
```

**這行做什麼？** 啟動 Beat scheduler。**Beat 負責「排程」**、**Worker 負責「執行」**——所以兩個都要跑。

  
### 2026 Celery 替代品（更簡單 / 更現代）

```text
=== 1. RQ（Redis Queue）===
- 比 Celery 簡單 5 倍、學 1 小時就會
- 適合 indie / 小 SaaS

=== 2. Dramatiq ===
- modern Python task queue
- 設計比 Celery 好（沒歷史包袱）

=== 3. Taskiq ===
- async-first（原生 async）
- FastAPI 親和度高、新興主流

=== 4. Inngest / Trigger.dev ===
- serverless 任務（不用自架 Redis）
- 按使用量計費、適合 indie
```

  
**indie 選擇心法：**

```text
- 小 SaaS（< 1000 用戶）：FastAPI BackgroundTasks（不用裝額外東西）
- 中 SaaS（需要 retry / 監控）：RQ（簡單夠用）
- 大型 / 複雜流程：Celery（業界標準、生態成熟）
- Serverless / 不想自架 Redis：Inngest（很適合 Vercel / Cloudflare 部署）
```

  
    **🚀 進階技巧**
    
2026 Python 後端即時 / 背景 stack：(1) **WebSocket** —— 聊天 / 協作 / 多人遊戲 (2) **SSE** —— AI 串流、單向通知（最常用）(3) **BackgroundTasks** —— 簡單背景（寄信 / log） (4) **Celery / RQ + Redis** —— 複雜任務（影片轉檔 / 大量處理 / 排程）。**indie 學習順序**：先用 BackgroundTasks（day 1 就有）→ 需要 retry / 排程再加 RQ → 需要重 task / 複雜流程才上 Celery。**不要過早優化**——10 個用戶不需要 Celery、就 BackgroundTasks 解決。

  

  
    
      ⚠️
      新手常見錯誤
    
    背景任務 3 大新手坑：(1) **在 endpoint 內直接跑慢操作** —— 例如 `/register` 內直接寄信、API 等 3 秒、user 以為當掉。**慢的事一律丟背景**。(2) **BackgroundTasks 跑重 task** —— BackgroundTasks 跟 API 同 process、跑 1 分鐘 task 會卡住整個 FastAPI、新請求進不來。**重 task 用 Celery / RQ**。(3) **Celery worker 沒重啟** —— 改 task code 後忘了重啟 worker、繼續跑舊版、debug 半天。**改 worker code 一定重啟 Celery worker（Ctrl+C 再起）**。
