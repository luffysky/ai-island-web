**RAG = Retrieval Augmented Generation**——「**先查資料、再生成**」。讓 LLM 能回答「**不在它訓練資料裡**」的問題。**2026 indie hacker 做 AI 產品的核心技巧**。

> 💡 **核心心法**：LLM 不知道你公司 / 客戶 / 私有資料、RAG 解這個問題。**比 fine-tune 簡單 10x、效果接近**。

  
### 為什麼需要 RAG？

**LLM 本身的限制：**

```text
🕰️ 知識截止
   → 訓練資料截到某時間

🔒 不知道你的私有資料
   → 公司文件 / 客戶資料 / 內部 wiki

😶 會編造（hallucinate）
   → 不知道時瞎掰

📏 context 有限
   → 不能把 10GB 文件塞 prompt

💸 更新困難
   → 要更新知識要 fine-tune（貴）
```

**RAG 解這些**：「**每次問問題前先去你的資料庫找相關資訊、再讓 LLM 回答**」。

  
### RAG 完整流程

```text
=== 準備階段（建立 index、做一次）===

1. 載入文件（PDF / Word / Markdown / 網頁 / DB）
2. 切段（Chunking）—每段 300-800 字 + overlap 50-100
3. 每段轉 embedding（向量）
4. 存進向量資料庫（含原文 + 向量 + metadata）

=== 查詢階段（每次使用者問）===

1. 使用者問題："請假規定是什麼？"
2. 問題轉 embedding
3. 向量 DB 找最相似的 5-10 段
4. 把找到的段落 + 問題 + system prompt 塞給 LLM
5. LLM 根據 context 回答 + 引用來源

→ 答案基於你的資料、不會編造、有來源、不需重訓 LLM
```

  
### RAG 真實範例

```text
=== 使用者問：「公司請假規定是什麼？」===

向量搜尋找到 3 段：

段落 1（員工手冊 p.45）：
「全職員工每年享有 14 天年假...」

段落 2（規章 §3.2）：
「請假需提前 3 天告知主管...」

段落 3（HR FAQ）：
「病假超過 3 天需附醫生證明...」

=== 塞給 LLM 的 prompt ===
「根據以下文件回答使用者問題。
不要根據文件以外的知識回答。
回答時引用來源。

文件：[段落 1] [段落 2] [段落 3]

問題：公司請假規定是什麼？」

=== LLM 回答 ===
「公司請假規定如下：
1. 年假：每年 14 天[1]
2. 申請：需提前 3 天告知主管[2]
3. 病假：超過 3 天需醫生證明[3]

來源：
[1] 員工手冊 p.45
[2] 規章 §3.2
[3] HR FAQ」
```

  
### Chunking 策略（關鍵設計）

**文件切成多大一段？這是 RAG 第一個要調的參數。**

```text
❌ 太大（> 2000 字）
   → 浪費 context
   → 不精準
   → 相似度不準

❌ 太小（< 100 字）
   → 失去 context
   → 一個概念被切碎
   → 答案不完整

✅ 推薦 300-800 字 + overlap 50-100
   → 平衡精準度 + context
   → overlap 避免概念被切碎
```

  
### 5 種 Chunking 方法

```text
1. 固定大小（Fixed Size）
   → 每 500 字切一次
   → 最簡單、最常用
   → langchain CharacterTextSplitter

2. 遞迴切（Recursive）
   → 先試 \n\n 切、不夠細 \n、再不夠 空格
   → 保留語意邊界
   → langchain RecursiveCharacterTextSplitter（推薦）

3. 語意切（Semantic）
   → 用 embedding 找語意斷點
   → 同個主題在一段
   → 慢、貴、適合長文

4. 結構切（Structural）
   → 按 markdown header / HTML tag 切
   → 適合結構化文件

5. 文件型態特定
   → PDF 按 page
   → Code 按 function
```

**這行做什麼？** 裝 langchain（RAG 框架）+ OpenAI integration + Chroma（本機向量 DB）。

  
📄 **寫進程式檔**：`rag.py`

```text
from langchain_community.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_community.vectorstores import Chroma
from langchain.chains import RetrievalQA

# === 1. 載入文件 ===
loader = PyPDFLoader("employee_handbook.pdf")
docs = loader.load()

# === 2. 切段 ===
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50,
    separators=["\n\n", "\n", " ", ""]
)
chunks = splitter.split_documents(docs)
print(f"切成 {len(chunks)} 段")

# === 3. 轉 embedding + 存向量 DB ===
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = Chroma.from_documents(chunks, embeddings)

# === 4. 建 retriever + QA chain ===
retriever = vectorstore.as_retriever(search_kwargs={"k": 5})

llm = ChatOpenAI(model="gpt-4o", temperature=0)
qa = RetrievalQA.from_chain_type(
    llm=llm,
    retriever=retriever,
    return_source_documents=True
)

# === 5. 查詢 ===
result = qa.invoke({"query": "請假規定是什麼？"})
print(result["result"])

# === 印來源 ===
for doc in result["source_documents"]:
    print(f"- {doc.metadata['source']} p.{doc.metadata['page']}")
```

  
### Hybrid Search（混合搜尋）

**只用 vector 不夠、結合「關鍵字 + 向量」效果最好：**

```text
=== 純向量 ===
✅ 語意相似（「咖啡」找到「拿鐵」）
❌ 精確匹配差（搜「2024」可能混進「2023」）

=== 純關鍵字（BM25）===
✅ 精確匹配強
❌ 同義詞找不到

=== Hybrid（混合）===
1. 向量搜 + 關鍵字搜 各拿 top 20
2. 用 Reciprocal Rank Fusion 合併
3. 取 top 10

→ 兩個優點都拿
→ 2026 RAG 主流
```

  
### Reranker（重新排序、效果提升 20%）

**情境**：向量搜拿到 20 個、但前 10 個哪個最相關？用 reranker 重排。

```text
import cohere

co = cohere.Client()

# === Step 1：向量搜拿 20 個 candidates ===
candidates = vector_search(query, top_k=20)

# === Step 2：用 reranker 重排（cohere rerank-3）===
results = co.rerank(
    model="rerank-multilingual-v3.0",
    query=query,
    documents=[c.content for c in candidates],
    top_n=5
)

# → 比純向量精準 20-30%
# → 但慢 + 多付 reranker 費用
```

  
### RAG 進階變體

```text
🔁 HyDE（Hypothetical Document Embeddings）
   → 先用 LLM 生「假想答案」、用它的 embedding 搜
   → 對短 query 效果好

🌳 Hierarchical RAG
   → 多層次：先搜大段、再搜細段
   → 大文件友善

🤖 Self-RAG
   → 動態決定要不要 retrieve
   → 簡單問題不查、複雜才查

📊 Graph RAG
   → 用 knowledge graph 增強
   → 適合複雜關聯資料

🔄 Adaptive RAG
   → 多次 retrieve、反覆精煉
```

  
### RAG 常見坑（5 大）

```text
1. ❌ Chunking 太大 / 太小
   ✅ 300-800 字 + overlap

2. ❌ 沒 metadata filter
   ✅ 用 user_id / date / category 過濾

3. ❌ 沒 reranker
   ✅ 加 reranker、精準度 +20-30%

4. ❌ Prompt 沒明確指示 LLM「只用 context 回答」
   ✅ 強調「不要根據文件以外的知識」

5. ❌ 沒引用來源
   ✅ 永遠引用、user 可驗證
```
