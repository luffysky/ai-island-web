**AI Eval：怎麼測 AI**——傳統測試：輸入 X 應輸出 Y。**AI 輸出不確定、怎麼測？**

> 💡 **2026 真相**：沒 Eval 的 AI 產品 = 賭博。**Eval 是「**從 demo 變 production**」的關鍵分水嶺**。

  
### 為什麼需要 Eval

```text
😵 改 prompt 後不確定變好變壞
   → 直覺：「應該更好」
   → 實際：可能 regression

🤷 換模型不知道效果差多少
   → 從 GPT-4o → GPT-4o-mini 省錢
   → 但準確率掉多少？

❓ 客戶說「答錯」但你不知道準確率多少
   → 1% 錯？10% 錯？

📉 無法量化迭代
   → 沒 baseline、改了不知道結果
```

  
### Eval 3 大類型

  
### 1. Code-based（程式可判斷、最簡）

```text
=== 測試：「請回 JSON {name, age}」===

判斷：
✅ 是否合法 JSON？
✅ 有 name 跟 age 嗎？
✅ type 對嗎？

簡單、可自動化
```

```text
import json

def test_json_output(query, response):
    try:
        data = json.loads(response)
    except:
        return False
    
    return "name" in data and "age" in data and isinstance(data["age"], int)
```

  
### 2. LLM-as-Judge（LLM 評分）

**用更強的 LLM 評估：**

```text
=== Judge Prompt ===
「以下是使用者問題、AI 回答、標準答案。
請評分（1-5）：
- 準確性（accuracy）
- 完整性（completeness）
- 相關性（relevance）

問題：[...]
AI 回答：[...]
標準答案：[...]

回 JSON：{accuracy, completeness, relevance, reasoning}」

→ 用 GPT-4 / Claude Opus 當裁判
→ 比人類便宜（1 萬次 < $10）
→ 夠準（跟人類評分相關性 0.7-0.9）
```

  
### 3. Human Eval（最準但貴慢）

**真人標記**——最準但貴慢。重要場景做。

```text
✅ 適合
- 高風險產品（醫療 / 法律）
- 模型 release 前最終 check
- 訓練 reward model

❌ 不適合
- 每次 prompt 改都跑
- 大量迭代
```

  
### Eval 8 大維度

```text
✅ Accuracy（準確性）
   → 答對嗎

✅ Relevance（相關性）
   → 跟問題相關嗎

✅ Faithfulness（RAG 用）
   → 是否基於 context、無編造

✅ Coherence（連貫性）
   → 流暢嗎、邏輯通順

✅ Safety（安全性）
   → 有害內容？

✅ Tone（語氣）
   → 符合品牌語氣

✅ Latency（延遲）
   → 速度

✅ Cost（成本）
   → 每次成本
```

  
### 實戰 Eval 流程

```text
=== 5 步驟 ===

1. 準備 50-200 個測試 case
   - 簡單問題
   - 邊界情況
   - 應拒絕的（傷害 / 無關）
   - 真實使用者問過的

2. 每次改 prompt / 模型 → 跑全部 case

3. 算分數：
   - Accuracy：85% → 88%（提升）

4. 看 regression：
   - 之前對的現在錯了？
   - 找出 fail case 分析

5. 持續累積測試集
```

### RAG Eval（特殊指標）

```text
✅ Retrieval Precision
   → 找到的 5 個有幾個相關

✅ Retrieval Recall
   → 相關的有幾個被找到

✅ Answer Faithfulness
   → 答案是否基於 context

✅ Answer Relevance
   → 答案是否回答了問題
```

  
**LlamaIndex Eval 範例（L19 教過）：**

```text
from llama_index.core.evaluation import FaithfulnessEvaluator, RelevancyEvaluator
from llama_index.llms.openai import OpenAI

eval_llm = OpenAI(model="gpt-4o")

faithfulness = FaithfulnessEvaluator(llm=eval_llm)
relevancy = RelevancyEvaluator(llm=eval_llm)

response = query_engine.query("請假規定？")

faith_result = faithfulness.evaluate_response(response=response)
rel_result = relevancy.evaluate_response(response=response)

print(f"Faithful: {faith_result.passing}")
print(f"Relevant: {rel_result.passing}")
```

  
### 持續 Eval（生產級）

**不只 deploy 前測、生產也要 sample 看：**

```text
=== 每天從生產 log 抽 1% 樣本 ===

1. 抽 100 筆 random sample
2. LLM-as-judge 打分
3. 異常案例告警
4. 累積成新測試集

→ 持續迭代
→ 早期發現問題（網站改版 / 反爬 / 模型更新）
```

  
### Eval 5 大常見坑

```text
1. ❌ 測試集太小
   50 個不夠代表性
   ✅ 200+ 比較穩

2. ❌ 測試集偏
   只測簡單、沒測 edge case
   ✅ 各種類型平均分布

3. ❌ 沒 baseline
   不知道改了好還壞
   ✅ 永遠記錄 baseline 分數

4. ❌ LLM-as-judge 偏好長答案
   要設計好 rubric
   ✅ 明確指示「不要看長度、看品質」

5. ❌ 測試集不更新
   使用者問新問題沒納入
   ✅ 定期 review log、加新 case
```

### 真實案例：客服 AI Eval

```text
=== 測試集 ===
- 100 個常見問題（FAQ）
- 50 個 edge case（怪問題）
- 30 個應拒絕（不相關 / 不當）
- 20 個真實使用者問題

=== 維度 ===
- Accuracy（答對率）
- Tone（友善程度）
- Safety（不出包）

=== 結果 ===
GPT-4o：95% / 90% / 98%
GPT-4o-mini + 我們 RAG：92% / 88% / 96%

→ 用 GPT-4o-mini + RAG、便宜 90%、效果只差 3-4%
→ 商業選擇：用便宜的
```
