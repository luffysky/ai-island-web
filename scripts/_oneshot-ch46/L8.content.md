**Prompt Engineering 入門**——「**跟 AI 講話的藝術**」。同樣的問題、不同 prompt、答案天差地別。**2026 仍是 LLM 使用最重要的技能**。

> 💡 **核心心法**：你越精確、AI 越精確。**模糊 prompt = 模糊答案**。

  
### 5 個基本原則

  
### 原則 1：明確（Be Specific）

```text
❌ 模糊：
「幫我寫程式」
→ AI 不知道要寫什麼、用什麼語言、目的、範圍

✅ 明確：
「用 Python + FastAPI 寫一個 GET /users/{id} endpoint、
從 PostgreSQL 撈使用者資料、找不到回 404、
返回 JSON 含 id / name / email、
含完整 type hint 跟 docstring」

→ AI 知道：語言、框架、行為、錯誤處理、格式、規範
```

> 💡 **「明確」的反面是「丟給 AI 自己想」**——AI 會「想」、但答案不一定是你要的。

  
### 原則 2：給 Context（背景資訊）

```text
❌ 沒 context：
「幫我修這個 bug」
（你以為 AI 知道、但 AI 沒看到你的 code）

✅ 完整 context：
「我有以下 Python 函式：

def sum_list(items):
    total = 0
    for item in items:
        total + item            # ← bug 在這
    return total

預期輸入 [1, 2, 3] 返回 6
實際返回 0

請找出問題並修正、並解釋為什麼。」

→ AI 看 code 找出 `total + item` 沒賦值給 total
→ 應該 `total += item`
→ 還會解釋為什麼錯
```

  
### 原則 3：給角色（Role Prompting）

```text
「你是有 10 年經驗的資深 Python 後端工程師、
熟悉效能優化跟資安。

請 review 以下程式碼、依優先級指出：
1. 安全漏洞
2. 效能問題
3. 可讀性問題

[code]」

→ AI 進入「工程師 review」模式
→ 比沒角色精準、結構化
```

> 💡 **2026 業界討論「role prompting 是否還有效」**——新版模型可能不需要。但**對複雜任務仍有效**、特別是需要特定領域知識（醫療 / 法律 / 財務）。

  
### 原則 4：給範例（Few-shot Learning）

```text
「請把以下句子轉成 Twitter 適合的格式（< 280 字、加 emoji、輕鬆語氣）：

範例 1：
原文：今天去咖啡店、買了一杯肯亞 AA、很好喝。
Twitter：☕ 早安！今天的肯亞 AA 太讚了～酸度漂亮、香氣濃郁、上班的動力 100% 💪

範例 2：
原文：學了 Python 三個月、終於寫出第一個爬蟲。
Twitter：🎉 三個月的努力沒白費！我的第一個 Python 爬蟲跑起來了～從零到爬蟲、Python 真的好上手 🐍

現在請轉換：
原文：剛剛部署了第一個 Next.js 網站到 Vercel。
Twitter：」

→ AI 學會風格 + 格式
→ 比純文字描述精準 10 倍
```

> 💡 **Few-shot 黃金數量**：2-5 個範例最佳。太多反而 token 浪費、效果不會更好。

  
### 原則 5：分步驟（Step-by-step / Chain of Thought）

```text
❌ 一次問：
「店裡有 23 個蘋果、買進 6 個、用了 9 個、剩多少？」
（AI 直接答、可能算錯）

✅ 要 AI 分步：
「店裡有 23 個蘋果、買進 6 個、用了 9 個、剩多少？
請一步一步算、最後給答案。」

→ AI 會：
Step 1：原本 23 個
Step 2：買進 6 個 → 23 + 6 = 29
Step 3：用了 9 個 → 29 - 9 = 20
答案：20 個

→ 比直接答正確率高 30-50%
```

> 💡 **這就是 Chain-of-Thought（CoT、思維鏈）**——下節 L9 詳教。

  
### 進階：用 XML tag 分區（Anthropic 推薦）

```text
「分析以下客戶評論、回傳結構化結果：

<review>
這家店食物很棒、特別是炒飯、但服務生態度不好、等了 30 分鐘才上菜。
</review>

<task>
請從評論中抽取：
1. 正面評論（quote 原文）
2. 負面評論（quote 原文）
3. 整體評分（1-5）
4. 建議改善（一句話）
</task>

<output_format>
JSON 格式、不要 markdown 包覆。
</output_format>
」

→ AI 清楚知道每段在做什麼
→ Claude 對 XML tag 反應最好
```

  
### 完整 Prompt 模板

```text
=== 推薦結構 ===

# Role
你是 [專業角色]

# Context
[背景資訊、目前狀況]

# Task
請 [明確任務]

# Constraints
- [限制 1]
- [限制 2]
- [限制 3]

# Examples（few-shot）
範例 1：[input → output]
範例 2：[input → output]

# Format
[輸出格式：JSON / Markdown / 純文字]

# Input
[使用者的實際輸入]
```

  
### 5 大常見錯誤

```text
1. ❌ 太模糊
   「幫我」、「處理一下」、「改一下」
   ✅ 改用具體動詞 + 範圍

2. ❌ 一次問太多
   「寫一個爬蟲、加上 DB、做 dashboard、上 Vercel」
   ✅ 一次一個任務、依序完成

3. ❌ 沒給 context
   假設 AI 知道你之前說過的事
   ✅ 每個 prompt 自包含

4. ❌ 沒指定格式
   AI 用 markdown / JSON / 純文字、隨機選
   ✅ 明確指定 format

5. ❌ 不檢查 AI 答案
   AI 會編造（hallucination）
   ✅ 重要事實驗證、別盲信
```

### 真實案例：好 vs 壞 prompt 對比

  
**爛 prompt：**

```text
「幫我寫個爬蟲」

AI 回答：
「請問你要爬什麼網站？要用什麼語言？要存哪？」
→ 又一輪溝通、效率差
```

  
**好 prompt：**

```text
「用 Python + httpx + BeautifulSoup 寫爬蟲、
爬 https://news.ycombinator.com 的所有文章、
抽取 title / url / score / comment_count、
存成 CSV 檔（UTF-8 BOM）。

要求：
- 加 retry（指數退避、最多 3 次）
- 加 User-Agent 偽裝成 Chrome
- 處理空欄位（用 N/A 填）
- 加 logging
- 異常時印錯誤訊息但繼續處理下一筆

直接給可執行的完整 code、不要切段。」

AI 回答：
直接給完整可跑的 code、含 retry / logging / 錯誤處理。
→ 一次到位
```

  
### 心法總結

```text
✅ 明確 > 模糊
✅ 給 context > 假設 AI 知道
✅ 給角色 > 預設模式
✅ 給範例 > 純文字描述
✅ 分步驟 > 一次到底
✅ 指定格式 > 讓 AI 隨機選
✅ 用 XML tag 分區 > 全混在一起
✅ 用 Prompt Generator > 自己慢慢調
```
