**LangChain / LangGraph 入門**——LangChain 是 2022-2024 最紅的 AI 框架。**2026 仍主流、但有爭議**——太大、太抽象。

> 💡 **2026 建議**：新專案優先 Vercel AI SDK / 純官方 SDK。LangChain 適合「**生態大 + prototype 快**」、生產級可考慮重寫。

  
### LangChain 是什麼

整合 LLM、工具、資料庫的「**瑞士刀**」：

```text
🤖 LLM 抽象（換模型一行）
📝 Prompt template
🔍 Vector DB 整合（30+ 種）
📄 Document loaders（讀 PDF / Word / Web）
🤝 Agent 框架
💭 Memory 管理
🔗 Chain（串聯多步驟）
```

### 基本使用

```text
from langchain_anthropic import ChatAnthropic
from langchain_core.prompts import ChatPromptTemplate

# === 1. 建 LLM ===
llm = ChatAnthropic(model="claude-opus-4-7")

# === 2. 建 prompt template ===
prompt = ChatPromptTemplate.from_messages([
    ("system", "你是專業翻譯"),
    ("user", "把下面翻成 {language}：{text}")
])

# === 3. 用 | 串接成 chain（LCEL 語法）===
chain = prompt | llm

# === 4. 呼叫 ===
result = chain.invoke({
    "language": "日文",
    "text": "今天天氣很好"
})
print(result.content)
```

> 💡 **LCEL（LangChain Expression Language）** = 用 `|` 串接、像 Linux pipe、2024+ 主流寫法。

  
### RAG with LangChain

```text
from langchain_community.document_loaders import WebBaseLoader
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings
from langchain_chroma import Chroma
from langchain.chains import RetrievalQA

# === 1. 載入文件 ===
loader = WebBaseLoader("https://example.com/docs")
docs = loader.load()

# === 2. 切段 ===
splitter = RecursiveCharacterTextSplitter(chunk_size=500)
chunks = splitter.split_documents(docs)

# === 3. 存進向量 DB ===
vectorstore = Chroma.from_documents(chunks, OpenAIEmbeddings())

# === 4. 建 RAG chain ===
retriever = vectorstore.as_retriever()
qa = RetrievalQA.from_chain_type(llm=llm, retriever=retriever)

# === 5. 用 ===
qa.invoke("請問請假規定？")
```

  
### LangGraph：Stateful Agent

**2024 LangChain 出的狀態機框架**——解決 Agent 流程控制問題。

```text
from langgraph.graph import StateGraph, END
from typing import TypedDict

# === 1. 定義 state schema ===
class State(TypedDict):
    question: str
    context: list
    answer: str

# === 2. 定義 nodes ===
def retrieve(state):
    docs = vectorstore.search(state["question"])
    return {"context": docs}

def generate(state):
    answer = llm.invoke(build_prompt(state))
    return {"answer": answer}

def should_continue(state):
    if not state["answer"]:
        return "retrieve"
    return END

# === 3. 建狀態圖 ===
workflow = StateGraph(State)
workflow.add_node("retrieve", retrieve)
workflow.add_node("generate", generate)
workflow.set_entry_point("retrieve")
workflow.add_edge("retrieve", "generate")
workflow.add_conditional_edges("generate", should_continue)

# === 4. 編譯 + 跑 ===
app = workflow.compile()
result = app.invoke({"question": "請假規定？"})
```

> 💡 **LangGraph 優勢**：(1) 顯式狀態機（debug 容易） (2) 條件分支 (3) loop 控制 (4) human-in-the-loop。

  
### LangChain 的批評（必看）

2024 業界開始討論「**不要用 LangChain**」：

```text
❌ 太抽象
   → 簡單事被包成複雜

❌ API 不穩
   → 頻繁 breaking change
   → 0.x 版本一直變

❌ 效能不好
   → 層層 wrapper、latency 高

❌ debug 困難
   → error stack 一堆 LangChain 內部
   → 很難找到自己 code 的問題

❌ 文件混亂
   → 版本對不上
   → 同一個 feature 有 3 種寫法
```

> 💡 **業界文章**：「Why we no longer use LangChain」、「LangChain considered harmful」這類在 2024 多次上 Hacker News 頭版。

  
### 2026 替代方案

| 工具 | 特色 |
|---|---|
| **純官方 SDK**（anthropic / openai） | 加自己的薄層、最乾淨 |
| **LlamaIndex** | RAG 專精（L19 教） |
| **Haystack** | 類似 LangChain 但乾淨 |
| **Vercel AI SDK** | TypeScript 主流（L18 教） |
| **DSPy** | Stanford 出、用 ML 思維優化 prompt |
| **Pydantic AI** | 型別友善 |
| **Mirascope** | 簡潔 Pythonic |

  
### 2026 選擇建議

```text
📚 學習 / 快速 prototype
   → LangChain（生態大、教學多）

🏭 生產 / 正式產品
   → 純官方 SDK 或 Pydantic AI

🤖 複雜 agent
   → LangGraph 或自己用狀態機

🔍 純 RAG
   → LlamaIndex 或自寫

🟦 TypeScript / Next.js
   → Vercel AI SDK
```

> 💡 **心法**：**不要「非用 LangChain 不可」**。用對的工具、不是用熱門工具。

  
### LangChain 適合 / 不適合場景

```text
✅ 適合
- 學習 AI 開發（教學多）
- 快速 prototype
- 簡單 RAG（用 RetrievalQA chain）
- 整合很多資料來源（document loaders）
- 學 agent 概念

❌ 不適合
- 生產級高效能需求
- 簡單 LLM call（直接用 SDK）
- 需要極度穩定（API 一直變）
- 團隊 debug 能力有限
```

  
### 真實案例

  
**何時用 LangChain 成功：**

```text
新創 MVP 階段：
- 1 週要 demo 給投資人
- LangChain 整合 RAG + Agent + 多 LLM
- 1 天搭起來
- 不在意效能優化
- 跑通就贏
```

  
**何時改成自寫：**

```text
產品上線後：
- 流量大、效能瓶頸
- Debug 困難
- API call 透明度低
- 改成純 SDK + 自寫薄層
- 開發成本 +50%、但維護成本 -70%
```
