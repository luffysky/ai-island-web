**Fine-tune vs RAG vs Prompt：怎麼選**——「**怎麼讓 LLM 學會我們公司的東西？**」三種方案、95% 場景選對能省 80% 時間。

> 💡 **2026 黃金路徑**：Prompt → Few-shot → RAG → Prompt Caching → Fine-tune（最後手段）。**95% indie 到 RAG 就夠**。

  
### 三種方案差異

| | Prompt | RAG | Fine-tune |
|---|---|---|---|
| **原理** | 每次塞 context | 動態查再塞 | 改變模型權重 |
| **準備時間** | 分鐘 | 幾小時 | 幾天 |
| **成本** | 低 | 中 | 高 |
| **更新資料** | 立即 | 立即 | 要重訓 |
| **大量資料** | ❌ 不行 | ✅ 行 | ✅ 行 |
| **學風格** | 少量可 | 不太行 | ⭐ 強 |
| **學新知識** | 當下 | 當下 | 內化 |

  
### Prompt（系統提示 + Few-shot）

**適合：**

```text
✅ 規則 / 風格不多（< 5000 字）
✅ 內容不常變
✅ 快速 prototype
✅ 不需要動態檢索
```

**範例：**

```text
system_prompt = """你是公司客服、回答時遵守以下規則：
1. 用繁體中文
2. 友善但專業
3. 不確定的事直接說「我需要轉接專員」
4. 不能提到競爭對手

範例 1：...
範例 2：...
"""
```

> ⚠️ **限制**：context window 限制、超過 100k 字就難。

  
### RAG（檢索增強生成）

**適合：**

```text
✅ 知識量大（千、萬篇文件）
✅ 需要引用來源
✅ 知識常更新
✅ 需要多租戶（不同使用者看不同資料）
```

**限制**：學不會「**風格**」、只能基於資料回答。

  
### Fine-tune（微調模型權重）

**適合：**

```text
✅ 需要學特殊風格 / 語氣
   → 醫生口吻 / 律師措辭

✅ 輸出格式很固定
   → 永遠回特定 JSON

✅ 專業領域術語
   → 醫療 / 法律 / 金融

✅ 要省 token
   → fine-tune 後 prompt 可變短

✅ 有 10000+ 高質量訓練資料
```

**限制：**

```text
❌ 準備資料困難
❌ 訓練要錢（$50-5000+）
❌ 更新麻煩（要重訓）
❌ 容易 overfit（過擬合）
❌ 可能讓模型變笨
   → catastrophic forgetting
```

  
### 實戰決策流程

```text
=== 黃金 4 步驟 ===

1. 先試 Prompt（最快、最便宜）
   ↓ 效果不夠

2. 加 Few-shot（給範例）
   ↓ 還不夠

3. 上 RAG（大量資料）
   ↓ 學風格不行

4. 才考慮 Fine-tune

→ 90% 場景到 RAG 就夠
→ 真的需要 fine-tune 的 < 5%
```

  
### Cost 對比範例（每月）

**情境：客服 SaaS、每月 100 萬次 query。**

```text
=== 純 Prompt（GPT-4o）===
- 每次 2k input + 500 output tokens
- 月成本：~$750

=== 純 RAG（GPT-4o + pgvector）===
- 每次多 1k context token
- 月成本：~$1000 + $50 vector DB

=== Fine-tune GPT-4o-mini ===
- 訓練：$50 一次
- 每次 only 500 token input（風格內化）
- 月成本：~$75

→ Fine-tune 後成本最低
→ 但前期投資 + 維護成本高
```

  
### 什麼時候 Fine-tune 才真值得

```text
✅ 大量重複 prompt
   → fine-tune 省 token、長期成本低

✅ 特殊風格 / 領域
   → prompt 怎麼寫都不對

✅ 固定格式輸出
   → 100% 穩定

✅ 已驗證產品有 PMF
   → 要降成本 / 提準確度
```

  
### 什麼時候別 Fine-tune

```text
❌ 還在 prototype 階段
   → 規則一直變

❌ 規則 / 知識常變
   → 重訓成本高

❌ 沒幾個高質量 example
   → < 1000 個都不夠

❌ 需要立刻 deploy
   → fine-tune 要幾天

❌ 用得不多（< 1 萬次 / 月）
   → 省的 token 抵不過訓練費
```

  
### 三種方案組合用（業界主流）

**情境：醫療 AI 助手**

```text
Layer 1：Fine-tune
   → 教模型醫療術語 + 醫生口吻
   → 一次性訓練

Layer 2：RAG
   → 接最新醫學論文 / 臨床指引
   → 每月更新

Layer 3：Prompt
   → System prompt 定義安全規則
   → Few-shot 教特殊 case

→ 三層組合、效果最好
```

  
### 2026 各家 Fine-tune API

  
**OpenAI：**

```text
from openai import OpenAI

client = OpenAI()

# === 1. 上傳訓練資料 ===
file = client.files.create(
    file=open("training.jsonl", "rb"),
    purpose="fine-tune"
)

# === 2. 建 fine-tune job ===
job = client.fine_tuning.jobs.create(
    training_file=file.id,
    model="gpt-4o-mini-2024-07-18",
    hyperparameters={"n_epochs": 3}
)

# === 3. 等訓練完 ===
# 通常 1-24 小時

# === 4. 用 fine-tuned 模型 ===
response = client.chat.completions.create(
    model="ft:gpt-4o-mini-2024-07-18:my-org::abc123",
    messages=[...]
)
```

```text
{"messages": [{"role": "system", "content": "你是醫療助手"}, {"role": "user", "content": "頭痛怎麼辦？"}, {"role": "assistant", "content": "建議先休息..."}]}

{"messages": [...]}
```

**最少 10 筆、推薦 100-1000 筆。**

  
### 決策表（依場景）

| 場景 | 推薦 |
|---|---|
| **MVP / 快速驗證** | Prompt + Few-shot |
| **企業內部知識庫** | RAG |
| **客服 AI** | RAG + Prompt |
| **多客戶 SaaS** | RAG（多租戶） |
| **特殊風格寫作** | Fine-tune |
| **醫療 / 法律專業** | Fine-tune + RAG |
| **量超大省 token** | Fine-tune |
| **規則一直變** | Prompt |

  
### 心法總結

```text
✅ Prompt 解決 60% 場景
✅ RAG 再解決 30%（共 90%）
✅ Fine-tune 解決最後 5-10%
✅ 別跳過階段、別過度工程

❌ 別「先 fine-tune 再說」
   → 95% 時候是 overkill

❌ 別忽略 prompt caching
   → 重複 prompt 便宜 90%

❌ 別認為「fine-tune = 一定好」
   → 可能 overfit / 退化
```
