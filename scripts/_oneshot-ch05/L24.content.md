**Utility Types（工具型別）= TS 內建的「型別變形工具」**—— 把現有的型別變一變、不用重打一份。`Partial`、`Pick`、`Omit`、`Record`、`ReturnType`...10+ 個現成工具、覆蓋 80% 日常變型需求。

> 💡 **這課的價值**：學會 Utility Types、你能用 1 行做到別人寫 20 行的型別操作——更新資料表單、API 回傳擷取、function 簽章複用、全部一行搞定。

  
### 1. 為什麼需要 Utility Types？

假設你有一個 `User` 型別：

📄 **寫進程式檔**：`types.ts`

```text
interface User {
  id: number;
  name: string;
  email: string;
  password: string;
  createdAt: Date;
}
```

**現實會遇到的「型別變化」**：

```text
1. 表單更新時：所有欄位都「可選」（只改 name 也行）
2. 公開 API：不能回傳 password
3. 建立新使用者：不需要 id 跟 createdAt（DB 自動產生）
4. 列表只回 id + name
```

每個情境都重打一份 interface？太蠢。Utility Types 就是來救命的。

  
### 2. `Partial<T>` —— 所有欄位變可選

```text
type UpdateUser = Partial<User>;

// 等於：
// {
//   id?: number;
//   name?: string;
//   email?: string;
//   password?: string;
//   createdAt?: Date;
// }

function updateUser(id: number, patch: UpdateUser) {
  // patch 可以只有 { name: "新名字" }
}

updateUser(1, { name: "Alice" });           // ✓
updateUser(1, { email: "a@b.c", name: "A" }); // ✓
updateUser(1, {});                            // ✓（什麼都不改也行）
```

**用途**：PATCH API、表單部分更新、可選 props。

  
### 3. `Required<T>` —— 所有欄位變必填

`Partial` 的反向操作：

```text
interface Config {
  host?: string;
  port?: number;
  ssl?: boolean;
}

type StrictConfig = Required<Config>;
// 等於 { host: string; port: number; ssl: boolean; }
// 全部都必填、沒有 ?
```

**用途**：把「設定預設值後」的 config 標成全必填，後面 code 不用一直 `config.host!`。

  
### 4. `Pick<T, K>` —— 只挑某幾個欄位

```text
type UserPreview = Pick<User, "id" | "name">;
// 等於 { id: number; name: string; }

type UserAuth = Pick<User, "email" | "password">;
// 等於 { email: string; password: string; }
```

**用途**：列表頁只顯示部分欄位、登入只需 email + password。

  
### 5. `Omit<T, K>` —— 去掉某幾個欄位

`Pick` 的反向：

```text
type PublicUser = Omit<User, "password">;
// 等於 User 但沒有 password

type CreateUser = Omit<User, "id" | "createdAt">;
// 等於 User 但沒有 id 跟 createdAt
// （DB 會自動產生）
```

**用途**：API 回傳隱藏敏感欄位、建立資料時不傳自動產生的欄位。

> 💡 **Pick 跟 Omic 是黃金組合**：Pick 是「白名單」、Omit 是「黑名單」。資料庫遷移、API 介面常常兩個交替用。

  
### 6. `Record<K, V>` —— 建立 key-value 物件

```text
type UserRoles = Record<string, "admin" | "user" | "guest">;

const roles: UserRoles = {
  alice: "admin",
  bob: "user",
  carol: "guest"
};

// 限定 key 為特定字串
type PageStats = Record<"home" | "about" | "blog", number>;
const stats: PageStats = {
  home: 1000,
  about: 200,
  blog: 500
  // 漏一個就編譯錯
};
```

**用途**：i18n 字典、設定表、頁面統計、enum-style 對應。

  
### 7. `ReturnType<T>` —— 取 function 的回傳型別

```text
function getUser(id: number) {
  return { id, name: "Alice", email: "a@b.c" };
}

type User = ReturnType<typeof getUser>;
// 等於 { id: number; name: string; email: string; }
```

**`typeof getUser`** = 取得 getUser 的型別（function 簽章）、再用 `ReturnType` 取它回傳的型別。

**用途**：API 函式回什麼、UI 元件就用什麼、不用同步維護兩份型別。

  
### 8. `Parameters<T>` —— 取 function 的參數型別

```text
function login(email: string, password: string, remember?: boolean) {
  // ...
}

type LoginArgs = Parameters<typeof login>;
// 等於 [email: string, password: string, remember?: boolean]

// 用途：包裝原本的 function
function loggedLogin(...args: Parameters<typeof login>) {
  console.log("calling login with", args);
  return login(...args);
}
```

  
### 9. `Awaited<T>` —— 解開 Promise 包裝

```text
async function fetchUser() {
  const res = await fetch("/api/user");
  return res.json() as Promise<User>;
}

type Result = ReturnType<typeof fetchUser>;
// = Promise<User>（還包著 Promise）

type UserData = Awaited<ReturnType<typeof fetchUser>>;
// = User（解開了！）
```

**用途**：寫 hook、寫 server action、處理 async function 回傳值。

  
### 10. `NonNullable<T>` —— 去掉 null / undefined

```text
type Maybe = string | null | undefined;
type Sure = NonNullable<Maybe>;  // = string

function process(input: Maybe) {
  if (input == null) return;
  const safe: Sure = input;  // 已經保證不是 null
}
```

  
### 11. `Exclude<T, U>` / `Extract<T, U>` —— 從 union 挑/排除

```text
type Status = "pending" | "approved" | "rejected" | "cancelled";

type DoneStatus = Exclude<Status, "pending">;
// = "approved" | "rejected" | "cancelled"

type ActiveStatus = Extract<Status, "pending" | "approved">;
// = "pending" | "approved"
```

**用途**：state machine、狀態子集、API 端點限定狀態。

  
### 12. 實戰組合：API + 表單 + DB

📄 **寫進程式檔**：`types/user.ts`

```text
// 1. DB 完整型別
interface UserRow {
  id: number;
  email: string;
  password_hash: string;
  name: string;
  avatar: string | null;
  created_at: Date;
  updated_at: Date;
}

// 2. API 回前端的（不能有 password_hash）
type UserDTO = Omit<UserRow, "password_hash">;

// 3. 建立新使用者（不需要 id + 時間戳）
type CreateUserInput = Omit<UserRow, "id" | "created_at" | "updated_at">;

// 4. 更新時所有欄位可選（但不能改 id）
type UpdateUserInput = Partial<Omit<UserRow, "id" | "created_at">>;

// 5. 列表只回 id + name + avatar
type UserListItem = Pick<UserDTO, "id" | "name" | "avatar">;
```

**一個 base type、衍生 5 個型別、IDE 自動補完全部正確**。

  
### 13. 5 大常見坑

```text
❌ 1. 重複手寫類似的 interface
   → 改一個欄位要改 5 個地方
   → 修：用 Pick / Omit 從 base 衍生

❌ 2. 把 Partial 用在「不該全選的場合」
   → 整個 User 都變可選、漏填欄位不會錯
   → 修：只在更新 API 用 Partial、新增用 Omit

❌ 3. ReturnType 後面忘了 typeof
   → ReturnType<getUser>（錯）
   → ReturnType<typeof getUser>（對）
   → typeof 是把「值」變成「型別」

❌ 4. Record 限定 key 但漏了一個
   → Record<"a" | "b" | "c", X> 物件少寫 c 就錯
   → 修：用 Partial<Record<...>> 容許部分

❌ 5. 巢狀 Utility Types 寫到看不懂
   → Awaited<ReturnType<typeof Partial<Omit<User, "a">>>>
   → 修：拆成多個 type alias 命名、註解清楚
```

  
### 🎯 重點回顧

- **Partial / Required** = 欄位變可選 / 全必填
- **Pick / Omit** = 白名單 / 黑名單挑欄位
- **Record<K, V>** = key-value 物件
- **ReturnType / Parameters** = 從 function 抽型別、配 `typeof`
- **Awaited / NonNullable / Exclude / Extract** = 細部變型工具
- **黃金原則**：永遠有一個 base type、其他都用 Utility 衍生
