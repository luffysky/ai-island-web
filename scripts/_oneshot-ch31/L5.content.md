**ES Modules（ESM、新模組系統）vs CommonJS（CJS、Node 傳統模組）**——`import` vs `require` 的世仇。**Node 20+ 完整支援 ESM、2026 業界主流轉 ESM**。但要懂兩者差別 + interop（互通）細節、不然 import 隨時噴錯。

> 💡 **這課的價值**：不懂 ESM / CJS 差別、Node 專案上 production 噴 `ERR_REQUIRE_ESM` 卡死。會這課後、你能正確設定 package.json、處理舊套件、寫出跨環境都能用的 module。

  
### 1. 兩個系統的歷史

```text
2009 Node.js 誕生
   → 用 CommonJS（CJS）
   → require / module.exports

2015 ES6 標準推出 ES Modules（ESM）
   → import / export
   → 瀏覽器原生支援

2017-2020 Node 慢慢加 ESM 支援
   → 早期一堆 bug、需要 .mjs 副檔名

2023+ Node 20 LTS 完整 ESM 支援
   → 業界開始全面轉

2026 主流框架（Next 15、Vite 5、Astro）已 ESM-first
```

  
### 2. 語法差別（一目了然）

**CommonJS（傳統）**：

📄 **寫進程式檔**：`math-cjs.js`

```text
// 匯出
function add(a, b) {
  return a + b;
}
module.exports = { add };

// 或：
exports.add = add;

// 匯入
const { add } = require("./math-cjs");
console.log(add(1, 2));
```

**ES Modules（新）**：

📄 **寫進程式檔**：`math-esm.js`

```text
// 匯出
export function add(a, b) {
  return a + b;
}

// 預設匯出
export default function multiply(a, b) {
  return a * b;
}

// 匯入
import multiply, { add } from "./math-esm.js";
console.log(add(1, 2));
```

  
### 3. 核心差別（不只是語法）

| 特性 | CommonJS | ES Modules |
|---|---|---|
| 載入 | 同步（runtime） | 非同步（parse 時就解析） |
| 載入時機 | 第一次 require 時 | 程式啟動時 |
| 動態 import | 直接 require(expr) | 用 import() 函數 |
| Top-level await | ❌ 不支援 | ✓ 支援 |
| Tree shaking | 差 | 好 |
| 副檔名 | 可省（`./math` ✓） | 強制寫（`./math.js`）|
| __dirname | ✓ 直接用 | ❌ 要自己算 |
| require.main | ✓ | ❌ 用 `import.meta` |

  
### 4. 怎麼告訴 Node 用哪個？

**方法 A：package.json 設 `type`**：

📄 **寫進 package.json**：

```text
{
  "name": "my-app",
  "type": "module",      // ← 整個 project 用 ESM
  "main": "index.js"
}
```

```text
{
  "type": "commonjs"     // ← 整個 project 用 CJS（預設）
}
```

**方法 B：用副檔名強制指定**：

```text
.js   → 看 package.json 的 type
.mjs  → 強制 ESM
.cjs  → 強制 CJS

// 混用範例：
src/index.mjs   ← 一定是 ESM
src/legacy.cjs  ← 一定是 CJS
src/other.js    ← 看 package.json
```

  
### 5. ESM 特殊規則

**(1) import 必須寫副檔名**：

```text
❌ import { add } from "./math";        // CJS 風格
✅ import { add } from "./math.js";     // ESM 規定

❌ import x from "./components/Button"; // 漏副檔名
✅ import x from "./components/Button.js";
```

> 💡 **TS 例外**：用 TS 時 `import "./math.js"`（即使檔案是 `.ts`）—— `tsc` / `tsx` 會處理副檔名替換。

**(2) 沒有 `__dirname` / `__filename`**：

```text
// CJS：直接用
console.log(__dirname);  // /home/user/project/src

// ESM：用 import.meta
import { fileURLToPath } from "url";
import path from "path";

const __filename = fileURLToPath(import.meta.url);
const __dirname = path.dirname(__filename);
console.log(__dirname);

// Node 20.11+ 新 API
console.log(import.meta.dirname);  // 直接拿到 dirname
console.log(import.meta.filename);
```

**(3) Top-level await（Game Changer）**：

```text
// ESM：可以！
const config = await fetch("/config.json").then(r => r.json());
console.log(config);

// CJS：不行、必須包 async function
async function main() {
  const config = await fetch("/config.json").then(r => r.json());
}
main();
```

  
### 6. ESM 用 CJS 套件（最常遇到）

**ESM 可以 `import` 大多數 CJS 套件**——Node 自動處理：

```text
// ESM 內：
import lodash from "lodash";       // CJS 套件（default import）
import { readFileSync } from "fs"; // Node 內建（dual）

// CJS 的 module.exports = { a, b } → 變成 default import
// 不是 import { a, b } from "lodash"（看套件支不支援）
```

**有些 CJS 套件不支援 named import**：

```text
❌ import { isEqual } from "lodash";    // 可能噴錯（依套件）
✅ import lodash from "lodash";
   const { isEqual } = lodash;

// 或用 lodash-es（已轉 ESM 版本）
✅ import { isEqual } from "lodash-es";
```

  
### 7. CJS 用 ESM 套件（最痛苦）

**CJS 不能直接 `require` 一個純 ESM 套件**——這就是 `ERR_REQUIRE_ESM` 噩夢。

```text
// CJS 內：
❌ const chalk = require("chalk");  // chalk v5 已純 ESM
   → 噴 ERR_REQUIRE_ESM

// 解法 1：用 dynamic import
✅ async function main() {
     const chalk = (await import("chalk")).default;
     console.log(chalk.red("hi"));
   }

// 解法 2：固定舊版（chalk@4 還是 CJS）
✅ npm install chalk@4

// 解法 3：把自己也轉 ESM
✅ package.json 加 "type": "module"
```

> 💡 **Node 22 開始實驗性支援 `require(ESM)`**（用 flag 開啟）——未來會徹底解決這個痛點。

  
### 8. 動態 import（兩邊都通）

**`import()` 函數**——CJS / ESM 都能用：

```text
// 同步條件載入
const mod = await import("./feature.js");

// 條件載入
if (user.premium) {
  const { premiumFeature } = await import("./premium.js");
  premiumFeature();
}

// 字串拼接（runtime 決定）
const lang = "zh-TW";
const messages = await import(`./i18n/${lang}.js`);
```

**用途**：

```text
✅ Code splitting（Vite / webpack 看到 import() 自動拆 chunk）
✅ 條件載入（不一定要用 feature 就不載）
✅ Lazy load（React.lazy()）
✅ CJS 內載入純 ESM 套件
```

  
### 9. TS + Node 設定（2026 推薦）

📄 **寫進 tsconfig.json**：

```text
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",        // ← Node 友善的 ESM
    "moduleResolution": "NodeNext",
    "esModuleInterop": true,
    "allowSyntheticDefaultImports": true,
    "strict": true
  }
}
```

📄 **寫進 package.json**：

```text
{
  "type": "module",
  "scripts": {
    "dev": "tsx watch src/index.ts",
    "start": "node --enable-source-maps dist/index.js",
    "build": "tsc"
  }
}
```

**用 tsx / ts-node**（不用 build 直接跑 TS）：

🖥️ **在終端機輸入**：

```text
npm i -D tsx
npx tsx src/index.ts
```

  
### 10. 何時該用哪個？

```text
新專案？
   → ESM 一律（"type": "module"）

老專案 / library 要 dual support？
   → 用 tsup / unbuild 產出 dual package
   → 同時提供 .cjs + .mjs

寫 CLI 工具？
   → ESM（Node 20+ 沒理由用 CJS）

寫 Cloudflare Workers / Deno / Bun？
   → 純 ESM、強制

寫舊系統整合 / Electron old？
   → 可能還是 CJS（看舊套件）
```

  
### 11. 5 大常見坑

```text
❌ 1. ESM 漏副檔名
   → import "./math" 噴 ERR_MODULE_NOT_FOUND
   → 修：永遠寫 .js（即使原檔是 .ts）

❌ 2. 用 __dirname
   → ESM 內報 ReferenceError
   → 修：import.meta.dirname（Node 20.11+）

❌ 3. CJS require 純 ESM 套件
   → ERR_REQUIRE_ESM
   → 修：dynamic import() 或自己轉 ESM

❌ 4. package.json 沒寫 type
   → 預設 CJS、寫 import 報 SyntaxError
   → 修：寫 "type": "module"

❌ 5. ts-node 跟 NodeNext 設定衝突
   → 報 "Unknown file extension .ts"
   → 修：改用 tsx（不卡）或 ts-node 加 --esm
```

  
### 🎯 重點回顧

- **CJS**：require / module.exports、同步、Node 傳統
- **ESM**：import / export、非同步、瀏覽器原生、2026 主流
- **package.json `"type": "module"`** = 整個專案用 ESM
- **ESM 規定寫副檔名 + 沒 `__dirname` + 支援 top-level await**
- **`import()` 函數兩邊都通**、Code splitting 必備
- **2026 新專案一律 ESM**、舊套件用 dynamic import 救
