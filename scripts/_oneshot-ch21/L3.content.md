**2FA / MFA（多因子認證）**——除了密碼、再加一道關卡：手機收 SMS、Google Authenticator 跳數字、Face ID 解 Passkey。**TOTP / SMS / WebAuthn / Passkey** 四種主流方式、安全等級差很多。**Passkey 是 2026 取代密碼的真命天子**。

> 💡 **這課的價值**：純密碼登入 = 高風險、純密碼 + 簡訊 = 中風險、Passkey = 低風險。會這課後、你能幫公司把帳號被盜事件降到 0、又不犧牲 UX。

  
### 1. 為什麼要 2FA？

```text
情境：純密碼登入
   駭客拿到 alice 的密碼（撞庫 / 釣魚 / DB 洩漏）
   → 直接登入 alice 帳號
   → 偷錢 / 偷資料 / 操控帳號

加 2FA 後：
   駭客有密碼但沒有 alice 的手機 / Face ID
   → 卡在第二關
   → 即使密碼洩漏也保住帳號

Google 統計：開 2FA 後、帳號被盜率降 99%
```

**名詞分清楚**：

```text
2FA（Two-Factor Authentication、雙因子認證）
   → 兩個因子（密碼 + 手機）

MFA（Multi-Factor Authentication、多因子認證）
   → 兩個以上因子（密碼 + 手機 + Face ID）

2FA 是 MFA 的子集、實務上常混用
```

  
### 2. 四大 2FA 方式速覽

| 方式 | 安全 | UX | 用戶 |
|---|---|---|---|
| **SMS（簡訊）** | 低 | 中 | 銀行、傳統服務 |
| **TOTP（驗證 app）** | 中 | 中 | Google Authenticator |
| **Push（app 通知）** | 中 | 好 | Duo、Microsoft |
| **WebAuthn / Passkey** | 高 | 極好 | Apple、Google、GitHub |

  
### 3. SMS（簡訊驗證碼）

**最普及、但最弱**。

**運作**：

```text
1. 使用者輸入密碼
2. server 產 6 位數隨機碼
3. SMS 寄到使用者手機
4. 使用者輸入碼
5. server 驗證
```

**為什麼弱？**

```text
❌ SIM swap 攻擊（社工 telco 把號碼轉到駭客手機）
❌ SS7 協定漏洞（簡訊可被攔截）
❌ 釣魚（假登入頁也能收到簡訊碼）
❌ 簡訊費（每則 NT$1-3、爆量燒錢）
❌ 國際漫遊 / 收不到的問題
```

> 💡 **NIST（美國國家標準局）2017 已建議停用 SMS 2FA**——但因為使用者熟悉、還是廣泛採用。**有比沒有好**、但不是最佳。

  
### 4. TOTP（Time-based One-Time Password、時間型一次密碼）

**Google Authenticator、Authy、1Password 都是 TOTP**。

**運作原理**：

```text
註冊時
   server 產 secret（隨機字串）
   給使用者 QR code（含 secret）
   使用者用 Authenticator 掃 QR

每 30 秒
   Authenticator app = TOTP(secret, current_time / 30)
   server 同樣公式算
   → 兩邊都得出同樣 6 位數字
   → 不用網路傳輸
```

**TOTP 的優勢**：

```text
✅ 完全離線（飛機模式也能用）
✅ 沒 SIM swap 風險
✅ 不用簡訊費
✅ 業界標準（RFC 6238）

⚠️ 但仍然能被釣魚（假頁要碼）
```

**實作（Node.js + speakeasy）**：

📄 **寫進程式檔**：`auth/totp.ts`

```text
import speakeasy from "speakeasy";
import qrcode from "qrcode";

// 1. 開啟 2FA：產 secret + QR
async function enable2FA(userId: string) {
  const secret = speakeasy.generateSecret({
    name: "AI 島 (alice@example.com)",
    length: 32
  });

  await db.users.update({
    where: { id: userId },
    data: { totpSecret: secret.base32, totpEnabled: false }
  });

  // 給前端 QR code 顯示
  const qr = await qrcode.toDataURL(secret.otpauth_url!);
  return { qr, secret: secret.base32 };
}

// 2. 驗證：使用者輸入 6 位數
function verify2FA(secret: string, token: string) {
  return speakeasy.totp.verify({
    secret,
    encoding: "base32",
    token,
    window: 1  // 容許 ±30 秒誤差
  });
}

// 登入流程
app.post("/login", async (req, res) => {
  const { email, password, totp } = req.body;
  const user = await findUser(email);

  if (!await bcrypt.compare(password, user.passwordHash)) {
    return res.status(401).end();
  }

  if (user.totpEnabled) {
    if (!verify2FA(user.totpSecret, totp)) {
      return res.status(401).json({ error: "Invalid 2FA code" });
    }
  }

  req.session.userId = user.id;
  res.json({ ok: true });
});
```

**Backup codes（救援碼）必備**：

```text
使用者掉手機 = 永遠登不進來
→ 開 2FA 時、產 10 個一次性 backup code
→ 顯示一次、要求使用者印出 / 存密碼管理器
→ 每個用過一次就失效
```

  
### 5. Push Notification 2FA

**手機 app 收推播、按「批准」就行**。Duo、Microsoft Authenticator、Google Prompt。

```text
1. 使用者輸入密碼
2. server 推播到 alice 註冊的手機
3. 手機跳「有人從 Taipei 登入 alice@example.com、批准？」
4. 使用者按 ✓
5. 手機回傳簽章給 server
6. server 驗證 → 通過
```

**優點**：UX 比 TOTP 好（不用打字）。**缺點**：要 client app、不適合 web。

  
### 6. WebAuthn / FIDO2（Passkey 的底層協定）

**WebAuthn = 瀏覽器 / OS 內建的「不可釣魚 2FA / 密碼替代」API**。

**運作（簡化）**：

```text
註冊時
   server 出 challenge + 使用者資訊
   browser 叫出 authenticator（Touch ID / Face ID / 硬體 key）
   authenticator 產一對 key pair：
     - private key（留在 Secure Enclave / TPM、永不離開）
     - public key（傳回 server 存）
   要使用者「按指紋 / Face ID 確認」

登入時
   server 出新 challenge
   browser 叫 authenticator 用 private key 簽 challenge
   要使用者「按指紋 / Face ID 確認」
   server 用 public key 驗簽
```

**為什麼是 phishing-resistant（抗釣魚）？**

```text
✅ 簽章包含「網域名稱」
   authenticator 只在 myapp.com 簽 myapp.com 的 challenge
   假網站 myapp-fake.com 無法叫 authenticator 簽
   → 釣魚攻擊直接失效

✅ private key 不離開裝置
   即使中間人攔截也只看到「簽完的 challenge」、沒有 secret 可偷
```

  
### 7. Passkey（FIDO2 + 跨裝置同步）

**Passkey = WebAuthn credential 可同步到 iCloud Keychain / Google Password Manager**。

```text
傳統 WebAuthn
   credential 只存在「這台裝置」
   換手機 = 重新註冊

Passkey
   credential 同步到雲端
   iPhone 註冊 → Mac / iPad / 其他 iPhone 都能用
   Google 同帳號的 Android / Chrome 都能用

→ UX 跟「密碼」一樣方便、但比密碼安全 100 倍
```

**實作（@simplewebauthn/server）**：

📄 **寫進程式檔**：`auth/passkey.ts`

```text
import {
  generateRegistrationOptions,
  verifyRegistrationResponse,
  generateAuthenticationOptions,
  verifyAuthenticationResponse
} from "@simplewebauthn/server";

const rpName = "AI 島";
const rpID = "yourapp.com";
const origin = `https://${rpID}`;

// 註冊：產生 options
app.post("/passkey/register/start", async (req, res) => {
  const user = req.session.user;
  const options = await generateRegistrationOptions({
    rpName,
    rpID,
    userName: user.email,
    attestationType: "none",
    authenticatorSelection: {
      residentKey: "preferred",
      userVerification: "preferred"
    }
  });

  await db.users.update({
    where: { id: user.id },
    data: { passkeyChallenge: options.challenge }
  });

  res.json(options);
});

// 註冊：驗證 client 回傳的 attestation
app.post("/passkey/register/verify", async (req, res) => {
  const user = req.session.user;
  const verification = await verifyRegistrationResponse({
    response: req.body,
    expectedChallenge: user.passkeyChallenge!,
    expectedOrigin: origin,
    expectedRPID: rpID
  });

  if (verification.verified) {
    await db.passkeys.create({
      data: {
        userId: user.id,
        credentialId: verification.registrationInfo!.credentialID,
        publicKey: verification.registrationInfo!.credentialPublicKey,
        counter: verification.registrationInfo!.counter
      }
    });
  }

  res.json({ verified: verification.verified });
});
```

**前端用法（瀏覽器 API）**：

```text
// 註冊
const options = await fetch("/passkey/register/start").then(r => r.json());
const cred = await navigator.credentials.create({ publicKey: options });
await fetch("/passkey/register/verify", {
  method: "POST",
  body: JSON.stringify(cred)
});

// 登入
const options = await fetch("/passkey/login/start").then(r => r.json());
const cred = await navigator.credentials.get({ publicKey: options });
await fetch("/passkey/login/verify", { method: "POST", body: JSON.stringify(cred) });
```

  
### 8. 為什麼 Passkey 取代密碼？

```text
密碼的缺陷           Passkey 解
─────────────────────────────────
重用                 每網站獨立 key pair
弱密碼               隨機 256-bit key
釣魚                 網域綁定、自動拒簽
keylogger            按指紋 / Face ID、沒按鍵
DB 洩漏              server 只存 public key
撞庫                 沒密碼可撞
記憶負擔             不用記、Face ID 一秒
忘記密碼流程         不存在
```

```text
✅ Apple iOS 16+ / macOS Ventura+ 全平台支援
✅ Android 9+ / Chrome 全平台支援
✅ Windows 10+ / Edge 支援
✅ GitHub / Google / Microsoft / Apple / Shopify / TikTok / DocuSign
   全都已上線 Passkey 登入
```

  
### 9. 5 大常見坑

```text
❌ 1. 只給 SMS 2FA
   → SIM swap 風險、不抗釣魚
   → 修：至少加 TOTP、最好加 Passkey

❌ 2. TOTP 沒給 backup code
   → 使用者掉手機 = 永遠登不進
   → 修：產 10 個一次性 backup code

❌ 3. WebAuthn 沒驗 origin
   → 釣魚網站偽造 challenge
   → 修：expectedOrigin / expectedRPID 必驗

❌ 4. 2FA 是「選擇性」
   → 99% 使用者不會開
   → 修：高敏感 app（金融）強制開、一般 app 上 nudge

❌ 5. Backup code 存明文
   → DB 洩漏 = 2FA 形同虛設
   → 修：DB 存 hash、用過刪掉
```

  
### 🎯 重點回顧

- **2FA / MFA** = 密碼之外加第二把鎖、降低 99% 帳號被盜
- **SMS** 最弱（SIM swap、SS7）、但比沒有好
- **TOTP** 業界主流、完全離線、必配 backup code
- **WebAuthn / Passkey** 不可釣魚、UX 極好、2026 主流
- **Passkey 跨裝置同步 + 取代密碼** = 認證未來
