**密碼 hash 三大演算法**：bcrypt（普及）、scrypt（記憶體硬）、argon2（2026 推薦）——絕對不要存明文。**OAuth 2.0 + PKCE** 是 mobile / SPA 安全標配。**Magic Link** 完整實作從 token 產生到一次性失效——全部一次學會。

> 💡 **這課的價值**：密碼存錯一次 = 公司新聞頭條。OAuth 寫錯 = 帳號被盜。Magic Link 沒做好 = token 被重放攻擊。這三個做對、認證系統 80% 就穩了。

  
### 1. 密碼絕對不能存明文

```text
❌ 災難等級錯誤
   DB.users.create({ email, password: "alice123" })

   → DB 被駭 / 員工偷看 / log 印出 → 全 user 密碼洩漏
   → 90% 的人重用密碼 → 駭客拿去撞其他網站
   → 公司面臨 GDPR / CCPA 罰款 + 集體訴訟

✅ 正確：永遠 hash
   const hash = await argon2.hash(password)
   DB.users.create({ email, passwordHash: hash })
```

> 💡 **Hash vs Encrypt**：encrypt（加密）可以解開、hash（雜湊）是單向、無法還原。密碼**永遠用 hash**、不是 encrypt。

  
### 2. 三大 hash 演算法比較

| 演算法 | 年份 | 抗 GPU | 抗 ASIC | 業界使用 |
|---|---|---|---|---|
| **MD5 / SHA256** | 1990s | ❌ | ❌ | 別用 |
| **bcrypt** | 1999 | △ | △ | 還能用 |
| **scrypt** | 2009 | ✓ | △ | OK |
| **argon2** | 2015 | ✓ | ✓ | **2026 推薦** |

**為什麼 SHA256 不能拿來 hash 密碼？**

```text
SHA256 是「快」的 hash（一秒可算十億次）
→ 駭客拿 GPU 一秒撞 100 億個密碼
→ 弱密碼瞬間被破

密碼專用 hash 要「慢」+「耗記憶體」
→ argon2 一次 hash 耗 64MB + 1 秒
→ 駭客撞 1 個密碼成本暴增
```

  
### 3. bcrypt 實作（Node.js）

📄 **寫進程式檔**：`auth/password.ts`

```text
import bcrypt from "bcrypt";

// hash 密碼（註冊時）
const hash = await bcrypt.hash(password, 12);
// 12 = work factor、越高越慢、建議 10-14

// 存進 DB
await db.users.create({
  data: { email, passwordHash: hash }
});

// 驗證（登入時）
const user = await db.users.findUnique({ where: { email } });
if (!user) return null;

const valid = await bcrypt.compare(password, user.passwordHash);
if (!valid) return null;
```

**bcrypt 的特色**：

```text
✅ 自動加 salt（防 rainbow table）
✅ work factor 可調（硬體變強就調高）
✅ 跨平台、相容性好
❌ 不抗 GPU / ASIC 暴力破解
❌ 密碼最長 72 bytes（多的截掉）
```

  
### 4. argon2 實作（2026 推薦）

```text
import argon2 from "argon2";

// hash
const hash = await argon2.hash(password, {
  type: argon2.argon2id,        // 預設、最強
  memoryCost: 65536,             // 64 MB
  timeCost: 3,                   // 迭代次數
  parallelism: 4                 // 平行 thread
});

// 驗證
const valid = await argon2.verify(hash, password);
```

**argon2 的三個變體**：

```text
argon2d  → 抗 GPU、但容易 side-channel attack
argon2i  → 抗 side-channel、但弱於 GPU
argon2id → 兩者混合（推薦預設）

→ 永遠用 argon2id
```

  
### 5. OAuth 2.0 + PKCE 完整流程

**PKCE（Proof Key for Code Exchange）** 唸 "pixy"、給 mobile / SPA 用、防 authorization code 攔截。

📄 **流程圖**：

```text
1. Client 產 code_verifier（隨機 43-128 chars）
   code_verifier = "M25iVXpKU3puUjFaYWg3T1NDTDQtcW1ROUY..."

2. 算 code_challenge = base64url(SHA256(code_verifier))
   code_challenge = "E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbu..."

3. 導使用者到 OAuth provider
   GET https://accounts.google.com/o/oauth2/v2/auth?
       client_id=xxx&
       redirect_uri=https://app.com/callback&
       response_type=code&
       scope=email+profile&
       code_challenge=E9Melhoa2...&
       code_challenge_method=S256&
       state=RANDOM_CSRF_TOKEN

4. 使用者同意後、Google redirect 回
   GET /callback?code=AUTH_CODE&state=RANDOM_CSRF_TOKEN
   → server 先驗 state（防 CSRF）

5. server 用 code + code_verifier 換 token
   POST https://oauth2.googleapis.com/token
   {
     grant_type: "authorization_code",
     code: "AUTH_CODE",
     redirect_uri: "...",
     code_verifier: "M25iVXpKU3puUjFa...",  ← 注意是 verifier、不是 challenge
     client_id: "xxx"
   }

6. Google 驗證 SHA256(verifier) == challenge → 回 access_token
```

**安全要點**：

```text
✅ 永遠用 PKCE（即使是 server-side、也加）
✅ state 是 CSRF token、必驗
✅ redirect_uri 必須精確匹配
✅ access_token 用 httpOnly cookie 存（不要 localStorage）
✅ refresh_token 加密存 DB
```

  
### 6. OAuth 實作（Node.js + Express）

📄 **寫進程式檔**：`auth/oauth-google.ts`

```text
import crypto from "crypto";

// 1. 產 code_verifier + challenge
function generatePKCE() {
  const verifier = crypto.randomBytes(32).toString("base64url");
  const challenge = crypto.createHash("sha256")
    .update(verifier)
    .digest("base64url");
  return { verifier, challenge };
}

// 2. 開始登入
app.get("/auth/google", (req, res) => {
  const { verifier, challenge } = generatePKCE();
  const state = crypto.randomBytes(16).toString("hex");

  // 存 verifier + state 到 session
  req.session.oauth = { verifier, state };

  const url = new URL("https://accounts.google.com/o/oauth2/v2/auth");
  url.searchParams.set("client_id", GOOGLE_CLIENT_ID);
  url.searchParams.set("redirect_uri", `${APP_URL}/auth/google/callback`);
  url.searchParams.set("response_type", "code");
  url.searchParams.set("scope", "email profile");
  url.searchParams.set("code_challenge", challenge);
  url.searchParams.set("code_challenge_method", "S256");
  url.searchParams.set("state", state);

  res.redirect(url.toString());
});

// 3. callback 處理
app.get("/auth/google/callback", async (req, res) => {
  const { code, state } = req.query;
  const { verifier, state: savedState } = req.session.oauth;

  if (state !== savedState) {
    return res.status(400).send("CSRF detected");
  }

  // 用 code + verifier 換 token
  const tokenRes = await fetch("https://oauth2.googleapis.com/token", {
    method: "POST",
    headers: { "Content-Type": "application/x-www-form-urlencoded" },
    body: new URLSearchParams({
      grant_type: "authorization_code",
      code: code as string,
      redirect_uri: `${APP_URL}/auth/google/callback`,
      client_id: GOOGLE_CLIENT_ID,
      client_secret: GOOGLE_CLIENT_SECRET,
      code_verifier: verifier
    })
  });
  const { access_token } = await tokenRes.json();

  // 取使用者資訊
  const userRes = await fetch("https://www.googleapis.com/oauth2/v2/userinfo", {
    headers: { Authorization: `Bearer ${access_token}` }
  });
  const profile = await userRes.json();

  // 建 / 找 user、發 session
  const user = await db.users.upsert({
    where: { email: profile.email },
    create: { email: profile.email, name: profile.name },
    update: {}
  });

  req.session.userId = user.id;
  res.redirect("/");
});
```

  
### 7. Magic Link 完整實作

📄 **寫進程式檔**：`auth/magic-link.ts`

```text
import crypto from "crypto";
import { Resend } from "resend";

const resend = new Resend(process.env.RESEND_API_KEY);

// 1. 寄 magic link
app.post("/auth/magic-link", async (req, res) => {
  const { email } = req.body;

  // 產 token（256-bit 隨機）
  const token = crypto.randomBytes(32).toString("hex");
  const tokenHash = crypto.createHash("sha256").update(token).digest("hex");

  // 存 hash 而不是 token 本身（防 DB 洩漏）
  await db.magicLinks.create({
    data: {
      email,
      tokenHash,
      expiresAt: new Date(Date.now() + 10 * 60 * 1000)  // 10 分鐘
    }
  });

  // 寄信
  await resend.emails.send({
    from: "noreply@yourapp.com",
    to: email,
    subject: "登入 AI 島",
    html: `
      <p>點此登入：</p>
      <a href="${APP_URL}/auth/verify?token=${token}">登入</a>
      <p>10 分鐘內有效。</p>
    `
  });

  res.json({ sent: true });
});

// 2. 驗證 token
app.get("/auth/verify", async (req, res) => {
  const { token } = req.query;
  const tokenHash = crypto.createHash("sha256").update(token as string).digest("hex");

  const link = await db.magicLinks.findUnique({ where: { tokenHash } });

  if (!link || link.usedAt || link.expiresAt < new Date()) {
    return res.status(400).send("Link expired or used");
  }

  // 標為已用（一次性）
  await db.magicLinks.update({
    where: { id: link.id },
    data: { usedAt: new Date() }
  });

  // 建 / 找 user、發 session
  const user = await db.users.upsert({
    where: { email: link.email },
    create: { email: link.email },
    update: {}
  });

  req.session.userId = user.id;
  res.redirect("/");
});
```

**安全要點**：

```text
✅ DB 存 SHA256(token)、不是 token 本身
✅ 10 分鐘 TTL
✅ usedAt 標記、一次性
✅ Rate limit（每 email 每 5 分鐘只能寄 1 次、防濫發）
✅ 不洩漏「email 是否已註冊」（永遠回 sent: true）
```

  
### 8. 5 大常見坑

```text
❌ 1. 密碼用 SHA256 / MD5
   → GPU 一秒撞十億個
   → 修：bcrypt / argon2id

❌ 2. OAuth 沒驗 state
   → CSRF 攻擊、可挾持登入
   → 修：每次產隨機 state、callback 驗

❌ 3. Magic Link DB 存 plain token
   → DB 洩漏 = 所有人帳號被盜
   → 修：DB 存 SHA256(token)

❌ 4. Magic Link 不限發信頻率
   → 攻擊者狂寄、被 email provider 封
   → 修：rate limit 每 email 5 分鐘 1 次

❌ 5. OAuth access_token 存 localStorage
   → XSS 一句 script 偷走
   → 修：httpOnly cookie
```

  
### 🎯 重點回顧

- **密碼 hash 必用 argon2id**（次選 bcrypt cost ≥ 12）
- **OAuth + PKCE** = mobile / SPA 標配、永遠驗 state
- **Magic Link** DB 存 SHA256(token) + 10 分鐘 TTL + 一次性
- **永遠 httpOnly cookie 存 token**、不要 localStorage
- **Rate limit + audit log + email 驗證** 是三大配套
