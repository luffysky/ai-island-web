**認證（Authentication）= 「你是誰」、授權（Authorization）= 「你能做什麼」**——兩個詞長得很像、意思完全不同。**2026 業界 4 大認證方式**：密碼、OAuth、Magic Link、Passkey——每個都要懂。

> 💡 **這課的價值**：認證授權是「使用者進門」最重要的一關、做爛 = 帳號被盜、合規被罰、產品死。會這課後、你知道為什麼 Apple / Google 都在推 Passkey、為什麼純密碼 2026 已經過時。

  
### 1. Authentication vs Authorization（兩個 Auth 的差別）

**這兩個英文都簡稱 Auth、但意思完全不同**：

| 縮寫 | 全名 | 中文 | 問題 |
|---|---|---|---|
| **AuthN** | Authentication | **認證** | 你**是誰**？ |
| **AuthZ** | Authorization | **授權** | 你**能做什麼**？ |

**生活類比**：

```text
🏢 進公司大樓
   1. 警衛看你的員工證 → 認證（你真的是員工 Alice）
   2. 你只能進 1-5 樓、不能進老闆辦公室 → 授權（你的權限）
```

**程式碼類比**：

```text
authN：「Alice 用對的密碼登入」→ 取得 session / token
authZ：「Alice 的 role 是 user、不能打 /admin」→ 檢查權限

兩個都要做、缺一不可。
```

> 💡 **本課專注 AuthN（認證）**——「你是誰」這一關。AuthZ（授權）有另外的章節（RBAC / ABAC / policy）。

  
### 2. 認證的本質：證明「你是你」

```text
3 種證明方式（稱「factor」、因子）：

1. Something you know（你知道的東西）
   → 密碼、PIN、母親婚前姓氏

2. Something you have（你擁有的東西）
   → 手機、硬體 key、信用卡
   → SMS 簡訊、Authenticator app、Passkey

3. Something you are（你本人的特徵）
   → 指紋、Face ID、虹膜、聲紋

MFA（多因子認證）= 同時要求 2+ 個因子
   → 密碼 + 手機 SMS = 2FA
   → 密碼 + Face ID + 硬體 key = 3FA
```

  
### 3. 認證 4 大方式（2026 業界）

| 方式 | 安全 | UX | 業界趨勢 |
|---|---|---|---|
| **密碼（Password）** | 低 | 差 | 衰退中 |
| **OAuth / Social Login** | 中 | 好 | 普及 |
| **Magic Link** | 中 | 好 | 上升 |
| **Passkey** | 高 | 極好 | 主流 |

  
### 4. 方式 A：密碼 + 密碼雜湊

**最古老、最普及、最弱**。

**為什麼弱？**

```text
❌ 使用者重用密碼（同一個 password 用 20 個網站）
❌ 弱密碼（123456、qwerty）
❌ 容易被釣魚（fishing 假登入頁）
❌ DB 外洩 → 全部密碼被破解（如果沒 hash 好）
❌ 工程師寫錯 → 明文存進 DB
```

**做對的話必須**：

```text
✅ bcrypt / scrypt / argon2 hash（不要 SHA256 / MD5）
✅ rate limit 防爆破
✅ 強密碼規則
✅ 配 2FA（不是純密碼）
✅ breached password check（haveibeenpwned）
```

  
### 5. 方式 B：OAuth / Social Login（用 Google 登入）

**讓使用者用 Google / Apple / GitHub / Facebook 登入**。

**優點**：

```text
✅ 使用者不用記新密碼（一鍵登入）
✅ 註冊轉換率高 3-5 倍
✅ Email 已驗證
✅ 安全交給 Google（他們花十億美金搞安全）
```

**缺點**：

```text
❌ 鎖死在 OAuth provider（Google 封你帳號 = 你失去所有 user）
❌ 需要實作 OAuth flow（PKCE / state / nonce）
❌ 使用者不一定有 Google 帳號（部分地區）
```

**業界做法**：永遠提供 **2-3 個 OAuth + 一個 Email magic link** 備援。

  
### 6. 方式 C：Magic Link（魔法連結）

**輸入 email → 收信 → 點連結登入、不用密碼**。

```text
1. 使用者輸入 alice@example.com
2. server 產 token、存 DB（5 分鐘 TTL）
3. 寄信：「點此登入 https://app.com/auth?token=xxx」
4. 使用者點 link
5. server 驗 token → 建 session → 登入

完全不用密碼！
```

**優點**：

```text
✅ 不用記密碼
✅ Email 已驗證（驗證了 email 才能進）
✅ UX 好（手機開信 + 跳 app 一路順）
✅ 安全（token 短期一次性）
```

**缺點**：

```text
❌ Email 服務掛 → 全部不能登
❌ 收信慢 → UX 變差
❌ Email 帳號被盜 → 你的 app 也被盜
```

**用戶**：Slack、Medium、Notion、Vercel 都用 Magic Link。

  
### 7. 方式 D：Passkey（FIDO2 / WebAuthn、2026 主流）

**Passkey = 用裝置的生物識別（Face ID / 指紋）取代密碼**——蘋果、Google、微軟聯手推。

**運作原理**（簡化版）：

```text
註冊時
   裝置產一對 key（public + private）
   public key 上傳 server
   private key 留裝置（Secure Enclave / TPM）

登入時
   server 出 challenge（隨機字串）
   裝置用 private key 簽（要 Face ID / 指紋解鎖）
   server 用 public key 驗

→ private key 從來沒離開過裝置
→ 釣魚網站騙不到（網域不對裝置不簽）
→ 沒密碼可洩漏
```

**Passkey 的優勢**：

```text
✅ 不可釣魚（網域綁定）
✅ 不會洩漏（沒密碼存 DB）
✅ UX 最好（Face ID 一秒）
✅ 跨裝置同步（iCloud Keychain / Google）
✅ 抗 phishing、抗 keylogger、抗 DB 洩漏
```

**現況**：Apple ID、Google、Microsoft、GitHub、Shopify、TikTok 都已支援。

  
### 8. 2026 業界主流登入頁長這樣

📄 **理想登入頁**：

```text
   📱 AI 島

   [ 用 Apple 登入 ]
   [ 用 Google 登入 ]
   [ 用 GitHub 登入 ]

   ─── 或 ───

   Email: [____________]
   [ 寄送登入連結 → Magic Link ]

   ─── 或 ───

   [ 用 Passkey 登入 → 一鍵 ]


   (沒有「密碼」欄位)
   (沒有「註冊」按鈕 — 第一次登入自動建帳號)
```

**Notion / Vercel / Linear / Resend 都長這樣**。

  
### 9. 完整認證系統的 checklist

📄 **production-ready 認證必備**：

```text
✅ 多種登入方式（OAuth + Magic Link + Passkey）
✅ 強制 HTTPS
✅ Rate limit（防爆破、防 magic link 濫發）
✅ Email 驗證
✅ Session 過期 + refresh token
✅ Logout 立刻清 server session
✅ Audit log（誰何時何地登入）
✅ 設備管理（顯示「目前登入的裝置」）
✅ 異常登入通知（新裝置寄 email）
✅ 帳號刪除流程（GDPR 必備）
✅ 2FA / MFA 選項
```

  
### 10. 5 大常見坑

```text
❌ 1. 純密碼登入、無 OAuth / Magic Link
   → 2026 已過時、轉換率 -50%
   → 修：至少給 Google + Email magic link

❌ 2. 自己寫認證
   → 一定有漏洞、合規難過
   → 修：用 Clerk / Auth0 / Supabase Auth / NextAuth

❌ 3. 沒 rate limit
   → /login 被撞密碼
   → 修：5 次 / 分鐘 + captcha

❌ 4. Session 沒過期 + 不能撤銷
   → 偷一次永久有效
   → 修：TTL + logout 清 server

❌ 5. 沒 audit log
   → 帳號被盜時、不知道從哪開始查
   → 修：每次登入記 IP、UA、時間
```

  
### 🎯 重點回顧

- **AuthN（認證）= 你是誰；AuthZ（授權）= 你能做什麼**
- **3 種證明因子**：知道的、擁有的、本人是的
- **4 大方式**：密碼（衰退）、OAuth（普及）、Magic Link（上升）、Passkey（主流）
- **2026 業界做法**：OAuth + Magic Link + Passkey 三合一、不要純密碼
- **自己寫認證 = 找死**——用 Clerk / Auth0 / Supabase Auth
