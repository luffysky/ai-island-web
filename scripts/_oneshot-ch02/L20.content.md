**Design Tokens（設計令牌）** = 設計系統的最底層「**原子**」——一組命名的設計常數。所有顏色、間距、字級、圓角都有名字、有層級。**讓設計師跟工程師有共同詞彙、改主題只改 token、不改元件**。

> 💡 **這課的價值**：學會 design token = 你能造自己的 Tailwind / Material UI。**這是設計系統的基石**、所有大廠（Apple / Google / Atlassian / Shopify）都這樣管樣式。

  
### 1. 為什麼需要 design token

```text
沒 token 的痛苦：
  → 設計師說「主色」、工程師說「#3b82f6」
  → 暗黑模式要改 100 個 color: #fff
  → 改間距要找 200 處 padding: 16px
  → 改字級要看每個元件 font-size

有 token：
  → 全網站用 var(--color-primary)
  → 切暗色 = 改幾個 semantic token
  → 改間距 = 改 1 個 --space-4
  → 改字級 = 改 1 個 --text-base
```

  
### 2. 三層 Token 結構（業界標準）

**頂級設計系統都採用 3 層結構**：

```text
1. Primitive（原始值層）
   → --blue-500: #3b82f6
   → --gray-100: #f3f4f6
   → 純色 / 純數值、不直接用

2. Semantic（語意層）
   → --color-primary: var(--blue-500)
   → --color-bg: var(--gray-50)
   → 給元件用的「意義名」

3. Component（元件層）
   → --btn-padding: var(--space-4)
   → --btn-radius: var(--radius-md)
   → 特定元件的設定
```

  
### 3. 完整三層範例

📄 **寫進程式檔**：`tokens.css`

```text
:root {
  /* ===== 1. Primitive（原始值）===== */
  
  /* 顏色色階 */
  --blue-50:  #eff6ff;
  --blue-100: #dbeafe;
  --blue-500: #3b82f6;
  --blue-600: #2563eb;
  --blue-900: #1e3a8a;
  
  --gray-50:  #f9fafb;
  --gray-100: #f3f4f6;
  --gray-500: #6b7280;
  --gray-900: #111827;
  
  /* 間距 */
  --space-1: 4px;
  --space-2: 8px;
  --space-3: 12px;
  --space-4: 16px;
  --space-6: 24px;
  --space-8: 32px;
  
  /* 圓角 */
  --radius-sm: 4px;
  --radius-md: 8px;
  --radius-lg: 16px;
  --radius-full: 999px;
  
  /* ===== 2. Semantic（語意）===== */
  
  --color-primary: var(--blue-500);
  --color-primary-hover: var(--blue-600);
  --color-text: var(--gray-900);
  --color-text-muted: var(--gray-500);
  --color-bg: var(--gray-50);
  --color-border: var(--gray-100);
  
  --space-card-padding: var(--space-4);
  --space-section-gap: var(--space-8);
  
  /* ===== 3. Component（元件）===== */
  
  --btn-padding-x: var(--space-4);
  --btn-padding-y: var(--space-2);
  --btn-radius: var(--radius-md);
  --btn-font-weight: 500;
}

/* 暗色主題：只改 semantic 層、不動 primitive */
[data-theme="dark"] {
  --color-text: var(--gray-50);
  --color-text-muted: var(--gray-500);
  --color-bg: var(--gray-900);
  --color-border: var(--gray-500);
}
```

  
### 4. 對應的元件 CSS

📄 **寫進程式檔**：`components.css`

```text
.btn {
  background: var(--color-primary);
  padding: var(--btn-padding-y) var(--btn-padding-x);
  border-radius: var(--btn-radius);
  font-weight: var(--btn-font-weight);
  color: white;
  border: none;
}

.btn:hover {
  background: var(--color-primary-hover);
}

.card {
  background: var(--color-bg);
  border: 1px solid var(--color-border);
  padding: var(--space-card-padding);
  border-radius: var(--radius-lg);
}

.text-muted {
  color: var(--color-text-muted);
}
```

> 💡 **觀察**：元件 CSS 完全不寫死數值——全部用 token。**這就是設計系統的力量**。

  
### 5. 三層的好處

```text
✅ 切換主題只改 semantic 層
   → primitive 不動、component 不動
   → 100% 主題隔離

✅ 改一個常數、全網站跟著變
   → 客戶說「主色從藍改紫」→ 改 --blue-500
   → 自動傳遞到所有用 --color-primary 的地方

✅ 設計師跟工程師有共同詞彙
   → 設計師：「我要 color/primary」
   → 工程師：「var(--color-primary)」
   → 完全對齊、沒誤會

✅ 可匯出給 Figma / Sketch
   → 用 Style Dictionary / Tokens Studio
   → CSS / Figma / iOS / Android 同步

✅ 未來換設計系統、元件不用改
   → 只改 token 對應、不動元件
```

### 7. Style Dictionary（跨平台 token 工具）

**問題**：Web 用 CSS、iOS 用 Swift、Android 用 XML——同一個顏色要寫三遍。

**解法**：用 **Style Dictionary** 從一份 JSON 自動產出所有平台格式：

📄 **寫進程式檔**：`tokens.json`

```text
{
  "color": {
    "primary": {
      "value": "#3b82f6",
      "type": "color"
    },
    "danger": {
      "value": "#ef4444",
      "type": "color"
    }
  },
  "space": {
    "small": { "value": "8px" },
    "medium": { "value": "16px" }
  }
}
```

🖥️ **在終端機輸入**

```text
pnpm add -D style-dictionary
style-dictionary build
```

**自動產出**：
- Web：`tokens.css`、`tokens.js`、`tokens.scss`
- iOS：`Colors.swift`
- Android：`colors.xml`

  
### 8. Figma Variables（2023+）

**Figma 從 2023 加入「Variables」功能**——直接管理 design token：

```text
在 Figma：
  1. 開 Local Variables
  2. 建立 Color / Number / String 等變數
  3. 套用到設計（用變數名取代寫死值）
  4. 切換 mode（亮 / 暗）即時看效果

匯出：
  → 用 Tokens Studio 外掛同步到 GitHub
  → 工程師端跑 Style Dictionary
  → CSS / Native 都自動同步
```

> 🚀 **Figma Variables + Style Dictionary = 設計師工程師零誤差**——這是 2026 大廠設計團隊的標準流程。

  
### 9. 實戰：完整元件用 token

📄 **寫進程式檔**：`button.css`

```text
.btn {
  /* 顏色 */
  background: var(--color-primary);
  color: var(--color-on-primary);
  
  /* 尺寸 */
  padding: var(--btn-padding-y) var(--btn-padding-x);
  font-size: var(--text-base);
  font-weight: var(--btn-font-weight);
  
  /* 外觀 */
  border: 1px solid var(--color-primary);
  border-radius: var(--btn-radius);
  
  /* 動畫 */
  transition: var(--transition-fast);
  
  /* 互動 */
  cursor: pointer;
}

.btn:hover {
  background: var(--color-primary-hover);
}

.btn--sm {
  padding: var(--space-1) var(--space-3);
  font-size: var(--text-sm);
}

.btn--lg {
  padding: var(--space-3) var(--space-6);
  font-size: var(--text-lg);
}
```

**結果**：元件完全跟 token 解耦——換主題、換尺寸、換字級、全部改 token 不改元件。

  
### 10. token 命名規範

```text
✅ 好命名：
   --color-primary           語意清楚
   --color-primary-hover     狀態明確
   --space-card-padding      情境明確
   --text-heading-1          語意層級

❌ 差命名：
   --blue                     太抽象（哪個藍？）
   --pad                      不知道指什麼
   --c1                        無語意
   --bigfont                   不規範

業界共識（從外到內）：
  --{category}-{property}-{variant}-{state}
  例如：--color-button-primary-hover
```

  
### 5 大常見坑

```text
❌ 1. 直接用 primitive token（--blue-500）
   → 改主題要改一堆地方
   → 修：永遠用 semantic（--color-primary）

❌ 2. token 命名不一致
   → --primary / --color-primary / --primaryColor 混用
   → 修：訂規則、團隊統一

❌ 3. 把所有東西都做成 token
   → token 爆炸、找不到要哪個
   → 修：只對「會重複用 3+ 次」的值做 token

❌ 4. 主題切換改 primitive 層
   → primitive 是「不可變原料」、不該動
   → 修：暗色 / 亮色只改 semantic

❌ 5. 沒跟設計師對齊命名
   → 工程師命名 vs Figma 命名不同
   → 修：共同維護 token 文件（Notion / Storybook）
```

  
### 🎯 重點回顧

- **Design Token = 設計系統的原子**——所有設計常數命名化
- **3 層結構**：Primitive → Semantic → Component
- **切主題只改 semantic 層**——primitive / component 不動
- **8 點網格 + 1.25 比例字級**——業界共識
- **Style Dictionary**——跨 Web / iOS / Android 自動產生
- **Figma Variables（2023+）**——設計師端管理 token
- **命名規範**：`--{category}-{property}-{variant}-{state}`
