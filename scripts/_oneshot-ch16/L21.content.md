**API 認證（authentication）5 大方式**：**API Key、Basic Auth、OAuth 2.0、JWT、Session**——每種都有「適合的場景」+「不能用的場景」。**選錯方法、不是安全爛就是 UX 爛**。

> 💡 **這課的價值**：90% 的 API 安全漏洞都是「認證選錯方法」。會這課後、你知道做手機 app 該用什麼、第三方串接該用什麼、長期 session 該用什麼——選對 = 一勞永逸。

  
### 1. 認證五大方式速覽

| 方法 | 強度 | 適合場景 |
|---|---|---|
| **API Key** | 低 | 公開 API、後端對後端 |
| **Basic Auth** | 低 | 內部工具、不上 production |
| **OAuth 2.0** | 高 | 第三方授權（用 Google 登入） |
| **JWT** | 中 | API stateless 認證 |
| **Session + Cookie** | 高 | 傳統 web app |

  
### 2. API Key（API 金鑰）

**最簡單的認證**——server 發一個字串、client 每次帶上。

📄 **client 送出**：

```text
GET /api/data HTTP/1.1
X-API-Key: sk_live_abc123xyz...

或：
GET /api/data?api_key=sk_live_abc123xyz...
```

**server 驗證**：

```text
const key = req.headers["x-api-key"];
const user = await db.apiKeys.findUnique({ where: { key } });
if (!user) return res.status(401).end();
```

**優點**：

```text
✅ 超簡單實作
✅ 適合 server-to-server 串接
✅ 可以 per-key 設不同權限 / 限額
```

**缺點**：

```text
❌ 只要洩漏就完全 takeover
❌ 沒有過期機制（除非手動 rotate）
❌ 不能識別「個別使用者」（一把 key 全 app 共用）
❌ 不能放前端（JS 看得到）
```

**何時用？**——OpenAI / Stripe / SendGrid 等服務的後端對後端串接。**永遠存環境變數、永遠不放 client side**。

  
### 3. Basic Auth（基本認證）

**HTTP 內建最古老的認證**——username + password Base64 編碼。

📄 **client 送出**：

```text
GET /api/data HTTP/1.1
Authorization: Basic YWxpY2U6c2VjcmV0
                    ↑ Base64("alice:secret")
```

**Base64 不是加密！只是換編碼、任何人都能解開**：

```text
🖥️ 在終端機輸入：

echo "YWxpY2U6c2VjcmV0" | base64 -d
→ alice:secret
```

**何時用？**

```text
✅ 內部工具、staging 環境快速保護
✅ 一定要配 HTTPS（不然密碼裸奔）
❌ 永遠不要用在 production 公開服務
```

  
### 4. OAuth 2.0（開放授權）

**第三方授權的標準**——「用 Google 登入」「用 Facebook 登入」就是 OAuth。

**核心概念**：你不交出密碼、只授權「特定權限」給應用程式。

**Authorization Code Flow（最常用）**：

```text
1. 使用者按「用 Google 登入」
   → app 把使用者導到 Google
   GET https://accounts.google.com/o/oauth2/v2/auth?
       client_id=xxx&
       redirect_uri=https://myapp.com/callback&
       scope=email+profile&
       response_type=code

2. 使用者在 Google 同意授權
   → Google 把使用者導回 myapp
   GET https://myapp.com/callback?code=AUTH_CODE_HERE

3. myapp 後端用 code 換 access_token
   POST https://oauth2.googleapis.com/token
   { code, client_id, client_secret, redirect_uri }
   → { access_token, refresh_token, expires_in: 3600 }

4. myapp 用 access_token 打 Google API
   GET https://www.googleapis.com/userinfo
   Authorization: Bearer ACCESS_TOKEN
   → { email, name, picture }
```

**為什麼安全？**

```text
✅ 使用者密碼從來沒進 myapp
✅ access_token 有 scope（只能讀 email、不能改）
✅ access_token 短期有效（1 小時）
✅ refresh_token 換新 token、不用使用者重新登入
```

**PKCE（Proof Key for Code Exchange）—— mobile / SPA 必加**：

```text
傳統 OAuth 假設 client_secret 是秘密
但 mobile app / SPA 沒辦法藏 secret

PKCE 解法：
1. client 產一個 code_verifier（隨機字串）
2. 算 code_challenge = SHA256(code_verifier)
3. 第一步請求帶 code_challenge
4. 換 token 時帶 code_verifier
5. server 驗證 SHA256(verifier) == challenge
→ 即使有人攔截 code 也換不到 token
```

  
### 5. JWT（JSON Web Token）

**JWT（JSON Web Token）= 一段加密簽章的字串、自己帶身分資訊**——server 不用查 DB 就能驗證。

**結構**：`header.payload.signature`

```text
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.        ← Header（演算法）
eyJzdWIiOiIxMjMiLCJuYW1lIjoiQWxpY2UifQ.      ← Payload（資料）
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c  ← Signature（簽章）
```

**Payload 範例**：

```text
{
  "sub": "user_123",     // 使用者 ID
  "name": "Alice",
  "role": "admin",
  "iat": 1700000000,     // 簽發時間
  "exp": 1700003600      // 1 小時後過期
}
```

**用 JWT 的認證流程**：

📄 **寫進程式檔**：`auth.ts`

```text
import jwt from "jsonwebtoken";
const SECRET = process.env.JWT_SECRET!;

// 登入時簽發
function login(user) {
  const token = jwt.sign(
    { sub: user.id, name: user.name },
    SECRET,
    { expiresIn: "1h" }
  );
  return token;
}

// 之後 client 帶 token
// Authorization: Bearer eyJhbGc...

// 驗證
function verify(token: string) {
  try {
    const payload = jwt.verify(token, SECRET);
    return payload;
  } catch {
    return null;
  }
}
```

**優點**：

```text
✅ Stateless（server 不存 session、scale 容易）
✅ 自帶資料（少打 DB）
✅ 跨服務通用（A 服務簽、B 服務驗）
```

**缺點**：

```text
❌ 簽出去不能撤銷（除非加 blocklist、就破壞 stateless）
❌ 變大（payload 多就肥）
❌ 不能放敏感資料（Base64 解開就看到）
❌ 容易被誤用（XSS、localStorage）
```

> 💡 **JWT 該存哪？**——存 **httpOnly cookie**、不要存 localStorage（XSS 會偷走）。

  
### 6. Session + Cookie（傳統最強）

**server 端建 session、cookie 只帶 session_id**。

```text
1. 登入成功
   server 產 session_id = "abc123"
   server 存 { abc123 → { userId: 1, role: "admin" } } 到 Redis
   Set-Cookie: session_id=abc123; HttpOnly; Secure; SameSite=Strict

2. 之後每次 request
   Cookie: session_id=abc123
   server 用 abc123 查 Redis、拿到 user

3. 登出
   server 刪掉 Redis 的 abc123
   → 立刻失效
```

**優點**：

```text
✅ 隨時可撤銷（刪 Redis 一筆）
✅ cookie 自動帶、不用 client 寫 code
✅ httpOnly 防 XSS
✅ SameSite 防 CSRF
```

**缺點**：

```text
❌ Stateful（要 Redis / DB）
❌ 跨 domain 麻煩
❌ mobile app 不愛 cookie
```

  
### 7. 該選哪個？決策表

```text
場景：傳統 web app（同網域 SSR）
   → Session + Cookie

場景：SPA + API（同網域）
   → JWT in httpOnly cookie

場景：Mobile app（iOS / Android）
   → OAuth 2.0 + PKCE、token 存 Keychain / Keystore

場景：第三方授權「用 Google 登入」
   → OAuth 2.0

場景：純 server-to-server 串接（後端打 OpenAI）
   → API Key

場景：內部 admin 工具、staging 保護
   → Basic Auth（配 HTTPS）
```

  
### 8. 安全清單（所有方法共通）

```text
✅ 全站 HTTPS（不商量）
✅ Token / session 有過期時間
✅ refresh token 機制（讓使用者不用一直登入）
✅ Rate limiting（防爆破）
✅ Token 存 httpOnly cookie（不要 localStorage）
✅ Audit log（誰在何時做了什麼）
✅ 強制 2FA / MFA for admin
```

  
### 9. 5 大常見坑

```text
❌ 1. 把 API Key 放在前端 JS
   → 任何人 view source 就拿到
   → 修：後端 proxy、key 留 server

❌ 2. JWT 存 localStorage
   → XSS 任何 script 都能偷
   → 修：存 httpOnly cookie

❌ 3. OAuth 沒加 PKCE
   → mobile / SPA 被攔截 code 就 takeover
   → 修：永遠加 PKCE

❌ 4. Session 沒過期 / 不能撤銷
   → 偷到一次永久有效
   → 修：設 TTL + logout 清 Redis

❌ 5. 401 vs 403 不分
   → 401 = 沒帶有效認證、403 = 認證有效但沒權限
   → 修：用對 status code
```

  
### 🎯 重點回顧

- **API Key** = 後端對後端、最簡單但風險高
- **Basic Auth** = 只給內部工具用、Base64 不是加密
- **OAuth 2.0** = 第三方授權標準、mobile / SPA 必加 PKCE
- **JWT** = stateless、自帶資料、存 httpOnly cookie
- **Session + Cookie** = 傳統最強、隨時可撤銷
- **永遠 HTTPS + httpOnly cookie + 過期 + rate limiting**
