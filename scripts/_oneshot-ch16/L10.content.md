**Cache（快取）= 把「算過 / 查過」的結果暫存起來、下次直接拿、不用再算一次**——backend 速度跟 cost 一翻兩瞪眼的差別。**Redis、Memcached、CDN** 三大主流方案 + **cache-aside / write-through / write-behind** 三種策略，覆蓋 95% 的場景。

> 💡 **這課的價值**：沒 cache 的系統打 DB 1000 次 / 秒、有 cache 的系統打 DB 10 次 / 秒——同樣硬體 cost 差 100 倍、回應時間從 200ms 變 5ms。

  
### 1. 為什麼需要 cache？

```text
場景：使用者列表頁、每次重新整理都打 DB
- 1000 個使用者 / 秒打 DB
- 每次 DB query 50ms
- DB CPU 90%、隨時掛

加 cache 後：
- 第 1 次 DB（50ms）→ 結果存 Redis（1ms 寫入）
- 後面 999 次直接讀 Redis（1ms）
- DB 從 1000 QPS 變 1 QPS
- 回應從 50ms 變 1ms
```

**Cache 是 backend 第一大加速武器、沒有之一**。

  
### 2. 三大 Cache 方案對比

| 方案 | 位置 | 用途 |
|---|---|---|
| **Redis** | 獨立 server | 應用層快取、session、leaderboard |
| **Memcached** | 獨立 server | 純 KV、簡單暫存 |
| **CDN** | 全球邊緣節點 | 靜態檔案、HTML、API 回應 |

  
### 3. Redis：最強通用 cache

📄 **寫進程式檔**：`cache.ts`

```text
import { createClient } from "redis";

const redis = createClient({ url: process.env.REDIS_URL });
await redis.connect();

// 寫入（with TTL）
await redis.set("user:1", JSON.stringify(user), {
  EX: 300  // 300 秒（5 分鐘）後自動過期
});

// 讀取
const cached = await redis.get("user:1");
if (cached) {
  return JSON.parse(cached);
}

// 刪除
await redis.del("user:1");
```

**Redis 為什麼強？**

```text
✅ in-memory（記憶體）+ 持久化（disk）
✅ 不只 KV、還支援 list / set / sorted set / hash / stream
✅ Pub/Sub、Lua script、transaction、cluster
✅ 全球業界主流（Twitter、GitHub、Instagram 都用）
✅ 響應 < 1ms
```

  
### 4. Memcached：更純粹的 KV

```text
特色
- 只做 in-memory KV、沒 disk
- 簡單、極快
- 沒有 Redis 的進階資料結構
- 重啟資料全沒

什麼時候選？
- 不需要進階資料結構、只要 KV
- 已經有現成 Memcached infra
- 否則 → 直接選 Redis（功能 Memcached 完全覆蓋 + 更多）
```

  
### 5. CDN：全球邊緣 cache（最大規模武器）

**CDN（Content Delivery Network、內容傳遞網路）= 把資料快取到「離使用者最近」的伺服器**。

```text
台灣使用者打 https://api.example.com/feed

❌ 沒 CDN：
   → 跨海到美國 origin server
   → 200ms 延遲 + 30 跳路由

✅ 有 CDN（Cloudflare / CloudFront / Fastly）：
   → 打台灣邊緣節點
   → 5ms 延遲、origin 完全沒打到
```

**CDN cache 控制**：

📄 **寫進 API response**：

```text
// 公開、可被 CDN cache 60 秒
res.setHeader("Cache-Control", "public, max-age=60");

// 私有、只有瀏覽器能 cache、不能進 CDN
res.setHeader("Cache-Control", "private, max-age=60");

// 完全不 cache
res.setHeader("Cache-Control", "no-store");

// stale-while-revalidate：回舊資料 + 背景重抓
res.setHeader("Cache-Control", "public, max-age=60, stale-while-revalidate=600");
```

  
### 6. 三大 Cache 策略

#### 策略 A：Cache-Aside（旁路快取、最常見）

```text
應用程式自己管 cache：

1. 讀取時
   → 先查 cache、有就回
   → 沒有就查 DB、結果寫 cache、再回
2. 寫入時
   → 寫 DB
   → 刪除（或更新）cache
```

**程式碼**：

📄 **寫進程式檔**：`user-service.ts`

```text
async function getUser(id: string) {
  // 1. 查 cache
  const cached = await redis.get(`user:${id}`);
  if (cached) return JSON.parse(cached);

  // 2. cache miss → 查 DB
  const user = await db.users.findUnique({ where: { id } });
  if (!user) return null;

  // 3. 寫 cache（5 分鐘）
  await redis.set(`user:${id}`, JSON.stringify(user), { EX: 300 });
  return user;
}

async function updateUser(id: string, patch: any) {
  // 1. 寫 DB
  await db.users.update({ where: { id }, data: patch });
  // 2. 刪 cache
  await redis.del(`user:${id}`);
}
```

**優點**：簡單、應用程式完全掌控。**缺點**：cache 跟 DB 可能短暫不一致。**業界最常用**。

#### 策略 B：Write-Through（同步寫入）

```text
寫入時：
1. 寫 cache
2. 同時寫 DB（同步）
3. 兩個都成功才回 OK
```

**優點**：資料永遠一致。**缺點**：寫入慢一倍。適合：讀多寫少、強一致場景。

#### 策略 C：Write-Behind（背後寫入）

```text
寫入時：
1. 先寫 cache
2. cache 立即回 OK
3. 背景非同步批次寫 DB
```

**優點**：寫入超快。**缺點**：cache 掛掉資料可能丟失。適合：log、analytics、計數器。

  
### 7. TTL（Time To Live、存活時間）—— cache 過期策略

```text
固定 TTL
- SET key value EX 300
- 5 分鐘後自動消失
- 簡單、最常用

無 TTL（永久）
- SET key value
- 適合：很少變的資料（國家列表、設定）
- 缺點：要手動管刪除

短 TTL + revalidate
- max-age=60, stale-while-revalidate=600
- 1 分鐘新鮮、10 分鐘容許舊但背景更新
- Next.js / Vercel 推的模式
```

  
### 8. Cache Invalidation（cache 失效）—— 最難的問題

> 名言：「電腦科學有兩個難題：cache invalidation、命名、跟 off-by-one error。」

```text
方法 1：TTL 自動過期（最簡單）
- 容許「短時間不一致」
- 適合：列表頁、商品卡片

方法 2：手動 delete on write
- 寫 DB 時、刪 cache
- 適合：使用者資料、profile

方法 3：版本號 / 標籤
- cache key 加 version：user:1:v3
- 全部失效 → 改 version
- Next.js 用的 revalidateTag()

方法 4：Pub/Sub 廣播
- 一台 server 改了 → 廣播給所有 server 清 cache
- 大規模分散式系統
```

  
### 9. 實戰：分層 cache 架構

📄 **寫進程式檔**：`get-feed.ts`

```text
async function getFeed(userId: string) {
  // L1: in-process memory cache（毫秒級）
  const mem = memCache.get(`feed:${userId}`);
  if (mem) return mem;

  // L2: Redis（5ms）
  const redis = await redisCache.get(`feed:${userId}`);
  if (redis) {
    memCache.set(`feed:${userId}`, redis, 30);  // 回填 L1
    return redis;
  }

  // L3: DB（50ms）
  const feed = await db.posts.findMany({ ... });

  // 回填兩層 cache
  await redisCache.set(`feed:${userId}`, feed, 300);
  memCache.set(`feed:${userId}`, feed, 30);

  return feed;
}
```

**多層 cache = 速度極致**。Browser → CDN → Redis → DB、每層 hit 都省掉下一層成本。

  
### 10. 5 大常見坑

```text
❌ 1. cache 沒 TTL
   → 一直留舊資料
   → 修：永遠設 EX（最壞也設個 1 小時）

❌ 2. cache stampede（雪崩）
   → cache 同時過期、1000 個 request 同時打 DB
   → 修：jitter（隨機 TTL 偏移）+ lock

❌ 3. cache 跟 DB 不一致
   → 寫 DB 沒清 cache、使用者看舊資料
   → 修：寫入後一定 del cache

❌ 4. cache key 不規範
   → user1 / user:1 / USER_1 混用、互相 miss
   → 修：用 prefix 規範（user:{id}）

❌ 5. cache 太大 / 沒 eviction
   → Redis 爆記憶體、OOM
   → 修：設 maxmemory + allkeys-lru
```

  
### 🎯 重點回顧

- **Redis** = 應用層通用快取王者、in-memory KV + 進階資料結構
- **Memcached** = 純 KV、簡單、Redis 上位取代
- **CDN** = 全球邊緣、靜態 + API 快取、最大規模武器
- **Cache-Aside** 業界 90% 用、Write-Through 強一致、Write-Behind 寫入極速
- **TTL + invalidation** 是 cache 一輩子的難題、用版本號 / Pub/Sub 解
- **多層 cache**：memory → Redis → DB、每層省下一層 cost
