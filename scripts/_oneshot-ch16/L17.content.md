**後端架構模式 + API 設計原則**：**Monolith（單體）/ Microservices（微服務）/ Serverless（無伺服器）/ API Gateway / BFF（後端為前端）**——5 種架構各有適用場景。**RESTful 6 大原則**讓你的 API 變成業界標準、不會被同事罵。

> 💡 **這課的價值**：選錯架構 = 系統 6 個月後重寫。會這 5 種模式 + 6 條 REST 原則、你寫的 API 可以撐到公司 IPO、不用大改。

  
### 1. 五大後端架構模式

| 模式 | 一句話 | 適合 |
|---|---|---|
| **Monolith** | 整個 app 一包 | 初創、小團隊 |
| **Microservices** | 拆成 N 個小服務 | 大公司、多團隊 |
| **Serverless** | function 級部署 | 流量不穩、快速迭代 |
| **API Gateway** | 統一入口 | 微服務必備配套 |
| **BFF** | 每個前端一份 API | 多端 app（web / mobile / TV） |

  
### 2. Monolith（單體架構）

**整個 app 包成一個 process / 一個 deploy unit**——所有功能在同一個 codebase。

```text
📦 my-app/
  ├─ users/        使用者模組
  ├─ posts/        文章模組
  ├─ payments/     支付模組
  ├─ notifications/通知模組
  └─ server.ts     一個 Node process 啟動全部
```

**優點**：

```text
✅ 開發超快（不用跨服務溝通）
✅ 部署簡單（一個 docker image）
✅ debug 容易（一個 log、一個 stack trace）
✅ transaction 直接用 DB（不用 distributed transaction）
```

**缺點**：

```text
❌ 規模大後啟動慢、build 慢
❌ 一個模組炸、整個服務掛
❌ 多團隊容易 merge conflict
❌ 局部熱點要整個 scale
```

> 💡 **Shopify、GitHub、Basecamp 至今主力都是 Monolith**——不要被「微服務 = 進步」騙了。初創 99% 該用 Monolith。

  
### 3. Microservices（微服務架構）

**拆成 N 個獨立 service、每個自己的 DB / 部署 / 團隊**。

```text
🌐 入口（API Gateway）
   ↓
   ├─ User Service（自己的 DB）
   ├─ Post Service（自己的 DB）
   ├─ Payment Service（自己的 DB）
   └─ Notification Service（自己的 DB）

各服務之間用 HTTP / gRPC / Message Queue 通訊
```

**優點**：

```text
✅ 各服務獨立部署、scale、技術選擇
✅ 多團隊平行開發、不打架
✅ 單一服務掛、其他正常運作
✅ 用對技術做對事（Python 寫 ML、Go 寫高並發）
```

**缺點**：

```text
❌ 複雜度爆炸（網路 / 認證 / log 聚合 / tracing）
❌ distributed transaction 很難
❌ debug 跨 10 個服務、頭痛
❌ infrastructure cost 暴增
```

  
### 4. Serverless（無伺服器架構）

**寫 function、平台幫你跑、按用量收費**——AWS Lambda / Cloudflare Workers / Vercel Functions。

📄 **寫進程式檔**：`api/hello.ts`

```text
export default async function handler(req: Request) {
  return new Response(JSON.stringify({ msg: "hi" }), {
    headers: { "content-type": "application/json" }
  });
}
```

**優點**：

```text
✅ 0 流量、0 cost
✅ 自動 scale 到 1000 個並行
✅ 不用管 server / patch / OS
✅ 部署超快（push → 30 秒上線）
```

**缺點**：

```text
❌ Cold start（冷啟動）—— 首次呼叫 100ms-1s
❌ 執行時間限制（10s / 60s / 15min 上限）
❌ vendor lock-in
❌ 大量呼叫單價高（不像長期跑的 VM）
```

  
### 5. API Gateway（API 閘道）

**所有 client 打的「統一入口」**、轉發給後端 service。

```text
📱 mobile / 🖥️ web / 🤖 ai-agent
              ↓
       [API Gateway]
       ├─ 認證（JWT / API key）
       ├─ 速率限制（rate limiting）
       ├─ 路由（/users → User Service）
       ├─ 監控（metrics / log）
       └─ Cache
              ↓
   各個 microservice
```

**工具**：Kong / AWS API Gateway / Cloudflare / 自寫 Express。

  
### 6. BFF（Backend For Frontend、後端為前端）

**每個前端有自己一份 API**——web、iOS、Android、TV 各一份 BFF。

```text
                    ┌─ web-bff（給網頁用、富資料）
microservices ─────┤
                    ├─ ios-bff（給手機用、精簡 + GPS）
                    └─ tv-bff（給電視用、超大 image url）
```

**為什麼？**——每個 client 需要的資料形狀不一樣、寫一個通用 API 會臃腫。

  
### 7. RESTful 6 大原則

**REST（Representational State Transfer、表現層狀態轉移）**= Roy Fielding 在 2000 年提的 API 設計風格。

#### 原則 1：以「資源」為核心、不是「動作」

```text
✅ GET /users/123              拿使用者 123
✅ POST /users                  建使用者
✅ DELETE /users/123            刪使用者

❌ GET /getUser?id=123          動作不要寫在 URL
❌ POST /deleteUser/123         method 已經表達動作
```

#### 原則 2：用對的 HTTP method

```text
GET    → 讀取（safe + idempotent）
POST   → 建立（non-idempotent）
PUT    → 完整更新（idempotent）
PATCH  → 部分更新
DELETE → 刪除（idempotent）
```

> 💡 **idempotent（冪等）**= 打一次跟打 100 次結果一樣。GET / PUT / DELETE 都是 idempotent、POST 不是。

#### 原則 3：用 HTTP status code

```text
200 OK              成功
201 Created         POST 建立成功
204 No Content      DELETE 成功、無回傳
400 Bad Request     參數錯
401 Unauthorized    沒登入
403 Forbidden       登入了但沒權限
404 Not Found       找不到
409 Conflict        資料衝突
422 Unprocessable   驗證失敗
500 Internal Error  server 炸
```

#### 原則 4：URL 用名詞、複數

```text
✅ /users           ✅ /posts          ✅ /users/123/posts
❌ /user            ❌ /getUserList   ❌ /api/v1/getAllUsers
```

#### 原則 5：版本化（versioning）

```text
方法 A：URL 路徑（最常見）
   GET /v1/users
   GET /v2/users

方法 B：Header
   Accept: application/vnd.example.v1+json

方法 C：Query
   GET /users?version=1
```

#### 原則 6：分頁、過濾、排序用 query string

```text
GET /posts?page=2&limit=20
GET /posts?status=published&sort=-created_at
GET /users?name=alice&age_gte=18

回 response 帶分頁資訊：
{
  "data": [...],
  "page": 2,
  "total": 1500,
  "next": "/posts?page=3"
}
```

  
### 8. 實戰：典型 RESTful CRUD

📄 **寫進程式檔**：`routes/posts.ts`

```text
// 列表 + 分頁 + 篩選
GET    /posts?page=1&status=published

// 取單一
GET    /posts/123

// 建立
POST   /posts
Body: { "title": "...", "content": "..." }
→ 201 Created + 新 post

// 完整更新（PUT）
PUT    /posts/123
Body: { "title": "...", "content": "..." }
→ 200 OK + 新 post

// 部分更新（PATCH）
PATCH  /posts/123
Body: { "title": "新標題" }
→ 200 OK + 新 post

// 刪除
DELETE /posts/123
→ 204 No Content

// 子資源
GET    /posts/123/comments
POST   /posts/123/comments
```

  
### 9. 該選哪個架構？決策樹

```text
你是初創 / MVP / 個人 side project？
   → Monolith（不用懷疑）

流量爆增、想極致省 cost？
   → Serverless

多團隊、複雜域、長期演進？
   → Microservices + API Gateway

多端 app、各自需求不同？
   → 加 BFF
```

  
### 10. 5 大常見坑

```text
❌ 1. 初創就上 Microservices
   → 1 個人切 5 個服務、debug 地獄
   → 修：先 Monolith、撐不住再拆

❌ 2. URL 動詞化
   → /getUser、/createPost
   → 修：用名詞 + HTTP method 表達

❌ 3. 全部回 200
   → 錯誤也回 200 + {ok: false}
   → 修：用對 status code（4xx / 5xx）

❌ 4. 不分頁 / 不限流
   → /users 一次回 10 萬筆、server 爆
   → 修：永遠加 limit 預設 20、max 100

❌ 5. 沒 versioning
   → 改 API 把所有 client 弄壞
   → 修：第一天就 /v1
```

  
### 🎯 重點回顧

- **Monolith** 初創必備、別被微服務騙
- **Microservices** 大團隊大規模才划算、複雜度爆炸
- **Serverless** 流量不穩 + 0 cost 啟動的神器
- **API Gateway** 微服務必備、統一入口
- **BFF** 多端 app 各自客製
- **RESTful 6 原則**：資源 + method + status + 名詞複數 + 版本 + 分頁
