**Rate Limiting（速率限制）= 限制使用者 / IP 在固定時間內能打 API 幾次**——防爆破、防 DDoS、保護 server 不被打死。**4 大演算法**：Fixed window、Sliding window、Token bucket、Leaky bucket——每個適合不同情境。

> 💡 **這課的價值**：沒有 rate limit 的 API、上線第 3 天就會被人 brute force 撞密碼、或被 bot 爬到 DB 掛掉。會這課後、你能用 Redis 寫出抗千萬 QPS 的 rate limit。

  
### 1. 為什麼必須 rate limit？

```text
❌ 不限制的後果
   → 駭客每秒打 1000 次 /login 撞密碼（brute force）
   → 競爭對手爬光你的價格資料
   → bot 註冊百萬假帳號
   → 一個惡意 user 把 server CPU 打到 100%
   → AI API key 一夜被燒爆 $10,000

✅ 加 rate limit 後
   → 每個 IP / user 每分鐘上限 60 次
   → 超過回 429 Too Many Requests
   → server 安全、攻擊成本爆增
```

  
### 2. 四大演算法速覽

| 演算法 | 一句話 | 適合 |
|---|---|---|
| **Fixed Window** | 每固定時間窗計次 | 簡單統計 |
| **Sliding Window** | 滑動時間窗計次 | 防邊界突破 |
| **Token Bucket** | 桶子裝 token、消費掉 | 容許突發、平均限流 |
| **Leaky Bucket** | 桶子漏水、固定速率 | 嚴格平滑流量 |

  
### 3. Fixed Window（固定窗口、最簡單）

**每 1 分鐘為一個窗口、計次 60 上限**。

```text
時間軸（每格 1 秒）
00:00 ┃────────────────┃ 60 秒 ┃
       count = 0 → 60 上限
       超過 60 → 拒絕

00:01 ┃────────────────┃ 新窗口
       count 重設為 0
```

**Redis 實作**：

📄 **寫進程式檔**：`rate-limit-fixed.ts`

```text
async function isAllowed(userId: string) {
  const now = Math.floor(Date.now() / 1000);
  const window = Math.floor(now / 60);  // 每 60 秒一個窗
  const key = `rl:${userId}:${window}`;

  const count = await redis.incr(key);
  if (count === 1) {
    await redis.expire(key, 60);  // 60 秒後自動清
  }

  return count <= 60;  // 60 次上限
}
```

**優點**：超簡單、Redis 一個 INCR 解決。

**缺點 — Boundary problem（邊界突破）**：

```text
00:59 打了 60 次（合法）
01:00 又打 60 次（合法）
→ 1 秒內打了 120 次、實際突破限制
```

  
### 4. Sliding Window（滑動窗口、修補 Fixed）

**不用整點切窗、而是「過去 60 秒」滑動計算**。

```text
現在時間 T、計算 T-60s ~ T 內的 request 數
記下每次 request 的時間戳、過期的丟掉

📦 sorted set: [1700000023, 1700000045, 1700000058, ...]
查詢 → 刪掉 < (now - 60) 的 → 算剩幾個
```

**Redis 實作**：

```text
async function isAllowed(userId: string) {
  const now = Date.now();
  const key = `rl:${userId}`;
  const windowStart = now - 60000;  // 60 秒前

  // 1. 刪掉過期的
  await redis.zRemRangeByScore(key, 0, windowStart);

  // 2. 加入這次的時間戳
  await redis.zAdd(key, { score: now, value: `${now}` });

  // 3. 算還有幾筆
  const count = await redis.zCard(key);

  // 4. 設過期（懶人清理）
  await redis.expire(key, 61);

  return count <= 60;
}
```

**優點**：完全沒有邊界問題、精確。**缺點**：每個 user 存一堆時間戳、佔記憶體。

  
### 5. Token Bucket（令牌桶、容許突發）

**想像有個桶子、固定速率裝 token、每次 request 消費 1 個**。

```text
🪣 桶子容量 = 100 token
   裝填速率 = 每秒 10 token

使用者打 API
   → 桶內有 token → 拿走 1 個 → 允許
   → 桶內沒 token → 拒絕

桶子滿了不再加（上限 100）
```

**特色**：

```text
✅ 容許「突發」—— 一次打 100 次（用光 token）後限制每秒 10 次
✅ 真實的 API 流量很多是這樣（使用者偶爾密集操作）
✅ Stripe / GitHub / AWS API 都用 Token Bucket
```

**簡化實作**：

```text
async function isAllowed(userId: string) {
  const key = `tb:${userId}`;
  const now = Date.now();
  const refillRate = 10;       // 每秒補 10 個
  const capacity = 100;        // 桶上限 100

  // 用 Lua script atomic 操作（避免 race condition）
  const lua = `
    local last = tonumber(redis.call("HGET", KEYS[1], "last") or 0)
    local tokens = tonumber(redis.call("HGET", KEYS[1], "tokens") or ARGV[1])
    local now = tonumber(ARGV[2])
    local elapsed = (now - last) / 1000
    tokens = math.min(ARGV[1], tokens + elapsed * ARGV[3])

    if tokens >= 1 then
      tokens = tokens - 1
      redis.call("HSET", KEYS[1], "last", now, "tokens", tokens)
      redis.call("EXPIRE", KEYS[1], 60)
      return 1
    else
      return 0
    end
  `;

  return await redis.eval(lua, {
    keys: [key],
    arguments: [`${capacity}`, `${now}`, `${refillRate}`]
  });
}
```

  
### 6. Leaky Bucket（漏水桶、嚴格平滑）

**桶子有個漏洞、固定速率漏水**——request 進來填水、桶滿就拒絕。

```text
🪣 桶子容量 = 100
   漏水速率 = 每秒 10 個

request 進來 → 加水
桶子每秒漏 10 → 處理 10 個 request
桶子滿 → 後續拒絕（不能突發）

VS Token Bucket：
- Token Bucket 容許瞬間爆量（只要有 token）
- Leaky Bucket 完全平滑（最多每秒 10 個出去）
```

**適合**：對「後端負載」要求嚴格、不能瞬間打爆（payment / SMS gateway）。

  
### 7. 實戰：Express middleware（Token Bucket）

📄 **寫進程式檔**：`middleware/rate-limit.ts`

```text
import { Request, Response, NextFunction } from "express";
import { redis } from "../lib/redis";

export function rateLimit(opts: {
  capacity: number;
  refillRate: number;  // per second
  keyBy: (req: Request) => string;
}) {
  return async (req: Request, res: Response, next: NextFunction) => {
    const key = `rl:${opts.keyBy(req)}`;
    const now = Date.now();

    const result = await redis.eval(LUA_SCRIPT, {
      keys: [key],
      arguments: [`${opts.capacity}`, `${now}`, `${opts.refillRate}`]
    });

    if (!result) {
      res.setHeader("Retry-After", "10");
      return res.status(429).json({
        error: "Too Many Requests",
        retryAfter: 10
      });
    }

    next();
  };
}

// 使用
app.use("/api/login",
  rateLimit({
    capacity: 5,
    refillRate: 0.1,   // 10 秒補 1 個 → 一分鐘 6 次
    keyBy: (req) => req.ip!
  })
);
```

  
### 8. 多層 rate limit（業界做法）

```text
🌐 CDN 層（Cloudflare）
   → 全球 IP 流量、bot 偵測
   → 每 IP 每分鐘 1000 次

🛡️ API Gateway 層
   → 每使用者每分鐘 100 次
   → 不同 endpoint 不同上限

🔧 應用層（critical endpoint）
   → /login 每 IP 每分鐘 5 次（防爆破）
   → /signup 每 IP 每天 3 次（防 bot）
   → /api/ai 每使用者每小時 100 次（保護 AI key）
```

  
### 9. 加 retry hint header（UX 友善）

```text
HTTP/1.1 429 Too Many Requests
X-RateLimit-Limit: 60
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1700000060
Retry-After: 30

→ Client 看到 429、知道 30 秒後重試
→ 不要瘋狂重試打爆 server
```

  
### 10. 5 大常見坑

```text
❌ 1. 用 in-memory 計數
   → server 重啟全清、多 instance 不同步
   → 修：用 Redis 共享狀態

❌ 2. keyBy 用 IP 但 client 在 proxy 後
   → 全公司同一 IP、互相 block
   → 修：用 X-Forwarded-For + cookie / user id

❌ 3. Fixed Window 邊界問題
   → 整點瞬間翻倍流量
   → 修：用 Sliding Window 或 Token Bucket

❌ 4. 沒回 Retry-After
   → client 瘋狂重試、放大攻擊
   → 修：永遠回 Retry-After + X-RateLimit-Reset

❌ 5. 對 admin 也限流
   → 老闆打 dashboard 被擋
   → 修：admin / internal 走 bypass list
```

  
### 🎯 重點回顧

- **Fixed Window** 最簡單、有邊界問題
- **Sliding Window** 精確、佔記憶體
- **Token Bucket** 容許突發、業界主流
- **Leaky Bucket** 完全平滑、嚴格場景
- **永遠用 Redis** + Lua script atomic
- **多層 rate limit**：CDN → Gateway → 應用層
- **回 429 + Retry-After** 是 UX 必備
