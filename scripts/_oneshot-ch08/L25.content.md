**React Performance 三大 Hook**：`React.memo`、`useMemo`、`useCallback`。**不是「永遠包」**——濫用反而拖慢、增加心智負擔。學會「何時該用、何時不用」才是真進階。

> 💡 **這課的價值**：80% 的人把 memo 包到處用、結果效能沒變甚至更差。**先 measure 再 optimize**——這課教你 measure 工具 + 真正該優化的時機。

  
### 1. 三大 Hook 一覽

```text
🧬 React.memo（元件層）
   作用：父 component 重 render、props 沒變 → 子不重 render
   時機：子元件 render 很貴、props 穩定

📦 useMemo（值層）
   作用：「快取一個計算結果」、依賴沒變不重算
   時機：計算很貴（>16ms）、結果穩定

🪝 useCallback（函式層）
   作用：「快取一個函式」、依賴沒變回同一個 reference
   時機：要傳函式給 memo 過的子元件、或當 useEffect 依賴
```

  
### 2. `React.memo` 詳解

```text
// 父 component 重 render、但 props 沒變 → 子不重 render
const MemoizedChild = memo(function Child({ name }: { name: string }) {
  console.log("render Child");
  return <div>{name}</div>;
});

function Parent() {
  const [count, setCount] = useState(0);
  return (
    <>
      <button onClick={() => setCount(c => c + 1)}>{count}</button>
      <MemoizedChild name="Luffy" />  {/* 不會重 render */}
    </>
  );
}
```

**沒 memo 時**：Parent state 變、Child 也跟著重 render。
**有 memo 時**：name 沒變、Child 跳過。

**何時 memo 沒用甚至更慢：**

```text
❌ 1. props 每次都是新的 reference
   <Child data={[1,2,3]} />  ← 每次 render 都是新陣列
   <Child onClick={() => fn()} />  ← 每次新 function
   → memo 永遠不命中、白浪費比較成本

❌ 2. children 是 JSX
   <Child>{<span>...</span>}</Child>  ← 每次新 JSX object
   → memo 不命中

❌ 3. Child 本來就很簡單（<div>{name}</div>）
   → render 本來就 < 0.1ms、memo 反而慢

✅ 1. 子元件包含一堆 hooks / 計算 / 子樹
✅ 2. props 是穩定值（string / number / primitive）
✅ 3. 父常 re-render、子內容真的不變
```

  
### 3. `useMemo` 詳解

```text
import { useMemo, useState } from "react";

function Search({ items }: { items: string[] }) {
  const [query, setQuery] = useState("");

  // 每次 render 都 filter、items 1 萬筆會卡
  // const filtered = items.filter(item => item.includes(query));

  // ✅ 用 useMemo 快取、query / items 沒變就不重算
  const filtered = useMemo(
    () => items.filter(item => item.includes(query)),
    [items, query],
  );

  return (
    <>
      <input value={query} onChange={e => setQuery(e.target.value)} />
      <ul>{filtered.map(i => <li key={i}>{i}</li>)}</ul>
    </>
  );
}
```

**useMemo 什麼時候真的有用：**

```text
✅ 計算 > 16ms（你已經 profile 證實了）
✅ 結果要傳給 memo 過的子元件當 prop
✅ 結果當 useEffect 依賴（避免 effect 一直跑）

❌ 計算 < 5ms（直接算更快）
❌ 結果只在 JSX 內用一次（沒影響）
❌ 依賴每次都變（memo 永遠不命中）
```

> ⚠️ **useMemo 本身有成本**：要儲存上次結果 + 比較依賴。簡單計算用 useMemo 反而慢。

  
### 4. `useCallback` 詳解

```text
// ❌ 沒 useCallback
function Parent() {
  const [count, setCount] = useState(0);
  const handleClick = () => console.log('clicked');  // 每次新 function
  return <MemoizedChild onClick={handleClick} />;
}

→ handleClick 每次新 reference、memo 不命中、Child 還是重 render

// ✅ 用 useCallback
function Parent() {
  const [count, setCount] = useState(0);
  const handleClick = useCallback(() => console.log('clicked'), []);
  return <MemoizedChild onClick={handleClick} />;
}

→ handleClick reference 穩定、memo 命中、Child 不重 render
```

**useCallback 90% 用在「配合 memo」**——單獨用通常沒效果。

  
### 5. 黃金法則：先 measure 再 optimize

**步驟 1：用 React DevTools Profiler**

```text
1. 裝 React DevTools Chrome 擴充
2. F12 → Profiler tab
3. 按 ⏺️ 錄製、操作網頁、按 ⏹️ 停止
4. 看：
   - 哪個 component render 最久（橙紅色）
   - 為什麼 render（hover 看 "Why did this render?"）
   - render 次數異常多嗎
```

**步驟 2：判斷該不該優化**

```text
🟢 < 16ms：60fps 順、不用優化
🟡 16-50ms：可優化、但其他事更重要先做
🔴 > 50ms：必須優化、使用者已感受到卡
```

**步驟 3：用對工具**

```text
慢的原因               解法
─────────────────────────────────
計算重複跑              useMemo
function 每次新建        useCallback
component 沒變還 re-render  React.memo
list 重排               key={item.id}（不要用 index）
list 太長               virtualization（react-window / TanStack Virtual）
資料抓太多              分頁 / infinite scroll
bundle 太肥             code split / lazy import
首屏 LCP 慢             Server Component / Suspense / streaming
```

  
### 6. 真實案例對比

**案例：1 萬筆資料的搜尋表單**

❌ **沒優化版**：

```text
function Search() {
  const [items] = useState(generate10000Items());
  const [query, setQuery] = useState("");
  const filtered = items.filter(i => i.name.includes(query));  // 每次 filter
  return (
    <>
      <input value={query} onChange={e => setQuery(e.target.value)} />
      {filtered.map(i => <Item key={i.id} item={i} onClick={() => console.log(i)} />)}
    </>
  );
}

→ 每打一個字、filter 1 萬筆 + render 1 萬個 Item
→ 卡 200-500ms
```

✅ **優化版**：

```text
const Item = memo(function Item({ item, onClick }) {
  return <div onClick={onClick}>{item.name}</div>;
});

function Search() {
  const [items] = useState(generate10000Items());
  const [query, setQuery] = useState("");
  
  const filtered = useMemo(
    () => items.filter(i => i.name.includes(query)),
    [items, query]
  );

  const handleClick = useCallback((item) => console.log(item), []);

  return (
    <>
      <input value={query} onChange={e => setQuery(e.target.value)} />
      {filtered.map(i => <Item key={i.id} item={i} onClick={() => handleClick(i)} />)}
    </>
  );
}

→ filter cache、Item memo、handleClick 穩定
→ 但 1 萬個 Item 還是太多、virtual list 才是真解
```

✅✅ **真正解 — virtual list（只 render 可見部分）**：

```text
import { useVirtualizer } from "@tanstack/react-virtual";

function Search() {
  const parentRef = useRef(null);
  const [items] = useState(generate10000Items());
  const [query, setQuery] = useState("");
  const filtered = useMemo(() => items.filter(i => i.name.includes(query)), [items, query]);

  const virtualizer = useVirtualizer({
    count: filtered.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 40,
  });

  return (
    <>
      <input value={query} onChange={e => setQuery(e.target.value)} />
      <div ref={parentRef} style={{ height: 400, overflow: 'auto' }}>
        <div style={{ height: virtualizer.getTotalSize(), position: 'relative' }}>
          {virtualizer.getVirtualItems().map(v => (
            <div key={v.key} style={{ position: 'absolute', top: v.start, height: v.size }}>
              {filtered[v.index].name}
            </div>
          ))}
        </div>
      </div>
    </>
  );
}

→ 只 render ~10 個可見 item、不論總筆數
→ 流暢 60fps
```

### 8. 5 大常見坑

```text
❌ 1. 「先 memo 再說」
   → 增加心智成本、可能更慢
   → 修：先 measure、確定慢才優化

❌ 2. useMemo 包基本值
   const a = useMemo(() => 1 + 2, []);  ← 寫了個寂寞
   → 修：只用於 expensive 計算

❌ 3. useCallback 但子元件沒 memo
   → useCallback reference 穩定、但子還是會 render（因為它沒 memo）
   → 修：兩個要一起用

❌ 4. memo 但 props 是 inline object/array
   <Child data={[1,2,3]} />
   → memo 永遠不命中
   → 修：把陣列 useMemo 出來

❌ 5. key={index} 在動態列表
   → 重排 / 刪除時 state 錯位
   → 修：用 stable id（item.id）
```

  
### 9. Performance checklist

```text
🔍 1. 開 React DevTools Profiler、看哪裡慢
🎯 2. > 50ms 才優化、< 16ms 不用碰
🧬 3. 重 component 包 memo + props 是 primitive
📦 4. expensive 計算用 useMemo（>16ms）
🪝 5. memo + 傳 function 配 useCallback
🔑 6. list 永遠用 stable key
📜 7. 列表 > 100 用 virtualization
✂️ 8. 大 bundle 用 React.lazy + Suspense
🏃 9. 首屏慢 → 用 Server Components
🤖 10. 試 React Compiler（auto memo）
```
