## 0. 任務背景：真實使用者回饋

Ch26《Python 基礎》（32 個 lesson）收到真實使用者回饋，指出三個具體問題。這些回饋極有價值——它們指出教材「預設讀者已有基礎」，對「完全零基礎」的人形成門檻。

**使用者回饋原話（重點）：**

1. 「我如果不知道 `input` 是使用者會輸入東西的話，我就不知道它幹嘛要有 Alice 寫在後面……完全沒有程式底子的會有很多說明搞不清楚。」
2. 「像 `$ uv run hello.py` 這一行，我不會知道那是描述程式在跑的句子，我也不知道哪些要打在終端、哪些要寫在程式上。」
3. 「它常常直接講程式的英文名稱，但不說中文意思是什麼，讀一讀就會被英文單字卡到。」（例：`docstring`、`comment`、`type hint`、`type checker`、`REPL`）

**問題的共同病根：** 教材的「預設讀者」是「已有一點概念」的人。它假設讀者知道什麼是終端機、知道 comment 就是「註解」、知道 type hint 大概是什麼。對「連門都還沒入」的人，這些假設每一個都是一道牆。

**本任務目標：** 在「不犧牲原有內容深度」的前提下，補上「連門都沒入的人」需要的那一層——讓 Ch26 對純零基礎者也能讀得下去。

## 0.5 最高原則（凌駕一切的第一鐵則）

> **預設讀者，是「從來沒接觸過任何程式碼的人」。**

改寫 Ch26 的每一行、每一段、每一個範例時，都要以這個假設為前提。不是「假設讀者大概懂一點」，是「假設讀者**完全沒寫過程式、沒用過終端機、不知道任何術語**」。

**改寫時的自我檢查（每段都要問）：**

> 「一個完全沒碰過程式的人，讀到這一行，會不會卡住？會不會出現一個他不認識、又沒人解釋的詞？」

## 3. 三條改寫規則（核心）

### 規則一：英文術語「英中對照 + 白話解釋」

**問題：** 教材直接寫英文術語，不給中文、不給解釋。零基礎者一卡住，整段就讀不下去。

**規則：** 每個英文術語**第一次在某個 lesson 出現時**，必須寫成「**英文 +（中文）+ 一句白話說明**」。同一 lesson 內再次出現可只用英文。

**範例：**

- ❌ 原本：「或當 docstring（function / class 內第一個）」
- ✅ 改成：「或當 **docstring（文件字串）**—— 寫在函式或類別開頭的一段說明文字，用來講『這段程式是做什麼用的』，給人看的」

- ❌ 原本：「但 2024+ 主流加 type hint（不強制、但好看）」
- ✅ 改成：「但 2024 年後主流會加 **type hint（型別提示）**—— 在變數或參數旁邊標註『這裡應該放什麼型別的資料』，像貼一張標籤；不強制寫，但寫了更清楚」

- ❌ 原本：「仍能跑、但 type checker 會抗議」
- ✅ 改成：「仍能跑，但 **type checker（型別檢查器）**—— 一種會幫你檢查『型別有沒有用錯』的工具 —— 會對你提出警告」

- ❌ 原本：「REPL：互動式 Python」
- ✅ 改成：「**REPL（讀取-執行-印出 循環，Read-Eval-Print Loop）**—— 一個『你打一行、它立刻給你結果』的互動視窗」

**執行方式：** 嶼築依「第 5 節術語對照表」進行。對照表已列出 Ch26 全章的術語與其「中文 + 白話」。逐 lesson 檢查：該 lesson 第一次出現的術語，套上對照表的寫法。

**重要——術語「無所不在」都要處理，不只正文：** 會卡住讀者的英文術語，不只出現在課文正文。它們常藏在角落：**程式碼框裡的註解、功能條列、表格、提示框**。這些地方一律比照處理。範例：

- ❌ 原本（藏在程式碼註解裡）：`# 不用 temp 變數！`
- ✅ 改成：`# 不用 temp（暫存用的）變數！`

- ❌ 原本（藏在功能條列裡）：「`!command`（執行 shell）」
- ✅ 改成：「`!command`（執行 shell —— shell 就是作業系統的命令列、可以下系統指令）」

- ❌ 原本（藏在條列裡）：「`?function`（看說明）」
- ✅ 改成：「`?function`（在 ipython 裡，於某個函式名稱前打問號，會顯示它的使用說明）」

只要是「一個沒寫過程式的人會卡住的詞」，不管它出現在課文、註解、列表還是表格，都要補中文與解釋。

**注意：** 不是每個英文字都要翻譯。變數名（`name`、`age`、`user`）、被當例子的東西不用。要處理的是「**程式/技術的專有名詞、概念詞**」——也就是「不解釋就會卡住理解」的那種詞。對照表已篩選好。判斷標準回到第 0.5 節最高原則：「完全沒碰過程式的人，看到這個詞會不會卡住？」

### 規則二：四種區塊「視覺分明」

**問題：** 程式碼、終端機指令、使用者輸入、程式輸出——四種完全不同的東西，目前全用同一種灰底程式碼框、長得一模一樣。零基礎者無法分辨「哪些要寫進檔案、哪些要打在終端機、哪些是電腦印出來的」。

**規則：** 這四種東西，在教材裡必須**標示清楚、一眼能分辨**。每一段這類內容前面，加一個「標籤行」說明它是什麼：

| 類型 | 標籤 | 意思 |
|---|---|---|
| 程式碼 | `📄 寫進程式檔（例：hello.py）` | 這段是要打字寫進 .py 檔案的程式 |
| 終端機指令 | `🖥️ 在終端機輸入` | 這行是要打在終端機/命令列的指令 |
| 使用者輸入 | `⌨️ 這時換你打字（然後按 Enter）` | 程式執行中，這是「你要打的」 |
| 程式輸出 | `💬 電腦會顯示` | 這是電腦印出來給你看的結果 |

**範例改寫——以 L2 的 `input()` 段落示範：**

❌ 原本（四種東西黏在一個框、無法分辨）：
```
# 問使用者
name = input("你叫什麼名字？ ")
print(f"嗨 {name}！")

$ uv run hello.py
你叫什麼名字？ Alice
嗨 Alice！
```

✅ 改成（標示清楚 + 加說明）：

```
📄 寫進程式檔 hello.py：

    name = input("你叫什麼名字？ ")
    print(f"嗨 {name}！")

說明：input() 的作用是「向使用者問問題、等他打字」。
括號裡的 "你叫什麼名字？ " 是程式要問的問題，會顯示在螢幕上。
使用者打的字，會被存進 name 這個變數裡。

🖥️ 在終端機輸入（執行這個程式）：

    uv run hello.py

💬 電腦會顯示（這行是程式問的問題）：

    你叫什麼名字？

⌨️ 這時換你打字，例如打 Alice 然後按 Enter：

    Alice

💬 電腦接著顯示（程式用你打的名字回應你）：

    嗨 Alice！
```

**關鍵差別：** 同樣的範例程式，加了「這段是寫進檔案的 / 這行是打在終端機的 / 這是你打的 / 這是電腦印的」標示與說明後，零基礎者就能跟上「先寫檔案 → 再去終端機執行 → 過程中我要打字 → 電腦回我」這個完整流程。

**關於終端機指令的 `$` 符號：** 教材裡終端機指令常寫成 `$ uv run hello.py`。那個 `$` 是「終端機提示符號」的慣例寫法，**不是要使用者打的字**。改寫時，在「環境預備課」要說明這個 `$` 是什麼；在各 lesson 的終端指令，建議**直接拿掉 `$`**、改用 `🖥️ 在終端機輸入：` 標籤 + 不含 `$` 的指令（如上例），避免零基礎者把 `$` 也打進去。REPL 的 `>>>` 同理——要說明那是 Python 互動視窗的提示符號，不是使用者要打的。

### 規則三：最前面加「環境預備課」

**問題：** Ch26 第一課 L1 直接進入「為什麼是 Python、安裝 uv」。但「連門都沒入」的人，在這之前還缺一層更基礎的概念：什麼是終端機？什麼是程式檔？「寫程式」和「執行程式」是不是同一件事？

**規則：** 在 Ch26 最前面，新增一個 lesson（建議命名 **L0：開始之前 —— 你需要知道的 5 件事**，或作為新的 L1、原 L1 順延）。這一課**不教 Python 語法**，只補「入門前的門」。

**這一課必須涵蓋（用最白話、大量比喻）：**

1. **什麼是「終端機」（Terminal / 命令列）。**
   - 它長什麼樣子（一個可以打字下指令的黑色/白色視窗）。
   - 怎麼打開它（Windows / Mac 各怎麼開）。
   - 比喻：終端機像「跟電腦直接對話的對講機」——你打一句指令，它做一件事。
   - 說明提示符號 `$`（或 Windows 的 `>`）：那是終端機「準備好了、等你打字」的記號，**不是你要打的字**。

2. **什麼是「程式檔」。**
   - 程式是寫在一個檔案裡的（例如 `hello.py`，`.py` 代表這是 Python 檔）。
   - 寫程式 = 用編輯器把程式碼打進這個檔案。
   - 比喻：程式檔像「食譜」——你把步驟寫下來，存成一張食譜。

3. **「寫程式」和「執行程式」是兩個不同動作。**
   - 寫：把程式碼打進 .py 檔（食譜寫好）。
   - 執行：去終端機打指令、叫電腦「照這個檔案跑一次」（照食譜做菜）。
   - 這是零基礎者最容易混淆的點，要講清楚、用比喻講透。

4. **這套教材的「四種區塊標示」怎麼看。**
   - 直接把規則二的四個標籤（📄🖥️⌨️💬）介紹給學員：「之後你會一直看到這四種標示，先認識它們」。
   - 這一課等於是「閱讀教材的使用說明」。

5. **遇到看不懂的英文術語怎麼辦。**
   - 告訴學員：教材會在術語第一次出現時給中文和解釋；看到不懂的詞先別慌、別跳過，那是正常的。
   - 鼓勵：用綠寶 AI 導師問「這個詞是什麼意思」。

**這一課的精神：** 它是「Ch26 的使用說明書 + 最低限度的環境概念」。讓一個完全沒碰過程式的人，讀完這課，有最基本的地圖感，知道接下來在做什麼。

## 5. Ch26 術語對照表

嶼築依此表執行「規則一」。表中每個術語給「中文 + 一句白話」。術語**第一次在某 lesson 出現**時，套用此寫法。

> 此表涵蓋 Ch26 主要技術術語。若改寫中發現表上沒有、但同樣會「卡住零基礎者」的術語，嶼築應比照格式補充，並於回交時列出新增項。

**基礎概念**
- `Terminal / 終端機`：跟電腦直接下指令對話的視窗
- `shell`：作業系統的命令列環境，可以下系統指令（終端機裡跑的就是 shell）
- `REPL`：讀取-執行-印出循環——打一行、立刻看結果的互動視窗
- `temp（暫存）`：臨時用一下、用完就不要的東西（如「暫存變數」）
- `comment（註解）`：寫在程式裡給人看的說明文字，電腦執行時會略過
- `docstring（文件字串）`：寫在函式/類別開頭、說明「這段做什麼」的文字
- `variable（變數）`：給一個值取的名字，方便之後使用
- `type / 型別`：資料的種類（數字、文字、真假值…）
- `type hint（型別提示）`：在變數旁標註「這該放什麼型別」的標籤，不強制
- `type checker（型別檢查器）`：幫你檢查型別有沒有用錯的工具
- `syntax（語法）`：程式語言的「文法規則」
- `indent / indentation（縮排）`：每行程式前面的空白，Python 用它來分辨程式的層次
- `IDE / editor（編輯器）`：用來打字、寫程式碼的軟體（如 VS Code）
- `ipython`：一個比預設更好用的互動式 Python 視窗（有自動完成、語法上色等）

**型別與容器**
- `int（整數）`、`float（浮點數／小數）`、`str（字串／文字）`、`bool（布林值／真假值）`
- `list（清單）`：一排可以裝很多東西、有順序的容器
- `tuple（元組）`：像 list、但內容不能改
- `set（集合）`：一堆不重複的東西、沒有順序
- `dict（字典）`：用「鍵-值」成對存放的容器，像查字典
- `comprehension（生成式）`：用一行精簡語法快速產生 list/dict 的寫法

**流程與函式**
- `if / else（條件判斷）`：「如果…就…否則…」的分支
- `for / while（迴圈）`：重複做某件事
- `function（函式）`：把一段程式打包、取名字，之後可重複呼叫
- `def`：Python 用來「定義一個函式」的關鍵字
- `parameter / argument（參數 / 引數）`：傳給函式的輸入值
- `return（回傳）`：函式把結果送回去
- `lambda（匿名函式）`：沒有名字的小型函式
- `*args / **kwargs`：讓函式能接收「數量不固定」的參數的寫法

**模組與套件**
- `module（模組）`：一個 .py 檔，裝著可重複使用的程式
- `import（匯入）`：把別的模組/套件拿進來用
- `package（套件）`：別人寫好、可安裝來用的程式庫
- `uv / pip`：Python 的套件管理工具，用來安裝套件
- `venv（虛擬環境）`：幫每個專案隔出獨立的套件空間，互不干擾

**例外與物件**
- `exception（例外／錯誤）`：程式執行時發生的問題
- `try / except（例外處理）`：「試著做、如果出錯就改做別的」
- `OOP（物件導向程式設計）`：把資料和功能包成「物件」來組織程式的方式
- `class（類別）`：物件的「設計藍圖」
- `instance（實例）`：照類別這張藍圖造出來的具體東西
- `inheritance（繼承）`：新類別沿用舊類別的功能
- `self`：類別方法裡，代表「物件自己」
- `decorator（裝飾器）`：在不改原函式的情況下、幫它加上額外功能的寫法
- `generator（生成器）`：一次只產生一個值、需要時才算的特殊函式

**資料科學與 ML**
- `NumPy`：做大量數字運算的套件
- `array（陣列）`：一整排數字資料
- `Pandas`：處理表格資料（像 Excel）的套件
- `DataFrame`：Pandas 的表格資料結構
- `machine learning（機器學習，ML）`：讓電腦從資料中「學出規律」的技術
- `model（模型）`：機器學習「學完之後」得到的成果，可拿來預測
- `supervised / unsupervised（監督式 / 非監督式學習）`：有沒有「標準答案」來教電腦
- `train / test split（訓練/測試集切分）`：把資料分兩份，一份教電腦、一份考電腦
- `feature（特徵）`：拿來預測的「線索欄位」
- `overfitting / underfitting（過擬合 / 欠擬合）`：學得太死板 / 學得太隨便
- `accuracy / precision / recall / F1 / AUC`：評估模型好不好的各種分數指標
- `neural network（神經網路）`：模仿大腦神經元結構的運算模型
- `deep learning（深度學習）`：用很多層神經網路的機器學習
- `backpropagation（反向傳播）`：神經網路「修正自己」的核心演算法
- `PyTorch / scikit-learn（sklearn）`：做機器學習/深度學習的套件
- `CNN / RNN / Transformer`：三種神經網路架構（影像 / 序列 / 注意力）
- `fine-tune（微調）`：拿別人訓練好的模型、再用自己的資料調整

**後端與部署**
- `API`：程式之間互相要資料、傳資料的介面（像外送員）
- `FastAPI / Django`：用 Python 寫網站後端的框架
- `ASGI`：讓 Python 後端能處理非同步請求的介面標準
- `async / await（非同步）`：讓程式「等待某件事時、先去做別的」的寫法
- `Pydantic`：幫後端驗證「收到的資料格式對不對」的套件
- `dependency injection（依賴注入，DI）`：把「需要的東西」自動送進函式的設計
- `SQLAlchemy`：用 Python 操作資料庫的工具
- `JWT / OAuth`：使用者登入驗證的機制（通行證）
- `WebSocket`：讓前後端能「即時雙向」溝通的技術
- `Docker`：把程式和環境打包成「容器」、到哪都能跑的工具
- `Uvicorn / Gunicorn`：負責「實際運行」後端程式的伺服器工具
