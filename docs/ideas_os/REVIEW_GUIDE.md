# REVIEW_GUIDE.md — How to Review Ideas OS Docs (for Codex)

> You (Codex) are the reviewer for every Ideas OS documentation file generated by Claude Code.
> Review **one document at a time**. Do not rewrite the doc — produce a structured review.
> Goal: each doc must be **implementation-ready** — an engineer can build from it with minimal clarification.

---

## 0. Before you review — read these first (canonical sources)

Read in this order, then judge the target doc against them:

1. `00_LOCKED_DECISIONS.md` — immutable, owner-approved decisions. **The target doc must not contradict this.**
2. `ADR/` — decision history (one ADR per decision; immutable once Accepted).
3. `IODL_MASTER_SPEC.md` / `MASTER_SPEC.md` / `MASTER_OUTLINE.md` — the generation spec the doc must satisfy. (IODL's 18-doc tree is authoritative for the file list.)
4. `README.md` — structure + workflow.

The platform is **ai-island-web** (Next.js 15 App Router + Supabase + Zeabur + Cloudflare R2). Reviews must be grounded in that reality, not generic.

## 2. Structure gate (required sections present & non-empty)

Every doc must contain these sections, each with real content:

`Purpose · Overview · Terminology · Design Goals · Core Concepts · Business Rules · User Flow · Mermaid Diagram(s) · Database Considerations · API Considerations · Permission Model · UI Considerations · Edge Cases · Security · Performance · Future Expansion · Implementation Notes`

Entity-heavy docs (04–12) also need, per entity: Definition · Lifecycle/State Machine · Ownership · Metadata · Permission · Version · Lineage · Examples.

**Fail the structure gate if any required section is missing or empty.**

---

## 3. Quality gate (reject if ANY of these)

- [ ] Contains `TODO`, `TBD`, `Coming Soon`, `Placeholder`, or an empty heading.
- [ ] Under ~300 lines / lacks equivalent engineering depth (the longer master spec asks ≥500 for entity docs).
- [ ] No Mermaid diagram where one clearly applies (flowchart / sequence / ER / state).
- [ ] Database content lacking: columns, **PK, FK, indexes, constraints, RLS, migration note, example row**.
- [ ] API content lacking: **method, route, permission, request, response, errors** (+ pagination/rate-limit where relevant).
- [ ] No explicit **Permission Matrix** (workspace roles × actions) where the feature touches workspace data.
- [ ] No Edge Cases / Security section with real, specific items (not generic filler).
- [ ] MVP vs Future not separated.
- [ ] **Template filler**: generic sentences that would read identically if the topic were swapped. (This was the failure mode of the old `02_product` stubs — watch for it.)

---

## 4. Consistency gate (must align with locked decisions)

Reject / flag if the doc violates any locked decision (see `00_LOCKED_DECISIONS.md` for the authoritative list). Key invariants:

- **Ownership:** new durable entities use `workspace_id`. Personal-only data may use `user_id`. Existing `user_id` systems (profiles, xp, lesson_progress, blog, course/AI-Island learning data) are **not** modified.
- **Roles:** workspace roles = Owner / Manager / Contributor / Viewer, **separate** from platform roles (Platform Owner/Admin/Moderator/Support). No workspace "admin".
- **AI:** reuse existing `ai_models` / `ai_api_keys` / `user_api_keys` + existing usage tables (`ai_usage_daily` / `ai_model_usage`). New layers only: Model Router, Cost Manager, `agent_runs`. Agents are **resources, not members**. Agents return validated structured output.
- **Economy:** Z Coin = existing platform Z 幣 (`point_transactions`/ledger). **Dust ≠ money**, separate ledger, no payout/conversion. Workspace wallet = shared Z 幣 allowance.
- **Work ≠ Blog:** Works live in a new `works` table; blog is a publish target; article Works sync to blog draft only on publish.
- **Admin boundary:** `/admin/idea-fragments` is preserved (not migrated/deleted). Shared logic via Creator Engine Shared Services. Creator Island fragment system is separate, user-facing.
- **Marketplace phase 1:** internal Z Coin only — no real-money payout / KYC.
- **n8n:** optional automation layer; core flows must not depend on it.
- **Language:** docs/code/db/API English; user-facing UI Traditional Chinese (and UI copy must not leak raw English system terms — use the glossary: Workspace→工作空間, Fragment→碎片, Work→作品, etc.).
- **Route/flag:** Creator Island = `/creator-island`, gated by `feature_creator_island_enabled` (existing `app_settings` `feature_*_enabled` pattern).

If the doc proposes changing a locked decision, that is **not** an inline edit — it requires a **new ADR**. Flag it as such.

---

## 5. Codebase-grounding gate (this is where generic docs fail)

For every concrete reference, verify against the actual repo:

- **Existing tables/libs cited** (e.g. `ai_models`, `ai_api_keys`, `user_api_keys`, `point_transactions`, `app_settings`, `idea_fragments`, `profiles`) — do they exist and is the usage correct? Flag invented names presented as existing.
- **New tables/routes/services** — are they clearly labeled NEW, and do they follow existing conventions (snake_case tables, `/api/...` routes, `src/lib/...` services, RLS pattern matching `idea_fragments_migration.sql`)?
- **Supabase 1000-row truncation** — any "fetch all rows" pattern must paginate (`.range()`), per the platform's known gotcha. Flag unpaginated `select('*')` on large tables.
- **Naming** — consistent across docs (a table/route/agent name must be identical everywhere it appears).

---

## 6. Output format (what your review returns)

```md
# Review: <NN_DOC_NAME>.md

Verdict: PASS | REVISE | REJECT

## Blocking issues (must fix before implementation)
- [gate §X] <file:line or section> — <problem> — <suggested fix>

## Non-blocking improvements
- <section> — <suggestion>

## Consistency vs locked decisions
- <OK, or list of violations with the ADR/decision they break>

## Codebase-grounding
- <OK, or list of invented/incorrect references>

## Missing required sections
- <none, or list>
```

- **PASS** = implementation-ready, no blocking issues.
- **REVISE** = usable but has blocking issues; list them precisely so Claude Code can fix in one pass.
- **REJECT** = structurally wrong (template filler, wrong lane, contradicts locked decisions wholesale).

Be specific and cite locations. Prefer one precise actionable fix over vague commentary. Do not pass a doc that merely "looks complete" — verify depth against §3–§5.
