## 7. 建議推進順序（雪鑰版）

不依區分組、依**價值 × 風險 × 依賴**重排：

| Sprint | 內容 | 估時 |
|---|---|---|
| **後 S1**（剛需止血）| P4-10 錯誤日誌 → P4-14 API rate limit → P4-11 DB 健康 widget | ~1 週 |
| **後 S2**（內容效率）| P4-01 章節後台編輯 + P4-02 版本記錄（合包） | ~3 週 |
| **後 S3**（客服剛需）| P4-07 工單 + P4-21 AI 對話審核 | ~1.5 週 |
| **後 S4**（運維小品）| P4-12 cache clear + P4-13 env panel + P4-17 changelog | ~4 天 |
| **後 S5**（內容延伸）| P4-03 SEO 預覽 + P4-04 排程發布 + P4-06 timeline | ~1 週 |
| **後 S6**（增長）| P4-08 referral + P4-09 segment 存檔 + P4-05 KPI 報表 | ~2 週 |
| **後 S7**（行銷工具）| P4-15 批次 email + P4-16 Web Vitals | ~1.5 週 |
| **後 S8**（多角色）| P4-19 教師 role + P4-20 作業批改 | ~2 週 |
| **後 S9**（高風險晚做）| P4-18 A/B 測試平台 | ~2 週 |

**為什麼「剛需止血」先**：

- P4-10 沒做 = 線上每次 bug 都靠用戶告才知道、是觀測黑洞
- P4-14 沒做 = AI 端點被惡意 user 燒錢、無防線
- P4-11 沒做 = DB 滿了 / 慢查詢無感、爆才知道

這三項都不大、合包 1 週、CP 最高。
