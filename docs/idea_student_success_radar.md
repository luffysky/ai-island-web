## 1. 為什麼要做這頁（痛點）

現在後台的資料很全，但**全是散的、而且都是「看」不是「做」**：

| 你想知道 | 現在要去哪 | 問題 |
|---|---|---|
| 誰快不來了 | `/admin/churn` | 只給名單、看完還要自己一個個處理 |
| 大家卡在哪一章 | `/admin/cohort`、首頁 Top10 章 | 是「整體」數字，不是「這個人」卡住 |
| 某人學得好不好 | `/admin/users/[id]` | 一次只能看一個人，無法掃全班 |
| 想鼓勵 / 送 Z 幣 / 發訊息 | zcoin、broadcasts、crm 各一頁 | 看到問題的地方≠能行動的地方，要切頁 |

**核心缺口：沒有一頁能「掃全班 → 標出風險 → 當場救」。**
這頁就是要把「**發現問題**」和「**採取行動**」收在同一個畫面，把現有的 churn / cohort / gamification / zcoin / crm / ai-tutor 串成一條動線。

教育平台的命脈是**完課率與留存**，這頁直接服務這兩個數字。

## 3. 各區塊詳細

### ① 健康總覽 4 連卡
四個分群的即時人數 + 本週變化箭頭。點任一張 → ② 清單自動套用該分群篩選。

- 🟢 **健康**：分數 ≥ 70
- 🟡 **觀察**：40–69（有疲態，但還在）
- 🔴 **高風險**：< 40 且 7 天內仍有活動（**還救得回來，優先**）
- ⚫ **已沉睡**：> 30 天沒登入（另開「喚醒活動」批次，不混進待救清單）

### ② 今日待救清單（這頁的主角）
每張卡 = 一個學員，**按「健康分數 × 可救性」由 AI 排序**（快流失但還在線的排最上，因為 CP 值最高）。

每張卡顯示：
- 健康分數 + 趨勢箭頭（▲▬▼，跟上週比）
- **一句話診斷**：自動歸因「卡 ChXX 幾天 / streak 斷了 / quiz 答對率下滑 / 互動歸零」
- **行動鈕（就地執行、不跳頁）**：
  - `💬 發鼓勵`：開 modal，可選罐頭話術或自寫，走站內通知 + （選配）LINE/Email
  - `🪙 送 Z 幣`：快捷 +50/+100，自動寫入 ledger（`reason=radar_rescue`，明細「學習關懷」→ 直接接上既有 `/me/ledger`）
  - `🤖 生成專屬建議`：呼叫 AI，吃這位學員的「卡在哪+答錯什麼」生成客製學習路徑，可一鍵當鼓勵訊息送出
  - `👤 檔案`：開 `/admin/users/[id]`（既有頁）

### ③ 卡關熱點
「章節 × 目前卡住人數」長條圖。**和首頁「完成 Top10」相反——這裡看的是「進去了但出不來」**。
- 點章節 → 篩出卡在這章的所有人
- 連續多人卡同一章 = 內容該補強的訊號 → 旁邊給「→ 編輯這章」連到 `/admin/chapters/[id]`

### ④ 本週之星
進步最多的學員（XP 增幅 / 完課數 / streak）。一鍵「公開表揚」推到全站動態（正向循環，也順手測試 FriendsFeed）。

## 4. 健康分數怎麼算（演算法）

分數 0–100，四個面向加權（**全部用 DB 既有欄位、不需新表也能算**）：

| 面向 | 權重 | 取自 | 邏輯 |
|---|---|---|---|
| 🔥 **動能 Momentum** | 35% | `profiles.streak_days`、`last_active_at` | 連續天數越高分；最近沒登入扣分 |
| 📈 **進度 Progress** | 30% | `lesson_progress`（近 14 天完課速度） | 完課變慢/停滯扣分；卡同一章 N 天重扣 |
| 🎯 **理解 Mastery** | 20% | `quiz_attempts` 近 N 次答對率趨勢 | 答對率下滑是「快放棄」的早期訊號 |
| 💬 **參與 Engagement** | 15% | 部落格/論壇/AI 導師互動次數 | 從互動到完全安靜 = 警訊 |

- 每個面向各自 0–100，加權平均 = 健康分數。
- **趨勢箭頭** = 本週分數 vs 上週分數。
- 「可救性」= 風險高（低分）**且** 7 天內仍有任何活動 → 排序往前。

> MVP 可先只用「動能 + 進度」兩項（純 `profiles` + `lesson_progress`，最好取數），準了再加 mastery / engagement。

## 5. 資料來源 & 後端

複用既有表，**不需新 migration 就能出 MVP**：

- `profiles`（streak_days, last_active_at, xp, level, created_at）
- `lesson_progress`（chapter_id, 完成時間 → 算速度/卡關）
- `quiz_attempts`（答對率趨勢）
- `point_transactions` / ledger（送 Z 幣寫入，接 `ledger-labels.ts` 的 `radar_rescue`）
- AI：複用 `ai-tutor-prompt.ts` 既有 mastery context 生成建議

**注意既有雷**：算卡關要掃 `lesson_progress`，**務必用 `fetchAllLessons` 式分頁**，別直接 `.select('*')`（PostgREST 1000 筆截斷，CLAUDE.md 有記）。

### API（新）
- `GET /api/admin/radar?bucket=risk&range=week` → 回各學員健康分數 + 診斷（owner-only、server 端算分）
- `POST /api/admin/radar/action` → `{ userId, type: 'encourage'|'grant'|'advice', payload }` 統一行動入口，內部分派到既有通知 / ledger / AI

### 效能
- 全班逐人算分頗重 → **每日 cron 預算一次**寫進輕量表 `student_health_daily`（user_id, date, score, breakdown jsonb, diagnosis），頁面只讀當天快照（即時卡用 `analytics_sessions`）。這也讓「趨勢箭頭」有歷史可比。

## 6. 權限 / 安全
- 整頁 + 所有 API：**owner-only**（沿用 `is-owner.ts` 的 `checkOwner`）。admin 點不進去。
- 每次「送 Z 幣 / 發訊息」都寫 `audit_logs`（actor、target、action=`radar_rescue`）。
- 不外洩學員私訊內容；診斷只用聚合學習數據。
