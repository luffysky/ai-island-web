## 0. 一句話定義

把現有的 AI 島（70 章、5 副本、討論區、寵物、AI 導師），從「網頁選單」升級成「一座**真 3D、可走動、有 AI 角色陪伴**的島嶼」。使用者操控角色在島上移動，走到節點進入學習內容。

**v0 範圍 = 一座小島的垂直切片**，不是完整 6 區域島嶼。先證明技術可行、體驗成立，再擴張。

## 1. 鐵則（嶼築必須遵守）

1. **3D 島嶼絕不取代經典模式，兩者永久並存對等。** 島嶼模式（3D）與經典模式（現有清單式網站）是兩個對等的入口，服務兩種使用者需求。不可把島嶼做成「唯一入口」，也不可把經典模式降級成「備用」。詳見第 2 節「雙模式架構」。
2. **不重做任何現有內容**。章節、副本、討論區、寵物、AI 導師的資料與頁面全部保留。3D 島嶼只是「新的入口層」，點節點後**跳轉或開啟現有頁面**。
3. **不切換技術棧**。沿用 Next.js 15 + React 19 + TypeScript。3D 用 React Three Fiber，融入現有專案。
4. **3D 島嶼是獨立路由 `/island`**，不取代現有首頁。現有 `/`、`/chapters`、`/courses`、`/forum` 全部保留可用。
5. **手機必須能跑**。每一批開發都要在行動裝置實測 FPS。效能不過關就降規格，不堆功能。低階裝置自動 fallback 到經典模式。
6. **分批交付**。每批結束都要可執行、可看見成果、`tsc --noEmit` 通過。禁止「做到一半無法執行」的中間狀態。
7. **低多邊形（low-poly）風格**。不做寫實。第一版優先用免費 3D 資源，不自建模型。
8. 沿用現有設計 token（`src/lib/utils.ts` 的 `STAGE_COLORS`、`globals.css` 的 CSS 變數）。

## 2. 雙模式架構（核心產品原則）

AI 島提供兩種對等的學習模式，使用者自由選擇、隨時切換。

### 2.1 兩種模式

| | 🏝️ 島嶼模式 | 📋 經典模式 |
|---|---|---|
| 路由 | `/island` | 現有 `/`、`/chapters`、`/courses`、`/forum` 等 |
| 形態 | 3D 沉浸式、可走動、有 AI 角色陪伴 | 清單式、選單導覽、快速 |
| 適合 | 想享受過程、探索、有陪伴感的人 | 想有效率、快速看完就走的人 |
| 狀態 | v0 新建 | **已存在，原封不動保留** |

兩者**地位對等**——不是「島嶼為主、經典為輔」，也不是反過來。它們是服務不同需求（與不同時刻）的兩個入口。

### 2.2 共用同一份資料與進度（關鍵）

兩種模式只是「外觀與導覽方式」不同，**底層完全共用**：

- 章節內容、副本、討論區 → 同一份資料
- 學習進度（`lesson_progress`）、XP、等級（`profiles`）→ 同一套
- 寵物、成就、Z-coin → 同一套

在島嶼模式完成 ch05，切到經典模式它也是完成的，反之亦然。**模式不分家，進度只有一份。**

→ 這代表：**經典模式不需要重做，它已經存在且完整**。v0 要做的只是「島嶼模式」本身，加上「兩模式並存、可切換、記住偏好」這層銜接。

### 2.3 模式選擇與記憶

- **首次進站**：給一個輕量選擇畫面——「想怎麼學？🏝️ 逛島嶼 ／ 📋 快速開始」
- **記住選擇**：存進 `profiles` 新欄位 `preferred_mode`（值：`'island'` | `'classic'`），預設 `'classic'`
- **之後進站**：直接帶到使用者偏好的模式
- **隨時可切換**：島嶼模式內有「切換經典模式」出口；經典模式（如 TopNav）有「進入島嶼」入口。切換不影響任何進度。

### 2.4 低階裝置自動 fallback

若偵測到裝置 WebGL 效能不足 / 不支援，或 3D 場景 FPS 持續過低：
- 自動提示並導向經典模式，確保使用者「一定學得到東西」
- 經典模式不依賴 WebGL，任何裝置都能用

→ 這個設計讓 3D 島嶼變成「**成功是加分、失敗也不傷本體**」的安全擴充：島嶼就算在某些裝置跑不動，那些使用者照樣能用經典模式完整學習。

### 2.5 資料庫變更

```sql
-- profiles 加一欄記住模式偏好
ALTER TABLE public.profiles
  ADD COLUMN IF NOT EXISTS preferred_mode TEXT NOT NULL DEFAULT 'classic';
  -- 值：'classic' | 'island'
```

這是 v0 唯一的資料庫變更（島嶼本身只讀現有資料，不新增表）。

## 3. 技術選型

| 用途 | 套件 | 說明 |

| 3D 渲染 | `three` | WebGL 底層 |

安裝指令：
```bash
npm install three @react-three/fiber @react-three/drei
npm install -D @types/three
```

**3D 模型來源（v0 不自建模）**：
- Kenney.nl（CC0 免費低多邊形資源，有 nature kit、city kit）

- Poly Pixel / Sketchfab 的免費 CC0 模型
- 模型放 `public/island/models/`

## 4. v0 範圍定義（極度克制）

### 做什麼

一座**小島**，島上有：

| 元素 | 數量（v0） | 對應現有系統 |
|---|---|---|
| 可走動的玩家角色 | 1 | — |
| 章節入口節點 | 3 個（示範用：對應 ch01、ch02、ch03） | `/chapters/[id]` |
| 副本入口節點 | 1 個（對應 ai-writing 副本） | `/courses/ai-writing` |
| 村莊入口節點 | 1 個（對應討論區「新手提問」版塊） | `/forum/questions` |
| AI 居民 NPC | 1 個（站在島上、走近會對話） | 接現有綠寶 AI（`/api/ai/chat`） |
| 跟隨寵物 | 1 隻（沿用現有 pet 系統的物種與名字） | `src/lib/pet-species.ts` |

### v0 不做什麼（明確排除，避免範圍膨脹）

- ❌ 完整 6 大區域地圖（v0 只有一座小島）
- ❌ 自建 3D 模型（用免費現成資源）
- ❌ 多人即時同場（看不到其他玩家）
- ❌ 角色自訂、換裝
- ❌ 載具、騎乘
- ❌ 日夜循環、天氣系統
- ❌ 過場動畫、劇情演出
- ❌ 環境音樂、音效（v0 先靜音，第 4 批再評估）

這些都是「v1 以後」的事。v0 只證明：**3D 島嶼能在現有專案裡跑起來、能走動、能進入現有內容、有 AI 角色陪伴**。

## 5. 分批開發計畫

### 第 1 批 — 能站上去的島

**目標**：一塊 3D 地面 + 一個能走動的角色 + 相機跟隨。打開 `/island` 能看到、能走。

**內容**：
- 新路由 `src/app/island/page.tsx`（client component，動態 import 3D 場景避免 SSR 問題）
- 3D 場景容器 `src/components/island/IslandCanvas.tsx`
- 一塊低多邊形地面（島嶼地形，可先用簡單幾何體或免費地形模型）
- 玩家角色（免費低多邊形角色模型，或先用膠囊體佔位）
- 操控：桌機 WASD / 方向鍵移動；手機虛擬搖桿（用 `nipplejs` 或自製）
- 相機第三人稱跟隨角色
- 基本光照（環境光 + 平行光）+ 天空盒
- 開發期顯示 FPS（`Stats`）

**驗收**：`/island` 打開，能用鍵盤/觸控讓角色在島上走動，相機跟著，手機 30 FPS 以上。

---

### 第 2 批 — 節點與進入內容

**目標**：島上放上 5 個入口節點，走近能進入現有的章節/副本/討論區。

**內容**：
- 節點資料設定檔 `src/lib/island-map.ts`：定義每個節點的 3D 座標、類型、對應目標。
  ```ts
  // 範例結構
  interface IslandNode {
    id: string;
    type: "chapter" | "dungeon" | "village";
    position: [number, number, number];  // 3D 座標
    label: string;
    target: string;       // 點擊後導向的路由，如 "/chapters/1"
    icon: string;         // 顯示用 emoji
  }
  ```
- 節點 3D 物件 `src/components/island/MapNode.tsx`：地上的發光標記/建築物，頭頂浮 label
- 走近節點 → 浮出互動提示（「按 E 進入」/ 手機顯示按鈕）
- 觸發 → 用 Next.js router 導向對應頁面（章節用 `/chapters/[id]`、副本 `/courses/[slug]`、村莊 `/forum/[boardSlug]`）
- 章節節點依 `STAGE_COLORS` 上色

**驗收**：走到節點能進入對應的現有頁面，回到 `/island` 角色位置保留。

---

### 第 3 批 — 進度連動與村莊

**目標**：島嶼反映使用者的真實學習進度；村莊節點呈現社群活絡感。

**內容**：
- 讀取現有 `lesson_progress` 表 + `profiles`（XP/等級），判斷每個章節節點狀態：
  - 已完成 → 節點點亮/有特效
  - 進行中 → 節點發光（下一個該做的）
  - 未解鎖 → 節點灰暗
- 進度資料透過現有 Supabase client 抓取（沿用 `gamification.ts` 的模式）
- 村莊節點顯示該討論版塊的活絡度（最近主題數，呼叫現有 `/api/forum/threads`）
- 島上顯示玩家當前 Lv（用現有 `profiles.level`）
- HUD（畫面疊層）：左上角顯示玩家頭像/Lv/XP，右下角小地圖（可選）

**驗收**：登入後進 `/island`，看得出自己學到哪、哪些解鎖了。

---

### 第 4 批 — AI 角色與生命感

**目標**：島上有 AI 居民 NPC + 跟隨寵物，讓島「活起來」。

**內容**：
- **跟隨寵物**：沿用 `src/lib/pet-species.ts` 的物種設定，3D 寵物模型跟著玩家角色移動（平滑跟隨、不穿模）
- **AI 居民 NPC** `src/components/island/IslandNPC.tsx`：
  - 站在島上固定位置的角色
  - 走近 → 浮出對話泡泡 / 開啟對話面板
  - 對話接現有 AI 系統（`/api/ai/chat`），但帶不同的 system prompt——這個 NPC 是「島上居民」人設，不是綠寶導師本身
  - NPC 用途：給學習提示、引導下一步、閒聊增加沉浸感
- 節點微動畫（浮動、旋轉、發光呼吸）
- 進入區域的轉場效果
- 環境氛圍微調（光暈、霧、粒子，效能允許下）

**驗收**：島上有寵物跟著走、有 NPC 能對話，整體「想待著」。

### 6.3 導覽整合

實作第 2 節「雙模式架構」的銜接層：

- **首次進站選擇**：使用者第一次造訪（`profiles.preferred_mode` 尚未設定或為預設），顯示一次性的輕量選擇——「想怎麼學？🏝️ 逛島嶼 ／ 📋 快速開始」。選擇後寫入 `profiles.preferred_mode`。
- **進站導向**：之後每次進站，依 `preferred_mode` 決定預設落點（`island` → `/island`，`classic` → 現有首頁）。
- **島嶼 → 經典**：島嶼模式的 HUD 提供「📋 切換經典模式」出口，點了回到現有網頁版並更新偏好。
- **經典 → 島嶼**：`TopNav` 加「🏝️ 島嶼」連結；現有首頁 `/` 放一個明顯的「進入島嶼」入口按鈕。
- **切換不影響進度**：兩模式共用同一份資料，切換只換外觀，學習進度/XP/寵物全部延續。
- **fallback**：偵測 WebGL 不支援或效能過低時，提示並導向經典模式（見第 7 節效能要求）。

### 6.4 SSR 注意事項

R3F / Three.js 不能 SSR。`/island/page.tsx` 必須：
```tsx
import dynamic from "next/dynamic";
const IslandCanvas = dynamic(() => import("@/components/island/IslandCanvas"), {
  ssr: false,
  loading: () => <載入畫面 />,
});
```

## 7. 效能要求（硬性，每批驗收）

| 裝置 | 最低 FPS | 備註 |
|---|---|---|
| 桌機 | 50+ | |
| 中階手機 | 30+ | 這是底線，跌破就要降規格 |

**效能保險措施**：
- 低多邊形模型，單一模型面數控制在合理範圍
- 善用 instancing（重複物件如樹木、草）
- 視距裁切（遠的不渲染）
- 貼圖壓縮、尺寸節制
- 提供「畫質設定」（高/中/低），低階裝置自動降
- 進場有 loading 畫面，模型預載

**若手機效能無法達標**：優先降畫質、減物件，不是加開發時間硬撐。必要時 v0 可接受「手機版簡化、桌機版完整」。

## 8. 風險與對策

| 風險 | 對策 |
|---|---|
| 範圍膨脹（想做完整島嶼） | 嚴守 v0 範圍，第 4 節「不做什麼」清單不可違反 |
| 手機效能爆掉 | 第 1 批就在手機測，低多邊形 + instancing，必要時降規 |
| 3D 模型來源卡住 | 先用幾何體佔位也能做完整批次，模型可後補 |
| 開發時間失控 | 分 4 批，每批可獨立交付。某批做太久就停下重新評估 |
| 跟現有專案衝突 | 島嶼只新增檔案、只讀現有資料，不改現有邏輯 |

## 9. 成功的定義

**v0 成功 = 這句話成立**：

> 使用者打開 `/island`，看到一座 3D 小島，能操控角色走動，島上的寵物跟著他，走近 AI 居民能對話，走到節點能進入真實的章節/副本/討論區，整個過程在手機上順暢不卡。

做到這句，就證明了「3D 沉浸式島嶼」這條路在你的專案上**走得通**。然後才談擴張成 6 區域、村莊細化、AI 角色擴編的 v1。

**v0 沒做到也有價值**——它會告訴你卡在哪（效能?範圍?資源?），這些是規劃完整版前必須知道的事。

- 一次只做一批，做完跑 `tsc --noEmit` 確認無誤、能 `npm run dev` 啟動看到成果，再進下一批。
- 每批結束依「回交格式」回交（批次編號、file scope、diff 摘要、驗證方式與結果、效能數據、已知風險、待釐清項目），交 Gate 線裁定。
- Gate 裁定 NOT READY 時，補件重交，不跳過。
- 3D 模型若還沒備好，用 Three.js 內建幾何體（box、sphere、capsule、plane）佔位，先把邏輯與互動做完，模型後替換。
- 不確定範圍時，**選小的做**——寧可 v0 太陽春，不要做不完；說不清的標「待釐清」回報董事長，不自行假設。
- 所有新檔案集中在 `src/components/island/`、`src/app/island/`、`src/lib/island-map.ts`、`public/island/`，不散落、不污染現有結構。
- 涉及使用者可見 UI / 文案的批次，交付前送煙汐體驗審校。

# 附錄 A · 第 1 批 — 逐步實作拆解

> 這是「第 1 批：能站上去的島」的手把手步驟。嶼築照此順序做，每步做完可驗證、回交 Gate 線。

### 第 1 批的最終樣子

打開 `/island` → 看到一塊 3D 地面、一個角色站在上面 → 用 WASD（桌機）或虛擬搖桿（手機）能讓角色走動 → 相機第三人稱跟著角色 → 角落顯示 FPS。**這批沒有節點、沒有 NPC、沒有寵物**——純粹「能站上去、能走」。

### 步驟 1：裝套件

```bash
npm install three @react-three/fiber @react-three/drei
npm install -D @types/three
```
裝完跑 `npm run dev` 確認專案還能啟動。

### 步驟 2：建路由骨架

建 `src/app/island/page.tsx`：
- client component（`"use client"`）
- 用 `next/dynamic` 動態 import 3D 場景、`ssr: false`、給一個 loading 畫面
- 頁面本身只負責「全螢幕容器 + 動態載入 IslandCanvas」

```tsx
"use client";
import dynamic from "next/dynamic";

const IslandCanvas = dynamic(
  () => import("@/components/island/IslandCanvas").then(m => m.IslandCanvas),
  { ssr: false, loading: () => <載入中畫面 /> }
);

export default function IslandPage() {
  return <div style={{ width: "100vw", height: "100dvh" }}><IslandCanvas /></div>;
}
```

驗證：`/island` 能開、看到 loading 畫面（此時還沒有 3D 內容）。

### 步驟 3：建最小 3D 場景

建 `src/components/island/IslandCanvas.tsx`：
- R3F 的 `<Canvas>` 包整個場景
- 加環境光 `<ambientLight>` + 平行光 `<directionalLight>`
- 加一塊地面：先用 `<mesh>` + `<planeGeometry>` 或 `<boxGeometry>`（綠色，當草地）
- 加 drei 的 `<Sky>` 當天空
- 開發期加 drei 的 `<Stats>` 顯示 FPS
- 相機先用預設

驗證：`/island` 看到一塊綠色地面 + 藍天 + 左上角 FPS 數字。**這步先不要有角色。**

### 步驟 4：加角色（佔位）

建 `src/components/island/Player.tsx`：
- 角色先用 `<capsuleGeometry>`（膠囊體）當佔位，不要急著找模型
- 放在地面中央、站在地上（y 座標 = 膠囊高度一半）
- 用 `useRef` 拿到角色的 mesh ref，之後移動要用

驗證：地面中央站著一個膠囊體。

### 步驟 5：鍵盤操控（桌機）

建 `src/components/island/useIslandControls.ts`：
- 監聽 `keydown` / `keyup`，記錄 WASD / 方向鍵狀態
- 在 R3F 的 `useFrame` 裡，依按鍵狀態更新角色位置（用 delta time，不要用固定值，確保不同 FPS 速度一致）
- 角色移動時面向移動方向

驗證：桌機按 WASD，膠囊體在地面上移動。

### 步驟 6：相機跟隨

在 `IslandCanvas` 或獨立元件：
- 第三人稱相機：相機位置 = 角色位置 + 固定偏移（後上方）
- 每幀用 `lerp`（線性插值）平滑跟隨，不要硬綁（硬綁會很僵硬）
- 相機看向角色

驗證：角色走動時，相機平滑跟著，視角是第三人稱後上方。

### 步驟 7：手機虛擬搖桿

- 偵測觸控裝置 → 顯示一個虛擬搖桿（畫面左下角）
- 可用 `nipplejs` 套件，或自製一個簡單的 div 搖桿
- 搖桿輸出方向 → 餵給跟鍵盤同一套移動邏輯
- 搖桿是 HTML 疊層（不是 3D），用絕對定位蓋在 Canvas 上

驗證：手機開 `/island`，左下角有搖桿，拖曳能讓角色走動。

### 步驟 8：效能驗收

- 桌機 + 手機各測一次，看 `<Stats>` 的 FPS
- 桌機需 50+、手機需 30+
- 不達標 → 簡化地面、移除不必要的光源/陰影，不要加東西

### 第 1 批完成檢查清單

- [ ] `npm install` 三個 3D 套件成功
- [ ] `/island` 路由能開、不影響其他頁面
- [ ] 看得到地面 + 天空 + 角色
- [ ] 桌機 WASD 能走
- [ ] 手機虛擬搖桿能走
- [ ] 相機平滑第三人稱跟隨
- [ ] 桌機 50+ FPS、手機 30+ FPS
- [ ] `tsc --noEmit` 通過
- [ ] 現有頁面（`/`、`/chapters` 等）完全不受影響

全部打勾 → 嶼築依回交格式回交 → Gate 線（玄樞）裁定 READY → 才進第 2 批。

# 附錄 C · 後續版本路線（v1 ~ v3）

> v0 完成並驗證後，依序往下。每個版本一樣「分批、可交付、不破壞前一版」。
> 各版詳細規格由董事長指示、於該版開工前另寫；此處先定方向與範圍，讓整條路線清楚。

## v1 · 完整島嶼（6 大區域 + 村莊系統）

**前提**：v0 已驗證 3D 在專案中可行、效能可接受。

**目標**：從「一座小島」擴張成「完整的 AI 島」——對應現有 6 個 stage。

範圍：
- **6 大區域地形**：基礎之地 / 互動王國 / 後端深淵 / 多語大陸 / 商業港口 / AI 紀元，各有不同地貌與配色（用 `STAGE_COLORS`）
- **全部章節節點上島**：70 章全部變成島上節點，依 stage 分布在對應區域
- **區域解鎖**：跟著學習進度，區域漸次開放（呼應現有的 stage 解鎖邏輯）
- **村莊系統**：討論區 11 版塊做成島上的村莊聚落，可走進去、有建築、有居民走動
- **5 副本入口**：做成地城/洞窟造型的入口
- **區域間移動**：路徑、橋、傳送點
- **正式 3D 模型**：此階段開始替換佔位幾何體為正式低多邊形模型（地形、建築、角色）

分批建議：① 6 區域地形與相機 ② 70 章節點分布 + 進度解鎖 ③ 村莊聚落 ④ 副本入口 + 區域轉場 ⑤ 模型替換與美術打磨

## v2 · AI 角色生態（島上居民）

**目標**：島嶼從「空間」變成「有居民的世界」。

範圍：
- **多個 AI 居民 NPC**：不同村莊有不同居民，各有人設與專長（學習提示、鼓勵、解惑、閒聊）
- **AI 導師升級**：綠寶在島上有實體、會走動、會主動關心進度
- **寵物深化**：寵物在島上有更多互動、會跟 NPC 互動、會表達情緒
- **角色對話系統**：對話接 AI，但每個角色有獨立 system prompt 與記憶
- **動態事件**：島上偶發小事件（居民求助、限時任務），連動現有任務/成就系統
- **群體感**：顯示「最近島上有多少人在學習」之類的氛圍資訊（非即時多人，是統計呈現）

## v3 · 社交與共存（多人感）

**目標**：讓島嶼有「不是一個人」的感覺。

範圍：
- **他人足跡**：能看到其他學習者的足跡 / 名牌 / 最近活動（非即時同步，輕量呈現）
- **互動點**：村莊裡能看到其他人的討論貼文以 3D 形式呈現
- **協作元素**：組隊、共學房間（呼應藍圖的「組隊」概念）
- **島嶼個人化**：使用者能擺設、裝飾自己的小空間
- （評估）**即時多人同場**：技術與成本最重，最後評估是否值得做

## 版本路線總原則

1. **每個版本都不破壞前一版**——經典模式永遠並存、永遠可用
2. **每個版本獨立可交付**——v1 沒做完，v0 照樣是完整可用的產品
3. **效能優先於功能**——任何版本，手機跑不動就降規格，不犧牲流暢度
4. **內容不重做**——所有版本都只是「現有內容的新呈現層」，章節/副本/討論區資料永遠是同一份
5. **一個版本一個版本完成**——不跳級、不並行，v0 驗證過才開 v1
