# Lesson 01｜AI 為什麼也會卡 Bug

## 🎯 本課目標

理解一件反直覺的事：**就算你用 Claude、Codex、ChatGPT 寫程式，bug 還是會發生，而且 AI 自己常常解不掉。** 學完這課，你會知道 bug 不是「你太笨」或「AI 不夠強」，而是寫程式的本質，並建立「我和 AI 是一個團隊，要一起 debug」的正確心態。

## 🧠 核心觀念

很多人剛開始用 AI 寫程式時，心裡有個期待：「我用的是最強的 AI，它寫的程式應該不會有 bug 吧？」

然後第一次遇到畫面一片空白、按鈕沒反應、紅字噴滿整個螢幕的時候，就會陷入恐慌：「是不是我哪裡用錯了？」「是不是這個 AI 不夠強？我要不要換一個？」

真相是：**bug 跟「AI 強不強」幾乎沒有關係。** bug 來自三個 AI 無法完全掌控的地方——

第一，**AI 看不到你的全部。** 你的專案有幾十個檔案、一個資料庫、一堆設定。你貼給 AI 的，永遠只是其中一小塊。AI 是在「資訊不完整」的情況下幫你寫程式，就像請一個師傅幫你修房子，但他只能從門縫看進去。

第二，**程式是「組合」出來的，組合會產生意外。** 一段程式自己跑沒問題，另一段自己跑也沒問題，兜在一起卻爆炸——這是寫程式最常見的狀況。AI 寫的每一塊可能都對，但「兜起來」這件事充滿變數。

第三，**環境會變。** 你電腦上跑得好好的，部署到網路上就壞了；昨天好好的，今天套件更新了就壞了。這些 AI 在「寫程式的當下」根本預測不到。

所以結論是：**bug 是寫程式的常態，不是異常。** 連 Google、Apple 那種頂級工程師團隊，每天的工作有很大一部分就是在 debug。你會遇到 bug，不代表你失敗，代表你正在做一件「真正的工程」。

## 👶 超新手理解版

想像你請了一位超強的廚師（AI）幫你做菜。這位廚師廚藝世界第一。

但是——你只透過電話跟他溝通。他看不到你的廚房、不知道你的瓦斯爐其實有一個火口是壞的、不知道你冰箱裡的牛奶過期了、不知道你家的鍋子是不沾鍋還是鐵鍋。

他憑著你電話裡講的話，給你一套完美的食譜。你照著做，結果菜燒焦了。

是廚師不夠強嗎？不是。是因為**他看不到你的廚房**，而你也沒發現「火口是壞的」這個關鍵資訊。

寫程式跟 AI 的關係就是這樣。AI 很強，但它在「隔著電話、看不到你全部廚房」的情況下工作。菜燒焦了（出現 bug），不是要換一個廚師，而是你要學會走進廚房、看看到底哪個火口壞了——然後把這個資訊告訴廚師，他就能幫你修正。

**這門課，就是在教你怎麼「走進廚房、看清楚狀況」。**

## 🏗 工程師版本

從工程角度講清楚為什麼 AI 必然會產生 bug：

**1. Context window 的限制。** LLM 一次能讀進去的內容有上限。一個中型專案的程式碼動輒數萬行，你不可能、也不該全部貼給 AI。AI 永遠是在「部分 context」下生成程式碼，它對你專案其他部分的狀態是用「推測」補完的，推測就有錯的可能。

**2. 訓練資料的時間切片問題。** AI 的知識停在某個時間點。你用的框架（Next.js、Nest、某個套件）可能在那之後改版了 API。AI 用「它記得的舊寫法」生成程式碼，套件已經改成新寫法——這就是一種 bug 來源。

**3. 狀態與副作用（state & side effects）。** 程式不是純函式的堆疊。它有狀態（資料庫、記憶體、使用者輸入）、有副作用（網路請求、檔案讀寫）。AI 能寫出邏輯正確的程式碼，但「這段程式碼跑的當下，狀態長什麼樣」它無法預知。

**4. 整合複雜度（integration complexity）。** 單元正確 ≠ 整體正確。前端、後端、資料庫、第三方服務各自都對，介面（interface）對接時的格式、時序、權限不一致，就會出問題。AI 通常一次只處理一個單元，整合層的 bug 它最難預防。

所以資深工程師看待 AI 寫程式的心態是：**AI 是一個產出速度極快、但需要被驗證的程式碼來源。** 它不取代「驗證、測試、除錯」這些工程動作，它只是讓「生成」這一步變快。Debug 能力，反而因為 AI 而變得**更**重要——因為你生成程式碼的速度變快了，產生 bug 的速度也變快了。

## 🧩 真實世界案例

Luffysky 在做 AI 島的時候，請 AI 寫一個「使用者完成課程後加經驗值」的功能。AI 給的程式碼邏輯完全正確：偵測到完成 → 呼叫加經驗值的函式 → 畫面跳出慶祝動畫。

結果上線後，慶祝動畫從來沒出現過。

AI 的程式碼錯了嗎？沒有。問題出在那個函式裡有一行 `if (amount < 50) return`——經驗值少於 50 就直接結束。而一般課程的經驗值是 10~30。這行 `if` 是**很久以前另一個情境**留下來的，AI 在寫新功能時根本不知道這行的存在（它沒看到那個檔案的那一段）。

這就是「AI 看不到你全部廚房」的真實版本。AI 寫的新程式碼是對的，但它撞上了專案裡一個它看不到的舊設定。**這種 bug，AI 自己永遠 debug 不出來，因為它根本不知道那行 code 存在——除非你把它找出來、貼給 AI 看。**

## ❌ 常見死亡方式

- **以為換一個更強的 AI 就會沒 bug。** 從 ChatGPT 換到 Claude 再換到 Codex，bug 還在——因為 bug 不在 AI，在你看不到的那個環節。
- **遇到 bug 就整段砍掉重寫。** 把會動的部分一起砍掉，新寫的可能產生新 bug，越搞越亂。
- **把 bug 當成「我不適合寫程式」的證據。** 然後放棄。其實你只是還沒學 debug 而已。
- **無限次地對 AI 說「還是不行，再修」。** 不給新資訊，AI 只能用同樣的盲區再猜一次，當然還是錯。

## 🔍 如何定位問題

這一課還不教具體的定位工具（那是後面幾章的事）。這課要你建立的是定位問題的**起點心態**：

遇到 bug，先在心裡問自己一句話——**「AI 現在缺了什麼資訊，導致它解不出來？」**

不是問「AI 怎麼這麼爛」，而是問「我能補給它什麼？」。可能是錯誤訊息、可能是某個它沒看到的檔案、可能是「實際發生的狀況」跟「它以為的狀況」不一樣。Debug 的本質，就是**你當 AI 的眼睛**，把它看不到的東西找出來、餵給它。

## 🛠 Debug SOP

本課的 SOP 是一個「心態切換」流程，遇到 bug 時照做：

1. **停。** 不要立刻叫 AI 再改一次。
2. **深呼吸，告訴自己：bug 是正常的，不是我的失敗。**
3. **問自己：「這個 bug，AI 是不是缺了某個它看不到的資訊？」**
4. **把目標從「叫 AI 修好」改成「我要找出 AI 缺的那塊資訊」。**
5. 帶著這個心態，進入後面章節教的具體 debug 動作。

## 🤖 Claude Prompt

當你意識到 AI 可能在「資訊不足」下打轉，可以這樣對 Claude 說：

```
我們遇到一個 bug 卡住了。在你再次嘗試修改之前，我想先確認：
為了正確診斷這個問題，你還需要我提供哪些資訊？
請列出一份清單，例如：錯誤訊息、相關檔案、實際的執行結果、
或我預期的行為 vs 實際的行為。
我會一項一項補給你。
```

這個 prompt 的作用：把 AI 從「盲目再試一次」拉回「先盤點缺什麼資訊」。

## 🤖 Codex Prompt

對 Codex（在你的 SnowRealm 協作體系裡作為執行者）可以這樣下：

```
[診斷模式]
目前有一個 bug 尚未定位。請不要直接修改任何程式碼。
請先輸出：
1. 你目前對這個 bug 的成因假設（可以有多個）
2. 每個假設需要什麼證據才能確認或排除
3. 你建議我提供哪些檔案或資訊
在我提供足夠資訊前，禁止動手修改。
```

重點是那句「禁止動手修改」——先診斷、再動手，這在後面章節會反覆強調。

## 🧪 練習題

1. 回想你自己（或想像）用 AI 寫程式時遇過的一個 bug。試著用「廚師看不到廚房」的比喻，說出 AI 當時「看不到」的是什麼。
2. 找出一個你曾經「叫 AI 一直修、修不好」的經驗。如果重來一次，你會補給 AI 哪一個它原本沒有的資訊？
3. 用自己的話，向一個完全不懂程式的朋友解釋：「為什麼用最強的 AI 寫程式，還是會有 bug？」能講到對方點頭，代表你懂了。

## ✅ 驗收標準

- 你能說出 AI 產生 bug 的至少三個原因（看不到全部、組合產生意外、環境會變）。
- 遇到 bug 時，你的第一反應不再是「換 AI」或「砍掉重寫」，而是「AI 缺了什麼資訊？」。
- 你能用一個生活比喻，向新手解釋「AI 為什麼也會卡 bug」。

## 📌 重點整理

- bug 是寫程式的**常態**，不是你的失敗，也不代表 AI 不夠強。
- AI 產生 bug 的根本原因：它**看不到你的全部專案**、程式是**組合**出來的會有意外、**環境會變**。
- AI 自己常常 debug 不出來，因為它不知道「它看不到的那塊」存在。
- 你的角色是**當 AI 的眼睛**——把它看不到的資訊找出來、餵給它。
- 遇到 bug 的正確第一念頭：「AI 現在缺了什麼資訊？」

# Lesson 02｜不要瞎改

## 🎯 本課目標

學會 debug 最重要的紀律：**在搞清楚原因之前，不要動手改。** 你會理解「瞎改」為什麼會讓事情越來越糟，並學會用「一次只改一個地方」的方式取代亂槍打鳥。

## 🧠 核心觀念

遇到 bug，人的本能反應是「趕快改改看」。改一行、重跑、還是壞、再改一行、再重跑……這個行為叫**瞎改（shotgun debugging，亂槍打鳥除錯）**。

瞎改感覺上很「積極」，好像你在努力解決問題。但它其實是 debug 裡最危險的行為，原因有三個：

**第一，你會製造新 bug。** 你改的每一行，都可能在「修舊問題」的同時「造新問題」。改到後來，畫面上有三個 bug，你已經分不清哪個是原本的、哪個是你改出來的。

**第二，就算它「好了」，你也不知道為什麼好。** 瞎改瞎到某次突然正常了，你鬆一口氣。但你不知道是哪個改動讓它好的。下次再壞，你完全沒有累積到任何經驗，又要從頭瞎改一次。

**第三，你會把「會動的部分」也改壞。** 焦慮之下，人會忍不住連沒問題的地方也動。本來只有一個 bug，瞎改之後變成五個。

正確的紀律是：**先理解，再動手。** 在你能用一句話說出「這個 bug 是因為 ___」之前，不要改任何東西。聽起來慢，實際上快非常多——因為你不會製造新問題，也不會白繞路。

## 👶 超新手理解版

想像你家的電燈不會亮。

**瞎改的人**：跑去亂轉燈泡、亂拍牆壁、亂按所有開關、把電線扯一扯……燈還是不亮，而且現在連旁邊本來會亮的燈也壞了，因為你剛剛亂扯到它的線。

**正確的人**：先站著想一下。「燈不亮，可能是燈泡壞了、可能是開關壞了、可能是跳電了。」然後一個一個查：先看電箱有沒有跳電（沒有）、再換一顆好燈泡試試（還是不亮）、再檢查開關……一次只查一個，每查一個就排除一個可能。

第二種人看起來慢吞吞，但他**五分鐘就會找到真正的原因**，而且不會把別的燈弄壞。第一種人忙得滿頭大汗，搞了一小時，問題更多。

Debug 就是這樣。**忍住「趕快亂改」的衝動，先想清楚「可能是什麼」，再一個一個查。**

## 🏗 工程師版本

瞎改在工程上的正式名稱是 **shotgun debugging**，相對的正確做法是 **hypothesis-driven debugging（假設驅動除錯）**。

假設驅動除錯的流程是：

1. **觀察現象**：精確描述 bug。不是「壞了」，而是「點擊登入按鈕後，畫面停在登入頁，網址沒變，console 沒有紅字」。
2. **提出假設**：根據現象，列出可能的原因。例如「① 登入請求根本沒送出 ② 請求送出了但失敗 ③ 登入成功了但跳轉邏輯有問題」。
3. **設計驗證**：對每個假設，想一個「能確認或排除它」的最小動作。例如要驗證假設①，去 Network 面板看請求有沒有發出。
4. **一次驗證一個**：做一個驗證動作，得到結果，**排除或確認一個假設**。
5. **收斂**：重複到只剩一個假設，那就是 root cause（根本原因）。
6. **這時候才動手改**——而且只改那一個地方。

這個流程的核心紀律有兩條：

- **One change at a time（一次只改一個變因）。** 如果你一次改三個地方然後它好了，你永遠不知道是哪個改動生效的。一次改一個，結果才有意義。
- **可回復（reversible）。** 每個改動都要能還原。後面 Chapter 12 講的 Git，就是讓「還原」變簡單的工具。在學到 Git 之前，至少每次改之前先把原本的程式碼複製一份留著。

「先理解、後動手」不是龜毛，是工程效率的根本。瞎改的期望值是負的——它平均而言會增加你的總時間。

## 🧩 真實世界案例

一個常見場景：使用者做了一個購物網站，「加入購物車」按鈕沒反應。

**瞎改版本**：他先改按鈕的程式碼、沒用；再改購物車的狀態管理、沒用；再改 API、沒用；中途把按鈕的樣式也動了，結果按鈕跑版了；最後他把整個購物車元件砍掉重寫，花了一整天，重寫的版本又有新 bug。

**假設驅動版本**：同一個 bug。他先觀察——「按鈕點下去，完全沒反應，console 沒紅字」。提出假設：① 按鈕的點擊事件根本沒綁上 ② 綁上了但函式內部出錯 ③ 函式跑完了但購物車畫面沒更新。他先驗證假設①：在點擊函式的第一行加一句 `console.log("按鈕被點了")`。重跑、點按鈕——console 沒有印出那句話。**假設①確認，②③直接排除。** 問題範圍從「整個購物車系統」瞬間縮小到「按鈕的事件綁定」。五分鐘後他發現是按鈕的事件名稱拼錯了。改一個字，解決。

同一個 bug，一個花一天還更糟，一個花十分鐘。差別只在「有沒有忍住不瞎改」。

## ❌ 常見死亡方式

- **一次改五個地方，然後重跑。** 好了不知道為什麼好，壞了不知道為什麼壞。
- **焦慮性亂改。** 心裡慌，手就停不下來，連沒問題的程式碼也動。
- **改之前不留原版。** 改壞了回不去，原本只是一個 bug，現在連「正常的版本」都救不回來。
- **對 AI 說「全部重寫」。** 把會動的一起丟掉，等於用新 bug 換舊 bug。
- **改了沒驗證就繼續改。** 一口氣改十個地方才重跑，結果根本不知道是哪個改動的效果。

## 🔍 如何定位問題

這一課的定位原則是「**先別改，先描述**」。在你動手之前，強迫自己完成這句填空：

> 「這個 bug 的現象是 ___；我目前的假設有 ___、___、___；我打算先驗證 ___，因為 ___。」

如果這句話你填不出來，代表你還沒準備好動手——你現在改任何東西都是瞎改。把這句話填出來，再開始。

## 🛠 Debug SOP

1. **凍結。** 看到 bug，手離開鍵盤，不要改任何東西。
2. **描述現象。** 用具體的句子寫下「發生了什麼」，不要用「壞了」這種模糊詞。
3. **列假設。** 寫下 2~4 個可能的原因。
4. **挑一個最好驗證的假設**，設計一個最小的驗證動作。
5. **驗證、得到結果、排除或確認。**
6. 重複 4~5，直到只剩一個原因。
7. **只改那一個地方。** 改之前先備份原本的程式碼。
8. **改完立刻驗證**——好了才繼續，沒好就還原、重新想。

## 🤖 Claude Prompt

當你或 AI 開始有「瞎改」的傾向，用這個 prompt 拉回紀律：

```
請先不要修改程式碼。我們改用假設驅動的方式 debug。
請根據目前的現象，列出 2~4 個可能的原因（假設）。
針對每個假設，告訴我一個「最小的驗證動作」可以確認或排除它。
我們會一次驗證一個，不要一次改多個地方。
```

## 🤖 Codex Prompt

```
[紀律：one change at a time]
針對這個 bug，請遵守以下規則：
1. 一次只提出「一個」修改，並說明它對應哪一個假設。
2. 修改範圍盡可能小，只動必要的那幾行。
3. 不要順手重構、不要動與此 bug 無關的程式碼。
4. 提出修改後，等我驗證結果，再決定下一步。
```

## 🧪 練習題

1. 找一段你最近遇到的 bug 描述（自己的或網路上的）。把它從「壞了」這種模糊說法，改寫成一句具體的現象描述。
2. 針對那個 bug，寫出 3 個可能的原因（假設），以及每個假設「怎麼驗證」。
3. 反思：你過去 debug 時，最常犯的「瞎改行為」是哪一種？（一次改太多？焦慮亂改？不留原版？）寫下來，這是你要特別提醒自己的點。

## ✅ 驗收標準

- 遇到 bug，你能忍住「立刻改」的衝動，先完成「現象描述 + 假設列表」。
- 你理解「一次只改一個地方」的原因，並能說出「一次改多個」的壞處。
- 你每次改程式碼前，會先確保原本的版本留得住（備份或之後學的 Git）。

## 📌 重點整理

- **瞎改（亂槍打鳥）是 debug 最危險的行為**：製造新 bug、好了不知為何好、把好的也改壞。
- 正確做法是**假設驅動除錯**：觀察現象 → 列假設 → 設計驗證 → 一次驗一個 → 收斂出根本原因 → 才動手。
- 兩條鐵律：**一次只改一個地方**、**每個改動都要能還原**。
- 動手前先完成填空：「現象是___，假設有___，我先驗證___」。
- 慢即是快——先理解再動手，平均比瞎改快非常多。

# Lesson 03｜如何判斷問題層級

## 🎯 本課目標

學會在遇到 bug 時，快速判斷「問題大概出在哪一層」。一個網站從你看到的畫面、到背後的資料，中間有好幾層。能判斷層級，你就能把「整個網站壞了」這種茫然，縮小成「問題在某一層」，debug 範圍立刻小很多。

## 🧠 核心觀念

一個現代網站（像你的 AI 島）不是一塊鐵板，它是**分層**的。從使用者看得到的、到最底層的資料，大致分成這幾層：

1. **畫面層（UI）**：你眼睛看到的東西——按鈕、文字、圖片、排版。
2. **狀態層（State）**：畫面背後「記在記憶體裡的資料」——例如「現在登入的是誰」「購物車裡有幾樣東西」。
3. **溝通層（API / Network）**：前端跟後端之間傳遞資料的管道——「外送員」。
4. **後端邏輯層（Server）**：伺服器上負責處理、運算、判斷的程式。
5. **資料層（Database）**：真正存放資料的地方——「冰箱」。

bug 一定發生在**某一層**（有時是兩層交界）。如果你能判斷「這個 bug 大概在第幾層」，你就不用整個網站亂找。這就是 debug 效率的關鍵——**先分層，再聚焦**。

判斷層級不需要精確，只要大方向對。「問題應該在 API 那層附近」就足以讓你少找 80% 的地方。

## 👶 超新手理解版

想像你在餐廳點了一份餐，結果上錯菜了。

這間餐廳有好幾個環節：**你（客人）→ 服務生 → 廚房 → 食材冰箱**。

上錯菜，可能是：
- **服務生**寫錯單（溝通層出錯）
- **廚房**看錯單做錯菜（後端邏輯出錯）
- **冰箱**裡根本沒有你要的食材（資料層出錯）
- 或者其實菜是對的，只是**端到你面前**擺盤擺得讓你認不出來（畫面層出錯）

如果你只會大喊「這間餐廳爛透了！」——沒有用，問題還是在。
但如果你能判斷「我覺得是服務生寫錯單」，餐廳經理就能直接去查點單記錄，馬上找到問題。

Debug 一個網站也一樣。「整個網站壞了」是沒用的抱怨。「我覺得問題在『資料從後端送到前端』這一段」——這就是有用的判斷，能讓你直接殺到那一層去查。

## 🏗 工程師版本

分層判斷的工程意義，是把問題空間（problem space）快速縮小。

一個全端應用的請求生命週期大致是：

```
使用者操作
   ↓
UI 元件（畫面層）
   ↓
前端狀態 / 事件處理（狀態層）
   ↓
發出 API 請求（溝通層）
   ↓
後端路由 → 控制器 → 服務邏輯（後端邏輯層）
   ↓
資料庫查詢（資料層）
   ↓
回傳資料，原路返回
```

判斷層級的核心問題是：**「資料流在哪一站斷掉、或變質了？」**

幾個快速判斷的訊號：

- **畫面整個空白、或排版爆掉，但 console 有 render 相關紅字** → 偏畫面層 / 狀態層。
- **畫面正常，但資料是空的或舊的** → 可能是狀態層沒更新，或 API 沒拿到資料。
- **打開瀏覽器 Network 面板，看到 API 請求是紅色的（4xx / 5xx）** → 問題在溝通層或後端。
- **API 請求是綠色的（200）、而且 response 裡有正確資料，但畫面還是沒顯示** → 問題在前端（拿到資料卻沒正確用）。
- **API 回傳的資料本身就是空的或錯的** → 問題在後端邏輯層或資料層。

注意一個關鍵技巧：**API 這一層是「分水嶺」。** 打開 Network 面板看那個 API 請求——
- 如果請求**成功且資料正確**，那 bug 在「前端拿到資料之後」，你可以**完全不用看後端和資料庫**。
- 如果請求**失敗、或資料就是錯的**，那 bug 在「後端那一側」，你可以**先不用懷疑前端畫面**。

光是這個判斷，就能把你要找的範圍**砍掉一半**。Chapter 4 會專門教你怎麼用 Network 面板做這件事，這課先建立「分層」的概念。

## 🧩 真實世界案例

AI 島有個「排行榜」頁面。某天使用者回報：排行榜整頁空白。

如果不分層，他可能會從前端排行榜元件開始亂看，看不出問題再去翻後端，再去翻資料庫，繞一大圈。

分層判斷的做法：他打開瀏覽器 F12 的 Network 面板，找到那個抓排行榜資料的 API 請求。看到它是**紅色的、狀態 500**。

這一眼就定位了——**問題在後端那一側**（500 是伺服器內部錯誤）。前端的排行榜元件完全不用看，因為它根本還沒拿到資料、連表現的機會都沒有。範圍直接從「整個排行榜功能」縮到「後端的排行榜邏輯」。後來發現是後端一個資料庫查詢的欄位名稱打錯了。

如果那個 API 請求是**綠色 200、而且 response 裡明明有排行榜資料**，那結論會完全相反——問題在前端，後端不用看。

**同一個「整頁空白」，看一眼 API 在哪一層出錯，就決定了你接下來要往哪個方向走。**

## ❌ 常見死亡方式

- **把「整個網站壞了」當成問題描述。** 太大、無從下手，等於沒描述。
- **不分層，從第一個檔案開始一個一個翻。** 像大海撈針。
- **bug 明明在後端，卻一直改前端畫面。** 因為沒判斷層級，方向從一開始就錯。
- **跳過 Network 面板。** 那是判斷「前端問題還是後端問題」最快的工具，不看它等於蒙著眼睛。
- **假設「畫面壞了所以是畫面的問題」。** 畫面是「結果」，壞掉的原因可能在任何一層。畫面空白，很可能是資料層沒給資料。

## 🔍 如何定位問題

判斷層級的標準動作，是問自己這三個問題，順序很重要：

1. **「畫面有沒有拿到資料？」** 畫面如果是空的，先別管排版——先確認「資料到底有沒有送過來」。
2. **「打開 Network 面板，相關的 API 請求是成功還是失敗？」** 這一步把問題切成「前端側」或「後端側」。
3. **「如果 API 成功，response 裡的資料對不對？」** API 成功但資料是錯的 → 後端邏輯／資料層。API 成功且資料正確 → 前端沒用好資料。

走完這三問，你大概就知道 bug 在哪一層了。

## 🛠 Debug SOP

1. **先描述現象**（延續 Lesson 02 的紀律）。
2. **打開瀏覽器 F12 → Network 面板**（Chapter 4 會詳教，現在先知道它在哪）。
3. **找到跟這個 bug 相關的 API 請求。**
4. **看那個請求的顏色 / 狀態碼**：
   - 紅色 / 4xx / 5xx → 問題在溝通層或後端，**先不看前端畫面**。
   - 綠色 / 200 → 點開看 response 的資料。
5. **看 response 的資料對不對**：
   - 資料是空的或錯的 → 後端邏輯層 / 資料層。
   - 資料是對的 → 問題在前端（拿到了卻沒正確顯示）。
6. **鎖定層級後**，才進到那一層去做細部 debug。

## 🤖 Claude Prompt

```

請幫我做「分層判斷」——先不要給修改方案。
請告訴我：這個 bug 最可能在哪一層（畫面層 / 狀態層 / API溝通層 /
後端邏輯層 / 資料層），以及我可以做哪一個檢查動作來確認層級。
我會去檢查，再把結果回報給你。
```

```

2. 標出每一站「最可能出問題」的點。
3. 建議我從哪一站開始檢查，以及檢查方法。

```

## 🧪 練習題

1. 用你自己的話，把一個網站的五個層級（畫面、狀態、API、後端、資料）各用一個生活比喻說出來。
2. 給你三個 bug 現象，判斷它「最可能在哪一層」：
   (a)「API 請求是綠色 200，response 裡有正確的 10 筆資料，但畫面上一筆都沒顯示。」
   (b)「點按鈕後 Network 面板出現一個紅色 500 的請求。」
   (c)「畫面排版整個跑掉、文字疊在一起，但資料內容是對的。」
3. 想一個你遇過的 bug，事後回想——它其實在哪一層？你當時有沒有判斷對？

> 練習題 2 參考方向：(a) 前端（拿到資料卻沒顯示）(b) 後端邏輯／資料層（500 是伺服器錯誤）(c) 畫面層（資料對、純粹是排版 CSS 問題）。

## ✅ 驗收標準

- 你能說出一個網站的五個層級，並各用一個比喻解釋。
- 遇到 bug，你會先用 Network 面板把問題切成「前端側」或「後端側」，而不是從頭亂翻。
- 給你一個 bug 現象，你能講出「它最可能在哪一層、為什麼」。

## 📌 重點整理

- 一個網站是**分層**的：畫面層 → 狀態層 → API 溝通層 → 後端邏輯層 → 資料層。
- bug 一定在某一層（或兩層交界），**先判斷層級，能砍掉大部分搜尋範圍**。
- **API（Network 面板）是分水嶺**：請求失敗 → 後端側；請求成功且資料正確 → 前端側。
- 判斷三問：畫面有沒有拿到資料？API 成功還失敗？response 資料對不對？
- 「整個網站壞了」不是有用的描述；「問題在 ___ 層」才是。

# Lesson 04｜Debug Mindset

## 🎯 本課目標

把前面三課的觀念，收斂成一套可以長期帶著走的「除錯心態」。這一課不教新技術，而是把 Debug 思維內化成你的習慣，讓你之後面對任何 bug——包括這門課沒教過的——都有穩定的心理素質和方法論去面對。

## 🧠 核心觀念

走完前三課，你已經知道：bug 是常態（L1）、不要瞎改（L2）、要分層判斷（L3）。這一課把它們捏成一個完整的心態。

成熟的 Debug Mindset，有五個支柱：

**1. bug 是訊息，不是攻擊。** bug 不是程式在「整你」，它是程式在「告訴你」某個地方跟你想的不一樣。紅字、錯誤訊息——那些不是在罵你，是線索。你越把 bug 當敵人，越焦慮、越容易瞎改；你越把它當訊息，越冷靜、越能讀懂它要告訴你什麼。

**2. 你是偵探，不是工人。** Debug 的主要工作不是「動手改」，而是「查案」。八成的時間在觀察、推理、縮小範圍，動手改可能只是最後一行。把自己想成偵探——蒐證、推理、鎖定兇手——而不是急著動工的工人。

**3. 相信「一定有原因」。** 沒有「莫名其妙」的 bug。電腦是完全照規則跑的機器，它不會無緣無故。當你覺得「這完全沒道理」，正確的解讀是「我還沒看到那個道理」。這個信念能讓你撐住——因為你知道，只要繼續查，原因一定在那裡等你。

**4. 縮小範圍，永遠在縮小範圍。** Debug 的每一個動作，目的都應該是「讓問題的可能範圍變小」。從「整個網站」縮到「某一層」，再縮到「某個檔案」，再縮到「某一行」。你做的每件事，都該問：「這讓範圍變小了嗎？」如果沒有，那個動作就是浪費。

**5. 卡住時，缺的是資訊，不是努力。** 當你卡很久，正確的反應不是「更用力地試」，而是「我還缺哪個資訊？」。可能是還沒看某個錯誤訊息、還沒檢查某一層、還沒問對問題。Debug 卡關，幾乎都是資訊缺口，不是努力不夠。

這五個支柱，會讓你從「怕 bug 的人」變成「能跟 bug 共處、並穩定解決它的人」。

## 👶 超新手理解版

把 Debug 想成你在當**偵探**。

一個好偵探遇到案件，他不會衝進去亂抓人（瞎改）。他會：

- **冷靜**——因為他知道「案件一定有真相」（一定有原因）。
- **蒐證**——看現場、找線索、聽證詞（看錯誤訊息、看 Network、看 console）。
- **推理**——根據證據縮小嫌疑犯範圍（分層、排除假設）。
- **鎖定**——證據都指向同一個人，破案（找到 root cause）。
- **最後才動手**——逮捕兇手（改那一行 code）。

爛偵探才會一進門就大喊「你們都有嫌疑！」然後亂抓。好偵探安靜、有耐心、相信真相一定存在、每一步都讓嫌疑範圍變小。

而且——**好偵探遇到查不下去的時候，不會更大聲地吼，他會去找「還缺哪條線索」。** 你 debug 卡住的時候也一樣：不是更用力，是去找那個還沒拿到的線索。

你不需要很聰明才能 debug。你需要的是**偵探的耐心和方法**。這個，是可以學的，而你正在學。

## 🏗 工程師版本

把五個支柱對應到工程實務：

**bug 是訊息** → 對應「錯誤訊息要逐字讀」。資深工程師看到 error 的第一件事是**完整讀完它**，包括 stack trace（錯誤發生的呼叫堆疊）。新手常常瞄一眼就跳過去——而答案常常就寫在那段紅字裡。Chapter 2 會專門教讀錯誤訊息。

**你是偵探** → 對應 hypothesis-driven debugging（L2 教過）。工程師的 debug 時間分配大致是「觀察與推理 80%、動手修改 20%」。如果你發現自己一直在改、很少在想，那方向就錯了。

**相信一定有原因** → 對應 determinism（決定性）的信念。電腦是決定性系統，相同輸入必然產生相同輸出。所謂「有時好有時壞」的 bug，不是真的隨機，而是有一個你還沒辨識出來的變因（時間、順序、某個非同步狀態）。把「隨機」重新理解為「有一個我還沒找到的變因」。

**縮小範圍** → 對應 binary search 式的除錯思維。每個 debug 動作最好都像二分搜尋——一刀下去排除一半。Chapter 5 的「切片法」、Chapter 6 的「最小重現」、Chapter 12 的 `git bisect`，本質上都是這個思維的工具化。

**卡住是缺資訊** → 對應 observability（可觀測性）。工程上解決「卡住」的方法是「增加可觀測性」——加 log、加中斷點、看監控。卡住時不是寫更多程式，而是想辦法**看見更多**。Chapter 3 的各種 console 技巧，就是在提升可觀測性。

這五個支柱不是雞湯，它們各自對應一套具體的工程實踐，後面每一章都是在把某個支柱「工具化」。這一課先把心法立起來，後面的章節你就會知道「為什麼要學這個工具」。

## 🧩 真實世界案例

一個對照組故事。兩個人，同樣在 debug 同一個 bug：「網站部署到線上後，登入功能壞掉，但本地測試一切正常。」

**A（沒有 Debug Mindset）**：他第一反應是「線上環境爛掉了」「部署平台有問題」——把 bug 當成外部攻擊。然後開始瞎改登入相關的程式碼，改一行、重新部署、等三分鐘、還是壞、再改一行。改了一下午，登入沒修好，還改出了兩個新問題。他最後說「玄學」「不知道為什麼」，放棄。

**B（有 Debug Mindset）**：他先冷靜——「本地正常、線上壞，代表問題出在『兩個環境的差異』，一定有原因。」他把自己當偵探，開始縮小範圍：本地和線上，差在哪？環境變數、網域、資料庫連線……他一個一個排除。他打開線上的瀏覽器 Network 面板（蒐證），看到登入 API 回傳一個跟 cookie 有關的錯誤。線索出現了。他查到是線上環境少設了一個跟 cookie 網域有關的設定。改一個設定，解決。全程一小時，沒有製造任何新問題。

同一個 bug。A 把它當「玄學」，B 把它當「一定有原因、只是我還沒找到的案件」。**差別不在技術高低，在心態。** B 用的每一個具體動作（看環境差異、看 Network、讀錯誤訊息），這門課後面都會教。但如果沒有那個心態打底，再多工具也會被焦慮和瞎改浪費掉。

## ❌ 常見死亡方式

- **把 bug 當人身攻擊**，越 debug 越生氣，情緒一上來就開始瞎改。
- **說「這是玄學」「莫名其妙」然後放棄。** 這句話的真正意思是「我不想再找了」，不是「真的沒原因」。
- **卡住就更用力地試同樣的方法。** 卡住要的是新資訊、新角度，不是更多力氣。
- **跳過錯誤訊息。** 答案常常就寫在那段紅字裡，你卻沒讀完。
- **一直在「改」，很少在「想」。** debug 時間分配嚴重失衡，方向錯了。
- **一個人悶頭鑽牛角尖。** 卡很久時，把問題完整講給別人（或 AI）聽，光是「重新講一遍」就常常讓你自己發現盲點——這叫小黃鴨除錯法（rubber duck debugging）。

## 🔍 如何定位問題

這一課的「定位」是一個總結性的自我檢查。每次 debug 卡住時，跑一遍這份心態檢查表：

- 我現在是冷靜的，還是焦慮到在瞎改？
- 我最近五個動作，是在「縮小範圍」，還是在原地打轉？
- 我有沒有逐字讀完錯誤訊息？
- 我相不相信「這個 bug 一定有原因」？還是已經在心裡說「玄學」了？
- 我卡住了——那我「還缺哪一個資訊」？我能去哪裡拿到它？

只要這份檢查表能誠實跑完，你通常就會發現下一步該往哪走。

## 🛠 Debug SOP

這是整個 Chapter 1 的總 SOP，把四課收在一起：

1. **遇到 bug：先停。** 不焦慮、不瞎改。告訴自己「bug 是訊息，一定有原因」。（L1、L4）
2. **描述現象。** 用具體句子寫下「發生了什麼」。（L2）
3. **判斷層級。** 用 Network 面板把問題切成前端側或後端側。（L3）
4. **列假設。** 在鎖定的層級裡，寫下 2~4 個可能原因。（L2）
5. **一次驗證一個假設**，每次都讓範圍變小。（L2、L4）
6. **收斂到 root cause**——能用一句話說出「bug 是因為 ___」。
7. **只改那一個地方**，改前備份。（L2）
8. **改完立刻驗證。** 好了，結案；沒好，還原、回到步驟 4。
9. **卡住時**，不要更用力——問「我還缺哪個資訊」，去把它找出來。（L4）

這套 SOP 會貫穿整門課。後面每一章，都是在強化其中某一步。

## 🧪 練習題

1. 用你自己的話，把 Debug Mindset 的五個支柱寫一遍（bug 是訊息、你是偵探、一定有原因、縮小範圍、卡住是缺資訊）。寫完闔上文件，看能不能默背出來。
2. 回想一次你「把 bug 當玄學、放棄了」的經驗。用現在學到的心態重新看它——如果一定有原因，你會去哪裡找那個還沒看到的線索？
3. 把 Chapter 1 的總 SOP（9 步）抄到一張紙、或存成手機備忘錄，貼在你寫程式的地方。下次遇到 bug，照著走一遍。
4. 小黃鴨練習：找一個你現在手上沒解決的 bug，假裝對著一隻玩偶（或對 AI）把它「從頭到尾完整講一遍」——現象、你試過什麼、你卡在哪。觀察自己講到一半有沒有突然想到新方向。

## ✅ 驗收標準

- 你能默背 Debug Mindset 的五個支柱，並用自己的話解釋每一個。
- 你能完整覆述 Chapter 1 的總 SOP（從「先停」到「卡住時找缺的資訊」）。
- 遇到 bug 時，你能察覺自己的情緒狀態（冷靜 or 焦慮），並在焦慮時主動停下來。
- 你不再用「玄學」「莫名其妙」當結論——你相信並願意去找「那個還沒看到的原因」。

## 📌 重點整理

- Debug Mindset 五支柱：**bug 是訊息**、**你是偵探**、**一定有原因**、**永遠在縮小範圍**、**卡住是缺資訊不是缺努力**。
- Debug 的時間分配應該是「觀察推理 80%、動手改 20%」——你是偵探，不是急著動工的工人。
- 「玄學」「莫名其妙」不是結論，是「我還沒找到原因」的代名詞。
- 卡住時，不要更用力試同樣的方法——去找你還缺的那個資訊（加 log、看訊息、換角度、講給別人聽）。
- Chapter 1 總 SOP 9 步是整門課的骨架，後面每一章都在強化其中一步。
