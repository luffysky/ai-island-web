## 1. P0 — HTTP Security Headers（目前完全沒有）

掃描全專案：`Content-Security-Policy`、`Strict-Transport-Security`、`X-Frame-Options`、`X-Content-Type-Options` **一個都沒設**。這是最容易補、回報最大的一層——能一次擋掉點擊劫持、MIME 嗅探、降低 XSS 影響面、強制 HTTPS。

**修復：在 `next.config.mjs` 加 `headers()`**

```js
const securityHeaders = [
  { key: "X-Frame-Options", value: "DENY" },                       // 防點擊劫持（被嵌 iframe）
  { key: "X-Content-Type-Options", value: "nosniff" },             // 防 MIME 嗅探
  { key: "Referrer-Policy", value: "strict-origin-when-cross-origin" },
  { key: "Permissions-Policy", value: "camera=(), microphone=(), geolocation=()" },
  { key: "Strict-Transport-Security", value: "max-age=63072000; includeSubDomains; preload" },
  { key: "X-DNS-Prefetch-Control", value: "on" },
];

const nextConfig = {
  // ...既有設定
  poweredByHeader: false,                                          // 不洩漏框架
  async headers() {
    return [{ source: "/:path*", headers: securityHeaders }];
  },
};
```

- [ ] 加上以上 headers
- [ ] HSTS 上線前先確認全站都跑 HTTPS（Zeabur 預設有），`preload` 是不可逆承諾，先用短 `max-age` 測一週再拉長

### CSP（Content-Security-Policy）—— 要做但要小心
CSP 是擋 XSS 最強的一層，但這專案有 Three.js、framer-motion（inline style）、Supabase、GA4，**直接上嚴格 CSP 會把網站打爆**。做法：
- [ ] 先用 **Report-Only 模式**（`Content-Security-Policy-Report-Only`）上線，收 violation report 不擋
- [ ] 觀察一兩週、把合法來源加進白名單後，再轉成強制模式
- [ ] 走 Next.js nonce 機制（middleware 產生 nonce、`script-src 'nonce-xxx'`）取代 `unsafe-inline`
- [ ] 起手白名單：`default-src 'self'`；`img-src 'self' data: *.supabase.co lh3.googleusercontent.com`；`connect-src 'self' *.supabase.co`；`frame-ancestors 'none'`

> CSP 是「持續調」的東西，不是一次設好。Report-Only 先行是關鍵,別省這步。

## 2. P0 — 公開 / 認證端點的 Rate Limit（防暴力破解 + API 濫用）

### 🔴 2-1：對外 API `/api/v1/chat` 沒有 rate limit
**檔案：** `src/app/api/v1/chat/route.ts`
目前只有「月配額」，沒有「每分鐘速率」。攻擊者可在配額內瞬間爆量打、或拿外洩的 key 燒你的 AI 成本。
- [ ] 加 per-key 的短窗 rate limit（如每分鐘 20 次），用既有 `rateLimit(key, limit, windowMs)`
- [ ] key 驗證失敗也要 rate limit（per-IP），防有人拿一堆 `aii_xxx` 來暴力試 key

### 🔴 2-2：登入 / 註冊沒有應用層 rate limit
`src/app/api/auth/*`、login、signup 沒看到 `rateLimit` 呼叫。Supabase Auth 本身有一些保護，但自家的 ensure-profile、magic-link、OAuth callback 仍可被刷。
- [ ] 對 login / signup / 寄信類 endpoint 加 per-IP rate limit（如每 IP 每 15 分鐘 5 次失敗就鎖）
- [ ] 失敗計數可寫進既有 `rate_limit_hits` 表

### ⚠️ 2-3：In-memory rate limit 在多 instance 會失效
`src/lib/rate-limit.ts` 自己註明了：memory store 只在單 instance 有效，Zeabur 多開就各算各的。
- [ ] 若 Zeabur 之後 scale 到多 instance → 換 **Upstash Redis**（有免費額度、邊緣友善）做分散式 rate limit
- [ ] 現階段單 instance 可先用 memory，但在報告 / 程式碼留 TODO

**建議做一個共用 wrapper**（搭配《漏洞報告》的 `admin-guard`）：
```ts
// src/lib/with-rate-limit.ts 概念
export async function enforceRateLimit(req, { key, limit, windowMs }) {
  const r = rateLimit(key, limit, windowMs);
  if (!r.ok) return NextResponse.json(
    { error: "too_many_requests" },
    { status: 429, headers: { "Retry-After": String(r.retryAfter ?? 60) } }
  );
  return null;
}
```

## 3. P1 — 輸入驗證（zod 裝了但只有 2/236 條 API 在用）

掃描：`zod` 已安裝，但只有 2 條 API route import 它。其餘 234 條多半直接 `await req.json()` 後就信任 body。這是注入 / 髒資料 / 邏輯繞過的溫床。

**修復方向：**
- [ ] 所有「吃 body 的」API route（POST/PUT/PATCH）都定義 zod schema、`schema.safeParse(body)`，失敗回 400
- [ ] 統一一個 helper：
```ts
// src/lib/validate.ts 概念
export async function parseBody<T>(req: NextRequest, schema: z.ZodSchema<T>) {
  const raw = await req.json().catch(() => null);
  const r = schema.safeParse(raw);
  if (!r.success) return { ok: false as const, response: NextResponse.json(
    { error: "invalid_input", details: r.error.flatten() }, { status: 400 }) };
  return { ok: true as const, data: r.data };
}
```
- [ ] 優先補：金流、AI、blog/forum 發文、admin 寫入類 route（高風險）
- [ ] 字串欄位記得 `.max()` 限長度（防超大 payload 塞爆 DB / token）

## 4. P1 — 機器人 / 濫用防護（CAPTCHA + 蜜罐）

掃描：沒有任何 captcha（turnstile / hCaptcha / reCAPTCHA）。註冊、發文、求助表單會被機器人灌爆（spam 帳號、廣告貼文、燒簡訊/email 額度）。

**修復方向：**
- [ ] 在「無登入也能觸發、且會消耗資源」的入口加 **Cloudflare Turnstile**（免費、隱私友善、比 reCAPTCHA 輕）：
  - 註冊 / 登入
  - blog / forum 發文（尤其匿名或新帳號）
  - 任何寄 email / LINE 推播的表單
- [ ] 表單加**蜜罐欄位**（hidden input，真人不會填、機器人會）——零成本擋掉一大票笨 bot
- [ ] 新帳號設「冷卻期 / 信任分數」：剛註冊的帳號發文頻率更嚴格（你 OrderEase 的 no-show 信用分機制可以複用思路）

## 5. P1 — 認證 / 帳號層加固

- [ ] **Owner 帳號上 MFA**：林董帳號（後台最高權限）強烈建議開 Supabase MFA / TOTP。一個帳號失守 = 整個後台失守
- [ ] **Admin slug 不是安全機制**：`console-x7k2` 只是 obscurity（《漏洞報告》P1-2 已列），真正的門是 role 檢查。移除硬編 fallback、缺 env 時 fail-closed
- [ ] **Session 安全**：確認 cookie 是 `httpOnly` + `secure` + `sameSite=lax/strict`（Supabase SSR 預設有，驗一下沒被覆寫）
- [ ] **敏感操作二次確認**：刪資料、env 變更、impersonate、退款 → 走 `requireOwner()`（已附 `admin-guard.ts`），並寫 `audit_logs`
- [ ] **impersonate 功能**（`/admin/impersonate`）是高風險特性：確認每次都寫稽核、有時限、結束自動還原、且只有 owner 能用

## 7. P2 — 注入 / SSRF / 其他

- [ ] **SQL injection**：用 Supabase client 的參數化查詢（`.eq()`/`.in()`）基本免疫；但若有任何 `.rpc()` 拼字串或原生 SQL，逐一檢查參數化
- [ ] **SSRF**：任何「吃使用者 URL 去 fetch」的功能（如 marketing/scrape、OG 圖抓取、頭像 URL）→ 白名單 domain、禁止內網 IP（`169.254.*`、`127.*`、`10.*`、`192.168.*`）、禁 redirect 到內網
- [ ] **開放重導**：登入 `?next=` 參數要驗證只能是站內相對路徑（防 open redirect 釣魚）
- [ ] **檔案上傳**（media_assets / 頭像）：驗 MIME + 副檔名 + 大小上限、存到 Supabase Storage 而非可執行路徑、圖片重新編碼去除 metadata/嵌入 payload

## 8. P2 — 監控 / 偵測 / 事件應變

你已經有 `audit_logs`、`error_logs`、`breach_incidents`、`rate_limit_hits` 表——把它們接起來變成「會叫的警報」：
- [ ] **異常告警**：既有 `.github/workflows/anomaly-check.yml` → 擴充偵測「短時間大量 401/403/429」「單 IP 高頻」「service_role 異常呼叫」，命中就 LINE / email 通知林董
- [ ] **登入異常**：同帳號短時間多地登入、大量失敗 → 記 `audit_logs` + 告警
- [ ] **AI 成本異常**：單 key / 單用戶 token 暴衝 → 自動暫停 + 通知（保護你的 API 帳單）
- [ ] **breach 流程演練**：`breach_incidents` 表 + 台灣個資法頁面已備，寫一份「外洩了第一小時做什麼」的 SOP（停 key、輪替、通知、記錄）

## 9. P3 — 邊界 / 基礎建設

- [ ] **Cloudflare 擋在 Zeabur 前面**（如果還沒）：免費版就給你 DDoS 緩解、WAF 規則、Bot Fight Mode、速率限制、隱藏真實 origin IP——這是 CP 值最高的外層防線
- [ ] **隱藏 origin**：套 Cloudflare 後，Zeabur 的真實位址別外洩（否則被繞過 CF 直打）
- [ ] **依賴漏洞掃描**：開 GitHub Dependabot + `npm audit`，定期修有 CVE 的套件
- [ ] **備份**：Supabase 定期備份 + 確認還原流程可用（勒索 / 誤刪的最後一道防線）

## 附註

- 全部為**防禦性**措施。CSP（#2）與 rate-limit 數值（#3#4）需在 staging 實測調整,先寬後緊、避免誤殺真實使用者。
- 對個人營運的產品，**最高 CP 值是 #1（headers）+ #3#4（rate limit）+ #12（Cloudflare）**——這三項就能擋掉絕大多數自動化攻擊與意外爆量。
- 進階弱點（SSRF、CSP nonce、分散式 rate limit）建議基礎三項就位後,再由雪鑰逐條深入。
