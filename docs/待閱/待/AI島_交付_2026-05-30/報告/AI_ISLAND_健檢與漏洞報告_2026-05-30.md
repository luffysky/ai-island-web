## 2. P0 — 安全關鍵（請雪鑰優先處理）

### 🔴 P0-1：UGC 的 HTML 清洗用 regex 黑名單，可被繞過（XSS）

**檔案：** `src/lib/rich-html.ts`（僅 14 行）
**影響面：** Blog 文章、論壇貼文等使用者產生內容，最終以 `dangerouslySetInnerHTML` 渲染於：
- `src/app/blogs/[userSlug]/[articleSlug]/page.tsx`
- `src/app/forum/thread/[id]/page.tsx`
- `src/app/me/resume/ResumeClient.tsx`

**問題：** `sanitizeRichHtml()` 是用一連串 `.replace(/<script>.../, "")`、移除 `on*=`、`javascript:`、`iframe/object/embed` 的**黑名單字串清洗**。這類做法在安全上是已知不可靠的——畸形標籤、屬性拆解、編碼變形、`<svg>` / `<math>` 向量、mutation XSS 都可能繞過正規表達式。一旦繞過，惡意 script 會在其他登入使用者的瀏覽器中執行（可竊 session、代操作）。

**修復方向（不要再補 regex）：**
- [ ] 改用 server 端**白名單**清洗器：`sanitize-html` 或 `isomorphic-dompurify`
- [ ] 明確列出允許的標籤 / 屬性（p, h1–h6, ul, ol, li, a[href], img[src], code, pre, blockquote, strong, em, table…）
- [ ] `a` 標籤強制 `rel="noopener nofollow ugc"`、`href` 只允許 http/https/mailto
- [ ] **寫入時就清洗一次**（存進 DB 前），渲染時再清洗一次（雙層）
- [ ] `src/data/chapters/*.json` 內含 `dangerouslySetInnerHTML` 的章節 HTML 屬於受信任作者內容，可走較寬鬆的白名單，但仍建議過同一清洗器

> ⚠️ 雪鑰請注意：報告中刻意不提供繞過字串。實作完成後請用 sanitize-html 的測試套件 / OWASP 的 XSS 測試向量做單元測試驗證，不要自行手寫攻擊字串。

### 🔴 P0-2：後台 guard 重複複製、無單一共用 helper

**現況：** 101 條 `src/app/api/admin/*` route 全部都有某種授權檢查（已逐條確認，沒有完全裸奔的），但實作方式分散：
- 2 條用各自 local 的 `gateAdmin()` 函式（如 `src/app/api/admin/kanban/route.ts`）
- 21+ 條用 inline `["admin","owner"].includes(profile.role)` 檢查
- 其餘各種 `getUser()` + 手寫 role 判斷

**風險：** 沒有單一 source of truth，下一條新 admin route 很容易忘記加 gate 就上線（這是這類後台最常見的破口來源）。屬於「結構性風險」而非當下已開的洞。

**修復方向：**
- [ ] 在 `src/lib/` 新增共用 `requireAdmin()` 與 `requireOwner()`（回傳 `{ ok, userId } | { ok:false, status, body }`）
- [ ] 用 `getUser()`（已驗 token，非僅讀 cookie）+ `profiles.role`/`is_owner` 判斷
- [ ] 把 101 條 admin route 的 inline / local gate 全部換成這個共用 helper
- [ ] 加一條 lint 規則或 CI 檢查：`src/app/api/admin/**` 的 route 必須 import `requireAdmin`

**File Scope：** 新增 `src/lib/admin-guard.ts`；修改 `src/app/api/admin/**/route.ts`（逐條替換，不動商業邏輯）

### 🔴 P0-3：RLS「空 policy」9 張表需逐表確認意圖

**來源：** `rls-audit-report.md`（你們自己的掃描工具，2026-05-25）

被標為 🔴 致命「RLS 開但 0 policy」的 9 張表：
`achievements`、`admin_events`、`ai_api_keys`、`ai_usage_daily`、`analytics_snapshots`、`audit_logs`、`broadcasts`、`seo_pages`、`seo_redirects`

**重要釐清（不要無腦全部加 policy）：**
RLS 開 + 0 policy = **只有 service_role 能讀寫，一般使用者完全讀不到**。對系統 / 後台專用表來說，這其實是「預設拒絕」的**安全正確狀態**：

| 表 | 判斷 | 建議動作 |
|---|---|---|
| `audit_logs` / `admin_events` | 系統稽核 | 維持空 policy（正確），文件註記「by design」 |
| `ai_api_keys` | 金鑰表 | **維持空 policy**（絕不可開給使用者） |
| `ai_usage_daily` / `analytics_snapshots` | 後台統計 | 維持空 policy；前台若需顯示走 service_role API |
| `broadcasts` / `seo_pages` / `seo_redirects` | 後台內容 | 若前台需讀取 → 補 `SELECT` policy；否則維持 |
| `achievements` | 成就「目錄」 | ⚠️ **多半需要補 public/auth `SELECT` policy**，否則使用者看不到可解鎖的成就清單（這是功能性 bug，不是安全洞） |

**修復方向：**
- [ ] 逐表決定「deny-all 是對的」還是「需要 read policy」，把結論寫進 migration 註解
- [ ] `achievements` 最可能需要：`CREATE POLICY achievements_read ON achievements FOR SELECT USING (true);`（或限 `auth.role() = 'authenticated'`）
- [ ] 不確定的表先在 staging 用一般使用者帳號實測前台是否該頁正常

## 3. P1 — 重要（盡快但非緊急）

### 🟡 P1-1：Telegram webhook 的 secret 驗證是「可選」
**檔案：** `src/app/api/telegram-webhook/route.ts`（line 219–224）
secret 只有在 `TELEGRAM_WEBHOOK_SECRET` env 有設時才驗。沒設就完全跳過 → 任何人知道 endpoint 就能偽造 update。
- [ ] 改成 production 強制要求 secret；缺 env 時 fail-closed（回 500 而非放行）
- [ ] 確認線上 env 已設 `TELEGRAM_WEBHOOK_SECRET`

> 對照組：`stripe/webhook` 用 `constructEvent` 驗簽（✅ 正確）、`v1/chat` 用 `aii_` API key SHA-256 比對 + 月配額（✅ 正確）——這兩個是好範本。

### 🟡 P1-2：Admin slug 硬編 fallback `console-x7k2` 散落 25 處
**檔案：** `src/middleware.ts` line 4 等 25 個檔
`const ADMIN_SLUG = ... || 'console-x7k2'`。真正的 gate 是 `src/app/admin/layout.tsx` 的 server 端 role 檢查（✅ 這個是對的，slug 只是 obscurity），但 fallback 字串已在 repo 公開。
- [ ] 移除硬編 fallback，改成 env 缺失時 fail-closed
- [ ] 收斂成單一常數 `src/lib/admin-href.ts`（已存在），其餘 24 處改 import

### 🟡 P1-3：22 條 RLS policy 是 `FOR ALL` 但缺 `WITH CHECK`
**來源：** `rls-audit-report.md`（ai_conversations、bookmarks、notes、pets、quiz_attempts、user_api_keys… 等）
Postgres 對 `FOR ALL` 缺 `WITH CHECK` 時會 fallback 用 `USING`，多數情況安全，但升級 / 某些 client 版本會擋。
- [ ] 為這 22 條補上明確 `WITH CHECK (auth.uid() = user_id)`（與 USING 對齊）

### 🟡 P1-4：Codex handoff 未收尾的 migration
**來源：** `docs/Codex_to_Claude.md`（2026-05-22）
- [ ] `supabase/breach_and_email_migration.sql` 尚未完整套用（線上 `breach_incidents` 是舊 schema、缺欄位、重跑卡在 policy exists）
- [ ] `supabase/interaction_analytics_migration.sql` 尚未跑
- [ ] **修法：** 把這兩份 migration 內所有 `CREATE POLICY` 前面都加 `DROP POLICY IF EXISTS ... ;`，做成可重複執行（idempotent），就不會卡 policy exists
