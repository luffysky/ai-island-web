## 0. 一句話結論

> 最高 CP 值的兩件事：**(A) `src/data/chapters.ts` 把 76 個 JSON（8.7MB）全靜態 import，且被 client component 拉進去 → 可能整包進前端 bundle**；**(B) `next.config.mjs` 還是空的，加一行 `optimizePackageImports` 幾乎零風險就能瘦身。** 這兩個做完,首屏 JS 體積應該有感下降。其餘是漸進式優化。

## 1. P0 優化 — bundle 體積（最有感）

### 🔴 OPT-1：章節資料全靜態 import，且滲進 client bundle

**檔案：** `src/data/chapters.ts`（166 行，`import ch01 from "./chapters/ch01.json"` … 一路到 ch75）
**證據：**
- `src/data/chapters/` 共 **8.7MB**
- `chapters.ts` 把 76 個 JSON **全部靜態 import** 成一個 barrel
- **client component** `src/components/me/SkillRadar.tsx`（`"use client"`）直接 `import { chapters } from "@/data/chapters"`

**問題：** 只要任何 client component import 這個 barrel，打包器就傾向把整棵章節內容（含每章的 lesson HTML）塞進前端 JS chunk。使用者其實只需要「章節 metadata」（id、標題、技能標籤、順序）來畫雷達圖,不需要每一章的完整教材內文。

**修復方向：**
- [ ] 拆成兩層：
  - `src/data/chapters-meta.ts` — 只含輕量 metadata（id / title / slug / skills / order / 章節數），給 client 與列表頁用
  - 完整 lesson 內文 → 走 server component / DB（`chapters` 表已存在）/ 或 per-chapter 動態 import（`await import(\`./chapters/ch\${id}.json\`)`），只在進到該章頁時載入
- [ ] `SkillRadar.tsx`、`CareerProgress.tsx` 等 client 端改 import metadata 版
- [ ] 確認 `seo/page.tsx`、`certificates/page.tsx` 這類 server 端可維持完整 import（不進 client bundle 就沒事）

> 預期：這是本報告體積影響最大的一條。做完用 bundle analyzer 對照前後。

## 2. P1 優化 — 設定層快速勝利（低風險、高回報）

### 🟡 OPT-2：`next.config.mjs` 幾乎是空的，補 `optimizePackageImports`

**現況：** 只設了 `reactStrictMode` 跟 `images.remotePatterns`。
這些 barrel 套件（lucide-react、recharts、framer-motion、date-fns、@codemirror/*）預設會把整包拉進來。Next 15 的 `optimizePackageImports` 會自動改成 per-export tree-shake。

**修復方向：**
```js
// next.config.mjs
const nextConfig = {
  reactStrictMode: true,
  poweredByHeader: false,
  experimental: {
    optimizePackageImports: [
      "lucide-react",
      "recharts",
      "framer-motion",
      "date-fns",
      "@tiptap/react",
    ],
  },
  images: { /* 既有設定保留 */ },
};
```
- [ ] 加上以上 `optimizePackageImports`（lucide-react 尤其有感、整個 icon set 很大）
- [ ] 加 `poweredByHeader: false`（移除 `X-Powered-By` 洩漏框架版本）
- [ ] 視需要加 security headers（CSP / X-Frame-Options / Referrer-Policy）的 `headers()`

### 🟡 OPT-3：裝個 bundle analyzer 才知道肥在哪
- [ ] `npm i -D @next/bundle-analyzer`、包一層 `withBundleAnalyzer`、加 `npm run analyze` script
- [ ] 跑一次找出最大的幾個 chunk，後續優化有依據（不要憑感覺）

## 3. P1 優化 — 程式碼分割（code-splitting）

### 🟡 OPT-4：452 個元件只有 5 處用 `next/dynamic`

**已做對的：** Monaco（`PlaygroundCard.tsx`）跟 3D 島（`IslandClient.tsx` 的 `IslandV0`）都已經 `next/dynamic` 動態載入 ✅ —— 這兩個最重的已經處理好了，給雪鑰參考這個模式。

**還可以做的：**
- [ ] TipTap 編輯器（blog/論壇發文）：只在編輯頁需要，動態 import + `ssr:false`
- [ ] recharts 圖表（dashboard / admin analytics）：圖表多在折疊下方，動態 import
- [ ] CodeMirror 的 12 個語言包（`@codemirror/lang-*`）：不要全靜態 import，依使用者選的語言 `await import()` 對應包，其餘不載

### 🟡 OPT-5：Monaco + CodeMirror 兩套編輯器並存
專案同時裝了 `@monaco-editor/react`（1 處用）與 CodeMirror（3 處用）。兩套都是體積大戶。
- [ ] 評估能否收斂成一套（CodeMirror 較輕、語言包模組化好）。若 Monaco 只為 PlaygroundCard 的某個特性而留，確認那個特性 CodeMirror 是否也能做 → 能的話移除 Monaco 省一大塊

## 4. P2 優化 — Server / Client component 再平衡

### 🟢 OPT-6：57% 元件是 `"use client"`，偏高
**現況：** 256 / 452 標了 `"use client"`。App Router 的優勢是預設 server component（不進 bundle、可直接查 DB）。許多「只是 fetch + 顯示」的頁面其實不需要 client。

**修復方向（漸進，不急）：**
- [ ] 抓出「標了 `use client` 但沒有任何 hook / event handler / 瀏覽器 API」的元件 → 拔掉 `"use client"` 轉 server component
- [ ] 把互動部分（按鈕、表單）抽成小的 client 子元件，外層維持 server
- [ ] 優先處理高流量頁面（首頁、章節列表、blog 列表）

> 這條收益實在但工程量大，建議排在 P0/P1 之後、分批做，別一次全改（參考之前 95 頁一次改的教訓 😅）。

## 5. P2 優化 — 資料層

### 🟢 OPT-7：104 處 `select("*")` 過度抓取
列表 / 表格類查詢用 `select("*")` 會把所有欄位（含大文本欄）一起撈。
- [ ] 把 hot-path 的查詢改成明確列欄位（`select("id, title, created_at")`），減少 payload、有機會走 index-only scan
- [ ] 優先：列表 / 分頁 / 後台表格類 API

### 🟢 OPT-8：RLS 的 admin 子查詢每列重跑
很多 policy 寫 `EXISTS (SELECT 1 FROM profiles WHERE id = auth.uid() AND role = 'admin')`，每筆 row 評估時都跑一次子查詢。
- [ ] 改用 `SECURITY DEFINER` 的 `is_admin()` / `is_owner()` SQL function（Postgres 會 cache stable function 結果），policy 改呼叫它
- [ ] 或把 role 放進 JWT custom claim，policy 直接讀 claim 不查表（最快，但要改 auth hook）
- [ ] 確認 `profiles.id` 是 PK、`lesson_progress(user_id)`、`ai_messages(conversation_id)` 等外鍵欄有 index

### 🟢 OPT-9：公開內容加快取 / ISR
章節列表、blog、排行榜這類「不常變、所有人看一樣」的頁面,目前若每次 SSR 都打 DB 很浪費。
- [ ] 公開讀取頁用 `export const revalidate = 300`（ISR）或 `unstable_cache` 包查詢
- [ ] `ai_response_cache` 表已存在 → 確認 AI 回應快取命中率,後台加個 hit-rate 指標
